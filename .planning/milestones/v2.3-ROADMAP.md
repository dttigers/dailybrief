### Phase 56: Push origin on phase-complete for backend phases

**Goal:** Structurally prevent the 53-04 class of bug — local main dozens of commits ahead of origin while Railway runs stale vigil-core code
**Depends on:** None
**Plans:** 1 plan

Plans:
- [x] 56-01: Register `workflow.deploy_targets`, implement `maybePushPhaseCommits()` in `phase.cjs`, opt dailybrief into `["vigil-core"]`, zero-behavior-change harness, cleanup stale Phase 55 dependency

**Details:**
- Modifies `~/.claude/get-shit-done/bin/lib/phase.cjs` — direct edit to the global GSD install (CLI chokepoint covers execute-phase, transition, and autonomous entry points in one place). Reapply via `/gsd-reapply-patches` after GSD updates
- New config key `workflow.deploy_targets: string[]` in `.planning/config.json`; dailybrief set to `["vigil-core"]`; empty array = no-op for every other project
- `maybePushPhaseCommits()` helper greps `git log --name-only` in the phase commit range for deploy_targets prefixes; auto-pushes on match, prints `◆ Pushed N commits to origin — Railway will redeploy`; push failure = phase-complete failure (no swallow, no prompt)
- Commit-range regex uses `\(${phaseNum}[^0-9]` non-digit boundary to prevent false-matching future 3-digit phases 560-569
- Deviation handled at harness time: `loadConfig` flattens `workflow.*` keys, so the helper reads `cfg.deploy_targets` not `cfg.workflow.deploy_targets`; fix patched `core.cjs` as well
- `~/.claude/get-shit-done/tests/phase-complete-push.sh` harness validates BOTH scenarios: Mac-only phase → no push, vigil-core phase → push fires
- **First real-world fire:** 2026-04-08 during Phase 53 completion — hook pushed 22 backlogged commits to origin/main automatically. The 53-04 bug would have recurred without this

### Phase 57: Cross-machine bootstrap script

**Goal:** Turn "switch machines and resume work on dailybrief" from an error-prone manual checklist into one command
**Depends on:** None
**Plans:** 2 plans (wave 1 parallel — no file overlap)

Plans:
- [x] 57-01: `scripts/bootstrap.sh` — 10-step D-08 orchestrator (pre-flight → op restore → vigil-core build → sync keys → launchctl load → install.sh delegation → health check → summary), 1P vault preamble, `--check` shim, HTTP 200-only health check
- [x] 57-02: `scripts/dailybrief-doctor.sh` — READ-ONLY drift doctor (ANTHROPIC 4-places, VIGIL bearer single-source, informational `/v1/health` row that doesn't affect exit code)

**Details:**
- 1Password CLI (`op`) is the canonical secrets transport — three vault items as full-file documents: `vigil-config` → `~/.config/dailybrief/config.json`, `vigil-gcal-tokens` → `google_calendar_tokens.json`, `vigil-vigilcore-plist` → vigilcore LaunchAgent plist
- Pre-flight fails loud with `brew install --cask 1password-cli` recipe if `op` missing (no auto-install of system-level tools)
- Existing scripts (`setup.sh`, `install.sh`, `build.sh`, `sync-anthropic-key.sh`) left unchanged; `bootstrap.sh` is a thin orchestrator that calls them
- Doctor is read-only (zero writes, zero launchctl operations, verified by T-57-05 negative-grep): reports drift across 4 ANTHROPIC_API_KEY locations (`config.json` canonical → `.env` → plist → Railway) and the VIGIL_API_KEY bearer single source. Exit 0 clean, exit 1 drift. Printing heals via `./scripts/sync-anthropic-key.sh`
- Research discovered the `DATABASE_URL` gap: local vigil-core has always run with `status: "degraded"` because Mac apps talk to `api.vigilhub.io` (Railway), not localhost. D-13 locked: health check is HTTP 200 ONLY (no JSON status parsing); doctor's `local vigil-core DB connected?` is INFORMATIONAL only
- Doctor self-test on current machine: **exit 0, no drift** — sync-anthropic-key.sh has been keeping things consistent
- Bootstrap self-test on current machine (without `op` installed): preflight fails loud with brew recipe, idempotent twice in a row
- **Not yet validated end-to-end on a real fresh machine** — deferred to the next time the user provisions a Mac

## Milestone Summary

### What shipped
- **Projects as first-class entities** — full round trip from PostgreSQL schema to Mac UI with optimistic updates, covering PROJ-01 through PROJ-07 (7/7)
- **Menu-bar update action** — replaced the manual `install.sh` dev loop, covering DEV-01 through DEV-04 (4/4)
- **Three infrastructure wins** — push-on-complete hook (structural fix for 53-04), bootstrap.sh + doctor.sh (cross-machine portability foundation), Phase 55 NO-OP verification (corrected a false belief in user memory)

### What didn't ship
- **Smart Photo Upload (Phase 54)** — 6 PHOTO-XX requirements deferred to v2.4 cleanly via `.planning/deferred-to-v2.4.md`. Not touched.

### Key decisions
- Promote 55/56/57 from backlog mid-milestone rather than finishing 54 first — trading scope completeness for structural safety in response to the 53-04 incident
- Phase 55 closes as NO-OP after investigation rather than being removed from roadmap — keeps the record of "this was investigated and found already-done"
- Phase 56 modifies `~/.claude/get-shit-done/` global install rather than project-local hooks — single CLI chokepoint covers all three phase-complete entry points
- Phase 57 accepts `status: "degraded"` on local vigil-core as steady state — matches reality (Mac apps use Railway), avoids scope creep into DATABASE_URL configuration

### Known issues / tech debt
- **Phase 54 not built** — deferred to v2.4, captured in deferred-to-v2.4.md
- **Phase 55 criterion #2 PARTIAL** — failed migration crash-loops instead of surfacing as a quarantined failed deploy. `preDeployCommand` hardening deferred in CONTEXT D-04 until first real prod incident
- **Phase 57 end-to-end bootstrap not validated** — `op` CLI not installed on current machine; happy path only runs when user next provisions a fresh Mac
- **Phase 57 plan's "break apiBaseUrl + restart" verification recipe is broken** — empty thought list can't trigger the assign path. Use Wi-Fi toggle instead. Worth updating the UI-SPEC template for future checkpoints

### Phase 56 hook — the moment it saved the day
During Phase 53 UAT completion on 2026-04-08, the user approved 13/13 verification steps. Running `gsd phase complete 53` triggered the newly-built push hook, which detected 22 backlogged `vigil-core/` commits and auto-pushed them to origin. **Without Phase 56, the exact 53-04 bug would have recurred** — the user would have approved 53 complete while Railway still ran stale backend code. This was the first real fire of the hook and it validated the entire Phase 56 investment in a single command.
