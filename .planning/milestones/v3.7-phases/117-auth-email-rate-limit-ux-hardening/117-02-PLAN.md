Retry-After header parsing semantics (HTTP/1.1 RFC 7231 §7.1.3):
- delay-seconds: positive integer (e.g., "120")
- HTTP-date (e.g., "Wed, 21 Oct 2015 07:28:00 GMT") — RARE in practice, NOT supported by this classifier (consistent with Phase 116.1's range-guard). If the header is non-numeric, fall through to the body-field fallback OR return retryAfter undefined.

    ```typescript
    export async function classifyFetchError(input: Response | Error | unknown): Promise<ErrorClass> {
      // Non-Response throws (TypeError, AbortError, DOMException) → network bucket.
      if (!(input instanceof Response)) {
        return { kind: 'network' }
      }

      const res = input
      if (res.status === 401 || res.status === 403) {
        return { kind: 'auth' }
      }

      // Phase 117 (AUTH-12 D-10): 429 rate-limited bucket. retryAfter sourced
      // from Retry-After header (preferred — HTTP-spec source of truth) with
      // fallback to body.retryAfter field. Range-guard mirrors the 502 branch
      // below: must be a finite positive number ≤ 86400. If neither source is
      // valid, return without retryAfter (countdown UI suppresses itself).
      if (res.status === 429) {
        // Try header first.
        const headerRaw = res.headers.get('Retry-After')
        if (headerRaw !== null) {
          // Only accept delay-seconds (RFC 7231 §7.1.3) — HTTP-date format not supported.
          // parseInt handles "120", "120 \n" etc; rejects "Wed, 21 Oct...".
          const parsed = parseInt(headerRaw, 10)
          if (Number.isFinite(parsed) && parsed > 0 && parsed <= 86_400 && String(parsed) === headerRaw.trim()) {
            return { kind: 'rate-limited', retryAfter: parsed }
          }
        }
        // Header missing or invalid — fall back to body.retryAfter field.
        try {
          const body = await res.clone().json() as { error?: string; retryAfter?: unknown }
          const ra = body?.retryAfter
          if (typeof ra === 'number' && Number.isFinite(ra) && ra > 0 && ra <= 86_400) {
            return { kind: 'rate-limited', retryAfter: ra }
          }
        } catch {
          // Body parse failure — return without retryAfter.
        }
        return { kind: 'rate-limited' }
      }

      if (res.status === 502) {
        // Try to extract retryAfter from body. Body shape from Plan 02: { error, retryAfter? }.
        try {
          const body = await res.clone().json() as { error?: string; retryAfter?: unknown }
          const ra = body?.retryAfter
          if (typeof ra === 'number' && Number.isFinite(ra) && ra > 0 && ra <= 86_400) {
            return { kind: 'upstream', retryAfter: ra }
          }
        } catch {
          // Body parse failure — fall through to no-retryAfter upstream.
        }
        return { kind: 'upstream' }
      }
      // 500, 503, 504, any other 5xx, and unexpected non-ok responses → server bucket.
      return { kind: 'server' }
    }
    ```

    Note the parseInt strictness: `String(parsed) === headerRaw.trim()` rejects values like "120abc" or "120.5" that parseInt would otherwise accept. This is critical for AUTH-12-CFE-RL-07 (HTTP-date string).

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Server 429 response → PWA classifier | The Response object reaches the classifier from the page's fetch call. The header value and body field are server-controlled — but the SAME server (vigil-core) is the trust source for the rest of the auth flow, so the classifier trusts both inputs subject to range-guard. |
| Header `Retry-After` → countdown UI seconds | A malicious server could in principle send `Retry-After: 86400` to lock the user out for a day. Range guard caps at 86400. Beyond that, the user can manually navigate away — the countdown is UX, not a hard lockout. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-117-02-01 | Tampering | Retry-After header value injection (CRLF, etc.) | mitigate | Header parsing uses parseInt + strict equality check `String(parsed) === headerRaw.trim()` — rejects any non-numeric trailing characters. Even if a header CRLF were to make it through fetch's parser (it shouldn't — fetch normalizes headers), the parseInt sees the digit prefix only and rejects on the equality check. |
| T-117-02-02 | Denial of Service | Server sends `Retry-After: 999999999` to lock user out | mitigate | Range guard `parsed <= 86_400` (1 day) caps the lockout duration. Beyond range, retryAfter is undefined and the page falls back to non-countdown UX (Submit re-enables immediately on next render). |
| T-117-02-03 | Information Disclosure | Body parse error path reveals server internals | mitigate | The try/catch around body parse silently swallows parse failures and falls through to `return { kind: 'rate-limited' }` (no retryAfter). No error message surfaces to the UI. |
| T-117-02-04 | Spoofing | Body retryAfter spoofs a different value than header | mitigate | Header is checked FIRST (HTTP-spec source of truth); body is fallback only. AUTH-12-CFE-RL-03 test pins this ordering. |
| T-117-02-05 | Tampering | Phase 116.1's 502 bucket regresses due to extension | mitigate | AUTH-12-CFE-REGRESSION-OTHERS test asserts 502+retryAfter=30 returns `{ kind: 'upstream', retryAfter: 30 }` post-edit. Triple-redundant since Phase 116.1's existing tests in client.test.ts (if any) would also catch it. |

## Residual Risks

- **HTTP-date Retry-After format unsupported** — accepted per Phase 116.1 precedent (consistent with the 502 bucket which only accepts numeric body field). Auth routes only emit delay-seconds format (`String(retryAfterSec)` in vigil-core), so no real-world impact.
- **Classifier doesn't distinguish between auth-route 429 and a hypothetical sports-route 429** — the 429-status check is global. Today, sports-route 429s collapse to 502 server-side (Phase 116.1 Plan 02), so this is moot. If a future endpoint emits 429 directly, all PWA error-handling that calls classifyFetchError will route into the rate-limited branch — likely the desired behavior.
