## Summary

Phase 116.1 cleanly differentiates sports upstream error classes across three layers (service → route → PWA). The classification correctness is sound: `UpstreamError` collapses 4 service kinds into HTTP 502 with optional `retryAfter`, and the PWA's 4-bucket taxonomy (`auth`, `upstream`, `server`, `network`) maps correctly per the documented invariants.

**Strengths verified:**
- T-73-01 invariant holds: provider name "BDL"/"balldontlie" appears only in the `console.log` line at sports-service.ts:355, never in `UpstreamError.message`, route response bodies, PostHog event payloads, or PWA error copy. Multiple tests assert this (sports-service.test.ts:580, 624; sports.test.ts:480, 505, 516).
- `Retry-After` header parsing is properly defense-in-depth — `parseInt` + `1..86400` range guard at sports-service.ts:366 prevents header injection via CRLF or oversized values; the same guard is mirrored client-side in `classifyFetchError` (client.ts:942).
- AbortController cleanup is correct in both success (line 349) and error (line 341) paths in `fetchJSON`.
- React countdown timer cleanup is comprehensive: per-league timer refs, unmount cleanup at SettingsPage.tsx:334-344, and clear-on-new-fetch at line 253-256 prevent setState-after-unmount and timer leaks.
- PostHog event payload is enum-only (league literal + `'error'` status + extracted kind via regex) — no URL, no headers, no API key.

**Concerns (none Critical):** Two warnings around brief-assembly telemetry coverage gaps and a minor robustness check, plus 5 info-level cleanups.

## Warnings

### WR-01: Catastrophic sports failure emits no PostHog telemetry

**File:** `vigil-core/src/services/brief-assembly-service.ts:578-599`
**Issue:** When `sportsR.status === "rejected"` (e.g., `withTimeout` fires the source-timeout, or `sportsService.fetchAllLeagues` itself throws synchronously), the code only writes a console.log line — no `sports_league_fetch_failed` PostHog event is emitted. This means the very failure modes most worth tracking for cohort/funnel analytics (entire-source failure, source-level timeout) produce zero telemetry. Per-league events fire only inside `fulfilled` responses, leaving a blind spot for the worst-case scenarios.

The CONTEXT D-06 ("emit a PostHog event per failed league") is technically satisfied per-league, but operationally the brief is now blind to "sports source died entirely" events.

**Fix:** Emit a per-league `'unknown'` error_class event when the entire source rejects, so dashboards can distinguish "BDL had a bad day" (per-league) from "our timeout fired" (catastrophic):
```ts
if (sportsR.status === "rejected") {
  console.log(`[brief-assembly] Sports source failed: ${sportsR.reason instanceof Error ? sportsR.reason.message : "unknown"}`);
  // Emit a single catastrophic event so dashboards see source-level failures.
  trackEventImpl(userId, "sports_league_fetch_failed", {
    league: "all",
    status: "error",
    error_class: sportsR.reason instanceof Error && sportsR.reason.message === "Source timeout"
      ? "timeout"
      : "unknown",
  });
}
```

### WR-02: `extractErrorClass` regex is fragile to message-format drift

**File:** `vigil-core/src/services/brief-assembly-service.ts:73-78`
**Issue:** The regex `/Upstream sports provider failed \((auth|server-error|timeout|rate-limited)\)/` is tightly coupled to the exact message string constructed in sports-service.ts:38. If anyone ever edits that message (e.g., to "Sports provider failure (timeout)"), telemetry silently degrades to `"unknown"` for all events — a hidden contract with no compile-time enforcement. Today, `LeagueResult.error` is a `string` (from `err.message`), losing the structured `kind` field that was already present on the `UpstreamError` instance in sports-service.ts.

The lossy `Error → string → regex extraction` round-trip exists because `LeagueResult.error` is typed as `string | undefined`. The cleaner shape would propagate `kind` as a structured field.

**Fix:** Extend `LeagueResult` to carry the kind directly instead of round-tripping through a stringified message:
```ts
export interface LeagueResult {
  status: "ok" | "error" | "off_season" | "disabled";
  error?: string;
  errorKind?: "auth" | "server-error" | "timeout" | "rate-limited";  // NEW
  data?: LeagueData;
}
```
Then in fetchAllLeagues' `settledToResult`, when the rejection is an `UpstreamError`, populate `errorKind: r.reason.kind`. brief-assembly's telemetry then reads `lr.errorKind ?? "unknown"` directly — no regex, no string parsing, no drift risk.

Until that refactor lands, add a code comment at sports-service.ts:38 warning that the message format is parsed downstream.

## Info

### IN-01: Redundant condition in countdown render guard

**File:** `vigil-pwa/src/pages/SettingsPage.tsx:1031-1032`
**Issue:** `const isCountingDown = retryCountdownValue !== null && retryCountdowns[league] !== null && retryCountdownValue > 0` — `retryCountdownValue` was just assigned from `retryCountdowns[league]` on line 1031, so the second clause is dead. Reads slightly confusing and could mask a future bug if someone reorders the lines.
**Fix:** Drop the redundant check:
```tsx
const isCountingDown = retryCountdownValue !== null && retryCountdownValue > 0
```

### IN-02: `res.json()` body read is not under AbortController coverage

**File:** `vigil-core/src/services/sports-service.ts:349-372`
**Issue:** `clearTimeout(timeoutId)` runs at line 349 immediately after the headers/status are received. If BDL sends headers fast but stalls the response body, `res.json()` at line 372 can hang indefinitely — the original 10s `FETCH_TIMEOUT_MS` no longer protects this phase. In practice BDL responses are small (<10KB) and unlikely to stall mid-body, but the AbortController contract is technically incomplete.
**Fix:** Defer the `clearTimeout` until after `res.json()` resolves, or wrap the JSON read in a separate timeout race. Low priority — BDL behavior makes this a theoretical issue.

### IN-03: Retry-After HTTP-date format silently ignored

**File:** `vigil-core/src/services/sports-service.ts:364-366`
**Issue:** RFC 7231 allows `Retry-After` to carry either delta-seconds OR an HTTP-date (e.g., `"Wed, 21 Oct 2026 07:28:00 GMT"`). The current code uses `parseInt`, which returns NaN on date strings → `retryAfter` becomes `undefined`. BDL is documented to use seconds, so this is fine today, but if BDL ever changes to date format, throttling guidance is silently lost.
**Fix:** Add a fallback `Date.parse()` branch when `parseInt` returns NaN, computing seconds-from-now and clamping to the same `1..86400` range. Document explicitly that we only support delta-seconds in a comment if you'd rather not implement.

### IN-04: Duplicate `if (err.retryAfter !== undefined)` block

**File:** `vigil-core/src/routes/sports.ts:32-40`
**Issue:** Two consecutive `if (err.retryAfter !== undefined)` blocks (lines 32 and 38) — one builds the body, one sets the header. Could be merged for readability.
**Fix:**
```ts
if (err.retryAfter !== undefined) {
  body.retryAfter = err.retryAfter;
  c.header("Retry-After", String(err.retryAfter));
}
```

### IN-05: Per-league fetch loop in `loadSportsSelections` ignores promise results

**File:** `vigil-pwa/src/pages/SettingsPage.tsx:309-312`
**Issue:** The fire-and-forget `for (const league of result.enabledLeagues) { void loadTeamsForLeagueImpl(league); }` parallelizes the prefetch fan-out, which is intentional. Worth a one-line comment noting that each call internally manages its own state, so concurrent fetches for distinct leagues are safe (different state slots, different timer refs). Without the comment, future readers may worry about race conditions.
**Fix:** Add a brief comment:
```tsx
// Fan out concurrently — each call writes to its own teamsByLeague slot and
// owns its own countdown timer ref, so no inter-league races.
for (const league of result.enabledLeagues) {
  void loadTeamsForLeagueImpl(league)
}
```
