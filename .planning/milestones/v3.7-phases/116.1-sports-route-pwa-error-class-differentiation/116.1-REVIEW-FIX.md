### WR-02: `extractErrorClass` regex is fragile to message-format drift

**Files modified:** `vigil-core/src/services/sports-service.ts`, `vigil-core/src/services/sports-service.test.ts`, `vigil-core/src/services/brief-assembly-service.ts`, `vigil-core/src/services/brief-assembly-service.test.ts`
**Commit:** 6d40a32
**Applied fix:**

Replaced the lossy `Error -> string -> regex extraction` round-trip with a structured `errorKind` field on `LeagueResult`:

1. Extracted `UpstreamErrorKind` as an exported type alias on sports-service.ts (`"rate-limited" | "server-error" | "timeout" | "auth"`).
2. Added `errorKind?: UpstreamErrorKind` to the `LeagueResult` interface alongside the existing `error?: string` (kept for backward compatibility / human-readable diagnostics).
3. Updated `fetchAllLeagues`'s `settledToResult` helper: when the rejection is an `UpstreamError`, the resulting `LeagueResult` now carries `errorKind: r.reason.kind` directly. Non-Upstream errors keep the existing `{ status: "error", error: String(reason) }` shape (errorKind absent).
4. In brief-assembly-service.ts: deleted the `UPSTREAM_KIND_RE` regex and rewrote `extractErrorClass` to read the structured field directly: `errorKind ?? "unknown"`. Updated the per-league telemetry call site to pass `lr.errorKind` instead of `lr.error`.
5. Added a code comment on `UpstreamError`'s `super(...)` line explaining that the message format is no longer authoritative — the structured `kind` field is.
6. Added 3 new sports-service tests asserting `errorKind` propagation through `settledToResult` for 401/429/500 status codes.
7. Updated the existing brief-assembly PostHog telemetry test fixture to include `errorKind` on the `error`-status leagues.

Result: telemetry no longer depends on the `Upstream sports provider failed (${kind})` message format. Future edits to that string will not silently break dashboards.

### WR-01: Catastrophic sports failure emits no PostHog telemetry

**Files modified:** `vigil-core/src/services/brief-assembly-service.ts`, `vigil-core/src/services/brief-assembly-service.test.ts`
**Commit:** 422c212
**Applied fix:**

When `sportsR.status === "rejected"` (entire sports source died — `withTimeout` fired or the service threw synchronously), brief-assembly now emits a single catastrophic `sports_league_fetch_failed` PostHog event in addition to the pre-existing console.log. Property shape:

```ts
{
  league: "all",                    // distinguishes from per-league mlb/nfl/nba/nhl events
  status: "error",
  error_class: "timeout" | "unknown" // matches "Source timeout" string from withTimeout, else "unknown"
}
```

The `userId` is the same one passed to `assembleAndRender`. T-73-01 invariant preserved — properties are enum literals only, no URL, no apiKey, no stack trace text. Reuses the existing injectable `trackEventImpl` (test seam from D-06) so the new branch is testable without mocking PostHog directly.

Added 2 new tests:
- `SPORTS-01b-brief-posthog-event-fires-on-source-timeout-WR01` — sports service hangs forever, `_sourceTimeoutMs=10` forces withTimeout to reject; asserts single event fires with `error_class: "timeout"`.
- `SPORTS-01b-brief-posthog-event-fires-on-synchronous-throw-WR01` — sports service throws `new Error("boom")`; asserts single event fires with `error_class: "unknown"`.

Result: dashboards can now distinguish per-league BDL failures (existing `league: "mlb" | "nfl" | "nba" | "nhl"` events) from catastrophic source-level failures (new `league: "all"` event). Operational blind spot for "sports source died entirely" closed.
