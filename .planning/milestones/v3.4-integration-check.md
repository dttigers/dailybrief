### Focus Area 4: vk_ Backcompat (AUTH-04)

`bearerAuth` middleware (`vigil-core/src/middleware/auth.ts`):

vk_ path (lines 50–87): Hashes token → queries `api_keys` → reads `row.userId` → `c.set("userId", row.userId)`. Defensive NULL guard at line 69 returns 500 rather than silently treating as "no data." vk_ path sets `userId` identically to the JWT path.

JWT path (lines 90–102): `verifyToken(token)` → `Number(claims.sub)` → integer validation → `c.set("userId", userId)`.

Both paths call `c.set("userId", <number>)` before `return next()`. Downstream routes call `c.get("userId")` which returns the same `number` type regardless of which path was taken. **CONNECTED — vk_ backcompat preserved.**

Live verification: go/no-go curl #1 confirmed `GET /v1/summary` with seed vk_ key → HTTP 200 with seed-user data.

### Focus Area 5: Google OAuth userId Round-Trip (AUTH-05)

`GET /v1/auth/google` (line 52, google-auth.ts):
- Reads `const userId = c.get("userId")` — requires bearer (route is behind auth middleware)
- Signs state JWT: `new SignJWT({ nonce, userId }).setExpirationTime("5m").sign(secret)`

`GET /v1/auth/google/callback` (line 86):
- Verifies state JWT: `jwtVerify(state, secret, { algorithms: ["HS256"] })`
- Checks `typeof payload.userId !== "number" || payload.userId <= 0` → rejects invalid
- Sets `verifiedUserId = verifyResult.userId`
- Upserts `oauth_tokens` with `userId: verifiedUserId` using composite unique `(userId, provider)`

The userId round-trip is cryptographically anchored — only a valid JWT_SECRET can produce a state JWT that passes the callback's jwtVerify. **CONNECTED — Google OAuth userId carry is correctly wired.**

### Work-Order-Status Route: Unscoped Table (WARNING)

**Route:** `GET /v1/work-orders/statuses` and `PUT /v1/work-orders/:caseNumber/status`
**File:** `vigil-core/src/routes/work-order-status.ts`
**Schema:** `work_order_statuses` table has NO `userId` column (confirmed in `vigil-core/src/db/schema.ts` lines 224–228)

This table stores `(caseNumber, status)` pairs as a global lookup — it has no user ownership column. Any authenticated user can read and write any case number's status. This is consistent with Phase 102's stated scope (11 tables with userId FK) — `work_order_statuses` appears to be intentionally shared/global. However:
- It's not in the isolation tests (confirming it was scoped out of AUTH-05 scrutiny)
- There is no comment in the route file acknowledging the intentional un-scoped design

**Impact:** At single-user scale (current), not a problem. When multi-user is live and user B can see user A's work order case numbers, they can also silently overwrite A's status. This is a future-phase concern, not a current blocker.

---

### Calendar Service: Seed-User-Only (INFO)

`calendar.ts` delegates to `createCalendarService()` which reads `oauth_tokens` keyed to the seed user's lookup. The route has no `c.get("userId")` call because it uses the service-level seed scoping. This means any authenticated user's JWT can trigger a calendar events fetch, but it will always return the seed user's calendar data. This is the documented Phase 102 carry-forward (per 102-05-SUMMARY.md "AUTH-06+ scheduler per-user fan-out"). **Not a bug — an explicit known deferral.**
