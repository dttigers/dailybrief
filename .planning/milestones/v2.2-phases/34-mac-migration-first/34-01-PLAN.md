<task type="auto">
  <name>Task 1: Add date range filtering to GET /thoughts</name>
  <files>vigil-core/src/routes/thoughts.ts</files>
  <action>
Add `after` and `before` query parameters to the GET /thoughts endpoint. These accept ISO 8601 date strings and filter on the `createdAt` column.

- `after`: Include thoughts created AFTER this date (`createdAt > ?`)
- `before`: Include thoughts created BEFORE this date (`createdAt < ?`)
- Both are optional and can be combined with existing filters
- Add them as conditions in the existing conditions array, same pattern as other filters
- Validate that values are parseable as dates (return 400 if not)

This enables the Mac app's `fetchFiltered(after:)`, `fetchTherapyThoughtsByDateRange(from:to:)`, and `fetchRecentTherapyThoughts(days:)` methods to work through the API.
  </action>
  <verify>
Start the vigil-core server and test:
- `curl "http://localhost:3001/v1/thoughts?after=2026-01-01T00:00:00Z"` returns only thoughts after that date
- `curl "http://localhost:3001/v1/thoughts?before=2026-12-31T00:00:00Z&after=2026-01-01T00:00:00Z"` returns thoughts in range
- `curl "http://localhost:3001/v1/thoughts?after=not-a-date"` returns 400
- Existing filters still work unchanged
  </verify>
  <done>GET /thoughts accepts `after` and `before` ISO 8601 query params for date range filtering, with validation</done>
</task>

<task type="auto">
  <name>Task 2: Add bulk therapy classification endpoint</name>
  <files>vigil-core/src/routes/bulk.ts</files>
  <action>
Add POST /thoughts/bulk/therapy-classify endpoint to the existing bulk routes file. This maps to the Mac app's `bulkUpdateTherapyClassification(ids:classification:)` method.

Request body: `{ ids: number[], classification: "selfLearnable" | "bringToTherapist" }`

Implementation:
- Validate `ids` is non-empty array of numbers
- Validate `classification` is one of the two valid values
- Use a single UPDATE statement with `id IN (...)` pattern (same approach as existing bulk endpoints)
- Set `modifiedAt` to current timestamp and `syncStatus` to 'pending'
- Return `{ updated: number }` with count of affected rows

Follow the exact same patterns as the existing bulk/delete and bulk/recategorize endpoints in the same file.
  </action>
  <verify>
Start vigil-core and test:
- `curl -X POST http://localhost:3001/v1/thoughts/bulk/therapy-classify -H "Content-Type: application/json" -d '{"ids":[1,2],"classification":"selfLearnable"}'` returns `{ updated: N }`
- Invalid classification returns 400
- Empty ids array returns 400
  </verify>
  <done>POST /thoughts/bulk/therapy-classify endpoint works with validation, matching existing bulk endpoint patterns</done>
