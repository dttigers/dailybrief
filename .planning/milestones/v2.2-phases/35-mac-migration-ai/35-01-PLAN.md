  <action>
    Create `Sources/JarvisCore/Services/AIServiceProtocols.swift` with 6 protocols, each requiring Actor conformance (matching ThoughtRepository pattern from Phase 34):

    1. `TriageProviding` — `func triage(_ content: String) async throws -> TriageResult`
    2. `InsightProviding` — `func generateInsights(thoughts: [Thought], lookbackDays: Int) async throws -> [Insight]`
    3. `ImageDescriptionProviding` — `func describe(imageData: Data, mediaType: ImageMediaType) async throws -> String` and `func describe(imageURL: URL) async throws -> String`
    4. `TherapyClassifyProviding` — `func classify(_ content: String) async throws -> TherapyClassificationResult`
    5. `TherapyPatternProviding` — `func detectPatterns(thoughts: [Thought]) async throws -> [TherapyPattern]`
    6. `TherapyPrepProviding` — `func generatePrep(thoughts: [Thought], patterns: [TherapyPattern]?) async throws -> TherapyPrep`

    All protocols must be `public` and require `Actor` conformance for concurrency safety.

    Then add conformance declarations to each existing service. Since each existing service is already an actor with matching method signatures, conformance should be a simple extension:
    ```swift
    extension TriageService: TriageProviding {}
    extension InsightService: InsightProviding {}
    ```
    etc. Add these extensions at the bottom of each service's own file.

    Check existing method signatures carefully — if any parameter names or types differ slightly from the protocol, adjust the protocol to match (the existing services are the source of truth for signatures).

    Do NOT add the `describe(imageURL:)` method to the protocol if ImageDescriptionService doesn't have it, or vice versa. Match what exists. If ImageDescriptionService has both overloads, include both in the protocol. If only one, only include that one.
  </action>
  <verify>
    `swift build` compiles without errors. All 6 existing services conform to their protocols.
  </verify>

<task type="auto">
  <name>Task 2: Create API-backed AI service implementations</name>
  <files>Sources/JarvisCore/Services/APIAIServices.swift</files>
  <action>
    Create `Sources/JarvisCore/Services/APIAIServices.swift` with 6 actor implementations, each using VigilAPIClient to call the corresponding Vigil Core endpoint instead of the Anthropic API directly.

    All actors take `VigilAPIClient` in their init (same pattern as APIThoughtStore).

    **APITriageService: TriageProviding**
    - POST to `/triage` with body `{"content": "..."}` 
    - Response: `{"category": "task", "confidence": 0.95}`
    - Map response to existing `TriageResult` struct

    **APIInsightService: InsightProviding**
    - POST to `/insights` with body `{"thoughts": [{id, content, category, createdAt}], "days": N}`
    - Response: `{"insights": [{type, title, message, confidence, relatedThoughtIds}]}`
    - Map snake_case `related_thought_ids` to camelCase. Create Codable DTOs as needed.
    - Map response to existing `Insight` array

    **APIImageDescriptionService: ImageDescriptionProviding**
    - POST to `/describe-image` with body `{"image": "<base64>", "mediaType": "image/jpeg"}`
    - Response: `{"description": "..."}`
    - For the imageURL variant, read file data and convert to base64 before posting
    - For the imageData variant, base64-encode the Data and post

    **APITherapyClassificationService: TherapyClassifyProviding**
    - POST to `/therapy/classify` with body `{"content": "..."}`
    - Response: `{"classification": "selfLearnable"|"bringToTherapist", "confidence": 0.9, "reasoning": "..."}`
    - Map to existing `TherapyClassificationResult`

    **APITherapyPatternService: TherapyPatternProviding**
    - POST to `/therapy/patterns` with body `{"thoughts": [{id, content, category, createdAt}]}`
    - Response: `{"patterns": [{theme, description, frequency, trend, relatedThoughtIds, confidence}]}`
    - Map to existing `TherapyPattern` array

    **APITherapyPrepService: TherapyPrepProviding**
    - POST to `/therapy/prep` with body `{"thoughts": [{id, content, category, createdAt}], "patterns": [...]}`
    - Response: `{"items": [...], "overallThemes": [...], "suggestedFocus": "..."}`
    - Map to existing `TherapyPrep`

    Use private Codable DTOs (nested structs) for JSON encoding/decoding where the existing model types don't directly match the API response shape. Follow the same pattern as APIThoughtStore's response DTOs.

    Read the Vigil Core route files to confirm exact request/response JSON shapes before implementing. The API base URL already includes `/v1` prefix, so endpoint paths are relative (e.g., `/triage` not `/v1/triage`) — check how VigilAPIClient constructs URLs to confirm.

    Error handling: map VigilAPIError cases to the existing service-specific error types (TriageError, InsightError, etc.) so callers see the same error types regardless of backend.
  </action>

    `swift build` compiles without errors. All 6 API-backed services conform to their protocols.
