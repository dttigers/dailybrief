## Decisions Made

- **Detect-only daemon handling in dev-setup.sh** — per RESEARCH Pitfall 2 (Aqua-session bootout fails over SSH), dev-setup.sh detects `com.jamesonmorrill.vigilcore` and adds the retirement command to MANUAL_NOTES. The actual bootout is owned by Plan 04 (human-action checkpoint on each Mac). This plan is run *after* Plan 04 in wave 2 — the daemon should already be gone — but the detection is defensive: if a user ever restores the plist (e.g., during reversibility testing), dev-setup.sh will catch it and print the retirement command without silently failing.
- **MANUAL_NOTES array accumulator** — dev-setup.sh accumulates manual follow-ups during execution (daemon bootout, .env Railway migration, Anthropic workspace creation) and prints them all in one yellow block at the end. Alternative considered: print each warning inline. Rejected because the preflight call between the main work and the notes block would scroll past any inline warnings.
- **.gitignore covers both `.env.bak.*` and `*.env.bak.*`** — the former matches files directly in the repo root, the latter matches the `vigil-core/.env.bak.*` case actually produced by dev-setup.sh. Both patterns are cheap and future-proof against moving the `.env` to other subdirectories.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] Removed literal `VITE_API_URL` string from instructional comment**

- **Found during:** Task 2 acceptance-chain run (`grep -q 'VITE_API_URL'` must return 0 matches)
- **Issue:** Plan's `<action>` block prescribed a comment reading `"Research finding #2: the actual env var is VITE_API_BASE (NOT VITE_API_URL)."` — that literal string contains `VITE_API_URL`, which the plan's own `<automated>` verification chain (`! grep -q 'VITE_API_URL'`) rejects. Identical pattern to Plan 01's "Railway proxy" comment-vs-regex collision.
- **Fix:** Rephrased the comment to `"Research finding #2: the correct env var name is VITE_API_BASE (the common wrong guess has no effect in this codebase)."` Preserves educational intent (warn devs that there's a common wrong guess) without naming the wrong string — acceptance regex resolves to 0 matches.
- **Files modified:** `scripts/dev-setup.sh` (line 85)
- **Verification:** `grep -q 'VITE_API_URL' scripts/dev-setup.sh` → exit 1 (no match); full acceptance chain `OK`.
- **Committed in:** `35e784b` (fix applied inline before first commit of this file)

**2. [Rule 2 - Missing Critical] Added `.env.bak.*` and `*.env.bak.*` to .gitignore**

- **Found during:** T-LOCAL-2 backup smoke test — verified `dev-setup.sh` creates `vigil-core/.env.bak.20260421-175504`; `git status` then showed this as an untracked file because existing .gitignore patterns (`.env`, `*.env`, `.env.local`, `.env.*.local`) don't match timestamped backups.
- **Issue:** Every future invocation of `scripts/dev-setup.sh` over an existing .env (i.e. every non-first-time run) creates a new timestamped backup. Without the ignore rule, each backup shows up in `git status`, polluting every downstream dev loop and risking accidental `git add` via a too-broad add command.
- **Fix:** Added lines `.env.bak.*` (for root-dir backups) and `*.env.bak.*` (for subdir backups like `vigil-core/.env.bak.*`) to `.gitignore` after the existing `.env.*.local` line. Verified the new pattern matches via `git check-ignore -v`.
- **Files modified:** `.gitignore` (+2 lines)
- **Verification:** `git check-ignore -v vigil-core/.env.bak.20260421-175504` → `.gitignore:14:*.env.bak.*`; smoke-test backup removed; `git status` clean for vigil-core/.
- **Committed in:** `b150f9c` (separate chore commit — Rule 2 auto-add)
