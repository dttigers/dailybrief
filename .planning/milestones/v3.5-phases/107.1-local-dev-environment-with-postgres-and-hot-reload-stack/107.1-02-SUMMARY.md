# Phase 107.1 Plan 02: Local Fixture Seed Script — Summary

Idempotent `scripts/seed-local.ts` for `vigil-core/` populates a freshly migrated `vigil_dev` with the minimum fixtures needed by every PWA screen (1 user, 1 vk_ key, 5 thoughts spanning all 5 category values, 1 project, 1 work order), plus a new `seed:local` npm script wired with `tsx --env-file=.env` so `DATABASE_URL` is actually visible to the Node process at invocation time.

## Task Breakdown

### Task 1: Create `vigil-core/scripts/seed-local.ts`

**Type:** `auto`
**Status:** Complete.
**Commit:** `b077ea1` — `feat(107.1-02): add idempotent local fixture seed script`

**File:** `vigil-core/scripts/seed-local.ts` (205 lines).

**Structure (matches plan spec verbatim):**

| Fixture | Count | Idempotency mechanism |
|---------|-------|----------------------|
| Seed user (`jamesonmorrill1@gmail.com`) | 1 | `onConflictDoNothing({ target: users.email })` |
| Project (`Vigil Local Dev Fixture`) | 1 | Count-guard: `existingProjects.length === 0` (no unique name constraint) |
| API key (`local-dev-seed-key`, vk_-prefixed) | 1 | Count-guard: `existingKeys.length === 0` (preserves vk_ across re-runs) |
| Thoughts (one per category) | 5 | `onConflictDoNothing({ target: thoughts.cloudKitRecordID })` — each fixture has a unique `seed-local-{cat}-1` recordID |
| Work order (`SEED-WO-001`) | 1 | `onConflictDoNothing({ target: workOrders.caseNumber })` (caseNumber is PK) |

**Thought categories covered (all 5 per D-06):** `task`, `therapy`, `idea`, `reflection`, `project`. All rows use `source: "text"` and distinct `cloudKitRecordID` values (`seed-local-task-1`, `seed-local-therapy-1`, etc.).

**Password hashing:** `@node-rs/argon2` (same lib the `/v1/auth/register` route uses). Seed password `dev-password-1234` — local-only, dev can change via `POST /v1/auth/login` once the server is up.

**Exit semantics:**
- Missing `DATABASE_URL` → exit 1 with friendly `ERROR: DATABASE_URL is not set or unreachable. Ensure vigil-core/.env has DATABASE_URL=postgresql://localhost:5432/vigil_dev …` (verified live — see Verification section below).
- Seed-user insert fails and lookup returns no row → exit 1 with `ERROR: seed user insert failed and lookup returned no row. Check migrations: npm run db:migrate must have run.`
- Any thrown exception bubbles to the top-level `.catch` which logs `seed-local: FAILED` + the error and exits 1.
- Otherwise → exit 0 with a 5-line completion summary.

**vk_ key handling:** First run only — generates a fresh `crypto.randomBytes(32).toString("hex")` key, prints it to stdout with an "unrecoverable" warning, stores only the sha256 hash. Re-runs hit the count-guard and print `api_key already seeded — N existing key(s), no new vk_ key generated` instead. This preserves the key the dev copied on first run; rotating on every invocation would silently invalidate it.

### Live DB seed — NOT FULLY COMPLETED (pre-existing schema drift discovered)

Attempted end-to-end run on local `vigil_dev` (Plan 01 infrastructure). Steps taken:

1. Applied all `drizzle/0000`–`0012` migrations via `psql -f` (13 tables created).
2. Ran `migrate-102-seed.ts` to create the `users.email` seed row + `vigil.seed_email` GUC needed by migration `0012`.
3. Ran my `seed-local.ts` against the migrated DB.

**Result:** Seed script correctly inserted the user, project, and api_key rows, then crashed at the `work_orders` INSERT with:

```
PostgresError: column "notes" of relation "work_orders" does not exist (code 42703)
```

**Root cause:** `vigil-core/src/db/schema.ts` (line 232–252) defines `work_orders.notes`, `.last_change_at`, `.last_change_summary`, and `.archived_at`, but **no migration file in `drizzle/` creates these columns**. `grep -l 'notes\|archived_at\|last_change_at' vigil-core/drizzle/*.sql` returns zero matches.

**Why this is out of scope for Plan 02:**

1. The plan's critical_context explicitly instructs "This plan must assume the `vigil_dev` Postgres 16 DB exists and is migrated. Do not re-run migrations."
2. The plan's `<verification>` section provides an explicit fallback: "If Plan 01 was deferred, skip step 2-4 and verify only type-check + static grep." The schema-drift scenario is functionally identical — live verification is blocked by factors outside the seed-script surface.
3. Fixing the drift requires generating a new migration file (`0013_work_orders_drift_repair.sql`) via `drizzle-kit generate`, reviewing the diff for additive-only safety, and probably a parallel audit of other tables. This is firmly Rule 4 territory (architectural) for a follow-on plan.

**Action taken:** Logged to `.planning/phases/107.1-local-dev-environment-with-postgres-and-hot-reload-stack/deferred-items.md` with full context, suggested fix, and impact on Plans 03 + 04. Reset `vigil_dev` to empty state (dropdb + createdb) to leave the database in the same shape Plan 03's `dev-setup.sh` will encounter it — so Plan 03 must also handle this blocker.
