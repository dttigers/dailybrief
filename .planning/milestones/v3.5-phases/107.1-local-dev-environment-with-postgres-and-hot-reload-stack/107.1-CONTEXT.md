# Phase 107.1: Local Dev Environment with Postgres and Hot-Reload Stack - Context

**Gathered:** 2026-04-21
**Status:** Ready for planning

<domain>
## Phase Boundary

Make local development on vigil-core safe, fast, and self-contained. Today `npm run dev` in [vigil-core/](vigil-core/) connects to **production Postgres via Railway's public proxy** ([vigil-core/.env:3](vigil-core/.env#L3)) — every local edit mutates prod data. There is no local Postgres, no seed strategy, no unified dev command, and a vestigial `com.jamesonmorrill.vigilcore` launchd daemon reserves :3001 and forces a `launchctl bootout` dance before local dev can start.

This phase delivers, in order:

1. A local Postgres 16 cluster on the dev machine (both iMac and MacBook Pro).
2. A repeatable first-time setup script that provisions the local DB, runs existing Drizzle migrations, and seeds minimal fixture data.
3. Retirement of the `com.jamesonmorrill.vigilcore` launchd daemon — Mac apps all hit `api.vigilhub.io`, so the daemon serves no real traffic today.
4. A root-level unified dev command (`npm run dev` at repo root) that starts vigil-core + vigil-pwa together with interleaved logs and Ctrl+C stops both.
5. Env hygiene: `vigil-core/.env` repointed at LOCAL values; prod secrets remain canonical only in Railway. Separate Anthropic dev key with low budget cap.
6. Extensions to `scripts/dailybrief-doctor.sh` so drift doctor reports on local DB reachability (informational only).

**Explicitly out of scope:**
- Docker Compose / containerized dev (not chosen — native Homebrew is the pragmatic match).
- 1Password integration in the new dev-setup script (user doesn't use 1Password).
- Reworking `scripts/bootstrap.sh` (stale, 1Password-dependent — captured as deferred idea).
- CI runner setup, test DB isolation beyond local dev, per-branch DB isolation.
- Any change to Railway prod infrastructure, migrations, or env variables.
- Mac apps / Monitor / G2 plugin / Safari extension — all continue to hit api.vigilhub.io untouched.

</domain>

<decisions>
## Implementation Decisions

### Local Postgres provisioning

- **D-01:** Use **Homebrew `postgresql@16`** as the local Postgres install method. Rationale: Docker Desktop is not installed and adds a ~700MB + always-running surface for zero functional benefit given the schema uses only built-in `tsvector` (no pgvector, uuid-ossp, or other extensions). Native fits the repo's existing pattern (launchd, SMAppService, brew).
- **D-02:** Pin local Postgres to **Railway's prod version**. Researcher must verify Railway's current managed Postgres version before planning lands (likely 16.x). Prevents subtle dialect drift.
- **D-03:** Both **iMac and MacBook Pro** run identical local Postgres setups, documented once in RUNBOOK.md. Solo dev switches machines; silent drift between them is not acceptable.
- **D-04:** DB lives at the **default Homebrew location** (`/usr/local/var/postgresql@16` on Intel Macs — both machines confirmed Intel per memory). Single cluster, single `vigil_dev` database inside. Wipe path: `dropdb vigil_dev && createdb vigil_dev && npm run db:migrate && npm run seed:local`.

### Schema & seed bootstrapping

- **D-05:** Local DB schema is maintained via **existing Drizzle migrations** (`npm run db:migrate` runs `drizzle/0000` through `drizzle/0012` locally). Same path prod uses. No `db:push` shortcut — we want any new migration to be exercised locally before it ships to Railway.
- **D-06:** Local DB is seeded via a new **`scripts/seed-local.ts`** fixture script that inserts: one seed user (allowlisted email), one `vk_` API key, ~5 sample thoughts spanning categories, one sample work order, one project. Enough to exercise every screen without dump/restore complexity. Idempotent or drop-and-reseed semantics — planner decides exact shape.
- **D-07:** First-time setup is a single **`scripts/dev-setup.sh`** bootstrap script (NOT an extension of the 1Password-dependent `scripts/bootstrap.sh`). Idempotent. Steps: ensure `postgresql@16` brew service, `createdb vigil_dev` if missing, run migrations, run seed, write `.env` and `vigil-pwa/.env.local` templates from `.env.example` if missing. Printed in README.md.
- **D-08:** Mid-milestone schema resync uses the **same `npm run db:migrate`** as prod — no special dev-only path. Daily rhythm: `git pull && npm ci && npm run db:migrate`.

### Daemon vs dev-server conflict

- **D-09:** **Retire the `com.jamesonmorrill.vigilcore` launchd daemon on all dev machines** (iMac matches the MacBook-unloaded pattern). Mac apps all point at `https://api.vigilhub.io/v1` ([Sources/JarvisCore/Config/AppConfig.swift:28](Sources/JarvisCore/Config/AppConfig.swift#L28), [Sources/DailyBriefMonitor/AppDelegate.swift:66](Sources/DailyBriefMonitor/AppDelegate.swift#L66)) — the daemon serves no real traffic today. Retirement: `launchctl bootout gui/$UID ~/Library/LaunchAgents/com.jamesonmorrill.vigilcore.plist` + delete the plist file. Reversible (a future phase could reinstate if a caller emerges).
- **D-10:** `npm run dev` runs a **pre-flight safety check** (NOT auto-heal). Checks: daemon loaded? :3001 in use by another process? local Postgres running? `.env` DATABASE_URL pointing at `localhost`? On any failure, print the exact fix command and exit 1. Explicit > magic — matches the "investigate to root cause" feedback pattern.
- **D-11:** Local dev vigil-core listens on **port 3001** (same port the retired daemon used). PWA `.env.local` keeps `VITE_API_URL=http://localhost:3001`. Google OAuth redirect URI (`http://localhost:3001/v1/auth/google/callback`) unchanged. Zero reconfiguration required for callers.
- **D-12:** Remove `vigil-core/dist/` from the daemon workflow. Since the daemon is retired, local `npm run build` is only needed for prod verification (Railway builds its own image). Stale `dist/` is no longer a hazard, but the local `.gitignore` already excludes it — no additional work needed beyond documentation.

### Hot-reload orchestration

- **D-13:** Unified dev runner lives in a new **root `package.json` + `concurrently`** devDep. The root `dev` script runs `concurrently --names core,pwa --prefix-colors blue,magenta "npm --prefix vigil-core run dev" "npm --prefix vigil-pwa run dev"`. Minimal install (~30 lines in `node_modules/concurrently`), colored prefixed logs, Ctrl+C kills both. No shell-script signal-handling to maintain.
- **D-14:** Root `dev` script **calls the pre-flight doctor check first** (from D-10), then hands off to `concurrently`. Fails loud before either server starts if the environment is wrong.
- **D-15:** The unified `dev` only starts **vigil-core + vigil-pwa**. Other surfaces (Monitor, Safari extension, G2 plugin) keep their current workflows untouched.
- **D-16:** **Does NOT auto-start `brew services postgresql@16`**. Postgres is a one-time `brew services start` and then stays up across reboots (brew services is sticky). Adding it to every dev invocation is toggle-churn.

### Env & secrets hygiene

- **D-17:** After this phase, **`vigil-core/.env` contains LOCAL-ONLY values** — localhost DATABASE_URL, dev JWT_SECRET, `VIGIL_ALLOWED_EMAILS` including the seed-user email for local register/login, dev Anthropic key, etc. Prod secrets remain **canonical only in Railway**. Matches RUNBOOK "secret-drift policy" and removes prod-credential blast radius from the dev disk.
- **D-18:** A new **dev-scoped Anthropic API key** with a **low monthly budget cap** (default $20/month, revisitable) is created and used in local `.env`. Bounds blast radius if a dev loop bursts requests. Updates `sync-anthropic-key.sh` and `dailybrief-doctor.sh` to recognize that local and Railway keys are EXPECTED to differ (drift doctor no longer flags this as a drift).
- **D-19:** `scripts/dailybrief-doctor.sh` gains an **INFORMATIONAL row** for local DB reachability and migration currency. Follows existing D-13 (doctor-script decision D-13) pattern: informational rows do NOT affect exit code. Makes doctor the one-stop "is my dev env sane?" check.
