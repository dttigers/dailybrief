**Researched:** 2026-04-21

## Summary

Phase 107.2 is a tightly-scoped 6-file change: one prod-safety guard in `vigil-core/src/index.ts`, one hostname param on `serve()`, one proxy-target flip in `vigil-pwa/vite.config.ts`, one `server.host: true` flag, one new preflight check, and a RUNBOOK section. Every technical unknown from CONTEXT.md has been resolved by inspecting the actual libraries installed in this repo (no assumptions from training data):

1. **`@hono/node-server` 1.19.12 (installed)** supports `hostname` in its options object verbatim — the internal `server.listen(port, hostname, ...)` call passes it straight through to Node's HTTP server [VERIFIED: repo source `node_modules/@hono/node-server/dist/types.d.ts:41` + `server.js:1`].
2. **`vite-plugin-pwa` 1.2.0 (installed)** defaults `devOptions.enabled` to `false` and the current `vite.config.ts` never sets it — so no dev service worker is registered. D-E3 stands [VERIFIED: repo source `node_modules/vite-plugin-pwa/dist/index.d.ts` DevOptions interface].
3. **`hono/cors` with `origin: "*"` and `credentials` unset** is a safe combination — the middleware conditionally sets `Access-Control-Allow-Credentials` ONLY when `opts.credentials` is truthy. The current config doesn't set credentials, bearer tokens are applied manually in `vigilFetch`, so D-D1's wildcard is functionally correct [VERIFIED: `node_modules/hono/dist/middleware/cors/index.js` source].
4. **macOS Application Firewall IS independent of Tailscale utun** and can block inbound TCP on 0.0.0.0 — preflight check 5 (D-B3) is therefore actionable, not informational [CITED: Tailscale GitHub issue #454, #3185].
5. **`socketfilterfw --getglobalstate` works on this machine**, `defaults read /Library/Preferences/com.apple.alf globalstate` does NOT (returns "does not exist" on macOS 15) [VERIFIED: live `bash` on iMac 2026-04-21].
6. **Vite HMR runs on the main dev-server port (5173) via its own WebSocket**, separate from the `/v1` proxy — so adding `ws: true` to the proxy block is neither required nor helpful for our use case [VERIFIED: Vite docs + repo grep].
7. **Vite auto-loads `vigil-pwa/.env.local`** in dev (it's the standard `.local` file, higher priority than `.env`) — but env vars used inside `vite.config.ts` itself must be read via `loadEnv()`, NOT `process.env` [CITED: vite.dev/guide/env-and-mode].

**Primary recommendation:** Ship Phase 107.2 as planned in CONTEXT.md. The locked decisions D-A1..D-E3 are all implementable exactly as written. Add one small refinement the planner should capture: the Vite proxy target env var (`VITE_DEV_API_TARGET`) must be loaded via `loadEnv()` in `vite.config.ts` (not `import.meta.env`) because it's consumed at Vite startup time, not at browser bundle time.

## User Constraints (from CONTEXT.md)

### Locked Decisions

**Access model (Area 1)**
- **D-A1:** Thin-client architecture. MacBook Pro browser connects to iMac's `npm run dev` stack over Tailscale. Single `vigil_dev` Postgres, single source of truth. Trade-off accepted: if iMac is off / not running `npm run dev`, laptop cannot dev.
- **D-A2:** Both vigil-core (3001) and Vite (5173) exposed over Tailscale. The Vite proxy forwards API calls server-side (see D-C1), so the browser sees same-origin traffic, but both ports need to bind to a non-localhost interface.

**Bind strategy + prod safety (Area 2)**
- **D-B1:** vigil-core binds `0.0.0.0` in dev (all interfaces — Tailscale + LAN + localhost).
- **D-B2:** Prod safety via `VIGIL_BIND_HOST` env var. If unset, vigil-core binds `127.0.0.1` (localhost only — safe default). Dev `.env` sets `VIGIL_BIND_HOST=0.0.0.0`. Because `.env` is gitignored (D-17 from Phase 107.1), Railway prod never inherits this and falls through to the localhost default behind Railway's proxy. Belt-and-suspenders with D-D2.
- **D-B3:** Preflight check #5 added to `scripts/preflight-check.sh`. If `VIGIL_BIND_HOST=0.0.0.0` AND macOS Application Firewall is disabled, warn loudly but don't block. Keeps Plan 03 fail-loud-informative culture.
- **D-B4:** `.env.example` gets a documented `VIGIL_BIND_HOST` line with explicit `# LOCAL ONLY — Railway MUST leave this unset` comment.

**PWA → vigil-core routing (Area 3)**
- **D-C1:** Vite proxy handles API routing. `vigil-pwa/vite.config.ts` rewrites the `/v1` proxy target from `https://api.vigilhub.io` to `http://localhost:3001`. Same-origin traffic to port 5173 — no CORS involvement, no `VITE_API_BASE` change needed for this phase.
- **D-C2:** Proxy target is env-aware via `VITE_DEV_API_TARGET`. Defaults to `http://localhost:3001`. Escape hatch for future scenarios (e.g., staging integration tests).
- **D-C3:** `vigil-pwa/.env.local.example` documents `VITE_DEV_API_TARGET` alongside the existing `VITE_API_BASE`.

**CORS policy (Area 4)**
- **D-D1:** Dev leaves `CORS_ORIGINS` unset — falls back to `origin: "*"`. With Vite proxy handling same-origin routing (D-C1), CORS is barely exercised in dev. Tailscale is the real perimeter.
- **D-D2:** Fail-closed prod guard in `vigil-core/src/index.ts`. At startup, if `NODE_ENV === 'production' && !process.env.CORS_ORIGINS`, log FATAL and `process.exit(1)`. Mirrors the existing JWT_SECRET boot-check pattern (src/index.ts:59).

**Bind / URL / service-worker (secondary)**
- **D-E1:** Vite binds via `server.host: true` in `vite.config.ts`. Always 0.0.0.0 in dev.
- **D-E2:** Primary URL from laptop: `http://jamesons-imac-2:5173` (Tailscale MagicDNS hostname). Tailscale IP literal `http://100.120.110.110:5173` documented as fallback in RUNBOOK.
- **D-E3:** Dev service worker concern deferred. VitePWA plugin registers SW only in prod builds. Research (this document, §Code Examples) verifies against `vite-plugin-pwa` 1.2.0 installed in the repo.

### Pattern 1: Env-gated bind hostname on `@hono/node-server`

**What:** Read `VIGIL_BIND_HOST` env var with a `127.0.0.1` default, pass it to `serve()` via the `hostname` option.

**When to use:** Every phase that changes network exposure in vigil-core. This phase is the first.

**Example:**

```typescript
// Source: verified against installed node_modules/@hono/node-server/dist/types.d.ts
// and node_modules/@hono/node-server/dist/server.js (which does
// `server.listen(options?.port ?? 3e3, options.hostname, () => ...)`).

const port = Number(process.env.PORT) || 3001;
const hostname = process.env.VIGIL_BIND_HOST ?? "127.0.0.1"; // safe default

serve({ fetch: app.fetch, port, hostname }, () => {
  console.log(`Vigil Core API running on ${hostname}:${port}`);
});
```

**Notes:**
- The default `127.0.0.1` (not `localhost`) avoids IPv6 resolution (`::1`) ambiguity that has historically bitten `@hono/node-server` (Vite has a known bug #10638 about not checking 0.0.0.0 when port-scanning; sticking to explicit IPv4 literal side-steps this class of issue).
- The Railway prod image has NO `VIGIL_BIND_HOST` set (gitignored `.env` never ships). Railway's own container networking means 127.0.0.1 is correct behind its proxy — it historically worked with @hono/node-server's default `localhost`.
- **Belt-and-braces:** Even if `VIGIL_BIND_HOST=0.0.0.0` accidentally ends up in Railway, the D-D2 prod CORS guard will `process.exit(1)` if `CORS_ORIGINS` is unset (and it never is in Railway — confirmed in RUNBOOK env var table).

### Pattern 2: Fail-closed prod CORS guard

**What:** At startup, after the existing JWT_SECRET guard (src/index.ts:59), add a parallel check: if `NODE_ENV === 'production'` AND `CORS_ORIGINS` is unset/empty, log FATAL and exit.

**When to use:** First line of defense against wildcard-CORS accidentally hitting prod. Second line is Railway having `CORS_ORIGINS` always set.

**Example:**

```typescript
// Source: modeled on src/index.ts:59-62 (JWT_SECRET fatal check)
// Inserted BEFORE `export const app = new Hono();` so exit(1) prevents
// the rest of app wiring.

if (process.env.NODE_ENV === "production" && !process.env.CORS_ORIGINS) {
  console.error(
    "FATAL: CORS_ORIGINS must be set in production — refusing to boot with wildcard CORS"
  );
  process.exit(1);
}
```

**Notes:**
- Place ABOVE the existing `const corsOrigins = process.env.CORS_ORIGINS ? ... : null;` (line 67) so the fallback-to-`*` path is unreachable in prod.
- NODE_ENV: Railway's Node build sets `NODE_ENV=production` by default for `npm start`. Verified via RUNBOOK §Deploy Sequence implicitly (the `node dist/index.js` production command runs under `NODE_ENV=production`).
- `process.env.CORS_ORIGINS` is the check — not `corsOrigins` (which is already parsed). Checking raw-env matches the pattern used by JWT_SECRET on line 59.

### Pattern 3: Vite proxy target via `loadEnv()`

**What:** Read `VITE_DEV_API_TARGET` from `.env.local` inside `vite.config.ts` using `loadEnv()`, then pass to the proxy config.

**When to use:** Whenever a Vite config value depends on an env var (vs. browser-bundle env vars which use `import.meta.env`).

**Example:**

```typescript
// Source: vite.dev/guide/env-and-mode "In vite.config, we need to use loadEnv to
// load these variables explicitly... process.env is NOT populated with env vars
// from .env files by default."
import { defineConfig, loadEnv } from 'vite'

export default defineConfig(({ mode }) => {
  // Third arg '' loads ALL env vars (no prefix filter). VITE_DEV_API_TARGET is
  // technically VITE_-prefixed so we COULD use default prefix, but being explicit
  // about '' documents the intent (it's a server-side config value, not a bundle value).
  const env = loadEnv(mode, process.cwd(), '')
  const apiTarget = env.VITE_DEV_API_TARGET ?? 'http://localhost:3001'

  return {
    server: {
      host: true,        // D-E1 — bind 0.0.0.0 in dev so Tailscale/LAN can reach it
      proxy: {
        '/v1': {
          target: apiTarget,
          changeOrigin: true,
          // ws: true NOT needed — /v1 routes are HTTP; HMR has its own WS channel.
        },
      },
    },
    plugins: [/* unchanged */],
  }
})
```

**Notes:**
- `loadEnv(mode, process.cwd(), '')` with the empty-string prefix loads every env var (any prefix). This lets the planner pick any name in the future without re-editing the loadEnv call. However, browser-bundle consumers must still use `VITE_` prefix.
- `VITE_DEV_API_TARGET` starts with `VITE_` — which is deliberate: it makes the var visible to `import.meta.env` IF any future code path needs to inspect it at runtime in the browser (which today it doesn't, because the proxy target is a server-side-only concern).
- **DO NOT** use `process.env.VITE_DEV_API_TARGET` — Vite does NOT populate `process.env` from `.env` files; only `loadEnv()` reads them.

### Pattern 4: Preflight Check 5 — bind state + macOS firewall

**What:** Add a 5th check to `scripts/preflight-check.sh` that:
1. Reads `VIGIL_BIND_HOST` from `vigil-core/.env`.
2. If set to `0.0.0.0`, also probes `socketfilterfw --getglobalstate`.
3. If firewall is off: green "PASS — 0.0.0.0 bind (firewall off, Tailscale reachable)".
4. If firewall is on: yellow WARN with instructions to allow node.js (non-fatal — this is the D-B3 fail-loud-informative trade-off).
5. If `VIGIL_BIND_HOST` unset or `127.0.0.1`: green "PASS — localhost bind (cross-machine disabled)".

**Example:**

```bash
# Source: modeled on scripts/preflight-check.sh:60-90 (Check 4 — DATABASE_URL localhost)
# and scripts/preflight-check.sh:14 (red/green/info helpers — reused verbatim).

check_bind_and_firewall() {
  info "Check 5: VIGIL_BIND_HOST sanity (and macOS firewall state if exposed)"

  BIND_VALUE=$(grep -E '^VIGIL_BIND_HOST=' "$ENV_FILE" 2>/dev/null | head -1 | cut -d= -f2- || true)
  BIND_VALUE="${BIND_VALUE// /}"   # strip whitespace

  if [[ -z "$BIND_VALUE" ]] || [[ "$BIND_VALUE" == "127.0.0.1" ]] || [[ "$BIND_VALUE" == "localhost" ]]; then
    green "  PASS — vigil-core binds localhost only (cross-machine dev disabled)"
    return
  fi

  if [[ "$BIND_VALUE" != "0.0.0.0" ]]; then
    # Anything else is a typo / unsupported value — fail loud.
    red "  FAIL — VIGIL_BIND_HOST='$BIND_VALUE' is not a supported value."
    red "  Fix: set VIGIL_BIND_HOST=0.0.0.0 (cross-machine) or remove the line (localhost default)"
    FAIL=1
    return
  fi

  # 0.0.0.0 — now check macOS firewall.
  FW_BIN="/usr/libexec/ApplicationFirewall/socketfilterfw"
  if [[ ! -x "$FW_BIN" ]]; then
    # Not macOS, or unusual install — informational only.
    green "  PASS — 0.0.0.0 bind; socketfilterfw not found (assumed non-macOS or firewall unmanageable)"
    return
  fi

  # NOTE: the doc-recommended safe pattern is grep-based, NOT direct $? of the command.
  # `socketfilterfw --getglobalstate` returns exit 0 whether firewall is on or off.
  if "$FW_BIN" --getglobalstate 2>/dev/null | grep -q 'disabled'; then
    green "  PASS — 0.0.0.0 bind (macOS Application Firewall off — Tailscale peers can reach :3001 and :5173)"
  else
    yellow "  WARN — 0.0.0.0 bind with macOS Application Firewall ENABLED."
    yellow "    If the laptop browser cannot load http://jamesons-imac-2:5173, allow node.js:"
    yellow "      System Settings → Network → Firewall → Options → + Add → /usr/local/bin/node (or nvm path)"
    yellow "    OR turn off the firewall for dev (trusted Tailscale-only machine):"
    yellow "      sudo $FW_BIN --setglobalstate off"
    yellow "    This is a WARN, not a FAIL — preflight continues."
  fi
}
```

**Notes:**
- **MUST use grep pattern matching, not exit-code check** — `socketfilterfw --getglobalstate` always exits 0 regardless of state. MacRumors forum thread confirmed (cited in Sources). Live-tested 2026-04-21 on iMac: output is literal "Firewall is disabled. (State = 0)" — grep-for-'disabled' pattern works.
- **Alternative `defaults read /Library/Preferences/com.apple.alf globalstate` REJECTED**: live-tested 2026-04-21 on iMac macOS 15 (Sequoia), returns "The domain/default pair ... does not exist" — Apple has moved the state out of that plist in recent macOS versions. `socketfilterfw` remains reliable.
- WARN (not FAIL) is deliberate per D-B3: the user may consciously want firewall-on for a daemon they don't trust, and the Tailscale-preferred fix (add a single-app rule) is their call.

### Anti-Patterns to Avoid

- **DO NOT bind 0.0.0.0 unconditionally.** Without the `VIGIL_BIND_HOST` env gate, a Railway deploy could silently expose the API to the container's outside interface. The gate is load-bearing.
- **DO NOT set `credentials: true` in the hono cors middleware.** It's incompatible with `origin: "*"` per the CORS spec; browsers reject. Current config correctly omits `credentials` — keep it that way.
- **DO NOT add `ws: true` to the `/v1` proxy.** HMR uses Vite's own WebSocket channel (not `/v1`). Adding it would only matter if we were proxying vigil-core's own WebSocket endpoint (we have none today).
- **DO NOT use `process.env.VITE_DEV_API_TARGET` in vite.config.ts.** Vite doesn't populate process.env from .env files. Use `loadEnv()`.
- **DO NOT rely on `import.meta.env.DEV` to gate the prod CORS check in vigil-core.** `import.meta.env` is a Vite-only API — unavailable in Node. Use `process.env.NODE_ENV === "production"` (matches the JWT_SECRET pattern).
- **DO NOT echo `VIGIL_BIND_HOST` or the firewall state as JSON.** Follow the existing line-based preflight output format so grep-based tests in CI can parse it.

## Common Pitfalls

### Pitfall 1: `hostname: 'localhost'` vs `hostname: '127.0.0.1'` IPv6 quirks

**What goes wrong:** Binding to `'localhost'` on recent macOS/Linux may resolve to `::1` (IPv6) first, causing IPv4 clients to see a "connection refused" on the port.

**Why it happens:** `server.listen(port, 'localhost', ...)` in Node delegates to DNS resolution of `localhost`, which varies by OS and `/etc/hosts` layout. Vite has a known issue (#10638) in this territory.

**How to avoid:** Use literal `'127.0.0.1'` as the default in the hostname fallback. CONTEXT D-B2's "127.0.0.1" wording already commits to this — planner should preserve it verbatim.

**Warning signs:** vigil-core starts but `curl http://localhost:3001/v1/health` returns "Connection refused" while `curl http://127.0.0.1:3001/v1/health` works.

### Pitfall 2: `import.meta.env.DEV` vs `process.env.NODE_ENV` confusion in prod guard

**What goes wrong:** A dev writes `if (import.meta.env.DEV) { ... }` in vigil-core/src/index.ts because muscle memory from PWA code. Build fails or silently misbehaves.

**Why it happens:** `import.meta.env` is Vite's compile-time replacement. In Node (vigil-core), the construct compiles to `undefined.DEV` at best, runtime error at worst.

**How to avoid:** In vigil-core (Node/Hono), only use `process.env.NODE_ENV === "production"`. Leave `import.meta.env` for vigil-pwa (Vite-compiled code).

**Warning signs:** `ReferenceError: import is not defined in this context` at startup, or `Cannot read properties of undefined (reading 'DEV')`.

### Pitfall 3: `socketfilterfw --getglobalstate` exit code is always 0

**What goes wrong:** Script writes `if socketfilterfw --getglobalstate; then echo "enabled"; else echo "disabled"; fi` and always reports "enabled" (because the command succeeds regardless of state).

**Why it happens:** `--getglobalstate` returns the state on stdout, not via exit code. Exit is always 0 unless the binary can't run at all.

**How to avoid:** Grep the output: `socketfilterfw --getglobalstate | grep -q 'disabled'`. [VERIFIED via MacRumors forum thread linked in Sources and live-tested on iMac 2026-04-21.]

**Warning signs:** Preflight always WARNs about firewall regardless of actual state, or always says "clean" when user expected a WARN.

### Pitfall 4: `VITE_API_BASE` set in .env.local shadows the proxy

**What goes wrong:** Phase 107.1 set `VITE_API_BASE=http://localhost:3001` in `vigil-pwa/.env.local` (per `107.1-VERIFICATION.md` key link "vigil-pwa API client → localhost:3001 via VITE_API_BASE env var"). If the laptop opens `http://jamesons-imac-2:5173`, the PWA reads `VITE_API_BASE=http://localhost:3001` and tries to call the LAPTOP's localhost:3001 — which has no vigil-core — instead of routing through the proxy.

**Why it happens:** The PWA's `api/client.ts` line 3 is: `export const API_BASE = (import.meta.env.VITE_API_BASE as string | undefined) ?? (import.meta.env.DEV ? '' : 'https://api.vigilhub.io')`. When VITE_API_BASE is set, the proxy is bypassed.

**How to avoid:** This is the **load-bearing subtlety** of CONTEXT D-C1. For the proxy (and therefore cross-machine access) to work, `VITE_API_BASE` must be EMPTY or UNSET in dev. The `import.meta.env.DEV ? ''` fallback then kicks in and API_BASE becomes `''`, meaning `fetch('/v1/...')` is relative to page origin and gets proxied by Vite.

**Planner action:** Phase 107.2 MUST either (a) remove `VITE_API_BASE` from `vigil-pwa/.env.local` OR (b) set it to empty string. The `.env.local.example` template should reflect this clearly:

```bash
# vigil-pwa/.env.local.example — LOCAL DEV ONLY

# VITE_API_BASE: Leave UNSET in dev so Vite's proxy (see vite.config.ts)
# routes /v1/* to the local vigil-core. Prod builds default to api.vigilhub.io.
# VITE_API_BASE=

# VITE_DEV_API_TARGET: What the Vite dev proxy forwards /v1/* to.
# Default http://localhost:3001 (vigil-core in same `npm run dev`).
VITE_DEV_API_TARGET=http://localhost:3001
```

**Warning signs:** Laptop browser loads `http://jamesons-imac-2:5173`, dashboard renders HTML but Network tab shows failed calls to `http://localhost:3001/v1/summary` (laptop's localhost, not iMac's).

**This was a close miss** — Phase 107.1's verification celebrated `VITE_API_BASE=http://localhost:3001` as correct for the same-machine case. It's correct for same-machine. For cross-machine, the value MUST be unset so the proxy (new in this phase) can do its job.

### Pitfall 5: `change-origin: true` breaks Google OAuth callback

**What goes wrong:** Hypothetical — if a future test flow exercises the Google OAuth callback through the Vite proxy, `changeOrigin: true` rewrites the `Host` header and Google's redirect matching breaks.

**Why it happens:** `changeOrigin: true` sets `Host: localhost:3001` on the upstream request. If vigil-core ever checks `c.req.header('host')` for OAuth state validation, the check fails.

**How to avoid:** In this phase, it's a non-issue — Google OAuth is mounted on vigil-core at `GOOGLE_REDIRECT_URI=http://localhost:3001/v1/auth/google/callback` (per vigil-core/.env.example), and Google redirects directly to that URL (NOT via the Vite proxy). But document for future readers.

**Warning signs:** Only surfaces if a future phase routes OAuth through the PWA proxy. Not an issue today.

### Pitfall 6: Vite HMR WebSocket failing on remote host

**What goes wrong:** Laptop loads `http://jamesons-imac-2:5173`, dashboard works, but code edits on the iMac don't hot-reload in the laptop browser. Browser console shows `[vite] failed to connect to websocket. your browser doesn't support websockets, or the connection is refused.`.

**Why it happens:** Vite HMR uses its own WebSocket at the same port (5173 by default), and the browser's WebSocket client tries to connect to the server's HMR host. With `server.host: true`, Vite usually does the right thing and the browser connects to `ws://jamesons-imac-2:5173`. But certain edge cases (proxy chains, TLS upgrades, Chrome's IPv4/IPv6 quirks on MagicDNS per Tailscale issue #14500) can cause it to try `ws://localhost:5173` instead.

**How to avoid:** 
1. **First:** trust Vite's auto-detection — leave `server.hmr` unconfigured in this phase.
2. **If HMR breaks in UAT:** add a minimal `server.hmr.host: 'jamesons-imac-2'` override. Do NOT add it preemptively.
3. **Document in RUNBOOK** as a known tweak.

**Warning signs:** Everything works except file edits don't trigger browser updates. Browser DevTools Console shows ws-failed message.

**Research confidence:** MEDIUM — this is a known Vite edge case but not a guaranteed-to-hit pitfall. Defer to UAT rather than speculatively configuring.

### Pitfall 7: Chrome on macOS + MagicDNS IPv4/IPv6 mixed resolution

**What goes wrong:** Laptop uses Chrome, IPv4-only network, MagicDNS hostname resolves to BOTH IPv4 and IPv6 but IPv6 connection hangs — dashboard never loads.

**Why it happens:** Known Tailscale issue #14500 (macOS 2025). Chrome's Happy Eyeballs implementation sometimes favors IPv6 and the fallback to IPv4 is slow.

**How to avoid:** If issue surfaces:
- Use Safari (Tailscale issue notes Safari is unaffected).
- OR use the Tailscale IP literal: `http://100.120.110.110:5173` (CONTEXT D-E2 fallback already documented).
- OR disable IPv6 in Chrome via `chrome://flags/#enable-ipv6` (overkill for dev).

**Warning signs:** Safari works, Chrome hangs indefinitely on the Tailscale URL. RUNBOOK should cross-link D-E2 fallback.

### Pitfall 8: `dev-reset.sh` doesn't honor VIGIL_BIND_HOST (out of scope, but adjacent)

**What goes wrong:** After 107.2 adds `VIGIL_BIND_HOST` to `.env`, `dev-reset.sh` wipes the DB but preserves the .env (correct — it only touches DB). No regression — but CR-01 from Phase 107.1 is still open (dev-reset.sh doesn't pin PGHOST=localhost). If CR-01 is fixed in this phase by the planner picking it up, they must ALSO verify the VIGIL_BIND_HOST line in `.env` is not accidentally stripped.

**How to avoid:** If the planner folds CR-01 into this phase (Claude's Discretion area), the fix must be additive to `.env` preservation, not a rewrite. Otherwise, leave CR-01 for its own follow-up.

**Warning signs:** `dev-reset.sh` reshapes `.env` and subsequent `npm run dev` fails Check 5 with "VIGIL_BIND_HOST unset after reset".

## Open Questions

All 10 questions from the research brief have been resolved in-document:

1. **@hono/node-server serve() hostname param** — RESOLVED [VERIFIED]. Works as-is on installed 1.19.12 per `node_modules/@hono/node-server/dist/types.d.ts` (`hostname?: string`) and `server.js` (passed to `server.listen`).

2. **vite-plugin-pwa SW in dev** — RESOLVED [VERIFIED]. `devOptions.enabled` defaults `false` on installed 1.2.0. Current `vite.config.ts` never sets `devOptions`. D-E3 confirmed.

3. **macOS App Firewall vs Tailscale** — RESOLVED [CITED]. Firewall IS independent and can block utun-inbound. Preflight check 5 must WARN (not just inform) when AppFirewall is on AND `VIGIL_BIND_HOST=0.0.0.0`. Tailscale GitHub issues #454, #3185 cited.

4. **Vite proxy `ws: true` for HMR** — RESOLVED [CITED]. HMR uses its own WebSocket channel on port 5173 (Vite docs). Proxy `/v1` `ws: true` is unrelated to HMR. Do not add.

5. **hono/cors wildcard + credentials** — RESOLVED [VERIFIED]. `cors({ origin: "*" })` without `credentials` set does NOT emit `Access-Control-Allow-Credentials` and does NOT break unauthenticated or Bearer-token flows. Bearer tokens work because `vigilFetch` sets the Authorization header manually (not via `credentials: 'include'`). Repo source verified.

6. **process.env semantics with tsx --env-file vs Vite** — RESOLVED [CITED]. tsx `--env-file=.env` populates `process.env` for vigil-core (Phase 107.1 Plan 05 verified this live). Vite does NOT populate `process.env` from `.env.local` — config must use `loadEnv()`. The client-side `import.meta.env` only exposes `VITE_`-prefixed vars, and it DOES auto-load `.env.local`. `VITE_DEV_API_TARGET` is VITE_-prefixed so both sides work.

7. **socketfilterfw reliability** — RESOLVED [VERIFIED]. Live-tested 2026-04-21 on iMac: `socketfilterfw --getglobalstate` returns "Firewall is disabled. (State = 0)" with exit 0 (regardless of state). `defaults read /Library/Preferences/com.apple.alf globalstate` returns error on macOS 15 — do NOT use. Use grep-pattern check.

8. **Tailscale-specific behavior** — RESOLVED [CITED]. MagicDNS works in Chrome and Safari on macOS, with one known edge case (Tailscale issue #14500 — Chrome + IPv4-only + MagicDNS). Fallback to IP literal addresses this. No other hono-specific or bind-specific Tailscale gotchas found.

9. **RUNBOOK structure** — RESOLVED [CITED]. Section structure in Example 4 matches the Phase 107.1 RUNBOOK entry shape (setup steps, daily workflow, known issues, triage table). Parity confirmed by reading `vigil-core/RUNBOOK.md` §Local Development.

10. **dev-reset.sh / dev-setup.sh interactions** — RESOLVED. No changes required for cross-machine dev. Both scripts touch disk state only (DB, env files) — neither binds ports. CR-01 from Phase 107.1 REVIEW (dev-reset.sh missing PGHOST=localhost pin) remains an orthogonal open item; Claude's Discretion area says the planner may fold it into 107.2 or leave for follow-up.

**One additional pitfall flagged beyond the 10 questions:**

- **Pitfall 4 (above): `VITE_API_BASE=http://localhost:3001` in `.env.local` SHADOWS the proxy.** Phase 107.1 set this value; for 107.2's proxy approach to work, it MUST be removed or emptied. This is the single most load-bearing planner action outside the 5 locked decisions.

### Known Threat Patterns for {Node/Hono + Vite + Tailscale}

| Pattern | STRIDE | Standard Mitigation |
|---------|--------|---------------------|
| Wildcard CORS in prod accidentally ships | Information Disclosure | D-D2 FATAL boot-guard (this phase) + Railway env var `CORS_ORIGINS` (existing) |
| 0.0.0.0 bind reaches Railway (container exposes extra surface) | Elevation of Privilege | D-B2 VIGIL_BIND_HOST gate (defaults 127.0.0.1) + gitignored `.env` + Railway proxy in front (existing) |
| macOS Application Firewall silently blocks Tailscale peer | Denial of Service (dev-side) | D-B3 preflight Check 5 surfaces the condition before `npm run dev` starts |
| Laptop browser connects to a malicious "jamesons-imac-2" on public network | Spoofing | Tailscale tailnet ACLs are the authority; MagicDNS only resolves within the authenticated tailnet. Out-of-scope for vigil-core hardening. |
| Chrome IPv4/IPv6 MagicDNS stall (Tailscale #14500) | Availability (dev-side) | D-E2 IP-literal fallback documented in RUNBOOK |
| Unauthenticated access to vigil-core's /v1 routes from a Tailscale peer | Information Disclosure | Existing bearer middleware (src/index.ts:105-111) — unchanged. Any /v1/* path except health, register, login, OAuth callback requires Bearer token. Wildcard CORS does not bypass bearer auth (CORS is origin-level; bearer is header-level). |

**Key security insight:** The phase DOES widen the attack surface in dev (0.0.0.0 bind instead of 127.0.0.1) — but only behind an env-gated default and only on a tailnet-protected machine. No production hardening is relaxed; the FATAL CORS guard actually TIGHTENS prod posture. Net posture: dev marginally wider (intentional), prod strictly tighter (net gain).
