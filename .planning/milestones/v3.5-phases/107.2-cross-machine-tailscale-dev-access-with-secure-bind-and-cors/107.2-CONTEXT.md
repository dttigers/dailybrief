# Phase 107.2: Cross-machine Tailscale dev access with secure bind and CORS — Context

**Gathered:** 2026-04-21
**Status:** Ready for research + planning

<domain>
## Phase Boundary

Make the iMac's local dev stack (vigil-core on :3001 + Vite on :5173) reachable from the MacBook Pro over Tailscale, **without weakening production security**. Scope is a thin-client model where the laptop browser points at the iMac's running `npm run dev` — no second DB, no mirrored state, no second vigil-core process.

**In scope:**
- vigil-core bind change gated by an explicit env var
- Vite dev server bind + proxy rewrite
- CORS prod fail-closed gate
- Preflight-check and doctor script updates that surface the new bind state
- RUNBOOK documentation of the cross-machine flow
- Laptop-side URL bookmark (Tailscale MagicDNS hostname)

**Out of scope (note and defer):**
- Full mirror mode (laptop running its own vigil-core + Postgres). If ever needed, it becomes Phase 107.3 or later.
- HTTPS / self-signed certs for dev. Tailscale WireGuard already encrypts transport; PWA service worker doesn't register in dev mode anyway.
- Beta-tester / third-machine access. Today's scope is user's two machines.
- ServiceNow / Gmail OAuth (already-blocked work-order pivot from memory).

</domain>

<decisions>
## Implementation Decisions

### Access model (Area 1)
- **D-A1:** **Thin-client architecture.** MacBook Pro browser connects to iMac's `npm run dev` stack over Tailscale. Single `vigil_dev` Postgres, single source of truth. Trade-off accepted: if iMac is off / not running `npm run dev`, laptop cannot dev.
- **D-A2:** **Both vigil-core (3001) and Vite (5173) exposed** over Tailscale. The Vite proxy forwards API calls server-side (see D-C1), so the browser sees same-origin traffic, but both ports need to bind to a non-localhost interface.

### Bind strategy + prod safety (Area 2)
- **D-B1:** **vigil-core binds `0.0.0.0`** in dev (all interfaces — Tailscale + LAN + localhost).
- **D-B2:** **Prod safety via `VIGIL_BIND_HOST` env var.** If unset, vigil-core binds `127.0.0.1` (localhost only — safe default). Dev `.env` sets `VIGIL_BIND_HOST=0.0.0.0`. Because `.env` is gitignored (D-17 from Phase 107.1), Railway prod never inherits this and falls through to the localhost default behind Railway's proxy. This is belt-and-suspenders with D-D2 below.
- **D-B3:** **Preflight check #5 added to `scripts/preflight-check.sh`.** If `VIGIL_BIND_HOST=0.0.0.0` AND macOS Application Firewall is disabled (`/usr/libexec/ApplicationFirewall/socketfilterfw --getglobalstate` → off), warn loudly but don't block. Keeps the Plan 03 fail-loud-informative culture.
- **D-B4 (implicit):** `.env.example` gets a documented `VIGIL_BIND_HOST` line with explicit `# LOCAL ONLY — Railway MUST leave this unset` comment.

### PWA → vigil-core routing (Area 3)
- **D-C1:** **Vite proxy handles API routing.** `vigil-pwa/vite.config.ts` rewrites the `/v1` proxy target from `https://api.vigilhub.io` (current prod target) to `http://localhost:3001`. The proxy runs server-side on the iMac; vigil-core is also on the iMac, so `localhost` resolves correctly. The browser on the laptop sees same-origin traffic against port 5173 — no CORS involvement, no `VITE_API_BASE` change needed for this phase.
- **D-C2:** **Proxy target is env-aware via `VITE_DEV_API_TARGET`.** Defaults to `http://localhost:3001`. Exposes an escape hatch if a future scenario needs the proxy to point elsewhere (e.g., staging on Railway for integration tests).
- **D-C3 (implied):** `vigil-pwa/.env.local.example` documents `VITE_DEV_API_TARGET` alongside the existing `VITE_API_BASE`.

### CORS policy (Area 4)
- **D-D1:** **Dev leaves `CORS_ORIGINS` unset.** Current behavior: falls back to `origin: "*"`. With Vite proxy handling same-origin routing (D-C1), CORS is barely exercised in dev. Tailscale is the real perimeter. Trade-off accepted: a curl/Postman origin check from the laptop against vigil-core directly will also succeed — fine for a two-machine dev setup.
- **D-D2:** **Fail-closed prod guard in `vigil-core/src/index.ts`.** At startup, if `NODE_ENV === 'production' && !process.env.CORS_ORIGINS`, log FATAL and `process.exit(1)`. Mirrors the existing JWT_SECRET boot-check pattern (src/index.ts:59). Stops the `'*'` fallback from ever reaching Railway, even if the env var is accidentally unset.

### Bind / URL / service-worker (secondary)
- **D-E1:** **Vite binds via `server.host: true`** in `vite.config.ts`. Always 0.0.0.0 in dev. Not env-gated because Vite dev server never runs in prod (prod serves static built assets via Railway/CDN). Keeps `npm run dev` a one-liner.
- **D-E2:** **Primary URL from laptop: `http://jamesons-imac-2:5173`** (Tailscale MagicDNS hostname, survives IP rotations). Tailscale IP literal `http://100.120.110.110:5173` documented as fallback in RUNBOOK.
- **D-E3:** **Dev service worker concern deferred.** VitePWA plugin registers SW only in prod builds; dev mode serves from memory without SW. Research phase should still verify this against the current `vite-plugin-pwa` version, but do NOT block the phase on it.

- Whether `dailybrief-doctor.sh` gains a Tailscale-reachability row or a dedicated `--remote` subcommand that SSHes to the other machine. Both are reasonable; pick the simpler one in planning.
- Whether the laptop needs any one-time setup at all (beyond joining the tailnet, which it already has). If yes, RUNBOOK documents it; if no, RUNBOOK says so explicitly.
