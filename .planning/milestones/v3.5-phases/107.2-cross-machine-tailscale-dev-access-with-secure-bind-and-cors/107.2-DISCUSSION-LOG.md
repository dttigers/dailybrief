## Area 1: Access model & what gets exposed

### Q1.1 — Stack model

| Option | Description | Selected |
|--------|-------------|----------|
| Thin client to iMac | Laptop browser hits iMac's running npm run dev. One DB, one source of truth. If iMac is off, laptop can't work. | ✓ |
| Full mirror on laptop | Laptop runs its own vigil-core + Postgres + Vite. Fully offline-capable but two DBs to keep in sync. | |
| Both — either mode works | Build both paths for per-session flexibility. | |

**User's choice:** Thin client to iMac
**Notes:** Matches the minimal-complexity default. Mirror mode deferred to a later phase if/when thin-client proves insufficient.

### Q1.2 — Expose what

| Option | Description | Selected |
|--------|-------------|----------|
| Both vigil-core and Vite | Laptop browser loads PWA from Vite (5173) and hits vigil-core (3001). Standard dev workflow. | ✓ |
| Only vigil-core | Laptop runs its own Vite locally, API points at iMac's vigil-core. Simpler bind change but laptop needs full vigil-pwa checkout. | |
| Only Vite | Incoherent for standard flow; included for completeness. | |

**User's choice:** Both vigil-core and Vite
**Notes:** Confirms thin-client architecture — laptop is a pure browser, no local npm run dev needed on laptop.

## Area 2: Bind strategy + prod safety

### Q2.1 — Bind scope

| Option | Description | Selected |
|--------|-------------|----------|
| 0.0.0.0 (all interfaces) | Accepts localhost + LAN + Tailscale. Simple. Local firewall limits reach. | ✓ |
| Tailscale IP only (100.120.110.110) | Surgical; LAN peers can't hit. IP can change on Tailscale reinstall. | |
| Both localhost + Tailscale IP | Two specific addresses; requires explicit multi-bind or two serve() calls. | |

**User's choice:** 0.0.0.0 (all interfaces)
**Notes:** Pragmatic. LAN hotel-wifi exposure noted as a real risk but mitigated by the firewall check in Q2.3.

### Q2.2 — Prod safety mechanism

| Option | Description | Selected |
|--------|-------------|----------|
| Explicit env var, default localhost | VIGIL_BIND_HOST; default 127.0.0.1 if unset. .env sets 0.0.0.0 locally; Railway never inherits since .env is gitignored. | ✓ |
| NODE_ENV=development gate | Implicit; relies on Railway setting NODE_ENV=production correctly. | |
| Both (belt-and-suspenders) | Env var AND refuse-to-boot if NODE_ENV=production + VIGIL_BIND_HOST=0.0.0.0. | |

**User's choice:** Explicit env var, default localhost
**Notes:** Simplest explicit mechanism. D-D2 (fail-closed CORS) already provides the second safety net, making the combined option redundant.

### Q2.3 — Preflight-check.sh extension

| Option | Description | Selected |
|--------|-------------|----------|
| Yes — warn if VIGIL_BIND_HOST=0.0.0.0 but firewall down | Visible in every npm run dev start. Follows Plan 03 fail-loud-informative culture. | ✓ |
| Yes — just print the bind state (no warning) | Informational only; user self-audits. | |
| No preflight change | Trust the env-var + prod gate. | |

**User's choice:** Yes — warn if VIGIL_BIND_HOST=0.0.0.0 but firewall down
**Notes:** Consistent with Plan 03 preflight pattern. Warn, don't block.

## Area 3: PWA → vigil-core routing

### Q3.1 — API routing method

| Option | Description | Selected |
|--------|-------------|----------|
| Vite proxy (rewrite target to localhost:3001) | Browser sees same-origin. Proxy runs on iMac; localhost resolves there. No CORS involvement, no VITE_API_BASE change. Service workers keep working. | ✓ |
| Direct — VITE_API_BASE = Tailscale hostname | Browser hits vigil-core directly; needs CORS. PWA service worker may not register over HTTP on non-localhost. | |
| Both: proxy for REST, direct for WebSocket/streaming | More complex; reserved for future if streaming added. | |

**User's choice:** Vite proxy (rewrite target to localhost:3001)
**Notes:** Same-origin routing keeps CORS untouched and sidesteps SW/HTTPS concerns.

### Q3.2 — Proxy target configurability

| Option | Description | Selected |
|--------|-------------|----------|
| Env-aware: read VITE_DEV_API_TARGET from .env.local | Defaults to localhost:3001; overridable. Prod builds never activate Vite dev server. | ✓ |
| Hardcode to http://localhost:3001 | Simpler; less configurable. | |

**User's choice:** Env-aware via VITE_DEV_API_TARGET
**Notes:** Keeps future flexibility (e.g., integration testing against staging) with ~3 lines of wiring.

## Area 4: CORS allowlist for dev

### Q4.1 — Dev CORS behavior

| Option | Description | Selected |
|--------|-------------|----------|
| Leave unset — falls back to '*' | Current default. Fine for dev; Tailscale is the real perimeter. Vite proxy handles most requests same-origin. | ✓ |
| Explicit allowlist | Tighter but requires manual edit per new Tailscale peer. | |
| Auto-discover from `tailscale status` | Always accurate; adds CLI + startup-code dependency. | |

**User's choice:** Leave unset — falls back to '*'
**Notes:** Pragmatic for two-machine dev setup. D-D2 below prevents the '*' fallback from bleeding into prod.

### Q4.2 — Prod CORS safety

| Option | Description | Selected |
|--------|-------------|----------|
| Fail-closed check in src/index.ts | NODE_ENV=production && !CORS_ORIGINS → FATAL + exit(1). Mirrors JWT_SECRET pattern. | ✓ |
| Document in RUNBOOK only | No runtime enforcement. | |
| Already handled — Railway sets CORS_ORIGINS | Assumes Railway has it set. | |

**User's choice:** Fail-closed check in src/index.ts
**Notes:** Matches the existing FATAL-on-missing-env idiom; stops the dev-`*`-fallback from ever reaching Railway even if an env var is accidentally unset.

## Secondary decisions

### Q5.1 — Vite bind method

| Option | Description | Selected |
|--------|-------------|----------|
| server.host: true in vite.config.ts | Binds 0.0.0.0 always. Vite dev server never runs in prod so no prod exposure path. | ✓ |
| --host flag in the dev script | Flag-based. Same effect; slightly more discoverable. | |
| Conditional via VIGIL_BIND_HOST env | Unifies env var across both services. | |

**User's choice:** server.host: true in vite.config.ts
**Notes:** Simplest. No env-gating needed for Vite since Vite dev server never ships.

### Q5.2 — URL target from laptop

| Option | Description | Selected |
|--------|-------------|----------|
| Tailscale MagicDNS hostname | http://jamesons-imac-2:5173. Survives IP rotations; human-readable. | ✓ |
| Tailscale IPv4 literal | http://100.120.110.110:5173. No DNS dependency; can break on IP reassignment. | |
| Both documented | RUNBOOK lists both. | |

**User's choice:** Tailscale MagicDNS hostname (primary); literal documented as fallback
**Notes:** MagicDNS confirmed working on tailnet. Literal IP captured in RUNBOOK as debug aid.

### Q5.3 — Service worker concern

| Option | Description | Selected |
|--------|-------------|----------|
| Don't worry — dev doesn't need SW | VitePWA registers SW in prod builds only; dev mode serves from memory. | ✓ |
| Switch dev to HTTPS with self-signed cert | mkcert + vite --https. Parity with prod. Heavy ceremony. | |
| Investigate during research | Research confirms whether VitePWA registers SW in dev. | |

**User's choice:** Don't worry — dev doesn't need SW
**Notes:** Researcher should spot-verify vite-plugin-pwa behavior on current version but not block planning on it.
