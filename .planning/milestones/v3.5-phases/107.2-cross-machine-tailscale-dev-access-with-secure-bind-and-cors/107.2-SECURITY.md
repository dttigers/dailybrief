## Accepted Risks Log

| Risk ID | Threat Ref | Rationale | Accepted By | Date |
|---------|------------|-----------|-------------|------|
| AR-107.2-01 | T-CM-3 | Dev leaves `CORS_ORIGINS` unset → Hono middleware resolves `origin: "*"`. Hono `cors` middleware sets `credentials=false` by default when `origin='*'` (verified in `node_modules/hono/dist/middleware/cors/index.js`); browsers reject credential-carrying requests to wildcard origins. Bearer tokens are manual `Authorization:` headers, which browsers enforce CORS on the same way. Tailscale ACLs form the real perimeter for cross-machine dev. Prod is belt-and-suspendered by the CORS FATAL guard at `vigil-core/src/index.ts:69-72` — the wildcard path can never reach Railway. | Jameson Morrill (phase owner) | 2026-04-22 |

*Accepted risks do not resurface in future audit runs.*

## Implementation Details — Not-a-Threat Notes

### Phase 107.2 hotfix: `server.allowedHosts` in `vite.config.ts`

`vigil-pwa/vite.config.ts:26-28` adds env-gated `allowedHosts` (default `true`, narrow scope via `VITE_ALLOWED_HOSTS`) to unblock Vite 5+ DNS rebinding protection for MagicDNS hostnames like `jamesons-imac-2`. The permissive `true` default is **consistent with the D-D1 "Tailscale is the real perimeter" philosophy** (same rationale as T-CM-3's accepted wildcard CORS): the dev server only runs behind Tailscale ACLs. Dev-only; Vite dev-server never runs in prod. Not added to the threat register because it doesn't introduce a new surface beyond what T-CM-3 already accepts.
