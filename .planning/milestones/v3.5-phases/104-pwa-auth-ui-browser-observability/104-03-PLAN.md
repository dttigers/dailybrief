<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| sessionStorage → PostHog | identifyUser sends userId + email to PostHog Cloud — no thought content |
| browser → GET /v1/me | Response shape { userId, email } used for display — no untrusted content reflected in DOM (React auto-escapes email) |
| Sign out → localStorage | clearKey must also clear legacy vigil_api_key from localStorage — ghost state risk if omitted |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-104-03-01 | Information Disclosure | App.tsx returning-session identify | mitigate | GET /v1/me is called with Bearer JWT from sessionStorage. If JWT is expired/invalid, the server returns 401. The catch() in the useEffect silently ignores 401 — no logout is forced (per RESEARCH.md open question #2). The vigilFetch auth guard on protected routes will redirect to /auth when the user navigates to any protected page. |
| T-104-03-02 | Information Disclosure | SettingsPage email display | mitigate | Email string from /v1/me is rendered as React text node (`{accountEmail ?? ''}`), never via `dangerouslySetInnerHTML`. XSS risk is absent. |
| T-104-03-03 | Elevation of Privilege | Sign out — clearKey scope | mitigate | clearKey() removes BOTH sessionStorage 'vigil_jwt' AND localStorage 'vigil_api_key'. Any old API key sessions are cleaned up on sign-out. No ghost-state sessions possible after sign-out. |
| T-104-03-04 | Information Disclosure | PostHog identify — property allowlist | mitigate | identifyUser(userId, email) sends only userId (number-derived string) and email (string). No thought content, no work order content, no other user-generated strings. Matches the property allowlist locked in STATE.md. |
| T-104-03-05 | Tampering | Vigil Account section — Sign out confirmation | accept | No confirmation dialog before sign out (matches Layout.tsx pattern). Data is not deleted — JWT is cleared from sessionStorage only. Re-login restores full access. Low irreversibility → confirmation not required. |
</threat_model>
