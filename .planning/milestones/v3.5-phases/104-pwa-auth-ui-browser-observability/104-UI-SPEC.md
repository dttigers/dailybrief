## Spacing Scale

Declared values (multiples of 4 only):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, status dot sizes (`w-2 h-2`), inline gap-1 |
| sm | 8px | Tight element spacing (`gap-2`, `mb-2`, `mt-2`) |
| md | 16px | Default field/section spacing (`p-4`, `gap-4`, `mb-4`, `mt-4`) |
| lg | 24px | Card padding vertical (`p-5` = 20px is the existing card standard — see exception below) |
| xl | 32px | Major section gaps (`mb-6`, `p-8` on auth card) |
| 2xl | 48px | Not used in Phase 104 scope |
| 3xl | 64px | Not used in Phase 104 scope |

**Exceptions:**
- Auth card uses `p-8` (32px) — matches existing AuthPage.tsx pattern (source: CONTEXT.md D-01)
- Settings card sections use `p-5` (20px) — existing Google card pattern to replicate for Vigil Account section (source: CONTEXT.md D-06)
- Touch targets: form submit buttons use `py-2` (8px vertical) + full-width `w-full` — matches existing teal-600 submit button pattern
- Form inputs use `px-3 py-2` (12px/8px) — matches existing AuthPage input pattern

## Interaction Contracts

### Auth Form — Mode Toggle
- Default state: Login mode — heading "Sign in to Vigil", submit "Sign In", toggle "Don't have an account? Sign up"
- Signup state: heading "Create your account", submit "Sign Up", toggle "Already have an account? Sign in"
- Toggle is a plain button styled as `text-sm text-gray-400 hover:text-gray-200` — no underline, no teal
- Toggle is placed below the submit button with `mt-4 text-center`
- Switching mode resets error state (`setError(null)`) but preserves email + password field values

### Auth Form — Loading State
- Submit button text changes: Login → "Signing in…", Signup → "Creating account…"
- All inputs and submit button: `disabled={loading}`
- No spinner graphic — text-only loading indicator

### Auth Form — Error State
- Single generic string for ALL 4xx responses from both login and register: `"Invalid email or password. Please try again."`
- This string is IDENTICAL regardless of wrong email, wrong password, duplicate account, or any other 4xx
- Rendered as: `<p className="mt-2 text-sm text-red-400">{error}</p>` — below submit button
- Network errors (catch block): same generic string — no "network error" differentiation

### Error Boundary — Fallback UI
- Full-screen: `min-h-screen bg-gray-900 flex items-center justify-center`
- Centered card: `bg-gray-900 rounded-lg p-8 w-full max-w-md mx-4 text-center`
- Heading: `text-2xl font-medium text-white mb-4` — "Something went wrong"
- Single action: "Reload" button using teal-600 pattern (`py-2 px-4 bg-teal-600 hover:bg-teal-800 text-white rounded font-medium`)
- Action: `window.location.reload()` — no navigation, no retry state
- No back link, no error details exposed to user

### Vigil Account Section — Settings Page
- Position: Inserted at TOP of SettingsPage content, before the Google section
- Section heading: "Vigil Account"
- Content when loaded: user email as `text-gray-400 text-sm`
- Loading state: `<p className="text-gray-400 text-sm">Loading…</p>` — same pattern as Google section
- Sign out button: `text-sm text-red-400 hover:text-red-300` — matches "Disconnect" button pattern in Google section
- Sign out action: `clearKey()` then `navigate('/auth')` — same as existing Layout.tsx handleSignOut

## Accessibility Notes

- All form inputs must have associated `<label htmlFor>` matching input `id` — matches existing AuthPage.tsx pattern
- Error paragraph uses `role="alert"` is not required for inline form errors (no live region needed at form-field level) — matches existing pattern
- Banner component in SettingsPage already has `role="alert"` — Vigil Account section does not add a new banner
- Error boundary heading is `<h1>` — ensures focus can be set programmatically on error
- Reload button is a standard `<button>` — no `<a>` workaround
