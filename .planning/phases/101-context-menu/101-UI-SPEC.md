# Phase 101 — UI Design Contract

> Visual and interaction contract for the per-thought-row context menu (right-click / long-press) and the undo toast it requires. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

This phase introduces **two new surfaces** to the Vigil PWA:
1. A **floating popover menu** (first of its kind — the existing `Layout` More dropdown is an inline non-portal variant; this is the first real portaled popover)
2. A **toast host** (first of its kind — nothing like it exists today)

Everything else reuses the established Vigil PWA dark-theme tokens from `vigil-pwa/src/index.css` and the interaction shapes already proven in `BulkActionBar.tsx` and `Layout.tsx`.

| Property | Value |

## Spacing Scale

Declared values (all multiples of 4, matching Tailwind v4 default `0.25rem` step):

| Token | Value | Usage in this phase |
|-------|-------|---------------------|
| xs | 4px (`gap-1`, `py-1`) | Submenu arrow-to-label gap; toast icon-to-text gap |
| sm | 8px (`gap-2`, `py-2`, `mb-2`, `mt-1`) | Menu-item vertical padding; submenu-to-parent offset |
| md | 16px (`px-4`, `mx-4`) | Menu-item horizontal padding; toast inner padding |
| lg | 24px (`bottom-6`) | Toast offset from viewport bottom on desktop |
| xl | 32px (`bottom-8`, safe-area) | Toast offset from viewport bottom on mobile (above browser chrome / home indicator) |

Exceptions (phase-specific):
- **Long-press tolerance:** 10 px movement threshold before the menu is canceled (D-02 in 101-CONTEXT.md). This is an input threshold, not a layout spacing value — NOT part of the spacing scale.
- **Menu min-width:** 192px (`min-w-48`) — fits the longest label "Move to category →" comfortably at 14px/500 weight. Matches the information-dense feel of `BulkActionBar`'s `min-w-[140px]`.
- **Menu max-width:** 280px (`max-w-[280px]`) — prevents submenu category/project lists from spanning the viewport on wide displays.
- **Menu z-index:** 50 (existing convention — `BulkActionBar.tsx:36` uses `z-50`; `Layout.tsx:138` uses `z-50`).
- **Toast z-index:** 60 — one above the menu so the "Undo" toast for a destructive action always sits above any transient menu that might be open underneath.

## Typography

All inherits `font-sans` (Inter). Two weights in play, matching `index.css` font loader.

| Role | Size | Weight | Line Height | Usage |
|------|------|--------|-------------|-------|
| Menu item | 14px (`text-sm`) | 500 (`font-medium`) | 1.5 (`leading-normal`) | Context-menu labels (Edit, Re-triage, Move to category, Add to project, Delete) |
| Submenu item | 14px (`text-sm`) | 400 (`font-normal`) | 1.5 (`leading-normal`) | Category / project names in nested submenu — less visual weight than root items so the hierarchy reads |
| Submenu header | 12px (`text-xs`) | 500 (`font-medium`) | 1.25 (`leading-tight`) | "← Categories" / "← Projects" back-affordance on mobile inline-expand (D-12) |
| Toast body | 14px (`text-sm`) | 400 (`font-normal`) | 1.5 (`leading-normal`) | "Thought deleted." |
| Toast action | 14px (`text-sm`) | 500 (`font-medium`) | 1.5 (`leading-normal`) | "Undo" button — weight lift matches `Layout.tsx` active-tab treatment |

Rationale: the existing codebase uses `text-sm` for row content (`ThoughtRow.tsx:262`) and `text-xs` for metadata (`ThoughtRow.tsx:211`). The menu sits just above a row visually, so it reads at the same `text-sm` cadence the user is already scanning — no new sizes introduced.

## Color

Dark theme. All values from `@theme` in `vigil-pwa/src/index.css`. No new hex values introduced.

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `gray-900` (#2C2C2A) | App shell background — already established in `Layout.tsx:62` |
| Secondary (30%) | `gray-900/80` with `border-gray-400/30` | **Menu surface + toast surface** — matches `BulkActionBar.tsx:36` exactly. Translucent warm-near-black over the list. |
| Submenu/alt surface | `gray-800` with `border-gray-700` | Nested submenu on desktop — matches `Layout.tsx:138` More dropdown so the submenu reads as "secondary to the parent menu," not another translucent layer |
| Accent (10%) | `teal-400` (#1D9E75) | Checkmark indicating current category/project (D-14); focused menu item ring (keyboard nav, D-21); toast "Undo" action text |
| Hover state | `gray-400/30` (warm gray at 30% opacity) | Menu-item hover background — matches `Layout.tsx:149` `hover:bg-gray-700/50` feel and `BulkActionBar.tsx:63` `hover:bg-gray-400/30` exactly |
| Text primary | `gray-50` (#F1EFE8) | Menu-item labels, toast body |
| Text secondary | `gray-400` (#888780) | Submenu header back-affordance ("← Categories"); "(current)" suffix on non-selected current item if checkmark not used |
| Destructive | `red-400` (Tailwind default #F87171) for text; **do NOT fill the row** — Delete menu item is red text on the standard menu surface, not a red-filled button. Matches the restraint of `ThoughtRow.tsx:217` `text-red-400` favorite heart. |
| Destructive filled | `bg-red-600 hover:bg-red-700` | Reserved for **non-menu** destructive UI only (e.g. `BulkActionBar`'s bulk-delete button). The Delete menu item is text-only red because the menu separator above it (`border-t border-gray-700`) already provides the "different kind of thing" visual signal. |

**Accent reserved for:**
1. Current-selection checkmark (teal-400) in category/project submenu (D-14)
2. Keyboard-focused menu item ring (teal-600 at 40% opacity)
3. Toast "Undo" action label (teal-400)

**Accent is NOT used for:** hovered menu items (those use neutral `gray-400/30`), submenu arrow glyphs (neutral `text-gray-400`), regular menu item labels (those use `text-gray-50`), toast border (neutral `border-gray-400/30`).

60/30/10 check:
- **60%** — `gray-900` app shell background dominates the view behind the menu.
- **30%** — `gray-900/80` + `gray-800` menu/submenu/toast surfaces cover secondary chrome.
- **10%** — teal appears only on checkmark, focus ring, and Undo action (literally 3 pixels' worth of color per open menu).

## Copywriting Contract

### Menu item labels (root)

| Action | Label | Rationale |
|--------|-------|-----------|
| Edit | `Edit` | One word, verb-first. Matches inline edit entry point. |
| Re-triage | `Re-triage` | Hyphenated because the existing `↻` button tooltip in `ThoughtRow.tsx:232` already uses "Re-triage with AI" — keep the user-facing vocabulary consistent. |
| Move to category → | `Move to category` + `→` glyph suffix | Arrow glyph signals "opens a submenu." Plain-English, scannable. |
| Add to project → | `Add to project` + `→` glyph suffix | Parallel structure to "Move to category". Verb "Add" (not "Assign") matches the user's mental model — they're putting a thought into a project, not assigning ownership. |
| Delete | `Delete` | One word, neutral verb. Red text does the "destructive" signal; no "!" or "X" prefix. |

### Submenu (mobile inline-expand back-affordance)

| Element | Copy |
|---------|------|
| Category submenu header | `← Categories` |
| Project submenu header | `← Projects` |

Desktop has no back-affordance — the submenu closes when the user hovers to another root item or clicks outside (D-11).

### Submenu empty states

| Submenu | Empty copy |
|---------|------------|
| Categories | (never empty — hardcoded 5 categories from `BulkActionBar.tsx:3` `['task', 'therapy', 'idea', 'reflection', 'project']`) |
| Projects | `No projects yet. Create one on the Projects tab.` — 14px, `text-gray-400`, centered in a 32px-tall submenu cell. Clickable is NOT required; it's informational. |

### Toast copy (destructive undo flow, D-15/D-16)

| Element | Copy |
|---------|------|
| Toast body | `Thought deleted.` |
| Toast action | `Undo` |
| Toast a11y role | `role="status"` with `aria-live="polite"` so screen readers announce the deletion + undo affordance without hijacking focus. |
| Replacement behavior | If a second delete fires while the first toast is still visible, the first toast is replaced silently (D-16). No "2 thoughts deleted" pluralization — each delete is a discrete event with its own 5-second window. The first deletion commits silently. |

### Destructive confirmation — **not used in this phase**

Delete does NOT use a confirm dialog (D-15). The 5-second undo toast IS the confirmation mechanism. This is a deliberate deviation from `BulkActionBar.tsx:25` `window.confirm(...)` — bulk deletions still use the native confirm because "delete N thoughts" is a larger blast radius; single-row delete uses undo-toast.

### Re-triage copy

Re-triage fires silently (D-18) — **no toast, no confirmation, no "Re-triaging…" label.** The menu closes immediately; the user sees the category pill update when the API returns (existing `ThoughtRow` behavior). If the user wants feedback, they watch the category pill flip.

### Primary CTA for this phase

**`Undo`** — the toast's action button is the primary new CTA Phase 101 introduces. It is the only user-facing button the phase adds (every other action is a menu item in an existing interaction). Visual weight: `text-sm font-medium text-teal-400`, no background fill, hover `text-teal-100`.

### Empty state (menu + submenu)

- Root menu is never empty — always shows 5 items.
- Category submenu is never empty — 5 hardcoded.
- Project submenu empty copy: `No projects yet. Create one on the Projects tab.` (see above).

### Error state (menu action fails)

If Delete / Move / Add-to-project / Re-triage API call fails:
- The optimistic UI update is **reverted** (row reappears for Delete; category reverts for Move; etc.)
- Toast appears with: `Couldn't {action}. Try again.` where `{action}` ∈ `delete`, `move`, `add to project`, `re-triage`. Lowercase, sentence-case.
- Toast uses the same surface as the undo toast; no "Undo" button (there's nothing to undo — the action didn't stick). Dismisses after 5s or on user tap.
- A11y: `role="alert"` (stronger than `role="status"` — errors should interrupt).

### Keyboard hints (D-21, optional polish layer)

No keyboard shortcut text is rendered next to menu items in this phase (no `⌘E`, no `Del`). The menu is small enough that discoverability via hover + right-click is sufficient, and adding shortcut hints would require designing the full shortcut system (deferred).

## Interaction Contract (phase-specific, consumed by planner + executor)

### Menu open/close choreography

| Event | Timing | Animation |
|-------|--------|-----------|
| Menu open (desktop right-click) | Immediate — no enter animation | `opacity-0` → `opacity-100` via Tailwind `transition-opacity duration-75` is acceptable but not required. Existing codebase does not animate popover entrances (see `Layout.tsx:136` More dropdown opens instantly). |
| Menu open (mobile long-press) | After the 500ms long-press timer fires (D-02) | Same as desktop — no additional enter delay. The 500ms already IS the "animation" from the user's perspective. |
| Menu close (any trigger) | Immediate — no exit animation | Remove from DOM. Matches existing dropdown behavior. |
| Submenu open (desktop hover) | 0ms delay — hover-intent timer NOT implemented in this phase | Keeps the code path simple; pattern can upgrade later if users report accidental submenu flashing. |
| Submenu close (desktop) | Immediate when hovering another root item or clicking outside | |
| Submenu expand (mobile) | Instant content swap (in-place replace per D-12) — no slide animation | Matches `BulkActionBar.tsx:57` popover reveal, which is also instant. |
| Toast enter | 150ms `translate-y-4 opacity-0` → `translate-y-0 opacity-100` | Matches the implicit "things that appear at the bottom of the viewport slide up" convention users expect from Gmail/iOS. `transition-all duration-150`. |
| Toast exit (5s auto-dismiss) | 150ms reverse of enter | |
| Toast exit (Undo clicked) | Immediate — no animation | User took action; no need to linger. |
| Toast exit (replaced by new toast) | Immediate swap of content in the same DOM node — no enter/exit | Per D-16, only one toast slot exists. Replacing contents in-place avoids a visual "flicker" between two toasts. |

### Focus handling

- **Desktop menu open:** focus moves to the first menu item (`Edit`). Arrow keys navigate (D-21).
- **Desktop menu close:** focus returns to the triggering `ThoughtRow` element (D-21).
- **Mobile menu open:** no focus shift — touch users don't use keyboard focus. Tap outside closes.
- **Toast open:** does NOT steal focus. The user's cursor/finger stays where it was. A11y announces via `aria-live="polite"`.

### Hit targets

- **Desktop menu item:** minimum height 36px (`min-h-9`) — above the 24px desktop minimum, below the 44px mobile minimum. Keeps the menu compact.
- **Mobile menu item:** minimum height 44px (`min-h-11`) — meets the iOS Human Interface Guidelines touch target. The menu is rendered at the same size on both; the extra 8px is always there. Not a spacing scale violation — 44 is a legitimate touch-target exception (noted in Spacing Exceptions above is implied; 44px is an a11y minimum, not a layout scale value).
- **Toast "Undo" button:** minimum 44px-wide hit region regardless of text width.

## Registry Safety

| Registry | Blocks Used | Safety Gate |
|----------|-------------|-------------|
| shadcn official | none (shadcn not initialized in this project) | not applicable |
| third-party | none | not applicable |

**Rationale:** `vigil-pwa` is a Tailwind v4 app without `components.json`. All UI is hand-rolled against the token layer. No external component library code enters the tree in this phase. This decision matches the existing `vigil-pwa/package.json` deps (no `@radix-ui`, no `@headlessui`, no `@shadcn/*`) and keeps the phase additive-only.

If a future phase wants to replace the hand-rolled menu with a shadcn-powered one (e.g. for keyboard-nav polish), that's a separate design-system-adoption phase — not this one.
