**One-liner:** PostgreSQL `work_order_statuses` table with `PUT /work-orders/:caseNumber/status` upsert and `GET /work-orders/statuses` map endpoint, seeded with 4 existing completed work orders.

## What Was Built

**Schema:** `workOrderStatuses` table with `case_number` text PK, `status` text NOT NULL DEFAULT 'open', `updated_at` timestamp with time zone.

**Route (`work-order-status.ts`):**
- `GET /v1/work-orders/statuses` — returns all rows as flat `{ caseNumber: status }` map; registered before the param route to avoid greedy matching
- `PUT /v1/work-orders/:caseNumber/status` — validates status against `["open", "inProgress", "done"]` allowlist (T-65-01 threat mitigation), destructures only `{ status }` from body (mass-assignment defense), upserts via `onConflictDoUpdate`

**Types:** `DrizzleWorkOrderStatus`, `NewWorkOrderStatus`, `WorkOrderStatusValue` added to `types.ts`.

**Migration:** `drizzle/0004_silent_reaper.sql` generated by drizzle-kit, applied to Railway PostgreSQL via public proxy URL.

**Seed:** 4 existing completed work orders (CS0353598, CS0355778, CS0354176, CS0354992) inserted with `status="done"` and `ON CONFLICT DO NOTHING` idempotency.

## Deviations from Plan

**None — plan executed exactly as written.**

The only notable deviation in approach: migration was applied via `DATABASE_PUBLIC_URL` (Railway proxy) rather than `db:push`, because `drizzle-kit push` hung when using the `railway.internal` URL and the local Postgres instance had no credentials configured. `db:migrate` with the public proxy succeeded cleanly.
