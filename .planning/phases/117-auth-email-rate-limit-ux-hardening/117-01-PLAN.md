<objective>
Raise per-route rate-limit caps on the 4 auth-email endpoints per CONTEXT.md D-03/D-04/D-05 so legitimate household-NAT retry patterns no longer trip 429, while preserving brute-force protection. Concretely:

- `verify-email`: per-IP cap `5 → 20` (D-03)
- `reset-password`: per-IP cap `5 → 20` (D-03)
- `resend-verification`: per-userId cap `3 → 5` (D-04)
- `forgot-password`: per-IP cap `5 → 20`, per-email cap stays at `5` (D-05 — enum-safety preservation)

Update tests to (a) assert the new caps trip at the new boundary and (b) assert the cap CONSTANTS themselves to lock the value against accidental future-drift. The forgot-password change requires splitting the single `RATE_LIMIT_MAX = 5` into TWO constants — `RATE_LIMIT_MAX_IP = 20` and `RATE_LIMIT_MAX_EMAIL = 5` — because the existing `takeSlot()` helper takes a single max and is shared by both axes.

Purpose: AUTH-13 satisfied. Phase 117 SC#2 ("legitimate user retrying any of the 4 flows from a single IP no longer trips the rate limit on routine retry patterns") satisfied. Brute-force protection structurally preserved (20/hr per-IP still hard-blocks 100-attempts/min abuse — SC#3). Forgot-password enum-safety unchanged because per-email cap is the dominant defense and 200-enum-safe response shape is untouched.

Output: 4 server route files with updated cap constants, 4 test files with updated cap assertions + new cap-value-lock tests.
</objective>
