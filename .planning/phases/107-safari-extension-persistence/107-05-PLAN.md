<objective>
Close gap_107_1: suppress the storyboard-instantiated container window on every launch of Vigil Capture.app so that only the one-time first-launch NSAlert is visible on cold install, and NO window is visible on Login Item boot launches. Preserves D-04 persistence-pill behavior — when the user explicitly opens the container via Safari's "Open preferences" button or via a manual post-boot double-click, the window appears with Main.html + pill.

Purpose: The prior revision of this plan proposed an UNCONDITIONAL `webView.window?.makeKeyAndOrderFront(nil)` at the end of `webView(_:didFinish:)`. That regressed the Login Item boot path — after `sudo reboot`, the OS silently launches Vigil Capture, the WKWebView loads ~50-300ms later, didFinish fires, window flashes on screen. That directly violates D-01 ("No Dock icon, no menu bar, no window on launch — ever"), SC#2, and the plan's own truth #1-2. It also violates the user's explicit directive in gap_107_1.fix_approach: "Re-show only when Safari's preference-open callback fires."

This revision gates the re-show on a launch-source heuristic. Research (§research_note below) confirmed no single Apple-documented signal cleanly distinguishes Safari-prefs launch from Login Item launch on all macOS versions in practice — there is no guaranteed Apple Event delivery for `SFSafariApplication.showPreferencesForExtension`, no custom URL scheme, and `NSApplicationLaunchIsDefaultLaunchKey` is legacy and not reliably populated by Safari. The pragmatic hybrid adopted here:

1. **Primary signal (opportunistic):** `application(_:open:)` callback — if Safari delivers ANY inbound open event (URL or Apple Event), flag the launch as user-initiated.
2. **Fallback heuristic:** `ProcessInfo.processInfo.systemUptime < 120` is almost certainly a Login Item boot launch → NEVER re-show. `systemUptime >= 120` AND firstLaunchAlertShown is already set → assume user-initiated (Safari-prefs or manual double-click) → ALLOW re-show.
3. **First-launch (install) exception:** on the one-shot install launch, only the NSAlert is shown — the `makeKeyAndOrderFront` gate stays closed because `firstLaunchAlertShown` is still false at the moment `webView(_:didFinish:)` fires. This keeps the gap_107_1 fix intact.

## Inbound Launch-Source Signal for Safari "Open Preferences" (post-revision research)

**Question investigated:** When the user clicks "Open preferences" in Safari → Settings → Extensions → Vigil Capture, what signal does the container app receive that would let it distinguish this launch from a Login Item boot launch or a plain double-click?

**Findings (from Apple docs + community sources, 2026-04-20):**

1. **`application(_:open:)` (NSApplicationDelegate):** Fires when another app (including the system) launches the receiver with URL context. Safari's `SFSafariApplication.showPreferencesForExtension(withIdentifier:completionHandler:)` is NOT documented to deliver a URL to the target container — it uses `LSLaunchURLSpec` / Launch Services internally to bring the container to the foreground. Some community reports indicate Safari delivers a `kInternetEventClass` / `kAEGetURL` Apple Event in certain macOS versions; others report no inbound signal at all. Unreliable as a sole detection mechanism.

2. **`NSApplicationLaunchIsDefaultLaunchKey` in `NSApplicationDidFinishLaunchingNotification` userInfo:** Legacy key from pre-macOS-13. Documentation coverage is thin and deprecated. When present and `false`, indicates the app was launched by another process (login item, Finder double-click from LaunchServices, or Safari — not discriminating between the three). Not useful for THIS distinction.

3. **`NSWorkspace.didLaunchApplicationNotification` userInfo:** Delivered to OTHER apps, not to the launching app itself. Not applicable for self-detection.

4. **`ProcessInfo.processInfo.systemUptime`:** Cocoa-documented, returns seconds since boot. A value < 120s (2 minutes) is a strong signal the launch is a Login Item boot launch (SMAppService registers as a LoginItemsEnabled scheduled-at-boot task; macOS dispatches these in the first 60-90s post-login). A value ≥ 120s on a single-user workstation that has been idle since login is almost certainly a user-initiated launch (Safari-prefs click or manual double-click).

5. **`NSApp.isActive` / `NSRunningApplication.current.activationPolicy`:** For an LSUIElement=true accessory app, activationPolicy is always `.accessory`; isActive toggles based on whether the app is frontmost. At the moment `applicationDidFinishLaunching` fires, isActive is NOT yet deterministic — Safari-prefs launches DO typically activate the target, but timing is race-prone.

**Conclusion — chosen strategy:**

Adopt the `application(_:open:)` hook as an OPPORTUNISTIC primary signal AND the `systemUptime >= 120` heuristic as the AUTHORITATIVE fallback. Either condition being true allows the re-show. The conservative first-launch exception (re-show only when `firstLaunchAlertShown` is already true) further tightens the gate so the install flow shows only the NSAlert.

**Rejected approaches and why:**

- **Pure `application(_:open:)`:** Would fail in practice if Safari doesn't deliver the open event on this macOS version. User clicks "Open preferences," container launches, no URL arrives, window never re-shows — breaks D-04.
- **Safari Extension IPC (XPC / shared UserDefaults) from the .appex to signal intent:** Requires modifying the SafariExtensionHandler, which the gap_107_1 scope explicitly excludes. The .appex can't even run when the user is in the macOS Settings pane interacting with Safari's own UI.
- **Pure `NSApp.isActive` check:** Race-prone at `applicationDidFinishLaunching` timing; unreliable.
- **Pure uptime heuristic (no `application(_:open:)`):** Works, but misses the signal when Safari DOES deliver it — leaves users who click "Open preferences" within 2 min of boot with no window.

**Tradeoff accepted:** If the user clicks "Open preferences" within 120s of boot AND Safari does not deliver an Apple Event / URL, the window will NOT re-show — the user sees the D-04 pill fail gracefully (no UI). This is an acceptable edge case: the only path that triggers it is a user rebooting, immediately logging in, and within 2 min clicking Safari → Settings → Extensions → Open Preferences. The preferred user action in that case is to wait 2 minutes or quit and re-open the app — both work.

**Post-ship verification:** Human-UAT checkpoint in Plan 04 covers the user-facing flows (reboot persistence + first launch + Safari-prefs re-open after >2min uptime). Edge case "Safari-prefs click within 120s of boot" is documented in 107-HUMAN-UAT.md as a known tradeoff, not a gap.
</research_note>

```swift
    // MARK: - Window reveal gating (gap_107_1 fix)

    /// Exposed to ViewController.webView(_:didFinish:) to decide whether to call
    /// `makeKeyAndOrderFront(nil)` on the container window. Gated on launch source:
    ///   1. application(_:open:) fired during launch (Safari-prefs opportunistic signal)
    ///   2. OR: systemUptime >= 120 AND firstLaunchAlertShown already true
    ///      (user-initiated launch on fully-booted machine, post-install).
    /// Both conditions false → Login Item boot launch OR first install launch →
    /// window stays hidden. See 107-05-PLAN.md §research_note for rationale.
    var shouldRevealWindow: Bool = false

    /// application(_:open:) is the NSApplicationDelegate inbound hook for
    /// URL/Apple Event deliveries. Safari's showPreferencesForExtension MAY route
    /// through this path (documented as best-effort — see §research_note). If we
    /// receive any open event during launch, treat it as Safari-prefs-driven and
    /// flip the re-show gate.
    func application(_ application: NSApplication, open urls: [URL]) {
        shouldRevealWindow = true
        os_log("application(_:open:) fired; shouldRevealWindow=true", type: .info)
    }

    /// Closes gap_107_1: LSUIElement=true (Plan 107-01) suppresses Dock icon + menu
    /// bar but NOT storyboard-instantiated windows. Main.storyboard's initial
    /// Window Controller (B8D-0N-5wS) loads its window on every launch.
    ///
    /// Strategy:
    /// 1. Order out all windows unconditionally (covers first-launch NSAlert flow
    ///    AND Login Item boot flow — both must be silent).
    /// 2. Compute shouldRevealWindow based on launch source heuristic:
    ///    - systemUptime < 120s → Login Item boot launch → stay hidden forever.
    ///    - firstLaunchAlertShown == false → first install launch → NSAlert only, stay hidden.
    ///    - systemUptime >= 120s AND firstLaunchAlertShown == true → user launch
    ///      (Safari-prefs click or manual double-click) → allow re-show when
    ///      webView.didFinish fires (D-04 preservation).
    /// 3. application(_:open:) can OVERRIDE to true at any time if Safari
    ///    delivers an inbound URL/Apple Event (opportunistic detection).
    ///
    /// Per RESEARCH Pitfall 2, the storyboard stays intact — ViewController
    /// still instantiates, WKWebView still loads Main.html, and the D-04
    /// persistence pill still renders in the gated path.
    private func suppressStoryboardWindows() {
        NSApp.windows.forEach { $0.orderOut(nil) }

        let uptime = ProcessInfo.processInfo.systemUptime
        let alertAlreadyShown = UserDefaults.standard.bool(forKey: firstLaunchAlertKey)
        // Gate: only allow re-show if system has been up ≥ 2 min (past Login Item
        // dispatch window) AND this is not the install-launch.
        if uptime >= 120 && alertAlreadyShown {
            shouldRevealWindow = true
            os_log("suppressStoryboardWindows: uptime=%.0f, alertShown=true, revealGate=OPEN",
                   type: .info, uptime)
        } else {
            shouldRevealWindow = false
            os_log("suppressStoryboardWindows: uptime=%.0f, alertShown=%{bool}d, revealGate=CLOSED",
                   type: .info, uptime, alertAlreadyShown)
        }
    }
```
