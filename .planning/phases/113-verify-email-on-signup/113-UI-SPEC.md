## Component Anatomy

### Component 1: VerifyEmailPage (`vigil-pwa/src/pages/VerifyEmailPage.tsx`)

**Pattern source:** ResetPasswordPage — same full-screen centered card shell, same no-fetch-on-mount constraint.

**Shell (all states):**
```
div.min-h-screen.flex.items-center.justify-center.bg-gray-900
  div.bg-gray-900.rounded-lg.p-8.w-full.max-w-md.mx-4
    [state-specific content]
```

**State 1 — Token present, pre-click (idle):**
```
h1.text-2xl.font-medium.text-white.mb-6      "Verify your email"
p.text-sm.text-gray-400.mb-4                 "Click the button below to confirm your email address."
button[type=submit].w-full.py-2.bg-teal-600.hover:bg-teal-800.text-white.rounded.font-medium.disabled:opacity-50
  "Confirm" (idle) | "Confirming…" (loading)
```

Button disabled states:
- `disabled` prop = true when `state === 'loading'`
- No disabled state for idle

**State 2 — Success (swap in-place after 200):**
```
h1.text-2xl.font-medium.text-white.mb-4      "Email verified"
p.text-sm.text-gray-300.mb-6                 "You can close this tab, or"
Link[to="/"].text-sm.text-teal-400.hover:text-teal-300   "Go to app"
```

**State 3 — Generic error (400 / 5xx / network — single bucket):**
```
h1.text-2xl.font-medium.text-white.mb-4      "This link is no longer valid"
p.text-sm.text-gray-300.mb-6                 "Verification links expire after 24 hours and can only be used once."
button.w-full.py-2.bg-teal-600.hover:bg-teal-800.text-white.rounded.font-medium
  "Request a new link"  [navigates to /settings if logged-in, /auth if logged-out]
div.mt-4.text-center
  Link[to="/"].text-sm.text-gray-400.hover:text-gray-200   "Back to app"
```

Detecting logged-in state for "Request a new link" destination: check `sessionStorage.getItem('vigil_jwt') !== null` — the same key used by `storeKey()` in `api/client.ts`. No API call needed.

**State 4 — Missing token (rendered immediately on mount if `?token` absent):**
```
h1.text-2xl.font-medium.text-white.mb-4      "This verification link is malformed"
p.text-sm.text-gray-300.mb-6                 "Please use the button in the email we sent you."
button.w-full.py-2.bg-teal-600.hover:bg-teal-800.text-white.rounded.font-medium
  "Back to app"  [navigates to "/"]
```

**State variable:**
```typescript
type VerifyState = 'idle' | 'loading' | 'success' | 'error' | 'missing_token'
```

Initialize to `'missing_token'` if `searchParams.get('token')` is null/empty; `'idle'` otherwise.

**API call:**
- Triggered by Confirm button `onClick` only — NEVER in a `useEffect`.
- Uses raw `fetch()` (not `vigilFetch`) — endpoint is unauthenticated; avoids 401-redirect side-effects.
- POST to `${API_BASE}/v1/auth/verify-email` with body `{ token }`.
- 200 → set state `'success'`. Any non-200 → set state `'error'`.

### Component 2: Settings Verify Banner (inline in `vigil-pwa/src/pages/SettingsPage.tsx`)

**Placement:** First element inside `<div className="p-4 max-w-2xl mx-auto text-gray-50">`, before the dismissible `{banner && ...}` block and before the `<h1>Settings</h1>`. This ensures it is always the top-most visible element on /settings when rendered.

**Visibility condition:** Rendered when `verifyState.emailVerifiedAt === null` (from `/v1/auth/me` response). Hidden once non-null.

**Banner anatomy:**
```
div[role="alert"].mb-4.rounded.border.border-warning-400.bg-warning-50.px-4.py-3.flex.items-center.justify-between
  p.text-sm.text-warning-400
    "Verify your email — we sent a link to {email}. Click it to confirm."
  button[type="button"].ml-3.text-sm ... [Resend lifecycle — see below]
```

Tailwind class set for banner wrapper:
```
mb-4 rounded border border-warning-400 bg-warning-50 px-4 py-3 flex items-center justify-between
```

Note: `border-warning-400` and `bg-warning-50` reference the @theme custom tokens defined in `index.css`. The Tailwind v4 `@theme` block maps them to `--color-warning-400` and `--color-warning-50`. Use these class names exactly.

**Resend button states (D-25):**

| State | Label | Classes | Disabled |
|-------|-------|---------|---------|
| Idle | "Resend" | `ml-3 px-3 py-1 bg-teal-600 hover:bg-teal-500 rounded text-white text-sm` | false |
| Sending | "Sending…" | same + `opacity-50 cursor-not-allowed` | true |
| Sent (10s) | "Sent! Check your inbox." | `ml-3 text-sm text-warning-400` (plain text, no button styles; or keep button shape with `disabled:opacity-50`) | true |
| Rate-limited (429) | button hidden; inline error shown | — | — |
| Error (5xx/net) | "Could not send. Try again." as inline text; button re-enables | `ml-3 text-sm text-red-400` | false (re-enable Resend after error) |

Rate-limited state: when 429 fires, hide the Resend button entirely and render:
```
span.ml-3.text-xs.text-red-400   "You've requested too many. Try again later."
```
Button stays hidden until next page mount (D-25: "Button stays disabled in this rendered state until next page load").

**Resend state variable:**
```typescript
type ResendState = 'idle' | 'sending' | 'sent' | 'rate_limited' | 'error'
```

**Sent → Idle timer:** `window.setTimeout(() => setResendState('idle'), 10_000)`. Clear on unmount.

**aria-live:** The Resend status region (the span/button that changes labels) must have `aria-live="polite"` on a wrapping element so screen readers announce state changes without interrupting. Apply to the `<div>` containing the Resend button.

**`/v1/auth/me` fetch (D-28):**
```typescript
// New useEffect in SettingsPage — parallel to existing /v1/me useEffect
useEffect(() => {
  vigilFetch('/v1/auth/me')
    .then((r) => r.ok ? r.json() : undefined)
    .then((data?: { id: number; email: string; emailVerifiedAt: string | null }) => {
      if (data) setMeData(data)
    })
    .catch(() => { /* silent — banner absent if /me fails */ })
}, [])
```

New state variables added to SettingsPage:
```typescript
const [meData, setMeData] = useState<{ id: number; email: string; emailVerifiedAt: string | null } | null>(null)
const [resendState, setResendState] = useState<ResendState>('idle')
```

The existing `accountEmail` from `/v1/me` continues unchanged. The new `meData.email` from `/v1/auth/me` is used for the banner interpolation. If `/v1/me` returns before `/v1/auth/me`, the banner can use `accountEmail` as a fallback for `{email}` interpolation.

## State Diagrams

### VerifyEmailPage State Machine

```
mount
  └─ token absent → MISSING_TOKEN (terminal display)
  └─ token present → IDLE

IDLE
  └─ Confirm click → LOADING
     ├─ 200 → SUCCESS (terminal display)
     └─ any error (400/5xx/network) → ERROR (terminal display)

LOADING
  └─ (pending fetch)
```

No transitions out of SUCCESS, ERROR, or MISSING_TOKEN. All three are terminal render states. In-place swap, no navigation.

### Settings Resend Button State Machine

```
IDLE
  └─ Resend click → SENDING
     ├─ 200 → SENT
     │   └─ 10s timeout → IDLE
     ├─ 429 → RATE_LIMITED (until page reload)
     └─ 5xx/network → ERROR → IDLE (re-enabled immediately)
```

## Notes

1. **`components.json` absent, shadcn not initialized.** This is by design — the project uses Tailwind v4 `@theme` tokens in `index.css`. Do not initialize shadcn for this phase.

2. **Warning tokens are custom.** `bg-warning-50` and `border-warning-400` are defined in the `@theme` block in `index.css` — they are project-specific tokens, NOT Tailwind's built-in `amber-*`. Use these exact class names.

3. **VerifyEmailPage uses raw `fetch()`, not `vigilFetch()`.** The `/v1/auth/verify-email` endpoint is unauthenticated (bypass-listed in bearerAuth). Using `vigilFetch` could inject a stale bearer token and trigger the 401-redirect handler on unexpected errors. Use `fetch(${API_BASE}/v1/auth/verify-email, ...)` directly.

4. **The `/v1/auth/me` call in SettingsPage is a second `/me` call alongside the existing `/v1/me` call** (line 92 of SettingsPage.tsx). Both fire on mount in parallel. This is the "smallest possible PWA change" per CONTEXT D-28. Do NOT consolidate or replace the existing `/v1/me` call — App.tsx and SettingsPage depend on its `{ userId, email }` shape for PostHog identity.

5. **"Request a new link" destination logic:** check `sessionStorage.getItem('vigil_jwt')` at click time. Non-null → navigate to `/settings`. Null → navigate to `/auth`. This is a runtime check at click time, not at render time, so it stays accurate even if the session expires between page mount and button click.

6. **No loading skeleton for the banner.** If `/v1/auth/me` has not responded yet, the banner simply does not render (condition is `meData?.emailVerifiedAt === null`). While `meData` is `null` (loading), the banner is absent — this prevents a flash of banner on verified users.

7. **Resend button re-enables after error (5xx/network), NOT after rate-limit.** D-25 is explicit: rate-limited state persists until next page load; 5xx/network re-enables immediately.
