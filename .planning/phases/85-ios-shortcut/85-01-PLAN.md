## Prereqs
- iOS 17+ / macOS 14+ (for the "Shortcut Input" magic variable with type branching)
- Your Vigil API key ready to paste
- 10–15 minutes uninterrupted — the action chain has ~15 steps

## Step 1 — Create the Shortcut

1. Open **Shortcuts.app**.
2. Tap **+** (top-right) to create a new Shortcut.
3. Tap the Shortcut name at the top (defaults to "New Shortcut") → rename to **Vigil Capture**.

## Step 2 — Configure share-sheet intake

1. Tap the **ⓘ** info icon (top of the editor, or bottom on iPhone).
2. Toggle **Show in Share Sheet** ON.
3. Under **Share Sheet Types**, deselect **Everything** and select ONLY:
   - **URLs**
   - **Text**
   - **Safari web pages**
4. (Leave **Receive [input type] from Share Sheet** visible — this creates the magic variable `Shortcut Input`.)
5. Dismiss the info panel.

## Step 3 — Config: api_key and base_url (two Text actions, per D-01)

Add these first so they're at the top and obvious to edit.

**Action 1: Text**
- Search the action library for **Text** → add it.
- In the Text body, leave **empty** for now (user fills in their key after install — this is D-01's "blank before sharing" convention).
- Tap the action's chevron to expand → tap the action name **Text** itself → the action gets a little title. Use the ••• / rename options if available to label it `api_key`. If your Shortcuts version doesn't support per-action labels, add a **Comment** action directly above the Text action with text `api_key` — same effect.

**Action 2: Set Variable**
- Search → add **Set Variable**.
- Variable Name: `api_key`
- Input: tap the input slot → select the magic variable that represents the output of the previous Text action ("Text").

**Action 3: Text**
- Add another **Text** action.
- Body: `https://api.vigilhub.io`
- Label/comment it `base_url`.

**Action 4: Set Variable**
- Variable Name: `base_url`
- Input: the previous Text action's output ("Text").

At this point your Shortcut has 4 actions. User edits only Action 1 (paste key) and Action 3 (only if pointing at a different vigil-core instance).

## Step 4 — Ask the user for an optional note

**Action 5: Ask for Input**
- Input Type: **Text**
- Prompt: `Note (optional)`
- Default Answer: (leave blank)
- **Allow Multiple Lines:** ON (if your iOS version exposes this toggle; harmless if missing)

**Action 6: Set Variable**
- Variable Name: `note`
- Input: **Provided Input** (the output of Ask for Input).

## Step 5 — Branch on Shortcut Input type (URL vs Text)

**Action 7: If**
- Condition: tap the input slot → select **Shortcut Input** → then the condition `has any value` is fine for the outer check, OR use **is of type → URL**. We want a three-way branch: URL (incl Safari web page), Text, nothing. Since Shortcuts' If only supports one condition, we nest:

Set the If to: **Shortcut Input** → **is** → **URL**

### Inside the If (Shortcut Input is URL):

**Action 8 (inside If): Get Details of Safari Web Pages**
- Search for **Get Details of Safari Web Pages** (yes, plural, the Safari-specific variant — it returns page title from a URL).
- Input: **Shortcut Input**
- Get: **Name**

(If that action is unavailable on your iOS version, fall back to **Get Details of URL** → Get: **Page Title**. The intent is the same: resolve the page title from the URL.)

**Action 9 (inside If): Set Variable**
- Variable Name: `title`
- Input: output of Action 8 (the page name/title).
  - Note: if the URL isn't a real webpage (e.g., shared from a non-Safari context with no reachable page), this may be empty. That's handled below.

**Action 10 (inside If): Text**
- Body — type the following exactly, substituting magic variables where indicated:
  ```
  [title]
  [Shortcut Input]

  [note]
  ```
  Where `[title]`, `[Shortcut Input]`, and `[note]` are inserted by tapping the variable-picker and selecting the magic variables (NOT typed as literal brackets). Actual line breaks between them. The blank line between URL and note is intentional (D-02).

  If `title` is empty (non-webpage URL), the first line renders blank → that's acceptable, user sees `\nURL\n\nnote` which is still readable. Don't over-engineer.

**Action 11 (inside If): Set Variable**
- Variable Name: `content`
- Input: output of Action 10 (the assembled Text).

### Otherwise branch (Shortcut Input is NOT URL — i.e., Text or empty):

**Action 12 (inside Otherwise): Text**
- Body:
  ```
  [Shortcut Input]

  [note]
  ```
  (Two magic variables with one blank line between. If Shortcut Input is empty and note is empty, this renders as `\n\n` = effectively empty, which the empty-content guard catches.)

**Action 13 (inside Otherwise): Set Variable**
- Variable Name: `content`
- Input: output of Action 12.

### End If

**Action 14: End If** (Shortcuts adds this automatically when the If closes).

## Step 6 — Empty-content guard (D-06)

**Action 15: If**
- Condition: `content` **is** `` (empty)
  - To set "is empty": tap the If → condition **has any value** → flip to **does not have any value** if available. If your iOS version only has equality operators, use **is** and leave the comparison value blank. Shortcuts treats a blank comparison as "empty string".
  - Alternative: use a **Count** action on `content` → **Characters** → If count **is** `0`. Either works.

**Action 16 (inside If): Show Notification**
- Title: `Vigil`
- Body: `Nothing to capture`

**Action 17 (inside If): Stop Shortcut**
- (Search for **Stop Shortcut** or **Stop This Shortcut**.)

**Action 18: End If**

## Step 7 — Build the JSON body (Dictionary, not raw text)

**Action 19: Dictionary**
- Add a **Dictionary** action (search "Dictionary").
- Tap **Add new item** twice to get two rows:

  Row 1:
    - Type: **Text**
    - Key: `content`
    - Value: magic variable `content`

  Row 2:
    - Type: **Text**
    - Key: `source`
    - Value: literal text `text`  ← **MUST be literal string "text", not "browser" (would 400 per vigil-core VALID_SOURCES)**

**Action 20: Set Variable**
- Variable Name: `request_body`
- Input: the Dictionary output.

## Step 8 — POST to the API

**Action 21: Text**
- Body: build the URL using magic variables:
  ```
  [base_url]/v1/thoughts
  ```
  Insert `base_url` as a magic variable; type the literal `/v1/thoughts` afterward.

**Action 22: Set Variable**
- Variable Name: `endpoint`
- Input: the Text output from Action 21.

**Action 23: Text**
- Body: build the Authorization header value:
  ```
  Bearer [api_key]
  ```
  (Literal `Bearer ` followed by a single space, then magic variable `api_key`.)

**Action 24: Set Variable**
- Variable Name: `auth_header`
- Input: Action 23's output.

**Action 25: Get Contents of URL**  ← **the network call**
- URL slot: magic variable `endpoint`
- Tap **Show More** to expose Method / Headers / Request Body.
- **Method:** POST
- **Headers:** tap **Add new header** twice:
    - Header 1: Key `Authorization`, Value: magic variable `auth_header`
    - Header 2: Key `Content-Type`, Value: literal `application/json`
- **Request Body:** tap the dropdown → select **JSON**.
  - In the JSON editor, tap **Add new field** and rebuild:
    - Key `content`, Type **Text**, Value: magic variable `content`
    - Key `source`, Type **Text**, Value: literal `text`
  - (You may notice you're duplicating what Action 19 built. That's fine — the Dictionary-to-Variable chain in Step 7 is a sanity-checking intermediate; the Request Body field is where Shortcuts actually serializes. Alternatively: skip Action 19/20 and build the JSON directly here. Either pattern works; the explicit Dictionary makes it inspectable during development.)
- **Tip:** if your Shortcuts version supports **File** type for Request Body with a magic-variable JSON dictionary, you can pass the `request_body` Dictionary directly. Both paths result in the same HTTP body.

## Step 9 — Handle response (D-06)

**Action 26: Get Dictionary from Input**
- Input: output of Action 25 (Get Contents of URL).
- This parses the response body as a dictionary. If the response is non-JSON (unlikely but possible on 5xx), this fails silently and the variable is empty.

**Action 27: If**
- Condition: the Shortcut's HTTP status… Shortcuts.app does NOT expose response status code from Get Contents of URL in older iOS versions. On iOS 17+, Get Contents of URL exposes a **Status Code** property via the output picker.
  - If Status Code is available: `Status Code` **is** `201` (or `is not` `201`, depending on branch direction). Best: check **Status Code** is **less than** `300` for 2xx success.
  - If Status Code is NOT available in your iOS version: fall back to checking whether the parsed dictionary from Action 26 has a field that only exists on success (e.g., `id` — the vigil-core response includes the created thought's id). Use: `Dictionary Value for "id"` → If `has any value`.

Pick whichever check your iOS supports. The GOAL is: success branch vs failure branch.

### Success branch:

**Action 28 (inside If): Show Notification**
- Title: `Vigil`
- Body: `Captured to Vigil`

### Otherwise (failure):

**Action 29 (inside Otherwise): Show Notification**
- Title: `Vigil`
- Body — assemble from status if available:
  ```
  Vigil capture failed: HTTP [Status Code]
  ```
  If Status Code is not available, use body:
  ```
  Vigil capture failed
  ```

**Action 30: End If**

## Step 10 — Final sanity checks in Shortcuts.app

1. Scroll to top: confirm Text action 1 = `api_key` (editable), action 3 = `base_url` = `https://api.vigilhub.io`.
2. Confirm share-sheet types: URL, Text, Safari web pages.
3. **Test inside Shortcuts.app before export:**
   - Paste your real API key into the first Text action.
   - Tap the ▶ play button at the bottom.
   - When prompted for input, paste a URL (e.g., https://example.com) OR a text selection.
   - Enter a note like "test from editor".
   - Confirm "Captured to Vigil" notification.
   - Hit the Vigil PWA (or curl `{base_url}/v1/thoughts`) — confirm the thought exists with the right content.
4. **Test the empty-content guard:**
   - Run the Shortcut again with no input (or tap through the share-sheet dry-run). Leave the note blank when prompted.
   - Confirm "Nothing to capture" notification appears and nothing hits the API.
5. **Test the error path:**
   - Temporarily change the api_key Text action to `bad-key-999`.
   - Run it — confirm failure notification.
   - Restore your real key.

If any test fails, fix the offending action before moving to Task 2. Common issues:
- **"Text" action vs "Get Text from Input"** — two different actions. You want plain **Text** for the config values and URL/auth assembly.
- **Magic variable picker** — if you accidentally typed `[api_key]` as literal text, the Shortcut will try to send the literal string. Delete, tap the variable picker icon (the little chip button above the keyboard), and pick the variable.
- **Dictionary vs JSON in Request Body** — make sure the Request Body dropdown is JSON, not Form or File.
- **Status Code vs response body** — adjust the If in Action 27 based on what your iOS version exposes.

**Before exporting or sharing, clear your API key from the Shortcut:**

## B — Export .shortcut file

**On Mac (easiest):**
1. In Shortcuts.app, right-click the **Vigil Capture** tile in your library → **Export…** (or File → Export…).

**On iPhone/iPad:**
1. Long-press the Shortcut tile → **Share** → **Save to Files**.

3. AirDrop the file to your Mac OR upload it somewhere Claude can access it locally.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| User's device → Shortcuts.app | User pastes API key into a local editor; key lives in the Shortcut in iCloud-synced Shortcuts library |
| Shortcut → Vigil API | Bearer key crosses a local Shortcut action boundary into an HTTPS request |
| Exported .shortcut / iCloud link → public distribution | If key is NOT blanked, it leaks to anyone with the file or URL |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-85-01 | Information Disclosure | API key in exported .shortcut | mitigate | Task 2 Step A mandates blanking the key before export. README in plan 85-02 re-emphasizes. User memory documents "key sprawl already accepted" — this adds one more location but with an explicit "blank before sharing" discipline. |
| T-85-02 | Information Disclosure | API key in iCloud share link | mitigate | Same blanking rule — iCloud link snapshots the Shortcut's current state. Blank first, copy link second. |
| T-85-03 | Spoofing | Bearer token over HTTPS | mitigate | base_url defaults to `https://api.vigilhub.io`; user would have to deliberately change to http:// to break this. |
| T-85-04 | Tampering | Magic-variable injection into content | accept | User-shared text is the content; there's no server-side template injection (vigil-core treats content as opaque string). No XSS risk in a notification context. |
| T-85-05 | Denial of Service | Accidental repeated shares | accept | Vigil-core rate limits 100 req/60s/IP. Manual share-sheet invocation can't realistically exceed that. |
</threat_model>
