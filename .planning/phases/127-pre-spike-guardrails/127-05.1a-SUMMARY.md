# Phase 127 Plan 05.1a: GUARD-03 AI Client Wrap Summary

GUARD-03 wiring lands at the Anthropic SDK chokepoint: the three `callClaude*` wrappers in `vigil-core/src/ai/client.ts` now require `userId: number` on their options object and wrap every `ai.messages.create` call in `withBudgetTracking(userId, ...)`. The direct-bypass site at `process-audio.ts:82-115` (`ai.beta.messages.create`) is now wrapped explicitly ‚Äî Pitfall 4 closed at both chokepoints. A pre-flight `await requireAiBudget(userId)` gate guards the `/v1/process-audio` handler before any AI work. A drift detector in `client.test.ts` pins both invariants so a future revert trips CI.

## What landed

**Task 1 ‚Äî client.ts wrapper widening (commit `0a25c95c`):**
- All three wrappers (`callClaude`, `callClaudeConversation`, `callClaudeMultimodal`) gain a REQUIRED `userId: number` field on their options object ‚Äî NOT optional. TypeScript compile fails if any call site omits it (which is the propagation enforcement mechanism).
- Each `await ai.messages.create({...})` call is wrapped: `await withBudgetTracking(userId, () => ai.messages.create({...}))`. The Anthropic SDK argument list is preserved byte-for-byte inside the arrow function.
- Import added: `import { withBudgetTracking } from "../lib/ai-budget.js";`
- The literal `withBudgetTracking(userId,` appears exactly 3 times in client.ts (one per wrapper); JSDoc references that initially contained the literal were softened to non-literal prose so the drift detector's strict `=== 3` assertion holds.

**Task 2 ‚Äî client.test.ts drift detector (commit `8c048c40`):**
- New file at `vigil-core/src/ai/client.test.ts` using node:test + assert/strict (matches vigil-core convention).
- **Test 1 (wrapper drift):** asserts `withBudgetTracking(userId,` matches in client.ts === 3. Catches any future revert that unwraps a wrapper OR adds a 4th wrapper without wiring it through `withBudgetTracking`.
- **Test 2 (signature drift):** asserts `userId: number` matches in client.ts >= 3. Catches anyone making userId optional.
- **Test 3 (cross-file Pitfall 4 scan):** walks `routes/` + `ai/` recursively (excluding `.test.ts`), finds every `await ai.messages.create` / `await ai.beta.messages.create` call, asserts `withBudgetTracking(` appears within the preceding 200 chars. Pinned RED before Task 3 with `routes/process-audio.ts:93 ‚Äî await ai.beta.messages.create`; green after Task 3 closes the wrap.

**Task 3 ‚Äî Pitfall 4 closure + userId propagation (commit `e0542b76`):**

- `vigil-core/src/routes/process-audio.ts`:
  - Added `await requireAiBudget(userId);` immediately after `const userId = c.get("userId");` ‚Äî pre-flight gate BEFORE body parse / AI work. Throws propagate to `app.onError` (Plan 05.1b adds the 429 branch).
  - The direct `ai.beta.messages.create({...})` call at lines ~93-115 is now wrapped in `await withBudgetTracking(userId, () => ai.beta.messages.create({...}))` ‚Äî closes T-127-03-E2 (the explicit direct-bypass site CONTEXT called out as the original Pitfall 4 example).
  - `ai.beta.files.upload` stays UNWRAPPED ‚Äî file upload is not token-billed.
  - Line ~156 `callClaude({...})` site now passes `userId`.

- userId propagation across the 9 wrapper call-site files:
  - `describe-image.ts` / `affirmation.ts` ‚Äî new `const userId = c.get("userId")` at handler entry (these handlers previously didn't need userId for their pre-127 logic).
  - `therapy.ts` (3 sites) / `prioritize.ts` / `thoughts.ts` auto-triage / `insights.ts` ‚Äî `userId` was already in scope; just added to the options object.
  - `triage.ts` POST handler gains `const userId = c.get("userId")`; `triageThought` helper signature widened to `(content: string, userId: number)`.
  - `chat.ts` ‚Äî passes `userId` to `callClaudeConversation`. (The `requireAiBudget` gate + `app.onError` 429 branch are Plan 05.1b's scope; they land separately).

- DI seam widening:
  - `brief-assembly-service.ts` `callClaudeFn` alias type widened to require `userId: number` on opts. The three helpers (`fetchAffirmation`, `fetchPrioritization`, `fetchInsights`) gain a `userId: number` parameter that `assembleAndRender(dateStr, userId)` forwards from its existing parameter.
  - `process-photo.ts` `triageFn` alias widened from `(content: string) => Promise<TriageResult>` to `(content: string, userId: number) => Promise<TriageResult>`. The `Promise.allSettled` call site at line ~486 now passes `userId` (already in scope from line 296).
  - `index.ts:323` scheduler `callClaudeFn: callClaude` ref automatically picks up the widened wrapper signature via TypeScript inference ‚Äî no edit needed at the assignment, but the scheduler's internal calls to `callClaudeFn` (via `assembler.assembleAndRender(dateStr, userId)`) already thread `userId` because `userId` was a pre-existing parameter of `assembleAndRender`.

## Re-scopes applied (vs CONTEXT.md)

- **chat.ts userId propagation lands NOW, not in Plan 05.1b.** The plan said "this plan does NOT touch chat.ts" but TypeScript-enforced signature widening means chat.ts's `callClaudeConversation` call MUST pass `userId` or the entire compile breaks. We applied the minimal change (add `userId,` to the options object) and explicitly carved out the `requireAiBudget` gate + `app.onError` 429 branch for Plan 05.1b. This is Deviation Rule 3 (auto-fix blocking issue) ‚Äî the type-level constraint left no other option.
- **triageThought helper signature is positional**, not options-object. The pre-existing signature was `triageThought(content: string)` ‚Äî positional. We widened to `triageThought(content: string, userId: number)` rather than refactoring to options-object form, both to minimize blast radius and to match the analog positional signature of `dbUpdateTriageFn(thoughtId, userId, result)` elsewhere in process-photo.ts.
