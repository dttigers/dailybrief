# Phase 116.1: Sports route + PWA error-class differentiation - Context

**Gathered:** 2026-04-29
**Status:** Ready for planning

<domain>
## Phase Boundary

Wrap BDL upstream failures in a structured 502 response on the server (no provider-name leak per T-73-01) and differentiate error classes in the PWA picker so users see useful diagnostics instead of opaque "Couldn't load teams." Brief generation surfaces per-league failures as visible placeholder lines instead of silent omissions. WR-01 (`encodeURIComponent` on `teamId`) folded in as cheap defense-in-depth.

**Out of scope:** WR-02 (`mapSports teamName` from env-var fallback — needs renderer + env-var deletion runbook coordination, defer to separate phase). New error UI for non-sports endpoints. Server-side rate limiting (this phase only handles UPSTREAM rate-limit signals from BDL). Retry logic / automatic backoff (only surface Retry-After to the user; let them click).

</domain>

<decisions>
## Implementation Decisions

### Status code + body contract (server)
- **D-01:** Single 502 contract for all BDL upstream failures (BDL 4xx, BDL 5xx, BDL 429, network failure, timeout). Body shape: `{error: "Upstream sports provider unavailable", retryAfter?: number}`. The `retryAfter` field (seconds) is ONLY present when BDL returned 429 with `Retry-After` header — proxied through. Error string is verbatim per the user's pasted scope.
- **D-02:** Set the HTTP `Retry-After` response header IN ADDITION to the body field when applicable (defense-in-depth — clients that read the header get the same data clients that read the body do).
- **D-03:** Add `AbortController.signal` with 10s timeout to `fetchJSON` in `sports-service.ts:294-305`. Timeout classifies as upstream-error (UpstreamError kind: `'timeout'`) and maps to 502 same as other failures.
- **D-04:** BDL 401/403 (auth fail — the local-env-gap symptom that triggered this phase) classifies as same 502. Server-side log line `BDL fetch failed: ${url} → ${status}` at `sports-service.ts:302` already gives ops the diagnostic. T-73-01 preserved: log records URL + HTTP status only, never the API key value.

### Brief-assembly fallback policy
- **D-05:** Per-league BDL failure in brief PDF renders the section header + a single-line placeholder `"{LEAGUE} data temporarily unavailable."` (e.g., "MLB data temporarily unavailable.") in the section body. Replaces today's silent omission via `mapSports` filter. Other (successful) leagues render normally. Plan-phase identifies the exact PDF renderer file and adds the placeholder branch.
- **D-06:** Add PostHog telemetry: `sports_league_fetch_failed` event with `{league, status, errorClass}` properties, fired server-side per failure. Uses existing PostHog Node SDK wiring from Phase 105. Replaces (not duplicates) the existing console log line — Railway log scrubbing has been the bottleneck, PostHog gives us cohort/funnel queries.
- **D-07:** All-leagues-failed scenario (every enabled league errors out): render section header + single block "Sports data temporarily unavailable. Try again on tomorrow's brief." in PDF. Distinct from Phase 116 D-17 (all-disabled → entire section omitted via `pdf-service.ts:281` guard) — failure ≠ user choice. New code path; reuses the placeholder template from D-05 with different copy.
- **D-08:** NO negative caching of failed BDL responses. Failures retried on next brief generation / next PWA picker action. Brief is once/day per user; teams cache is 24h; league cache is 5min — hammer concern doesn't apply. Negative cache would block recovery when BDL recovers fast.

### Route scope + classification architecture
- **D-09:** Apply 502 contract to all 4 BDL-touching code paths:
  1. `GET /v1/sports/teams/:league` (picker — the route that surfaced the bug)
  2. `GET /v1/sports/:league` (single-league preview)
  3. `GET /v1/sports` (aggregate — used by some legacy paths)
  4. `brief-assembly-service.ts:485` internal `deps.sportsService.fetchAllLeagues(sportsSelections)` call
  Brief gets the per-league placeholder path from D-05; routes (1)-(3) get the HTTP 502 + body + Retry-After header.
- **D-10:** Service layer is the single classification source. Add typed `UpstreamError` class with `{kind: 'rate-limited' | 'server-error' | 'timeout' | 'auth', retryAfter?: number}` exported from `sports-service.ts`. `fetchJSON` catches BDL response statuses + network errors + AbortController timeout and throws the typed error. Per-league fetchers + `fetchAllLeagues` + `fetchTeams` propagate the throw. Routes catch and map to HTTP (502 + body + Retry-After). Brief-assembly catches and maps to per-league placeholder + PostHog event. Single classifier, multiple consumers.
- **D-11:** Fold WR-01 (`encodeURIComponent` on `teamId`) into this phase — 4 BDL URL interpolation sites in `sports-service.ts` (lines 331-332, 410-411, 489-490, 568-569). ~4-line defense-in-depth fix that fits naturally next to the route try/catch work. WR-02 (`mapSports teamName` from env-var) deferred — touches renderer + needs Railway env-var deletion runbook coordination, belongs in its own phase.
- **D-12:** Test contract — three layers:
  1. **Server unit** (~4 tests in `sports.test.ts`): route returns 502 + correct body + `retryAfter` field present-or-absent + no `BALLDONTLIE` substring in body, one test per `UpstreamError` kind.
  2. **PWA component** (~4 tests in `SettingsPage.test.tsx`): `loadTeamsForLeagueImpl` renders distinct copy per error class (auth → vigilFetch redirect, upstream → "Sports data temporarily unavailable" + countdown, server-other → generic, network → "No network connection").
  3. **Brief-assembly integration** (~2 tests in `brief-assembly-service.test.ts`): per-league failure renders placeholder line in `data.sports`; all-leagues-failed renders the all-failed block.
  ~10 new tests total. Existing `SPORTS-01-*` tests must continue to pass (regression coverage).

### PWA error-class taxonomy + extraction
- **D-13:** Extract `classifyFetchError(input: Response | Error): ErrorClass` helper in `vigil-pwa/src/api/client.ts` next to `vigilFetch`. Returns `{kind: 'auth' | 'upstream' | 'server' | 'network', retryAfter?: number}`. Used by all sports helpers (`getSportsTeams`, `getSportsSelections`, `setSportsSelections`) AND any future PWA helpers that need error-class differentiation. Single source of truth in PWA. Tested in isolation in `client.test.ts`.
- **D-14:** Four error buckets with distinct user-facing copy:
  1. **`auth` (401/403):** Handled by vigilFetch's existing `redirectToGoogleAuth`/sign-out flow. NO new picker copy — PWA never reaches the picker error block for this case.
  2. **`upstream` (502 with optional retryAfter):** "Sports data temporarily unavailable." Render the live countdown from D-15 if `retryAfter` present.
  3. **`server` (500, 503 without retryAfter, 504, any other 5xx):** "Sports settings unavailable. Try again." Generic server error.
  4. **`network` (fetch throws — no Response object, e.g. DNS fail / offline):** "No network connection."
  All four render in the same red-block per-league error UI established in Phase 116 D-22 (`bg-red-900/20 border border-red-900/40 rounded p-3` at `SettingsPage.tsx:967-976`) with the existing Retry button.
- **D-15:** Retry-After UI surfacing — live countdown next to Retry button: "Sports data temporarily unavailable. Try again in 0:23". Ticks down per second via `setInterval` cleared on unmount. Retry button disabled (opacity-50) until countdown hits 0. After 0, button re-enabled and countdown text disappears. Consistent with the Retry-After countdown pattern that will land in Phase 117 AUTH-12 (forward-looking consistency).
- **D-16:** Inline-block only on GET failure — NO toast. Matches Phase 115 D-21 pattern (toast on PUT failure for save errors, inline on GET failure for read errors). Per-league error already has the inline block as its surface; toast on top would compound (4 leagues failing = 4 stacked toasts). Save-failure toast (PUT `/v1/sports/selections` rollback path) is unaffected — that path keeps its existing toast.

### Sequencing
- **D-17:** Plan-phase produces parallelizable plans:
  1. **UpstreamError class + service-layer try/catch** in `fetchJSON` + per-league fetchers + `fetchTeams` (depends on nothing).
  2. **Route layer 502 mapping + Retry-After header propagation** for all 3 sports routes (depends on 1).
  3. **PWA `classifyFetchError` helper + 4-bucket copy + countdown UI** in `loadTeamsForLeagueImpl` (depends on contract from 1+2).
  4. **Brief-assembly per-league placeholder render + PostHog `sports_league_fetch_failed` telemetry + all-failed copy** (depends on 1).
  5. **WR-01 `encodeURIComponent` fix** in 4 BDL URL sites (independent — can ride along in plan 1 or its own micro-plan).
  Plan-phase decides exact wave assignment; plans 1, 5 in wave 1; plans 2, 4 in wave 2; plan 3 in wave 2 or 3.

<specifics>
## Specific Ideas

- The user's local-env-gap symptom (missing `BALLDONTLIE_API_KEY` → BDL 401 → opaque "Couldn't load teams.") is the canonical bug we're closing. After 116.1, the same env gap will produce: server log "BDL fetch failed: …/teams → 401" + 502 response + PWA shows "Sports data temporarily unavailable." instead of generic. Ops/dev can immediately see "ah, env-gap or upstream auth issue" from the server log.
- "Render placeholder, don't silently omit" (D-05/D-07) is the same principle as the picker fix — the user should always see WHAT'S MISSING and WHY, not get a misleadingly-clean output.
- The 4-bucket PWA taxonomy (D-14) is calibrated to USER ACTIONABILITY, not to HTTP status precision: the user can act on "no network" (check wifi), can act on "temporarily unavailable" (wait + retry), can't act on "server error" (just try later), and never reaches the picker for auth (vigilFetch redirects them).
- Live countdown (D-15) future-proofs the pattern for Phase 117 AUTH-12 — same UX in two places means one mental model for the user.
- `UpstreamError` typed class (D-10) is the pattern enabler: every future "we proxy a third-party API" gets the same error contract for free.

</specifics>
