## Decisions Made

- `upstreamErrorToResponse()` defined at module level (outside factory function) — no closure needed, avoids re-defining per factory call
- `import("hono").Context` inline in helper signature rather than a top-level named import — keeps the import minimal since Hono's `Context` type is only needed in this one helper
- `retryAfter` absent from JSON body uses conditional assignment (`if (err.retryAfter !== undefined) body.retryAfter = ...`) — ensures the key is truly absent from serialized JSON, not present-but-undefined (T-116.1-02-03)

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] fetchLeague's generic catch block was not re-throwing UpstreamError**
- **Found during:** Task 1 GREEN (Test 4: `SPORTS-01b-route-league-network-failure-returns-502` failed with 200 instead of 502)
- **Issue:** `fetchLeague` in `sports-service.ts` has a top-level catch block that catches ALL errors (including `UpstreamError`) and converts them to `{ status: "error", error: "..." }` — returning a 200 LeagueResult to the route. Plan 01 had added D-10 re-throws only to the standings-only catch blocks inside each per-league fetcher, but not to this top-level `fetchLeague` catch. The route's try/catch never saw the throw.
- **Fix:** Added `if (err instanceof UpstreamError) throw err;` before the fallback `status: "error"` result in `fetchLeague`'s catch block, matching the D-10 pattern from Plan 01.
- **Files modified:** `vigil-core/src/services/sports-service.ts`
- **Verification:** Test 4 now passes; all 31 service tests still pass (non-Upstream errors still produce `status: "error"` fallback — backward compat preserved)
- **Committed in:** `1992dd0` (same feat commit as route implementation)
