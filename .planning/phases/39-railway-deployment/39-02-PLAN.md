<task type="auto">
  <name>Task 1: Deploy vigil-core to Railway with Postgres</name>
  <files></files>
  <action>
**Pre-flight:** Install Railway CLI if not present: `brew install railway`
Authenticate: `railway login` (opens browser — auth gate if needed)

**Create project and services:**

1. `railway init` — create new project named "vigil-core" (or link to existing)
2. Set root directory for the service to `vigil-core`:
   - `railway settings --root-directory vigil-core`
   - OR configure via `railway service` settings
3. Add PostgreSQL plugin:
   - `railway add --plugin postgresql`
   - This auto-provisions a managed Postgres instance and injects DATABASE_URL into the service environment
4. Set environment variables:
   - `railway variables set ANTHROPIC_API_KEY=$ANTHROPIC_API_KEY` — user must have this in their shell env or provide it
   - DATABASE_URL is auto-injected by Railway's Postgres plugin (do NOT set manually)
   - PORT is auto-injected by Railway (do NOT set manually)
5. Deploy:
   - `railway up` from the repo root (Railway uses the root-directory setting to find vigil-core/)
   - OR `cd vigil-core && railway up` if root-directory is already configured
6. Monitor deployment:
   - `railway logs` to watch build + migration output
   - Dockerfile CMD runs `node dist/db/migrate.js && node dist/index.js`
   - Migrations should complete, then server starts

**If Railway CLI auth gate:** Stop, present `railway login` instructions, wait for user to authenticate, then retry.

**If ANTHROPIC_API_KEY not in environment:** Stop, ask user to provide it, then set via `railway variables set ANTHROPIC_API_KEY=<key>`.

**Important:**
- Do NOT set DATABASE_URL manually — Railway's Postgres plugin injects it automatically
- Do NOT set PORT — Railway injects it automatically
- The Dockerfile already handles the build (tsc) and start (migrate + node)
  </action>
  <verify>
    - `railway status` shows deployed service
    - `railway logs` shows "[migrate] Migrations complete" and "Vigil Core API running on port ..."
  </verify>
  <done>vigil-core deployed to Railway, Postgres addon provisioned, migrations ran successfully, server is running</done>
</task>

<task type="auto">
  <name>Task 2: Generate API key and verify production health</name>
  <files></files>
  <action>
**Generate first API key for production use:**

Use `railway run` to execute commands in the production environment (with DATABASE_URL injected):

1. Generate API key: `railway run npx tsx scripts/generate-key.ts -- --name "g2-glasses"`
   - This connects to the production database and creates the first API key
   - SAVE the printed key — it's shown once and stored as a SHA-256 hash
   - The key will be needed for G2 plugin and Mac app configuration (Phases 41-42)

**Verify health endpoint:**

2. Get the deployment URL: `railway domain` or check `railway status` for the public URL
   - If no public domain assigned: `railway domain` to generate one
3. Test health (no auth required):
   ```bash
   curl https://<railway-domain>/v1/health
   ```
   Expected: `{"status":"ok","timestamp":"...","version":"0.1.0","database":"connected"}`

4. Test authenticated endpoint with the generated API key:
   ```bash
   curl -H "Authorization: Bearer <api-key>" https://<railway-domain>/v1/thoughts
   ```
   Expected: 200 response with thoughts array (empty initially)

5. Test auth rejection (no key):
   ```bash
   curl https://<railway-domain>/v1/thoughts
   ```
   Expected: 401 Unauthorized

**Record the production URL and API key** — note them in the summary for subsequent phases (41: G2 plugin, 42: Mac app migration).
  </action>
  <verify>
    - Health endpoint returns {"status":"ok","database":"connected"}
    - Authenticated request to /v1/thoughts returns 200
    - Unauthenticated request to /v1/thoughts returns 401
  </verify>
  <done>Production API key generated, health endpoint verified with database connected, auth working (200 with key, 401 without)</done>
</task>
