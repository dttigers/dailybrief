## Executive Summary

- **install.sh does NOT call build.sh** — it calls `swift build -c release` inline. build.sh is a standalone convenience script for building only. This means bootstrap.sh calling install.sh also triggers a full swift build, which is correct but slow (~60–120s on a cold machine). [VERIFIED: codebase read]
- **install.sh DOES touch the Monitor plist** — it regenerates `~/Library/LaunchAgents/com.jamesonmorrill.dailybriefmonitor.plist` from a heredoc every run. It does NOT touch `~/.config/dailybrief/config.json` at all. This means the 1P restore in step D-08.3 is safe to run before OR after install.sh — no ordering conflict. [VERIFIED: codebase read]
- **VIGIL bearer is stored ONLY in `~/.config/dailybrief/config.json` at `api_key`** — no UserDefaults, no Keychain, no separate file. SettingsViewModel reads it from config.json at load and writes it back on save. The doctor reads it with the same python3 pattern as sync-anthropic-key.sh. [VERIFIED: AppConfig.swift + ConfigLoader.swift]
- **Health endpoint is `GET /v1/health` with NO auth required — but ALWAYS returns HTTP 200 regardless of database state.** When DB is unavailable, the JSON body has `"status": "degraded"`. Bootstrap MUST parse the JSON body and check `status == "ok"`, not just check for a 200. [VERIFIED: live curl + health.ts read]
- **PORT is env-var driven (default 3001), injected via the plist `EnvironmentVariables:PORT`** — port 3001 is set in the plist, so it's stable for the health check URL. [VERIFIED: plist read]
- **`DATABASE_URL` is NOT in the plist, NOT in `~/.config/dailybrief/.env`, and NOT in the shell env.** Vigil-core currently runs locally with DB unavailable (health returns `degraded`). On a fresh machine, bootstrap.sh needs to either (a) add DATABASE_URL to the plist or to `~/.config/dailybrief/.env`, or (b) document that DATABASE_URL must be set manually — this is an **open question for the planner**. [VERIFIED: live env check + plist read]
- **`op` CLI is NOT installed on this machine** — the bootstrap pre-flight check is essential, not optional. The Railway CLI IS installed (v4.36.1). `gh` is NOT installed. [VERIFIED: command -v checks]

## Q1: install.sh contract

[VERIFIED: direct read of scripts/install.sh]

### File modification table

| File / Path | What install.sh does | Idempotent? |
|---|---|---|
| `$HOME/.local/bin/DailyBrief` | Copies from `.build/release/DailyBrief` (cp -f, overwrites always) | Yes — always overwrites |
| `$HOME/.local/bin/DailyBriefMonitor` | Copies from `.build/release/DailyBriefMonitor` (cp -f, overwrites always) | Yes — always overwrites |
| `$HOME/Library/LaunchAgents/com.jamesonmorrill.dailybriefmonitor.plist` | Regenerated from heredoc on every run (cat >) | Yes — always regenerates |
| `$HOME/Library/Logs/DailyBrief/` | mkdir -p | Yes |
| `$HOME/.local/bin/` | mkdir -p | Yes |
| `/usr/local/bin/dailybrief` symlink | Removed if present (stale cleanup) | Yes — one-time cleanup |
| `$HOME/Library/LaunchAgents/com.jamesonmorrill.dailybrief.plist` | Removed if present (old CLI LaunchAgent cleanup) | Yes — one-time cleanup |

**install.sh does NOT touch:**
- `~/.config/dailybrief/config.json` — never read or written
- `~/.config/dailybrief/.env` — not touched
- `~/Library/LaunchAgents/com.jamesonmorrill.vigilcore.plist` — not touched
- Any Keychain entries
- Any symlinks other than the `/usr/local/bin/dailybrief` cleanup

### LaunchAgent operations

- **Boots out** `com.jamesonmorrill.dailybriefmonitor` (ignores error if not loaded): `launchctl bootout "gui/$(id -u)/com.jamesonmorrill.dailybriefmonitor" 2>/dev/null || true`
- **Bootstraps** (loads) the freshly regenerated Monitor plist: `launchctl bootstrap "gui/$(id -u)" "$MONITOR_PLIST"`
- **Also boots out** `com.jamesonmorrill.dailybrief` (old CLI LaunchAgent) if the plist file exists — one-time migration
- **Does NOT touch** `com.jamesonmorrill.vigilcore` at all

### Build behavior

- Calls `swift build -c release` inline (NOT via build.sh). build.sh is a standalone script that also builds+codesigns the DailyBrief CLI binary only; install.sh builds both targets inline.
- No check for pre-existing build output — always rebuilds.
- Cold build time: approximately 60–120 seconds (first run, depends on dependency caching).
- On re-run with no source changes, swift's incremental build should skip most work — still calls `swift build -c release` but exits quickly.

### Exit codes and error handling

- Uses `set -euo pipefail` — any failing command exits non-zero immediately.
- The `launchctl bootout` and `launchctl print` calls use `|| true` / `2>/dev/null` to suppress errors on not-loaded states.
- No explicit `exit N` statements — failure propagates through `set -e`.

### Idempotency verdict

**Safe to run multiple times.** Every file write uses `-f` (force/overwrite). The bootout+bootstrap sequence is already wrapped with `|| true`. The only footgun is that it always rebuilds from source, so re-running is slow but correct.

### Ordering constraint for bootstrap.sh

`install.sh` never reads `~/.config/dailybrief/config.json`. The 1P secret restore (D-08.3) and install.sh (D-08.8) have **no ordering dependency** — running them in either order is safe. D-08 ordering is correct for other reasons (vigil-core plist needs to exist before `launchctl load`).

**Important caveat from install.sh comments:** install.sh must NOT be called from inside the running DailyBriefMonitor process (doing so would `bootout` the calling process mid-execution). This is fine for bootstrap.sh which runs in a terminal session.

## Q3: Health endpoint + log paths

[VERIFIED: vigil-core/src/index.ts + vigil-core/src/routes/health.ts + plist read + live curl]

### Confirmed health endpoint

**URL:** `http://localhost:3001/v1/health`

- Route defined in `health.ts` as `health.get("/health", ...)`, mounted at `/v1` in `index.ts` → resolves to `/v1/health`. [VERIFIED]
- No auth required (explicitly excluded from `bearerAuth` middleware in index.ts line 60).
- Port: driven by `process.env.PORT || 3001`. The plist injects `PORT=3001` as an EnvironmentVariable. On a fresh machine after 1P restore of the plist, the port will be 3001. [VERIFIED: plist read]

### Response schema

```json
{
  "status": "ok" | "degraded",
  "timestamp": "<ISO8601>",
  "version": "0.1.0",
  "database": "connected" | "unavailable"
}
```

**CRITICAL: HTTP status is always 200** regardless of `status` field value. [VERIFIED: `curl -s -o /dev/null -w "%{http_code}" http://localhost:3001/v1/health` returned `200` when live server had `status: "degraded"`]

### Bootstrap health check requirement

The bootstrap health poller MUST:
1. Check HTTP 200 (to confirm the process is up)
2. **AND** parse the JSON body and confirm `"status": "ok"`

A bare `curl -sf http://localhost:3001/v1/health` will succeed even when the DB is down. The correct check:

```bash
HEALTH=$(curl -sf http://localhost:3001/v1/health 2>/dev/null) && \
  /usr/bin/python3 -c "import json,sys; d=json.loads(sys.argv[1]); sys.exit(0 if d.get('status')=='ok' else 1)" "$HEALTH"
```

Or using a simpler grep:
```bash
curl -sf http://localhost:3001/v1/health 2>/dev/null | grep -q '"status":"ok"'
```

### Log file paths

From the plist (exact paths, hardcoded to the user's home dir — will be correct after 1P restore):

| Log file | What it captures |
|---|---|
| `~/Library/Logs/DailyBrief/vigilcore-stdout.log` | vigil-core stdout — startup messages, health check logs |
| `~/Library/Logs/DailyBrief/vigilcore-stderr.log` | vigil-core stderr — errors, DB failures, TypeScript panics |

The bootstrap fail-loud log tail should use `vigilcore-stderr.log` for errors and `vigilcore-stdout.log` for startup context. D-05 specifies "tail the last 50 lines" — both files exist in practice. [VERIFIED: ls ~/Library/Logs/DailyBrief/]

**Note:** Log paths in the plist are absolute paths (not `~`-expanded). The plist restored from 1Password will have the original machine's username hardcoded. On a different machine with a different username, these paths will be wrong. See Open Questions.

## Q5: Idempotency patterns

[VERIFIED: direct codebase read for install.sh; ASSUMED for launchctl + npm patterns based on macOS standard behavior]

### Per-step idempotency analysis

| Step (from D-08) | Re-run failure mode | Correct pattern | Rationale |
|---|---|---|---|
| Pre-flight checks | None — read-only | Check and report, always safe | |
| `op whoami` | None — read-only | Check and report | |
| `op document get --out <path>` | Overwrites existing file — DESIRED | Overwrite-always | Vault is authoritative; stale local file should be replaced |
| `mkdir -p ~/Library/LaunchAgents` | mkdir -p is idempotent | No guard needed | |
| `npm install` | Safe re-run — skips unchanged packages | No guard needed; npm is idempotent | node_modules already present: npm checks and skips |
| `npm run build` (tsc) | Rebuilds always — slow but safe | No guard needed; tsc is deterministic | |
| `~/.config/dailybrief/.env` creation | File already exists — skip | Guard: `if [[ ! -f "$ENV_PATH" ]]` | Only seed from template if missing; sync-anthropic-key.sh handles the update |
| `scripts/sync-anthropic-key.sh` | Rewrites .env line and plist key — safe | No guard; script is already idempotent | sync-anthropic-key.sh uses replace-or-append pattern |
| `launchctl load vigil-core plist` | Already loaded → error | **Unload-then-load:** `launchctl unload ... || true; launchctl load ...` | sync-anthropic-key.sh already uses this pattern |
| `scripts/install.sh` | Always rebuilds + bootout+bootstrap Monitor | Call unconditionally — install.sh handles its own idempotency | install.sh already declared idempotent |
| Health check poll | vigil-core already running → first poll succeeds immediately | No guard needed; polling loop exits on first success | |

### Key patterns

**Overwrite-always (1P restore):** Vault is always authoritative. Never skip document restore on re-run.

**Skip-if-absent (.env seed):** Only create `.env` from template if missing. Never overwrite an existing .env — sync-anthropic-key.sh handles updates.

**Unload-reload (LaunchAgent):**
```bash
launchctl unload "$PLIST_PATH" 2>/dev/null || true
launchctl load "$PLIST_PATH"
```
This is the pattern sync-anthropic-key.sh uses and install.sh uses (with bootout/bootstrap variants). bootstrap.sh uses the same for vigil-core plist.

**Note on `launchctl load` vs `launchctl bootstrap`:** install.sh uses the newer `launchctl bootout` / `launchctl bootstrap "gui/$(id -u)"` API. sync-anthropic-key.sh uses the older `launchctl unload` / `launchctl load`. Both work on current macOS but `bootout`/`bootstrap` is the modern form. For consistency with install.sh, bootstrap.sh should use `bootout`/`bootstrap` for the vigil-core plist. [ASSUMED: MEDIUM]

## Q6: vigil-core cold start time

[VERIFIED: live health check + connection.ts read]

### Current state (this machine)

Vigil-core is already running. Live health check at `http://localhost:3001/v1/health` responds immediately. The plist sets `RunAtLoad: true` so it started at login.

### Cold start analysis

The startup sequence in `index.ts` is:
1. `testConnection()` — runs `SELECT 1` against PostgreSQL. If DB is unavailable, logs a warning and continues (does not abort startup).
2. Hono app setup — synchronous, milliseconds.
3. `serve({ fetch: app.fetch, port }, callback)` — starts the HTTP listener.

**With DB available:** Connection happens in parallel with HTTP listener startup (testConnection is called but the result is not awaited before serve). The HTTP server is likely listening within 1–3 seconds on a warm machine.

**With DB unavailable (fresh machine with no DATABASE_URL):** The health endpoint will return `{"status": "degraded"}` almost immediately (testConnection returns `false` quickly when no DB configured). HTTP listener is up within ~2 seconds.

**Cold npm start (fresh node process, no JIT warmup):** Hono + Node.js startup typically takes 1–3 seconds. [ASSUMED: MEDIUM — typical Node.js/Hono startup; not benchmarked live]

### DATABASE_URL gap (critical for planner)

The vigil-core plist does NOT contain `DATABASE_URL`. `~/.config/dailybrief/.env` does NOT contain `DATABASE_URL`. On a fresh machine after 1P restore, vigil-core will start but report `"status": "degraded"`.

**This means the health check `status == "ok"` will fail on a fresh machine.** The 30-second timeout will expire and bootstrap will tail logs showing DB unavailable.

The planner must decide: Does `DATABASE_URL` need to be added to the plist (requiring the user to have it) or is `status: "degraded"` acceptable as "healthy enough" for bootstrap verification? This is the most important open question.

### Timeout recommendation

Given 1–3 second actual startup time: **30 seconds is generous and correct**. Polling interval of 1 second × 30 iterations is appropriate. No adjustment needed from D-12. The bigger concern is the DB status field check (above), not the timeout length.

## Open Questions for Planner

### Q7 — DATABASE_URL gap (HIGH PRIORITY — blocks health check design)

**Problem:** The vigil-core plist does not contain `DATABASE_URL`. `~/.config/dailybrief/.env` does not contain it either. On this machine, vigil-core is running with `"status": "degraded"` and DB unavailable. The current health check always returns HTTP 200 regardless.

**Options for planner to choose between:**

**Option A — Accept degraded as "bootstrap success" for health check:**
Bootstrap checks that the process is running (HTTP 200) but does not check `status == "ok"`. The user verifies DB separately. Doctor adds a row for DB connectivity.

**Option B — Add DATABASE_URL to the 1P-restored plist:**
The `vigil-vigilcore-plist` document stored in 1Password should include `DATABASE_URL` as an EnvironmentVariable. On a fresh machine, bootstrap restores the plist with DATABASE_URL included, and vigil-core boots connected. This is the cleanest solution but requires the user to update their 1P vault item.

**Option C — Add DATABASE_URL to `~/.config/dailybrief/.env` and seed it from 1P config:**
Less clean — DATABASE_URL is a vigil-core concern, not a dailybrief config concern. Not recommended.

**Recommendation:** Option B is correct. The `vigil-vigilcore-plist` 1P document is designed to be the complete plist. The current plist on disk is missing `DATABASE_URL` — this is a pre-existing gap that the bootstrap phase is the right time to fix. The planner should document that the user must add `DATABASE_URL` to the plist before uploading to 1Password.

### Q8 — Plist log paths are hardcoded to current username

The plist `StandardOutPath` / `StandardErrorPath` contain `/Users/jamesonmorrill/...`. When restored on a machine with a different username, these paths will be wrong (logs will go to `/Users/jamesonmorrill/...` which won't exist).

**Options:**
- Accept this — the user only ever sets up machines as themselves (single-user tool)
- Bootstrap post-restore: use sed/PlistBuddy to substitute `$HOME` into the plist after restore

The CONTEXT.md does not address this. Planner should decide whether to add a post-restore plist path-fix step. Given this is a single-user tool, accepting the hardcoded username is probably fine.

### Q9 — `op document get` exact flag syntax

`op` is not installed on this machine. The `--out` flag for `op document get` is [ASSUMED]. Planner should verify with `op document get --help` on a machine with op installed before hardcoding in tasks.

### Q10 — vigil-core LaunchAgent: `launchctl load` vs `launchctl bootstrap`

The existing `sync-anthropic-key.sh` uses the deprecated `launchctl unload`/`launchctl load` API. install.sh uses the newer `launchctl bootout`/`launchctl bootstrap "gui/$(id -u)"`. For the vigil-core plist load in bootstrap.sh, the planner should pick one and document it. Recommendation: use `bootout`/`bootstrap` to match install.sh (the more recently written script).

### Q11 — setup.sh is superseded / obsolete

`setup.sh` calls `build.sh`, creates config from `--setup` flag (interactive `read -p`), and installs the OLD `com.jamesonmorrill.dailybrief` LaunchAgent (the one that install.sh now cleans up). It is effectively dead code — the Monitor LaunchAgent replaced it, and bootstrap.sh will be the new entry point. The planner should note this but NOT modify setup.sh (D-03 — existing scripts unchanged). A comment in bootstrap.sh's header noting "replaces setup.sh" would be sufficient.
