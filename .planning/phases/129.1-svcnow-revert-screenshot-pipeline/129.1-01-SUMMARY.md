# Phase 129.1 Plan 01: work_orders maintenance_problem + department Summary

**One-liner:** Drizzle migration 0022 adds two nullable text columns (`maintenance_problem`, `department`) to `work_orders`, and the `/v1/work-orders/sync` route's strict 10-field sanitizer whitelist is extended to 12 fields so both columns survive end-to-end through `SanitizedWorkOrder` → sanitizer destructure → `dbInsertOrGet` INSERT VALUES → `dbUpsertLegacy` ON CONFLICT DO UPDATE SET — unblocking downstream plans 129.1-03 (screenshot endpoint) and 129.1-05 (PWA manual-create form).

## What Was Built

### Task 1 — Migration 0022 + schema.ts (commit `39cae67`)

- **`vigil-core/drizzle/0022_add_work_orders_maintenance_problem_department.sql`** — Two `ALTER TABLE "work_orders" ADD COLUMN IF NOT EXISTS` statements separated by `--> statement-breakpoint`. Header documents Phase 129.1 + WO-MANUAL-03 + downstream SCAP-* references. Nullable text, no default → metadata-only Postgres ALTER (no table rewrite, no row scan).
- **`vigil-core/src/db/schema.ts`** — Added two lines to the `workOrders` `pgTable` block, immediately after `clientCaptureId`:

  ```ts
  maintenanceProblem: text("maintenance_problem"),
  department: text("department"),
  ```

- **`vigil-core/drizzle/meta/0022_snapshot.json`** + **`_journal.json`** — Drizzle-generated; record the 18-column shape of `work_orders` (was 16, now 18). Both columns recorded as `notNull: false` (nullable).
- Confirmed `npx drizzle-kit generate` reports "No schema changes, nothing to migrate" — zero drift between schema declaration and migration.

### Task 2 — Sanitizer + dbUpsertLegacy round-trip (commits `6d2e829` RED → `09eb2e8` GREEN)

**RED phase** (commit `6d2e829`): added 3 failing tests to `vigil-core/src/routes/work-orders.test.ts`:

- `WO-MANUAL-03/T1` — `clientCaptureId` path: assert sanitizer payload reaches `dbInsertOrGet` with `maintenanceProblem === "HVAC"` + `department === "Bakery"`.
- `WO-MANUAL-03/T2` — Legacy path (no `clientCaptureId`): two-call replay with different values per call; assert each `dbUpsertLegacy` invocation receives the call-specific values.
- `WO-MANUAL-03/T3` — Omitted fields land as `null` (not `undefined`); pins the coerce-or-null pattern.

All three failed with `actual: undefined` (sanitizer dropped the new keys), confirming the 10-field whitelist was the bottleneck. The 12 pre-existing tests still passed — additive only.

**GREEN phase** (commit `09eb2e8`): four atomic edits to `vigil-core/src/routes/work-orders.ts`:

1. `SanitizedWorkOrder` interface (~lines 22-37): added `maintenanceProblem: string | null;` + `department: string | null;`.
2. Sanitizer destructure (~lines 87-110): added coerce-or-null entries before `syncedAt`:

   ```ts
   maintenanceProblem: wo.maintenanceProblem != null ? String(wo.maintenanceProblem) : null,
   department: wo.department != null ? String(wo.department) : null,
   ```

3. `dbUpsertLegacy` SET clause (~lines 348-362): added `maintenanceProblem: input.maintenanceProblem,` + `department: input.department,` — the legacy path's explicit field list is the only call site where new fields don't propagate via `values(input)` inference, so this edit is load-bearing for round-trip on case_number conflict.
4. Comment count updated: "10 known fields" → "12 known fields" in both the file-header invariant block (line 15) and the sanitizer block header (~line 81). The count is the auditable invariant that future maintainers grep against.

All 15 tests pass after GREEN. `dbInsertOrGet` inherits new fields automatically via `db.insert(workOrders).values(input)` — no per-field edit needed there.

**REFACTOR phase:** skipped — implementation was already minimal and clean.

## Deviations from Plan

None of substance. Two minor mechanical adjustments:

**1. [Mechanical] Drizzle-kit auto-generation overwrote initial migration content**

- **Found during:** Task 1 (after initial Write of the migration SQL)
- **Issue:** `npx drizzle-kit generate --name add_work_orders_maintenance_problem_department` was needed to update `_journal.json` and produce `0022_snapshot.json` (otherwise schema-vs-snapshot would drift). Drizzle's auto-gen overwrote my hand-crafted SQL with its terser form lacking `IF NOT EXISTS` and the header comment.
- **Fix:** Let drizzle-kit run (to populate journal + snapshot), then rewrote the SQL file to restore the `IF NOT EXISTS` + header pattern that mirrors 0021. The snapshot reflects column shape, not SQL text — so the rewrite does not re-introduce drift.
- **Files modified:** `vigil-core/drizzle/0022_add_work_orders_maintenance_problem_department.sql` (rewritten after auto-gen).
- **Commit:** `39cae67` (single commit captures the end-state).

**2. [Mechanical] `drizzle-kit generate --dry` flag unavailable in installed version**

- **Found during:** Task 1 verify
- **Issue:** Plan acceptance criteria reference `npx drizzle-kit generate --dry` but drizzle-kit 0.31.10 (the version locked in `package.json`) does not recognize `--dry`. The flag exists only in newer drizzle-kit releases.
- **Fix:** Used the equivalent signal — running `drizzle-kit generate` after the schema/migration edit returns "No schema changes, nothing to migrate" when there is zero drift (and would produce a new migration file otherwise). This is the documented drizzle-kit behavior when schema and applied migrations are in sync.
- **Files modified:** None (verification approach only).
- **Commit:** N/A.

Neither falls under Rules 1–4 (no bug, no missing functionality, no blocker, no architectural change).
