# Phase 129.1 Plan 05: PWA manual-create + draft-review + vigil-core commit/delete routes Summary

**One-liner:** Wires the PWA work-orders manual-create CTA + CreateWorkOrderModal + ReviewWorkOrderModal + 3 new useWorkOrders mutations + 2 new userId-scoped vigil-core routes (POST /v1/work-orders/:caseNumber/commit transitions pending_review→open with operator edits; DELETE /v1/work-orders/:caseNumber hard-deletes drafts), labels the `department` field as "Maintenance Location" in both modals per the Phase 129.1-03 operator decision, and ships 23 new tests across 3 files (7 vigil-core + 16 vigil-pwa) all green plus a clean vigil-core+vigil-pwa build.

## What Was Built

### Task 1 — POST /:caseNumber/commit + DELETE /:caseNumber + tests (commits `a2f9acd` RED → `7e8e0ae` GREEN)

**RED phase** (commit `a2f9acd`): added 7 failing tests to `vigil-core/src/routes/work-orders.test.ts`:

- `WO-MANUAL-05/T1` — commit happy path: pending_review → open with operator edits applied; UPDATE SET includes state=open + lastChangeAt + lastChangeSummary="Committed from review".
- `WO-MANUAL-05/T2` — commit 404 when no row matches (caseNumber, userId).
- `WO-MANUAL-05/T3` — commit 409 when row state !== "pending_review"; no UPDATE happens.
- `WO-MANUAL-05/T4` — commit edits to `maintenanceProblem` + `department` propagate to UPDATE SET.
- `WO-MANUAL-05/T5` — delete 204 happy path with empty body.
- `WO-MANUAL-05/T6` — delete 404 when no row matches.
- `WO-MANUAL-05/T7` — delete is hard-delete: mock captures DELETE on workOrders (NOT a soft-update); row removed from store.

Added a `MockDb` builder supporting drizzle's `select().from().where()`, `update().set().where().returning()`, `delete().where()` chains with an in-memory row store. The mock dispatches on the captured `from(table)` reference and exposes `updateCalls`/`deleteCalls` arrays for assertion. 6 of 7 tests failed at RED; T6 trivially passed via Hono's default 404 (still satisfies the explicit-404 contract once the route is wired).

**GREEN phase** (commit `7e8e0ae`): two route handlers added inside `createWorkOrdersRoute(deps)`:

1. **POST /work-orders/:caseNumber/commit** — userId from `c.get("userId")` (T-129.1-23). Body parsed via `c.req.json()` with empty-body fallback. SELECT-by-(caseNumber, userId); 0 rows → 404; existing row.state !== "pending_review" → 409. Builds SET object: `state: "open" + lastChangeAt + lastChangeSummary` plus 8 required-string fields (coerce via `String()` when `edits[f] != null`) + 2 nullable fields (`maintenanceProblem`/`department` via coerce-or-null when `f in edits`). UPDATE with `.returning()`, returns 200 with the row.

2. **DELETE /work-orders/:caseNumber** — userId from middleware. Explicit `caseNumber === "archived"` guard returns 404 (defense-in-depth; trie-prefer-literal should win the /archived path). SELECT then 404. Hard-delete: workOrderStatuses cleanup first (FK has ON DELETE RESTRICT), then workOrders delete. Returns 204 with empty body. Registered AFTER DELETE /work-orders/archived so the literal path wins.

22 of 22 work-orders.test.ts tests pass; `npm run build` exits 0.

### Task 2 — CreateWorkOrderModal + useWorkOrders.createWorkOrder + WorkOrdersPage CTA (commit `7ea8d6f`)

**CreateWorkOrderModal.tsx** (354 lines, new file): full overlay modal (`fixed inset-0 z-50 bg-black/50` + inner `bg-gray-900 max-w-lg w-full max-h-[90vh] overflow-auto`). Closes on Escape (window keydown listener), overlay outside-click, and Cancel button. 11 editable fields:

- `case_number` — text input, required, regex `/^[A-Z0-9]+$/` form-level validation, inline error "Case number must contain only A-Z and 0-9", disables Submit until valid.
- `store`, `trade`, `location`, `equipment`, `contact`, `maintenance_problem` — text inputs.
- `short_description`, `notes` — textareas.
- `priority` — select with `(none) / Low / Medium / High / Critical`.
- `department` — text input labeled **"Maintenance Location"** per the Phase 129.1-03 operator decision Q1 (Polaris field mapping). The underlying payload field name stays `department` for schema alignment.

Submit calls `onCreate(input)` then `onClose`; rejections surface inline (modal stays open so operator can retry).

**useWorkOrders.ts** additions (all 3 useCallbacks, return-object extended):

- `createWorkOrder(input)` — generates `clientCaptureId` via `crypto.randomUUID()` with non-crypto fallback (`cc-${Date.now()}-${random}`). POSTs to `/v1/work-orders/sync` with `state: "open"` + the clientCaptureId. Throws on non-2xx. Bumps `setFetchTick`.
- `commitDraft(caseNumber, edits)` — POSTs to `/v1/work-orders/${encodeURIComponent(caseNumber)}/commit`. Throws on non-2xx. Bumps fetch tick.
- `discardDraft(caseNumber)` — DELETEs `/v1/work-orders/${encodeURIComponent(caseNumber)}`. Throws on non-2xx. Bumps fetch tick.

`ManualCreateInput` type exported from the hook so both modals can share it without circular deps.

### Task 3 — ReviewWorkOrderModal + WorkOrderRow pending_review branch + page hosting (commit `02e3769`)

- `maintenanceProblem` + `department` read from the workOrder prop via index-signature cast (`woAny.maintenanceProblem`) — these fields aren't on `WorkOrderApiResponse` today; plan 129.1-03 will widen the response shape. Forward-compat seam: absent fields render as empty strings.

## Threat Model Mitigations

All 7 STRIDE entries from the plan's threat register addressed in implementation:

| Threat ID | Mitigation |
|-----------|------------|
| T-129.1-23 (Spoofing — userId injection via path param) | Both commit + delete handlers read userId via `c.get("userId")` and scope SELECT/UPDATE/DELETE by `(caseNumber, userId)`. `caseNumber` is path param (the WHAT); userId is from bearerAuth (the WHO). |
| T-129.1-24 (Tampering — mass-assignment via commit body) | Commit handler builds an explicit SET object with the 12-field editable whitelist (8 required-string via `String()` coerce; 2 nullable via coerce-or-null); never spreads `...edits`. Mirrors the sync sanitizer's "12 known fields" invariant from plan 01. |
| T-129.1-25 (Tampering — state transition bypass) | Commit handler enforces `row.state === "pending_review"` OR returns 409 `{error: "Work order is not a pending_review draft"}`. Prevents reopening committed/closed work orders via the commit endpoint. |
| T-129.1-26 (Info Disclosure — extras JSON in review modal) | Accepted in plan — same data already persisted to `notes` by plan 03; rendered to the row owner only via userId-scoped fetch. |
| T-129.1-27 (Repudiation — hard-delete via Discard) | Mitigate-via-UX: `window.confirm("Discard this draft? This cannot be undone.")` forces operator acknowledgement. No soft-delete because drafts are by-definition pre-committed. |
| T-129.1-28 (Tampering — case_number format bypass) | Form-level regex `/^[A-Z0-9]+$/` enforced before POST; live validation disables Submit; server has no regex check (acceptable because server still scopes by userId and sanitizer coerces). |
| T-129.1-SC (Supply chain — new npm deps) | n/a — zero new npm deps in this plan. PWA + vigil-core both used existing infrastructure (@testing-library/react, vitest, hono, drizzle). |

No new threat surface flags beyond the plan's `<threat_model>`.
