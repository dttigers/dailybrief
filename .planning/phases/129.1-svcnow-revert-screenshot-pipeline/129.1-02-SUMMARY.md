## Deviations from Plan

### [Rule 1 - Bug] Chrome popup.{html,css,js} baseline corrected to 9762ae3 (not 1788d97)

- **Found during:** Task 1 (baseline discovery)
- **Issue:** The plan names `1788d97` (Phase 84) as the Chrome baseline. But Chrome popup.{html,js} was modified in commit `27ef2b7` (Phase 94 — URL toggle / Cmd+Enter / triage poll) and in commit `9762ae3` (Phase 114-00 — lockstep header). Resetting all three popup files to 1788d97 would lose the Phase 94 capture-UX features that Safari was ported from in Phase 114-02/03, breaking the byte-equality requirement.
- **Fix:** Used `9762ae3` for Chrome popup.{html,css,js}. Used `1788d97` (Phase 84) for Chrome manifest.json (manifest wasn't touched in Phase 94, so 1788d97 and 5507ba2 are byte-identical for manifest).
- **Files modified:** `vigil-extension/popup.{html,css,js}`
- **Commit:** `1a16bbf`

### [Rule 1 - Bug] Per-file Safari baselines vs the plan's single `f201ed9`

- **Found during:** Task 1
- **Issue:** Phase 114 split the Safari port across three commits: `5ec89b3` (popup.html, 114-02), `cda0f5f` (popup.css, 114-02), `f201ed9` (popup.js, 114-03). Plan named only `f201ed9`. Using `f201ed9` for popup.html or popup.css would reach a state where those files don't yet have the Phase 114 lockstep header.
- **Fix:** Resolved each Safari file's baseline empirically from `git log --oneline -- <safari-file>` and chose the commit immediately before `ab99acd`. Manifest baseline is `75f3093` (Phase 84) since Phase 114 didn't touch the Safari manifest.
- **Files modified:** `vigil-safari-extension/Vigil Capture Extension/Resources/{manifest,popup.html,popup.css,popup.js}`
- **Commit:** `1a16bbf`

### [Rule 1 - Bug] Normalize Phase 114 D-02 lockstep header to path-neutral form

- **Found during:** Task 1 (post-reset diff)
- **Issue:** The Phase 114 D-02 lockstep header comment uses asymmetric relative paths — Chrome's header says `../vigil-safari-extension/.../popup.html` and Safari's says `../../../vigil-extension/popup.html`. Direct reset produces strings that differ by exactly the relative-path expression, making strict `diff -q` byte-equality structurally impossible.
- **Fix:** Replaced both sides with a single path-neutral form: `Keep in lockstep with sibling extension popup.{ext} — Phase 114 (D-02), Phase 129.1 revert`. Preserves D-02's lock-step intent while satisfying the byte-equality acceptance criterion.
- **Files modified:** `vigil-extension/popup.{html,css,js}` + `vigil-safari-extension/.../popup.{html,css,js}` (one line each, header only)
- **Commit:** `1a16bbf`

### [Rule 2 - Missing tooling] Add tsx to root devDependencies

- **Found during:** Task 2 verification
- **Issue:** `npm run test:extension` fails with `sh: 1: tsx: not found` because tsx is not in root devDependencies and there's no root `node_modules/.bin/tsx`. This is a pre-existing condition — the original `test:extension` script in the prior commit (Phase 129-07 etc) had the same problem. It only worked when run via `npm exec tsx` or with tsx globally installed.
- **Fix:** Added `tsx@^4.22.1` to root devDependencies. `npm install` ran cleanly. `npm run test:extension` now resolves tsx via `node_modules/.bin/tsx`. `package-lock.json` updated.
- **Files modified:** `package.json`, `package-lock.json`
- **Commit:** `0bc471a`

### [Documented conflict] Acceptance criterion overbreadth — absence-test references

- **Found during:** Task 2 verification
- **Issue:** Plan acceptance_criteria line 243 demands `! grep -qE 'popup-helpers|background\.service_worker|content_scripts|background\.js|content-script\.js' vigil-extension-tests/parity.test.ts`. But the same plan's `<behavior>` block demands an absence-test that asserts those three filenames do NOT exist on either side. An absence-test must name the files it's asserting absence of.
- **Resolution:** Followed the `<behavior>` block (which is the load-bearing intent). The grep filter is documented as an over-broad heuristic — its true purpose is to detect re-introduction of file LOADS (`readFileSync(.../background.js)`, etc.), which the rewritten test correctly avoids. The new parity.test.ts contains zero `readFileSync` calls for the SVCNOW-only files; only string literals inside the absence-test mention them.
- **No action needed.** Documented for traceability.
