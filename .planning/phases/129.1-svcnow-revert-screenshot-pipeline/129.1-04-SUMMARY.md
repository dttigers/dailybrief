# Phase 129.1 Plan 04: macOS LaunchAgent screenshot daemon Summary

**One-liner:** New self-contained npm package `scripts/vigil-capture/` ships a chokidar-based TypeScript daemon (`watcher.ts`) that watches `~/vigil-captures/` with `awaitWriteFinish`, base64-encodes new PNG/JPG screenshots, POSTs to vigil-core's `/v1/captures/screenshot` endpoint with a Bearer key sourced from macOS Keychain (not env, not plist), and atomically moves the file to `processed/<ts>-<workOrderId>.<ext>` on success or `failed/<ts>-<filename>` with a `.err` sidecar on failure; an idempotent `vigil-capture install|uninstall|status|tail` CLI wraps `launchctl bootstrap gui/$UID` LaunchAgent management; the LaunchAgent plist (modeled verbatim on `com.jamesonmorrill.vigilcore.plist`) keeps the daemon alive via `KeepAlive` with `LimitLoadToSessionType: Aqua` and `ProcessType: Background`; 7 node:test cases via TDD (RED→GREEN proven) cover shouldSkip filter behavior, processFile happy path, HTTP failure, network error, mediaType derivation, and UUID v4 clientCaptureId emission. Tasks 1, 3, 4 committed; Task 2 pre-approved by orchestrator; Task 5 is a blocking operator checkpoint (Mac hardware required) — executor returns structured checkpoint state to orchestrator.

| # | Case | What it pins |
|---|------|--------------|

| 4 | HTTP 422 | file moved to `failed/`; `.err` sidecar contains `HTTP 422` + response body |

Mock seams: per-test `mkdtempSync` capture dir + `mockResponse({ok, status, json, text})` + `fetchFn` closure that captures request bodies for assertions. No process / env state mutated.

**`scripts/vigil-capture/src/watcher.ts` (310 LOC):**

- **Exports:** `processFile`, `getApiKey`, `shouldSkip`, `SKIP_PATTERNS`, `scanOrphans`, types `ProcessFileDeps`.
- **`processFile(filePath, deps?)`:**
  1. `shouldSkip(filename)` → return early on dotfile / `.tmp.` / `.download` / `.icloud`.
  2. Extension allowlist: `.png` → `image/png`, `.jpg`/`.jpeg` → `image/jpeg`, else return.
  3. `readFile` → base64; `randomUUID()` → `clientCaptureId` (UUID v4).
  4. `getApiKeyFn()` from Keychain; POST `{imageBase64, mediaType, clientCaptureId}` to `${apiBase}/v1/captures/screenshot` with `Authorization: Bearer ${apiKey}` + `Content-Type: application/json`.
  5. 2xx → `mkdir(processedDir, {recursive})` + `rename` to `processed/<ts>-<workOrderId><ext>`; log `[OK]`.
  6. non-2xx → `mkdir(failedDir)` + `rename` + `writeFile` sidecar `HTTP <status>\n<body>\n`; log `[FAIL]`.
  7. throw (network / DNS) → same failed-path move with sidecar `NETWORK ERROR\n<msg>\n`; log `[ERR]`.
  8. 2xx-but-unparseable-JSON → treated as failure (response shape is contract).
  9. Catch-all outer try (fs / Keychain throw) → best-effort move to failed/; if move itself fails, log + leave file.
- **`scanOrphans(captureDir)`:** one-shot `readdir` + per-file `stat` + extension filter + sequential `await processFile`. Runs once at startup before chokidar takes over.
- **`getApiKey()`:** `execSync("security find-generic-password -s vigil-api-key -w")` per-call (no caching) — key rotation takes effect on next event. T-129.1-16 invariant.
- **Main entry guard:** `import.meta.url === \`file://${process.argv[1]}\`` plus suffix-match fallback for tsx-wrapped argv; ensures import-safety for tests. Chokidar watcher uses `awaitWriteFinish: {stabilityThreshold: 500, pollInterval: 100}`, `ignoreInitial: true`, `depth: 0`, dotfile ignore regex; `.on("add", processFile)` sequentially.

## Deviations from Plan

**None of substance — plan executed as written.** A few minor refinements worth noting:

1. **chokidar pinned to `^3.6.0`** as the plan specified. `npm view chokidar version` returned 5.0.0 (the new ESM-only major release from 2025), but the plan author explicitly chose the 3.x line and RESEARCH §663-669 assumes the 3.x API surface. Installed version 3.6.0 with `paulmillr` as a maintainer confirms canonical lineage. No deviation — just confirming the planner's choice held under live execute-time inspection.

2. **Main-entry guard uses both `import.meta.url` comparison AND argv[1] suffix detection.** Pure `import.meta.url === \`file://${process.argv[1]}\`` works when running compiled JS directly (`node dist/watcher.js`), but tsx-wrapped invocation (`tsx src/watcher.ts` or `tsx --test src/watcher.test.ts`) sets `process.argv[1]` to the tsx wrapper or a synthesized path — would falsely fire the main block in tests. Suffix-detection (`argv[1].endsWith("watcher.ts" | "watcher.js")`) handles both cases without adding a separate `bin/main.ts` entrypoint. Tests confirm import-safety: the 7-test run completes without the chokidar watcher ever starting.

3. **Catch-all in `processFile`** has a nested try around the move-to-failed/ step so that a failure during the recovery path (e.g. `~/vigil-captures/failed/` not writable) logs `[ERR]` and leaves the file in place rather than throwing out of `processFile`. This protects the chokidar event handler from crashing the whole daemon on a single bad file; KeepAlive would restart it, but a quiet log message + file-left-in-place is friendlier to the operator's mental model.

No Rule 1 (bug), Rule 2 (missing critical), Rule 3 (blocker), or Rule 4 (architectural) deviations were applied. Plan's behavior block and acceptance criteria mapped 1:1 onto the implementation.

## Decisions Made

1. **Pre-approved chokidar legitimacy (Task 2).** Orchestrator authorized upfront; live execute-time `npm view` reconfirmed `paulmillr` maintainer at 3.6.0. Pause skipped; recorded here for the audit trail.
2. **`^3.6.0` pin (not `5.x`).** Matches RESEARCH-assumed API surface; 5.x is ESM-only with breaking changes. Future plan can migrate when convenient.
3. **`/usr/local/bin/node` in plist, not `tsx`.** Production daemons run pre-compiled JS — no transpile-at-startup latency, no tsx runtime hop, no dependency on tsx being on PATH for launchd.
4. **Plist uses hard-coded `/Users/jamesonmorrill/...` paths** (mirroring `com.jamesonmorrill.vigilcore.plist`). Install CLI warns if the running operator's `homedir()` differs and points to the file to edit. Keeps the plist verbatim-auditable in source control.
5. **Sequential processing (not concurrent).** Single-operator volume ≤50/day per RESEARCH §691; sequential gives natural backpressure and predictable log ordering. No Promise queue needed.
6. **Orphan-scan-then-watch ordering.** Run `scanOrphans(CAPTURE_DIR)` BEFORE `chokidar.watch(...).on("add", processFile)` so crash leftovers process before live events. `ignoreInitial: true` on the chokidar config ensures no double-fire.
