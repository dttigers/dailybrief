# Phase 129.1 — SVCNOW Revert + Screenshot Pipeline + PWA Manual-Create

**What this phase delivers:**
1. Reverts the Chrome + Safari extensions from the Phase 129 ServiceNow assisted-capture popup back to the Phase 84 generic thought-capture popup. The `*.service-now.com/*` host-restriction, MV3 service worker, content-script drift detector, popup-helpers, and svcnow-view UI all go away.
2. Builds an operator-specific screenshot pipeline: launchd watcher on `~/vigil-captures/` → uploads Polaris screenshot to vigil-core → Anthropic vision extraction → draft work-order in `pending_review` state for operator review in the PWA.
3. Builds a PWA "Create work order" UI usable by all users: a form exposing every column on `work_orders` (including the 2 new columns added by this phase). Used by non-operator users for ad-hoc entries; operator uses it for non-Polaris entries.

**What it preserves from Phase 129:**
- Production migration 0021 (`client_capture_id` column + partial unique index) — used by every capture path for dedup.
- `vigil-core/src/routes/work-orders.ts` (`dbInsertOrGet` + `dbUpsertLegacy` split) — the write path.
- All G2-side work (lifecycle restore, DOUBLE_CLICK gesture, build-gate convention, terminology cleanup).

**What it intentionally does NOT do:**
- Multi-tenant generalized screenshot ingestion (operator-specific by design — see decision in `<decisions>`).
- Re-implement drift detection. The screenshot path is single-shot; drift is not a concern.
- Touch the G2 plugin, vigil-watch, or any non-extension/PWA/vigil-core code.

<decisions>

### Schema scope (DECISION-A)

**Add 2 new columns to `work_orders`:**

1. **`maintenance_problem` text** — From Polaris "Maintenance Problem" field (e.g., "Other"). Categorizes the issue type. Useful for filtering / dashboards / future reporting.

2. **`department` text** — A new field whose Polaris mapping needs **operator clarification at plan-phase time**. Three plausible mappings:
   - **Assignment Group** (e.g., "MT-Maintenance") — the ServiceNow routing team.
   - **Maintenance Location** (e.g., "Bakery") — the store area where the issue is.
   - **Service** (e.g., "MT-Trades") — the ServiceNow service category.

   Planner asks the operator which Polaris field `department` maps to before writing the migration.

**Rationale for scope:** Operator's stated requirement is "all i really need is CS#, description, location, and department" + `maintenance_problem`. CS#, description, location already exist as `case_number`, `short_description`, `location`. So only 2 net-new columns. Other Polaris fields (Service, Assignment Group, Assigned to, Opened, Time Worked, Resolution Code/Notes, After Hours, Maintenance Vendor) are intentionally NOT added — extraction may still pull them for operator review, but the schema doesn't store them. If specific fields prove load-bearing later, add in a follow-up phase.

### Preview-before-commit feedback loop (DECISION-C)

**Same `work_orders` table with new `state: 'pending_review'` value.**

- Screenshot uploads to `POST /v1/captures/screenshot` (new endpoint, modeled on `describe-image.ts`).
- vigil-core calls Anthropic vision with the screenshot + extraction prompt (extracts CS#, description, location, department, maintenance_problem + any others Claude pulls for operator review).
- Server writes a row to `work_orders` with `state: 'pending_review'` (using `dbInsertOrGet` so duplicate uploads dedupe).
- Returns the created row id + extracted fields to the macOS watcher.
- Operator opens PWA → sees a notification badge on the work-orders list for `pending_review` rows → opens the draft → edits any mis-extracted fields → clicks **Commit** (changes state to `open`) or **Discard** (deletes the row).
- Manual-create UI also writes with `state: 'open'` directly (no review step for manually-entered rows).

**Schema note:** `work_orders.state` is already a free-text column (per the schema dump in 129-08-DEPLOY-LOG.md). Adding `pending_review` as a new value requires no migration — just code-level handling in the PWA list view and a state-transition endpoint.

### Manual-create UI access (DECISION-D)

**Visible to everyone. Form exposes every column on `work_orders` including the 2 new columns from DECISION-A.**

- Button placement: `vigil-pwa/src/pages/WorkOrdersPage.tsx` adds a "Create work order" CTA near the existing list header.
- Modal form fields: case_number, store, short_description, trade, location, equipment, priority, contact, state, **department**, **maintenance_problem**. Plus a `notes` textarea (already exists as `notes` column per schema dump).
- POSTs to existing `/v1/work-orders/sync` with `state: 'open'` (manual entries skip the review step; they're operator-typed already).
- Form-level validation: case_number must match `/^[A-Z0-9]+$/` (general identifier shape; Polaris's strict `CS\d{7}` is NOT required — supports manual entry for non-Polaris work orders).

### macOS trigger mechanism (DECISION-E)

**launchd watcher on `~/vigil-captures/`.**

- Operator screenshots Polaris page (Cmd+Shift+4 → save to ~/vigil-captures/, OR drag from Preview/Screenshots, OR any tool that drops a file there).
- launchd agent (LaunchAgents plist) watches `~/vigil-captures/` for new file events.
- On new image file: validates it's a PNG/JPG, base64-encodes, POSTs to `https://api.vigilhub.io/v1/captures/screenshot` with operator's api key, moves file to `~/vigil-captures/processed/<timestamp>-<workOrderId>.png` on success or `~/vigil-captures/failed/` with an error sidecar on failure.
- Ambient: no hotkey to remember. Pairs naturally with macOS's screenshot tool's "save to a location" preference.
- Plan also includes a `launchctl unload`/`load` runbook so the operator can pause + resume the watcher.

### Single-user scope (DECISION-F)

**Screenshot pipeline is operator-only.** The vigil-core endpoint `/v1/captures/screenshot` is publicly authenticated (any valid api key), BUT:
- The launchd watcher is installed only on the operator's Mac (no enrollment flow for other users).
- The endpoint's per-user spend tracking + extraction cost stays sane because it's single-user in practice.
- The endpoint is NOT advertised in the PWA or documentation as a feature for all users. Non-operator users discover it only via support / direct ask.

If a second operator joins later, the launchd plist + `~/vigil-captures/` convention extends trivially. No code change to support a second user — just install the agent on their Mac.
