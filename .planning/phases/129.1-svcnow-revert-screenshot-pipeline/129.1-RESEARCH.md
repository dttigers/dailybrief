### Supporting (new — for macOS daemon)

| Library | Version | Purpose | When to Use |
|---------|---------|---------|-------------|
| `chokidar` | latest stable `[ASSUMED — verify with npm view chokidar version before install]` | Cross-platform file watcher with atomic-rename detection for the macOS daemon | macOS LaunchAgent's `WatchPaths` fires on directory change but does not give per-file event details; chokidar inside the daemon gives reliable per-file ADD events with `awaitWriteFinish` to avoid mid-write reads. |
| `tsx` | `^4.19.0` `[VERIFIED: vigil-core/package.json]` | Run the TypeScript daemon directly (no compile step) | Already used throughout repo. |

**Alternatively (no new npm deps):** Pure Node.js with `fs.watch` + manual debouncing. `fs.watch` on macOS is FSEvents-backed and works for directory-level events, but does NOT detect atomic-rename completion reliably for files in progress. **Recommendation:** chokidar with `{ awaitWriteFinish: { stabilityThreshold: 500, pollInterval: 100 } }` is the safe choice — handles the macOS Cmd+Shift+4 case where the file appears as `.com.apple.NSServicesRP.*.tmp` first, then atomically renames to the final `.png`.

### Alternatives Considered

| Instead of | Could Use | Tradeoff |
|------------|-----------|----------|
| TypeScript daemon (Node) | Shell script with `fswatch` CLI | Brittle; per-event handling in bash is fragile; lose retry logic + structured error sidecar. |
| TypeScript daemon | Swift CLI | Higher implementation cost; would not reuse anything from vigil-watch (separate repo) inside this monorepo. |
| chokidar | Pure `fs.watch` | Loses `awaitWriteFinish`; risk of reading partial files during screenshot save. |
| LaunchAgent (Aqua) | LaunchDaemon (root) | Watcher needs user's `~/vigil-captures/` and per-user API key — must be Aqua-scoped. LaunchDaemon would run as root and lose access to user home. |
| Inline new endpoint POSTs `dbInsertOrGet` directly | New endpoint POSTs internally to `/v1/work-orders/sync` | Direct call is cleaner (no HTTP-loopback; no Authorization header re-issue; consistent with describe-image.ts pattern of doing work in-process). |

#### Recommended revert strategy — DO NOT use `git revert` blindly

`git revert` of these commits in reverse order would touch the same files repeatedly with conflicts (e.g. 397fcfe touches content-script.js which 0d41d1e created; reverting 397fcfe first works, but then reverting 0d41d1e is a no-op for that file). **The cleaner approach is file-level reset to Phase 84 baseline, then a single revert commit:**

```bash
# Restore the 6 SVCNOW-affected files to their Phase 84 state
git checkout 1788d97 -- \
  vigil-extension/manifest.json \
  vigil-extension/popup.html \
  vigil-extension/popup.css \
  vigil-extension/popup.js

# Delete SVCNOW-only files that did not exist at Phase 84
rm vigil-extension/background.js
rm vigil-extension/content-script.js
rm vigil-extension/popup-helpers.js

# Mirror to Safari (path quoting required for space in dir name)
git checkout 1788d97 -- "vigil-safari-extension/Vigil Capture Extension/Resources/popup.html"   # IF 1788d97 has the Safari version (Phase 84 is Apr 14 — Safari port lands Phase 114 in v3.6, so use Phase-114 era for Safari instead of 1788d97)
# In practice: Safari baseline target is Phase 114 EXT-02 (commit f201ed9), which IS the Phase 84 popup ported. Use git log to confirm: `git log --oneline -- "vigil-safari-extension/.../popup.html"`.
```

**CRITICAL nuance:** Phase 84 (`1788d97`) was Apr 14 2026 — the Safari extension did not exist yet. The Safari extension was created in Phase 114 EXT-02 (commit `f201ed9`, May 6 2026) as a verbatim Chrome → Safari port of the Phase 84 popup. The correct revert target for Safari is therefore the **post-114, pre-129 state of the Safari files** — i.e. `git checkout <commit-before-ab99acd> -- vigil-safari-extension/...` for popup.{html,css,js,manifest.json}, and `rm` for the SVCNOW-only files created by ab99acd.

Recommended commit shape: **one revert commit per side** (one for vigil-extension, one for vigil-safari-extension) with a body that lists each reverted plan-commit by hash and the disposition (REVERT / PARTIAL REVERT / KEEP).

import { callClaudeMultimodal, getAIClient, parseAIJson } from "../ai/client.js";

export const screenshotCaptures = new Hono();

const VALID_MEDIA_TYPES = ["image/png", "image/jpeg"] as const;
type MediaType = (typeof VALID_MEDIA_TYPES)[number];

// Required extracted fields
interface RequiredExtracted {
  case_number: string;
  short_description: string;
  location: string;
  department: string;
  maintenance_problem: string;
}

// Opportunistic fields — logged for operator review, not persisted

  service?: string;
  assignment_group?: string;

  }

  if (!getAIClient()) {
    return c.json({ error: "AI service unavailable. ANTHROPIC_API_KEY not configured." }, 503);
  }

  const existing = await db.select()
    .from(workOrders)
    .where(and(
      eq(workOrders.userId, userId),
      eq(workOrders.clientCaptureId, body.clientCaptureId),
    ))
    .limit(1);
  if (existing.length > 0) {
    return c.json({
      workOrderId: existing[0].caseNumber,
      duplicate: true,
      extractedFields: {/* echo back stored fields */},
    }, 200);
  }

  // ── Vision extraction ──
  let extracted: { required: RequiredExtracted; extras: ExtraExtracted };
  try {
    const rawText = await callClaudeMultimodal({
      content: [
        { type: "image", source: { type: "base64", media_type: body.mediaType as MediaType, data: body.imageBase64 } },

  // (userId, clientCaptureId) so two screenshots with different clientCaptureIds but
  // same case_number will collide on the PK. Behavior options for the planner:
  //   (a) Use dbUpsertLegacy (ON CONFLICT (case_number) DO UPDATE) — overwrites prior row.
  //   (b) Detect case_number collision and return an error so operator decides.
  // Recommendation: (a) — operator can edit the draft to fix; overwrite is acceptable since
  // pending_review is by definition a draft. Plan-phase confirms with operator.
  const inserted = await db.insert(workOrders).values({
    caseNumber: extracted.required.case_number,
    userId,
    store: "",
    shortDescription: extracted.required.short_description,
    trade: extracted.extras.trade ?? "",
    location: extracted.required.location,
    equipment: extracted.extras.equipment ?? "",
    priority: extracted.extras.priority ?? "",
    contact: extracted.extras.contact ?? "",
    state: "pending_review",  // ← DECISION-C
    notes: JSON.stringify(extracted.extras),  // surface extras for operator review
    syncedAt: new Date(),
    clientCaptureId: body.clientCaptureId,
    // NEW COLUMNS from migration 0022:
    maintenanceProblem: extracted.required.maintenance_problem,
    department: extracted.required.department,
  }).onConflictDoUpdate({
    target: workOrders.caseNumber,

      shortDescription: extracted.required.short_description,
      location: extracted.required.location,
      state: "pending_review",
      notes: JSON.stringify(extracted.extras),
      maintenanceProblem: extracted.required.maintenance_problem,
      department: extracted.required.department,
      syncedAt: new Date(),

  }).returning();

    duplicate: false,
    extractedFields: extracted.required,
    extraExtractedFields: extracted.extras,

(2) Internal HTTP loopback adds an Authorization header round-trip and serialization cost. (3) The screenshot endpoint needs to write `state: 'pending_review'` and `maintenance_problem` / `department` — the sync route's sanitizer (line 85–104 of work-orders.ts) is a 10-field whitelist that would drop the new fields without a separate edit. Direct DB call keeps the sync route's sanitizer focused on the popup/manual-create path.

**Caveat on the above:** the `dbInsertOrGet` short-circuit by clientCaptureId in this endpoint duplicates the same logic already in `routes/work-orders.ts`. Planner may want to **extract `dbInsertOrGet` into a shared service** (`vigil-core/src/services/work-orders-service.ts`) so both routes share it. This is a clean refactor and worth raising in 129.1-03.

#### Auth + spend tracking

`bearerAuth` dispatcher in `vigil-core/src/index.ts` populates `c.get("userId")` for all `/v1/*` routes. The endpoint uses `callClaudeMultimodal({userId})` which calls `withBudgetTracking(userId, ...)` (Phase 127 GUARD-03) — per-userId spend accumulates in `ai_usage_daily` automatically. No extra wiring.

#### Image size limits

`describe-image.ts` has no explicit base64 size cap. The Anthropic API has a per-image size limit (~5MB base64 ≈ 3.75MB raw — `[CITED: anthropic vision docs as of training data — verify with anthropic.com/docs/vision]`). A Polaris case-page screenshot at retina resolution (~2880×1800 PNG) is typically 400-800KB raw → 540KB-1.1MB base64 — well within limits. The planner can add a `body.imageBase64.length > 7_000_000` guard (≈5.25MB base64) for safety with a `413 Payload Too Large` response.

**Note on WatchPaths vs in-process chokidar:** `WatchPaths` in launchd causes the daemon to be **launched** when a file event occurs in the named path, but it doesn't tell the daemon WHICH file changed — and re-spawning a Node process per screenshot is wasteful (cold-start ~200ms × N). The launchd `WatchPaths` key is useful for daemons that exit between events (a poll-and-exit pattern), not for a long-running watcher.

#### Daemon implementation sketch

```typescript
// scripts/vigil-capture/src/watcher.ts
import chokidar from "chokidar";
import { readFile, rename, mkdir, writeFile } from "node:fs/promises";
import { randomUUID } from "node:crypto";
import { execSync } from "node:child_process";
import path from "node:path";

const CAPTURE_DIR = process.env.VIGIL_CAPTURE_DIR ?? `${process.env.HOME}/vigil-captures`;
const API_BASE = process.env.VIGIL_API_BASE ?? "https://api.vigilhub.io";
const PROCESSED = path.join(CAPTURE_DIR, "processed");
const FAILED = path.join(CAPTURE_DIR, "failed");

async function getApiKey(): Promise<string> {
  // Reads from macOS Keychain
  return execSync('security find-generic-password -s vigil-api-key -w', { encoding: "utf8" }).trim();
}

async function processFile(filePath: string): Promise<void> {
  const filename = path.basename(filePath);

  // Filter junk: .DS_Store, hidden files, in-progress temp files
  if (filename.startsWith(".") || filename.includes(".tmp.") || filename.includes(".download")) return;

  const ext = path.extname(filename).toLowerCase();
  const mediaType = ext === ".png" ? "image/png"
    : (ext === ".jpg" || ext === ".jpeg") ? "image/jpeg"
    : null;
  if (!mediaType) return;

  const ts = new Date().toISOString().replace(/[:.]/g, "-");
  try {
    const buf = await readFile(filePath);
    const imageBase64 = buf.toString("base64");
    const clientCaptureId = randomUUID();
    const apiKey = await getApiKey();

    const res = await fetch(`${API_BASE}/v1/captures/screenshot`, {
      method: "POST",
      headers: {
        "Authorization": `Bearer ${apiKey}`,
        "Content-Type": "application/json",
      },
      body: JSON.stringify({ imageBase64, mediaType, clientCaptureId }),
    });

    if (!res.ok) {
      const errBody = await res.text();
      await mkdir(FAILED, { recursive: true });
      await rename(filePath, path.join(FAILED, `${ts}-${filename}`));
      await writeFile(path.join(FAILED, `${ts}-${filename}.err`), `HTTP ${res.status}\n${errBody}\n`);
      console.error(`[FAIL] ${filename} → ${res.status}`);
      return;
    }

    const { workOrderId } = await res.json() as { workOrderId: string };
    await mkdir(PROCESSED, { recursive: true });
    await rename(filePath, path.join(PROCESSED, `${ts}-${workOrderId}${ext}`));
    console.log(`[OK] ${filename} → ${workOrderId}`);
  } catch (err) {
    const message = err instanceof Error ? err.message : String(err);
    await mkdir(FAILED, { recursive: true });
    await rename(filePath, path.join(FAILED, `${ts}-${filename}`));
    await writeFile(path.join(FAILED, `${ts}-${filename}.err`), `${message}\n`);
    console.error(`[ERR] ${filename} → ${message}`);
  }
}

const watcher = chokidar.watch(CAPTURE_DIR, {
  ignored: /(^|[\/\\])\../,  // ignore dotfiles
  persistent: true,
  ignoreInitial: true,        // don't fire on existing files at startup
  depth: 0,                   // don't recurse into processed/failed/
  awaitWriteFinish: { stabilityThreshold: 500, pollInterval: 100 },  // wait for atomic-rename completion
});

watcher.on("add", processFile);

console.log(`[vigil-capture] watching ${CAPTURE_DIR}`);
```

#### Edge-case stress tests (for plan-phase checklist)

- **Simultaneous multi-file drops:** chokidar fires one `add` event per file with `awaitWriteFinish`. Sequential processing inside the daemon is fine — no concurrency primitive needed at single-user volume (≤50/day).
- **macOS Cmd+Shift+4 → save:** macOS writes to a temp file (`.com.apple.NSServicesRP.*.tmp`) then atomically renames. `awaitWriteFinish` + the dotfile-ignore filter handle this — chokidar fires `add` only when the rename lands.
- **iCloud-synced folder:** if the operator puts `~/vigil-captures/` in iCloud Drive, partial downloads (`.icloud` files) need filtering. Add `if (filename.endsWith(".icloud")) return;` to the junk filter. Recommended: keep `~/vigil-captures/` OUT of iCloud Drive to avoid this entirely.
- **Daemon crash mid-upload:** file remains in `~/vigil-captures/` (not yet moved to processed/failed). On daemon restart, chokidar with `ignoreInitial: true` won't pick it up. Mitigation: on startup, scan the root directory once for any leftover PNG/JPG and process them before starting the watcher.

const discardDraft = useCallback(async (caseNumber: string): Promise<void> => {
  // DELETE /v1/work-orders/:caseNumber (new endpoint — current vigil-core only has DELETE /v1/work-orders/archived).

  const res = await vigilFetch(`/v1/work-orders/${caseNumber}`, { method: 'DELETE' });
  if (!res.ok) throw new Error(`Failed to discard draft: ${res.status}`);
  setFetchTick(n => n + 1);
}, []);
```

## Risks & Landmines

### Risk 1: `case_number` PK collision when 2 screenshots of the same Polaris case are dropped

**What happens:** Operator screenshots CS0363817, fixes a field via the PWA, then screenshots the same case again with a different `clientCaptureId`. The (userId, clientCaptureId) dedup misses, and the INSERT collides on the `case_number` primary key.

**Mitigation in the proposed endpoint:** Use `onConflictDoUpdate` (overwrite). Operator's mental model is "the latest screenshot wins."

**Alternative:** Add a "dirty-edit" warning in the PWA if a `pending_review` row gets overwritten before the operator commits it. Probably overkill for v1 — discuss with operator in plan-phase.

### Risk 2: Anthropic vision failure modes on Polaris pages

**Modes to consider:**
- **Non-English content** (Polaris is English-only in this operator's deployment — confirm with operator).
- **Low-resolution screenshots** (operator's Mac is retina — risk is low if operator uses Cmd+Shift+4 default settings).
- **Polaris UI changes** that re-arrange fields — vision should adapt because the prompt describes fields by name, not position. But if Polaris ships a major UI overhaul, accuracy may degrade. **Mitigation:** the operator review step is the safety net.
- **Truncated case-page (operator only screenshots half the page):** vision will return empty string for invisible fields. The required-field validation (`case_number` non-empty) catches the worst case.

### Risk 3: Race on multi-screenshot drop in <1s

**Behavior:** chokidar with `awaitWriteFinish` fires `add` events sequentially as each file's writes settle. The daemon processes them in sequence (no Promise.all). Two screenshots arriving 100ms apart will fire two sequential POST calls — each with a unique `clientCaptureId` — and each gets its own work_order row. **No data loss; no double-write to the same row.**

**Edge case:** if the operator screenshots the SAME case twice in <1s (e.g. takes 2 screenshots of the same page), both get distinct clientCaptureIds → both hit the case_number PK collision → second insert overwrites. Same as Risk 1 — acceptable.

### Risk 4: Safari extension Xcode "Copy Bundle Resources" requirement after revert

**Documented in 129-05 SUMMARY:** after editing Safari resource files (popup.html, popup.css, popup.js, etc.), the Xcode project file references those resources. Adding or removing files (e.g. deleting `popup-helpers.js`, `background.js`, `content-script.js`) requires Xcode "Copy Bundle Resources" build phase to be updated — otherwise the Safari extension build will either FAIL (missing resource) or SILENTLY ship the file in the .app bundle from the prior build.

**Mitigation:** Operator manual step after the file-level revert. Plan-phase should include an explicit Xcode build step in the runbook: "Open `vigil-safari-extension.xcodeproj`, select Vigil Capture Extension target → Build Phases → Copy Bundle Resources → remove background.js, content-script.js, popup-helpers.js; rebuild." Add a SHIP-validation step: the produced `.app` bundle's `Contents/Resources/` must not contain those files (`ls vigil-safari-extension/build/.../Contents/Resources/`).

### Risk 5: launchd daemon survives Mac sleep/wake

**Behavior:** `LimitLoadToSessionType: Aqua` + `KeepAlive` means the daemon runs whenever the user is logged in. On Mac sleep, the daemon stays loaded; on wake, it resumes file-watching. **No special handling needed.** Confirmed pattern from existing vigilcore plist (model template).

### Risk 6: API key rotation breaks the daemon silently

If operator rotates their Vigil API key, the daemon (reading from Keychain) needs the new key in Keychain. **Mitigation:** install CLI `vigil-capture install` should prompt for / update the Keychain entry. The operator manually re-runs `security add-generic-password -s vigil-api-key -a $USER -w <new-key> -U` after rotation. Document in runbook.

### Risk 7: Drizzle migration 0022 column-add order

`schema.ts` Drizzle-defined fields and migration SQL must agree on column type and nullability. The proposed migration uses `text` (nullable), and `schema.ts` uses `text("maintenance_problem")` (no `.notNull()`) — consistent. Re-run safety via `IF NOT EXISTS` matches 0021 pattern. **No `drizzle-kit generate --dry` divergence expected** — but planner should run that command after editing `schema.ts` to confirm.

### Risk 8: Production deploy ordering

`drizzle migrate` runs idempotently on every Railway deploy. The endpoint code references `workOrders.maintenanceProblem` and `workOrders.department` — if the code deploys BEFORE the migration runs, INSERT will fail with "column does not exist". Drizzle's migrate-on-boot pattern ensures migration runs first. **No special ordering required as long as the existing deploy pipeline (Railway) is used.**

### Risk 9: `vigil-extension/popup.html` setup-view markup preservation

DECISION-B says "129-07 inline setup-view markup itself" must be KEPT. The Phase 84 baseline (`1788d97`) popup.html ALREADY HAS setup-view + capture-view markup (verified — see baseline check output). So a `git checkout 1788d97 -- vigil-extension/popup.html` is the right move: Phase 84 popup.html already has the setup-view pattern. The 129-07 contribution was adapting that pattern to the SVCNOW-view alongside it — when we revert to Phase 84, we naturally get setup-view + capture-view (the pre-SVCNOW pattern). The DECISION-B "keep setup-view markup" wording is referring to the conceptual pattern, which Phase 84 already had.

**Sanity check:** the Phase 84 popup.html had `<div id="setup-view">` and `<div id="capture-view">`. After 129-07, it had `<div id="setup-view">` and `<div id="svcnow-view">`. After revert, it has `<div id="setup-view">` and `<div id="capture-view">` again. The setup-view markup is structurally preserved in both states.
