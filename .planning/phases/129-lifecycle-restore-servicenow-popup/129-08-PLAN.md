Production code already has the post-129-04 `work-orders.ts` deployed (confirmed by UAT — the route's `dbInsertOrGet` path fires and references `client_capture_id`, which is what causes the HTTP 500). Code rollout outpaced schema migration; this plan closes that gap.

<!-- Migration content (already on disk; not changed by this plan) -->

`vigil-core/drizzle/0021_add_work_orders_client_capture_id.sql` (verified existing):
1. `ALTER TABLE "work_orders" ADD COLUMN IF NOT EXISTS "client_capture_id" text;`
2. `CREATE UNIQUE INDEX IF NOT EXISTS "uq_work_orders_user_client_capture_id" ON "work_orders" ("user_id", "client_capture_id") WHERE "client_capture_id" IS NOT NULL;`

Both statements are IF NOT EXISTS — idempotent, safe to re-run if the apply fails partway.

<!-- Rollback considerations -->

Rolling back this migration requires:
1. `DROP INDEX IF EXISTS "uq_work_orders_user_client_capture_id";`
2. `ALTER TABLE "work_orders" DROP COLUMN IF EXISTS "client_capture_id";`

Rollback is safe IF the popup-deployed extension is not actively sending clientCaptureId values in production traffic. Per UAT, no real popup submissions have succeeded yet (every one returns HTTP 500). Rollback restores prod to the pre-deploy state without data loss. After rollback, the route's `dbUpsertLegacy` path (no clientCaptureId branch) continues to work — verified by 129-04's `behavior` tests covering the omitted-clientCaptureId case.

<!-- HTTP probe shape (used pre/post to validate behavior) -->

The probe POST that currently returns HTTP 500 in production:
- URL: `https://api.vigilhub.io/v1/work-orders/sync`
- Method: POST
- Headers: `Authorization: Bearer <operator's prod API key>`, `Content-Type: application/json`
- Body: `{"workOrders":[{"caseNumber":"CS9999999","shortDescription":"Phase 129 deploy probe — GAP-129-C","priority":"Low","clientCaptureId":"<fresh-uuid>"}]}`
- Expected pre-deploy: HTTP 500, body `{"error":"Internal server error"}` (or whatever generic-error envelope the route returns for the throw at db.insert(...).values({clientCaptureId}) on the missing column).
- Expected post-deploy: HTTP 200, body `{"synced":1}` (first call — new clientCaptureId, isNew:true path). A second identical POST returns HTTP 200 `{"synced":0}` (dedup hit, isNew:false) — that's bonus validation of SVCNOW-04 against real prod.

The operator should use a CS# they don't normally use (e.g., `CS9999999` from the probe template above) to avoid colliding with real ServiceNow case data; the test row can be cleaned up after with a manual `DELETE FROM work_orders WHERE case_number = 'CS9999999' AND user_id = <operator's user_id>;`.
