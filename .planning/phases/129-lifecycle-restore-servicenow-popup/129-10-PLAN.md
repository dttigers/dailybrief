<objective>
Close GAP-129-G: G2-LIFECYCLE-02 screen-state restore does not work on real Even Hub + G2 hardware (plugin lands on HOME after force-quit, even for non-parameterized screens). The implementation in `screen-state-restore.ts` + `restoreScreenFn` + `pickInitialScreen` is correct on paper and passes all sim-side tests, but real hardware breaks it.

UAT identified THREE root-cause hypotheses; the plan structure is intentionally PHASED so the fix is conditional on the diagnostic finding (per the gap-closure mandate: "DO NOT prescribe a fix without the diagnostic result"):

- **H1 (storage-write):** `bridge.setLocalStorage` writes to `vigil:v3:lastScreen` may not actually persist across iOS Even Hub app force-quit. RESEARCH assumed write-through to disk via the SDK, but the Flutter WebView's localStorage backing may be in-memory or cleared on app process death.
- **H2 (restore-read):** Read returns null/undefined on relaunch (storage cleared OR wrong key name OR async-timing bug where pickInitialScreen runs before bridge hydration), and the fallback path lands on HOME.
- **H3 (launch-source-misclassification):** Per D-10, `source === 'glassesMenu'` precedence bypasses restore. If the iPhone-relaunch path is being classified as `glassesMenu` (rather than the cold-start-from-iPhone case the runbook implies), restore is INTENTIONALLY skipped by the D-10 guard.

This plan runs in three execution phases:
1. **Diagnostic** — add temporary console.logs at three sites; operator runs on hardware with Safari Web Inspector attached; captures evidence.
2. **Fix** — apply the H1/H2/H3-specific fix based on the captured evidence.
3. **Validation** — re-run the diagnostic; confirm the fix works.
4. **Cleanup** — remove the temporary console.logs.

This plan is `autonomous: false` because Phase 1's diagnostic and Phase 3's validation BOTH require the operator's iPhone + G2 + Safari Web Inspector setup — Claude cannot self-verify on hardware. Phases 2 and 4 are autonomous code edits, but the BRANCH between H1/H2/H3 fixes is decided by operator-supplied evidence.

Plan 129-09 must ship first so the operator can ENTER a non-HOME screen (specifically TASK_DETAIL) on hardware to seed the restore precondition for full-fidelity validation; without it, only the simpler "stay on WORK_ORDERS list, force-quit, relaunch" case from UAT Scenario 1 (degraded) is exercisable. Plan 129-12 must ship first so the build-gate process (full `npm run build` pre-SUMMARY) catches any TypeScript errors introduced by the fix — a class-of-issue Plan 129-02 originally suffered from (GAP-129-H).

Purpose: Make G2-LIFECYCLE-02 actually work on real hardware — the primary deliverable of Phase 129's G2 sub-feature.

Output: A completed diagnostic log + the fix code (conditional on H1/H2/H3) + cleanup of temporary logs.

<!-- H1/H2/H3 fix sketches (decision-tree) -->

Per gap-closure mandate: only ONE of these fixes lands, based on the diagnostic finding.

**H1 fix (storage-write fails):**
- Verify by attempting `bridge.getLocalStorage(LAST_SCREEN_LS_KEY)` IMMEDIATELY after setLocalStorage resolves (still in the same plugin session) and confirming it returns the expected value. If yes, the in-session write works but the persistence across force-quit doesn't — true H1.
- Fix options:
  a. Move the persistence channel to vigil-core: POST `/v1/g2-state` with the screen+args+savedAt; on relaunch, GET `/v1/g2-state` to retrieve. Pros: server-backed → guaranteed durable. Cons: requires network on every nav (latency, offline gap) AND a new API endpoint + DB column.
  b. Try a different SDK primitive. Check `@evenrealities/even_hub_sdk@0.0.9` `.d.ts` for any sibling persistence API (`setSharedPreference`? `setPersistentStorage`?). If one exists with stronger durability guarantees, swap to it.
  c. Use `setBackgroundState` as the persistence channel — but this is for in-session background→foreground, NOT cross-launch. Likely NOT a valid fix unless the SDK actually persists background snapshots to disk (research this in the SDK source if accessible).
- Decision rule for executor in Phase 2: Pick option (a) — server-backed — because it has the highest durability guarantee and the network latency is acceptable (fire-and-forget on write; await on read at init). Document the choice in 129-10-DIAGNOSTIC-LOG.md.

**H2 fix (restore-read fails):**
- Verify by capturing the Web Inspector log line `[diag GAP-129-G H2-read]` at relaunch. If `raw` is null/undefined → storage was cleared OR key name mismatched. If `raw` is a non-empty string but `parsed` throws → JSON corruption. If `raw` is fine but pickInitialScreen returns HOME anyway → control-flow bug in the TTL gate or screen mapping.
- Fix options:
  a. Key-name mismatch: read AND write must use the same `LAST_SCREEN_LS_KEY` constant from screen-state-restore.ts — verify both call sites use the import, not a duplicated string literal. (RESEARCH Probe 6 + Pitfall 2 caveat.)
  b. JSON parse error: the existing try/catch in launch-source-helpers.ts should already swallow it; if it's propagating, audit the try/catch boundaries.
  c. Async-timing bug: pickInitialScreen runs at init time, but the bridge may not have hydrated localStorage from disk yet. Fix: await a bridge-ready signal before calling getLocalStorage, OR poll with a short backoff (max 200ms total), OR move the restore-read into a deferred microtask after the bridge promise resolves.
- Decision rule for executor in Phase 2: The fix depends on which sub-cause the diagnostic identifies. Author the fix accordingly; document the choice.

**H3 fix (launch-source-misclassification):**
- Verify by capturing the Web Inspector log line `[diag GAP-129-G H3-source]` at relaunch. If the source string is `'glassesMenu'` when the operator clearly launched from the iPhone home screen → misclassification. If it's `'appMenu'` or `'iconClick'` or some other expected non-glassesMenu value → H3 is NOT the cause.
- Fix options:
  a. The D-10 guard in pickInitialScreen uses `source !== 'glassesMenu'`. If the iPhone-launch source string is somehow `'glassesMenu'`, tighten the guard to match only the genuine glassesMenu path. Look at Phase 124 G2-POLISH-06 / D-07 for the exact set of source strings the SDK can emit.
  b. Alternative: widen the cold-start branch — if the source is unknown or empty, treat as cold-start and run restore.
- Decision rule for executor in Phase 2: Use the source string the diagnostic captures; document the fix narrows or widens which set of sources qualify for restore.
