    **Scenario 1 — G2 force-quit-iPhone restore (G2-LIFECYCLE-02 / Success Criterion 1):**
    Operator: navigate to WORK_ORDER_DETAIL for any work order on G2 glasses → confirm screen displays → force-quit Even Hub iPhone app from the iOS app switcher → wait 30 seconds → re-open Even Hub from the home screen → reconnect glasses → expect G2 plugin to land on the SAME WORK_ORDER_DETAIL screen for the same work order. Pass = correct screen + correct work order id. Fail = lands on HOME or a different work order.

    **Scenario 1b — TTL expiry boundary (G2-LIFECYCLE-02 + D-05):**
    Same as Scenario 1, but wait 31 minutes before re-opening. Expect HOME (TTL expired per D-05). Pass = HOME. Fail = restored screen (TTL drift).

    **Scenario 2 — Phone-background→foreground Companion HUD cache (G2-LIFECYCLE-01 / Success Criterion 2 / D-11):**
    Operator: open Even Hub → start a Claude Code session (or have one already active) → verify Companion HUD shows the session card → swipe Even Hub to phone background (home button / app switcher) → wait 60 seconds → swipe back to Even Hub foreground. Expect the SAME session card to appear immediately (no flash of empty state, no re-fetch shimmer). Pass = no empty re-render. Fail = empty state visible for more than ~200ms.

    **Scenario 3 — Glasses-menu precedence (G2-LIFECYCLE-03 / Success Criterion 3 / D-10):**
    Operator: open Vigil plugin from G2 glasses menu (NOT from Even Hub iPhone app). Expect the screen the operator picked from the glasses menu (HOME by default, or any screen if the operator selected one). Pass = the glasses-menu-selected screen. Fail = any restored last-viewed screen (Phase 124 D-07 invariant violated).

    **Scenario 4 — Polaris CS# extraction (SVCNOW-01 / Success Criterion 4 part 1):**
    Operator: navigate to a live ServiceNow Polaris case page in Chrome → open browser DevTools → in Console, run `document.title` and record the exact string in 129-UAT-RESULTS.md → click the Vigil extension button → expect popup to open with the CS# from the title locked in the header (font-size larger than the description input). Pass = CS# matches the 7-digit number in the title. Fail = popup shows wrong CS# or "No case# detected" (indicates regex needs adjustment).

    **Scenario 4b — Multi-tab POST race (SVCNOW-04 / Success Criterion 4 part 2 / D-12):**
    Operator: open the same Polaris case page in TWO Chrome tabs → click Vigil extension button on tab 1 → type description "RACE-A" → click Send (do NOT close popup wait for window.close) → IMMEDIATELY repeat in tab 2 with description "RACE-B" → verify ONE row appears in the PWA work-orders list. (Reasoning: each popup generates a different `clientCaptureId`, so two rows ARE expected here — this scenario is the operator-friction acceptance check, not strict idempotency. For strict idempotency, use Scenario 4c.) Pass = both rows in PWA, no duplicate-of-itself. Fail = PWA shows duplicates of the same description text.

    **Scenario 4c — Retry-storm idempotency (SVCNOW-04 strict):**
    Operator: visit a Polaris case page → click Vigil button → in DevTools, set the network throttle to "Offline" → click Send → wait for the inline error → set network back to Online → click Send AGAIN (same clientCaptureId — popup did not reload) → expect HTTP 200 close on the retry, ONE row in PWA. Pass = exactly one row. Fail = two rows OR popup doesn't close on retry.

    **Scenario 5 — Safari parity smoke (SVCNOW-05 / Success Criterion 5):**
    Operator: on macOS, open Xcode → ensure the new files (background.js, content-script.js, popup-helpers.js) are added to the `Vigil Capture Extension` target's `Copy Bundle Resources` build phase (manual step from 129-05 SUMMARY) → build + install the Safari extension → enable in Safari Extensions → visit the same Polaris page used in Scenario 4 → click the extension button → expect the SAME SVCNOW form (case-number header, description, priority, Send) → submit and verify a row in PWA work-orders. Pass = identical Safari UI + working submit. Fail = visual divergence, missing form, or runtime JS error (likely `chrome.action.disable` API shim issue per RESEARCH Assumption A2).

    **Scenario 6 — Title drift banner (SVCNOW-02):**
    Operator: on a Polaris case page, click Vigil button → popup opens with CS# X → without closing the popup, in the SN tab, click a different case to trigger pushState navigation to a different case page → switch back to the popup → expect a yellow drift-banner showing "CS# changed — reopen popup". Pass = banner appears. Fail = banner missing or popup auto-updates (would mask the drift).

    Include a final section at the bottom of the runbook: "After running all scenarios, fill in `129-UAT-RESULTS.md` per scenario with PASS/FAIL + observed string for the Polaris `document.title`."
