<objective>
Close GAP-129-E (medium severity: terminology + scenario design issue). Phase 129's planning docs conflate two distinct domain concepts:

1. **G2 task screen (`TASK_DETAIL`)** — populated by the G2 plugin's `/v1/brief` openTasks, which are sourced from the `thoughts` table (taskStatus IN ('open', 'inProgress')). These are user-captured thought-tasks (e.g., "review the Q3 report" — a personal todo).
2. **ServiceNow work order (`work_orders` table + `/v1/work-orders/sync`)** — populated by Gmail import + the SVCNOW extension popup. These are tickets from ServiceNow Polaris, identified by `CS#` and stored in the PostgreSQL `work_orders` table.

The two are separate entity types with separate code paths. The Phase 129 implementation correctly used `task.id` (thought ID, not caseNumber) — confirmed by 129-02's executor-flagged deviation. But the planning docs and the UAT runbook used "work order" / "WORK_ORDER_DETAIL" language for the G2 task screen, which created operator confusion during UAT (operator's account had no "active work orders" visible on G2 because the G2 screen actually shows thought-tasks, not the SVCNOW work_orders table).

This plan is DOC-ONLY. No source code is modified except for one optional file rename (`work-orders.ts` → `tasks.ts`) that the executor may or may not include based on scope safety.

Purpose: Make the phase docs accurately reflect what was actually built so future Phase 129 follow-ups and future phases (130+) don't perpetuate the confusion. Surface the distinction clearly in the runbook so future operator-facing UAT runs don't repeat the GAP-129-E confusion.

Output: Three updated planning markdown files; optional source-file rename (gated by executor judgment).
</objective>
