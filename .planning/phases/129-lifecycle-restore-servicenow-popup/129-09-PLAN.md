<objective>
Wire a context-sensitive DOUBLE_CLICK gesture on the WORK_ORDERS screen that enters TASK_DETAIL for the top-of-list / first-visible task. This closes GAP-129-F: on real G2 hardware, the operator currently has NO way to reach TASK_DETAIL, which means the G2-LIFECYCLE-02 restore-to-TASK_DETAIL path (already correctly implemented in `restoreScreenFn` at main.ts:99-130) is unreachable through normal user interaction.

The existing `main.ts:316-322` wires `CLICK_EVENT` on a list item to `navigateToTaskDetail` — that handler is sim-only (Phase 45 retro: CLICK_EVENT is not reliably plumbed on G2 hardware). This plan keeps that wiring intact (sim-side test fidelity) and ADDS a hardware-reliable DOUBLE_CLICK path using the Phase 124 D-08 carve-out template proven on COMPANION.

Purpose: Unblock UAT Scenario 1 (full TASK_DETAIL restore flow). Without a hardware-validated entry gesture, the operator cannot set up the precondition for the restore test — no way to navigate INTO TASK_DETAIL means the localStorage write of `{screen: TASK_DETAIL, args: {id}}` never happens, which means restore-on-relaunch has no parameterized-screen payload to act on.

Output: Modified `navigation.ts` (new WORK_ORDERS DOUBLE_CLICK branch in `handleNavEvent`), modified `main.ts` test file (no source change to main.ts itself — the existing CLICK_EVENT wiring at lines 316-322 stays for sim fidelity), new unit tests in `navigation.test.ts` covering both empty-list and non-empty-list paths.

Note: This plan is `autonomous: true` because the wiring + sim-side test are deterministic. The hardware verification step (operator confirms DOUBLE_CLICK on real G2 enters TASK_DETAIL) lives in plan 129-13 (re-run UAT). Plan 129-10 (G2 restore diagnostic) depends on THIS plan having shipped so the operator can seed the restore precondition.
</objective>

```
// Phase 124 D-08 — context-sensitive DOUBLE_CLICK on Companion screen.
// ONLY DOUBLE_CLICK_EVENT is plumbed reliably on G2 (CLICK_EVENT is
// sim-only per Phase 45 retro; long-press is absent from OsEventTypeList
// in @evenrealities/even_hub_sdk@0.0.9 — see SEED-011).
// Priority: banner-ack → cycle-session → jump-Home.
if (
  currentScreen === Screen.COMPANION &&
  eventType === OsEventTypeList.DOUBLE_CLICK_EVENT
) {
  if (hasActiveBanner()) { ackBanner(); await refreshCurrentScreen(bridge); return }
  if (getActiveSessions().length >= 2) { cycleSession(); await refreshCurrentScreen(bridge); return }
  await navigateTo(Screen.HOME, bridge)
  return
}
```

The new WORK_ORDERS carve-out follows the same shape:
- Check `currentScreen === Screen.WORK_ORDERS && eventType === OsEventTypeList.DOUBLE_CLICK_EVENT` BEFORE the bottom default switch.
- Read `getLastFetchedTasks()` (already exported from navigation.ts — same source that `navigateToTaskDetail` uses).
- If `tasks.length === 0`: fall through to the default behavior (don't return early; let the bottom switch handle it → HOME).
- Else: call `await navigateToTaskDetail(0, bridge)` (index 0 = top-of-list / first-visible task per the convention this plan establishes); return.
