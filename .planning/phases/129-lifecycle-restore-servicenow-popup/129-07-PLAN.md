<!-- GAP-129-D reference: chrome.storage.session MV3 docs -->

chrome.storage.session is:
- Available in MV3 since Chrome 102 (early 2022); Safari WKWebExtension supports it as well.
- Cleared on browser quit (NOT cleared on tab close, NOT cleared on extension reload during browser session).
- Quota: 10MB total, far above the ~10 bytes needed for a CS# string.
- DOES NOT require an additional `permissions` entry beyond the existing `"storage"` permission already declared in manifest.json (the storage permission grants access to local + sync + session + managed sub-namespaces).

Storage key name: `vigil_last_case_number` (NOT `vigil_api_key` — that key continues to live in chrome.storage.local).

<!-- GAP-129-A: Why __tests__ folder breaks Chrome unpacked load -->

Chrome's extension manifest spec (https://developer.chrome.com/docs/extensions/reference/manifest) reserves any file or directory name starting with `_` for the extension system itself. Loading an unpacked extension whose root contains `__tests__/` triggers:

  "Could not load extension. Cannot load extension with file or directory name __tests__. Filenames starting with "_" are reserved for use by the system."

The fix is to move the tests OUT of the extension load root entirely. Two options were considered during UAT debrief:
- Option A: rename `__tests__/` → `tests/` (still in the extension root). REJECTED because the convention `tests` (no underscore) is closer to a tests dir name a user might want to ship in production, and we'd still need to remember to exclude it from the production build.
- Option B (chosen): move tests to a sibling directory at the repo root: `vigil-extension-tests/`. CLEAR separation; no risk of accidentally shipping the test files when packing the extension for store submission.

- The parity.test.ts file must update its relative paths from `../popup.html` (when in `vigil-extension/__tests__/`) to `../vigil-extension/popup.html` (when in `vigil-extension-tests/`).
