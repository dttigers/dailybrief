## Decisions Made

- **External probe is its own mode (not merged into --runtime).** LOCKED by RESEARCH Open Question 2. `--runtime` does xcodebuild + SMAppService probe; neither requires outbound network. `--external` does. Separating them keeps each mode's failure domain clean and allows Plan 107.3-01 to invoke only `--external` post-Railway-redeploy without re-running Xcode.
- **`--full` includes `run_external` after `run_runtime`.** The whole-suite ordering is: static (fast, offline) → runtime (builds + local launch) → external (network, live prod). Fail-fast on cheap checks first.
- **Default MODE line unchanged.** `MODE="${1:---static}"` stays — this plan adds capability, not a default-behavior flip. The `--full|""` alternation is preserved defensively (matches legacy semantics).
- **Probe URL, timeout, and fallback string all come directly from RESEARCH.md Code Examples.** No relitigating in this plan — the harness matches research verbatim so the signal stays consistent with the documented live response shape (`{"status":"ok","timestamp":"...","version":"0.1.0","database":"connected"}`).
