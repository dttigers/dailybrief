## Summary

Three independent paper-cuts from the MacBook Pro bootstrap session on 2026-04-22. Each fix is self-contained and small. No new dependencies are introduced.

**Fix 1 (vigil-core bind):** Railway injects `RAILWAY_SERVICE_ID` (and related vars) into every container runtime. There is NO bare `RAILWAY_ENVIRONMENT` var — it is `RAILWAY_ENVIRONMENT_NAME`. The cleanest fix is to detect Railway via `RAILWAY_SERVICE_ID` (always present on Railway, never present locally) and force `0.0.0.0` when that var is set, while keeping `127.0.0.1` as the default for all non-Railway environments. This eliminates the 502 class of bug while avoiding a global security bias flip.

**Fix 2 (install.sh silent-fail):** The root cause is confirmed: `set -euo pipefail` exits the whole script when `grep "Developer ID Application"` finds no match in the pipeline. The minimal correct fix is appending `|| true` to the `security | grep | head -1 | grep -o | tr -d` pipeline. An awk-based single-command replacement is cleaner still (awk exits 0 on no match). Either approach is safe. The `|| true` pattern does mask a real `security` binary failure, but that failure mode is so rare (security binary missing entirely) that it is acceptable.

**Fix 3 (doctor stale-drift):** The retirement marker `107.1-daemon-retirement.md` exists in the repo at `.planning/phases/107.1-local-dev-environment-with-postgres-and-hot-reload-stack/107.1-daemon-retirement.md`. The cleanest fix is: detect whether the plist file is absent, and if absent AND the retirement marker exists, print an informational-only row (like the local-DB row) rather than incrementing `DRIFT_COUNT`. This distinguishes "intentionally retired" from "accidentally deleted" without relying on env vars or any state outside the repo.

**Primary recommendation:** Three targeted single-function edits: 1) `vigil-core/src/index.ts` line 186, 2) `Scripts/install.sh` lines 28-32, 3) `scripts/dailybrief-doctor.sh` lines 113-126.

## Fix 1: Railway Bind Host Detection

### Railway-Injected Environment Variables (HIGH confidence)

Source: https://docs.railway.com/reference/variables (fetched 2026-04-22)

Railway automatically injects these vars into every container runtime. None require user configuration:

| Variable | Always Present? | Notes |
|---|---|---|
| `RAILWAY_SERVICE_ID` | YES | UUID — always injected, no exceptions noted in docs |
| `RAILWAY_PROJECT_ID` | YES | UUID |
| `RAILWAY_ENVIRONMENT_NAME` | YES | String name of the environment (e.g. "production") |
| `RAILWAY_ENVIRONMENT_ID` | YES | UUID |
| `RAILWAY_DEPLOYMENT_ID` | YES | UUID |

**Critical finding:** There is NO bare `RAILWAY_ENVIRONMENT` variable. The ROADMAP entry's suggested detection via `RAILWAY_ENVIRONMENT` would not work. The correct vars to test are `RAILWAY_SERVICE_ID` or `RAILWAY_ENVIRONMENT_NAME`.

**Best detection variable:** `RAILWAY_SERVICE_ID` — it is a UUID that is always set in Railway and never set locally. No false-positive risk.

### Three Fix Options Analysis

**Option A — Detect Railway and force 0.0.0.0 (RECOMMENDED)**

```typescript
// vigil-core/src/index.ts ~line 186
// Phase 107.3: Railway forces 0.0.0.0 (proxy cannot reach 127.0.0.1).
// Detect Railway via RAILWAY_SERVICE_ID (always injected; never present locally).
// Local dev: VIGIL_BIND_HOST in .env overrides explicitly.
// All other envs default to 127.0.0.1.
const isRailway = !!process.env.RAILWAY_SERVICE_ID;
const hostname = process.env.VIGIL_BIND_HOST ?? (isRailway ? "0.0.0.0" : "127.0.0.1");
```

Tradeoffs:
- Security bias stays conservative (local default is still 127.0.0.1)
- Railway path is automatic — no manual env var required
- Explicit `VIGIL_BIND_HOST` always wins, so local dev can still override to 0.0.0.0 for Tailscale
- The comment at lines 183-184 also needs updating to remove the false claim

**Option B — Flip default to 0.0.0.0 globally**

Would break the security invariant Phase 107.2 established: local dev accidentally binding to all interfaces. Not recommended.

**Option C — Add external HTTP probe to verify harness**

Useful as a regression test but does not fix the underlying default. Do alongside Option A, not instead.

## Fix 2: install.sh Silent-Fail Under `set -euo pipefail`

### Root Cause (HIGH confidence — verified live)

```bash
# Current code (Scripts/install.sh lines 28-32):
IDENTITY=$(security find-identity -v -p codesigning \
           | grep "Developer ID Application" \
           | head -1 \
           | grep -o '"Developer ID Application: [^"]*"' \
           | tr -d '"')
```

With `set -euo pipefail`, when `grep "Developer ID Application"` finds no match it exits 1. `pipefail` propagates that exit code from the pipeline, so the assignment to `IDENTITY` fails, and since there's no `|| true`, the entire script exits. The `if [[ -z "$IDENTITY" ]]` guard at line 34 is never reached.

Verified: `bash -c 'set -euo pipefail; VAR=$(echo x | grep nomatch); echo $VAR'` → exit 1 with no output.

### Fix Option A — `|| true` on the whole pipeline (minimal diff, LOW risk)

```bash
IDENTITY=$(security find-identity -v -p codesigning \
           | grep "Developer ID Application" \
           | head -1 \
           | grep -o '"Developer ID Application: [^"]*"' \
           | tr -d '"' || true)
```

Verified: With `|| true`, a no-match grep yields an empty string and the script continues. The existing `if [[ -z "$IDENTITY" ]]` guard then fires and prints the remediation message.

**Masking risk:** `|| true` will also swallow a real `security` binary failure (e.g., the binary doesn't exist). In practice, if `security` is missing, macOS itself is broken — not a meaningful risk.

### Fix Option B — awk replacement (single command, no masking risk from grep)

```bash
IDENTITY=$(security find-identity -v -p codesigning \
           | awk 'match($0, /"Developer ID Application: [^"]*"/) {
               s=substr($0, RSTART+1, RLENGTH-2); print s; exit
             }')
```

Verified live: extracts `Developer ID Application: Jameson Morrill (5H57ADQS8G)` correctly when cert is present; returns empty string (exit 0) when no match.

**Advantage over `|| true`:** awk exits 0 on no match, so `pipefail` never triggers. The `|| true` pattern is not needed at all. A failure in `security` itself (not awk) would still propagate correctly since awk won't run in that case.

**Recommendation:** Use Option B (awk) — it is safer and produces the same output format. The diff from the current code is small. The existing `if [[ -z "$IDENTITY" ]]` guard works unchanged with either option.

### doctor.sh already uses `|| true` correctly for this same pattern (lines 215-219)

```bash
DEVID_DOCTOR=$(security find-identity -v -p codesigning 2>/dev/null \
               | grep "Developer ID Application" \
               | head -1 \
               | grep -o '"Developer ID Application: [^"]*"' \
               | tr -d '"' || true)
```

The doctor script is already correct (it uses `|| true`). Only `install.sh` is broken.

```bash
# 3. Plist EnvironmentVariables:ANTHROPIC_API_KEY
```

The `else` branch increments `DRIFT_COUNT` whenever the plist is missing, including when it is intentionally absent.

### Informational Row Pattern (from existing doctor code — HIGH confidence)

The local-DB row and `/v1/health` row use a separate section and a `printf` that does NOT call `print_row` (no `DRIFT_COUNT` increment). The pattern is:

```bash
# Informational rows (D-13) — do NOT affect exit code
printf "%-38s | %s\n" "TARGET" "VALUE"
...
printf "%-38s | %s\n" "local vigil_dev DB" "connected (${DB_STATUS} users)"
```

### Recommended Fix — Three-Way Branch

Replace the current two-way `if/else` with a three-way check:

```bash
RETIREMENT_MARKER="$REPO_DIR/.planning/phases/107.1-local-dev-environment-with-postgres-and-hot-reload-stack/107.1-daemon-retirement.md"

if [[ -f "$VIGILCORE_PLIST" ]]; then
    # Plist exists: check value for drift
    ANTHROPIC_PLIST=$(/usr/libexec/PlistBuddy -c "Print :EnvironmentVariables:ANTHROPIC_API_KEY" "$VIGILCORE_PLIST" 2>/dev/null || echo "")
    if [[ "$ANTHROPIC_PLIST" == "$ANTHROPIC_CANONICAL" ]]; then
        MATCH="✓"
    else
        MATCH="✗"
        DRIFT_COUNT=$((DRIFT_COUNT + 1))
    fi
    print_row "plist EnvironmentVariables" "$(prefix8 "$ANTHROPIC_PLIST")" "$(last_modified "$VIGILCORE_PLIST")" "$MATCH"
elif [[ -f "$RETIREMENT_MARKER" ]]; then
    # Plist absent AND retirement marker exists → intentionally retired (Phase 107.1-04)
    # Print informational-only row (same pattern as local-DB row); do NOT increment DRIFT_COUNT
    printf "%-38s | %s\n" "plist EnvironmentVariables" "retired (Phase 107.1-04) — see 107.1-daemon-retirement.md"
else
    # Plist absent AND no retirement marker → unexpected absence, count as drift
    print_row "plist EnvironmentVariables" "(missing)" "(missing)" "✗"
    DRIFT_COUNT=$((DRIFT_COUNT + 1))
fi
```

**Why this is better than just dropping the row:** If someone accidentally deletes the plist without retiring the daemon properly (a real regression), the third branch catches it. The middle branch only fires when both conditions are true (plist gone AND retirement on record), which is the definition of "intentionally absent."

**Why informational printf instead of print_row:** `print_row` uses the table format with a MATCH column (`✓`/`✗`). The informational section uses a two-column `printf` without a MATCH column. This is consistent with the existing D-13 pattern.

**Note on table section placement:** The three-way branch should stay in the `ANTHROPIC_API_KEY drift check` section (the existing drift table). The informational `printf` within the drift-check table is slightly inconsistent with the existing style, but is clearer than moving the row to a completely different section. Alternatively, the row can be dropped entirely (simplest) — but the three-way branch is more defensive.

## Architecture Patterns

### Bash `set -euo pipefail` Safety Rules (HIGH confidence — verified)

| Pattern | Safe? | Notes |
|---|---|---|
| `$(cmd | grep pattern)` | UNSAFE | grep exits 1 on no match; pipefail propagates |
| `$(cmd | grep pattern \|\| true)` | SAFE | empty string on no match; masks cmd failure |
| `$(cmd | awk '/pattern/ {print}')` | SAFE | awk always exits 0 |
| `$(cmd \| grep pattern; true)` | UNSAFE | `;` doesn't help inside `$()` with pipefail |
| Splitting to `_line=$(... \|\| true)` then second grep | SAFE | each command isolated |

### Railway env-var detection pattern (HIGH confidence)

```typescript
// Canonical: use RAILWAY_SERVICE_ID (always injected, UUID, never local)
const isRailway = !!process.env.RAILWAY_SERVICE_ID;

// Alternative: RAILWAY_ENVIRONMENT_NAME (string the user named the env)
// Less reliable because the user could name a non-prod env "production" etc.
// RAILWAY_SERVICE_ID is more reliable as a presence-only signal.

// Do NOT use:
// - RAILWAY_ENVIRONMENT (does not exist — not in Railway's variable list)
// - NODE_ENV (set by user convention, not by Railway infrastructure)
```

## Common Pitfalls

### Pitfall 1: Testing only the local path for Railway bugs

**What goes wrong:** Phase 107.2-01 ran `NODE_ENV=production` locally and concluded Railway prod was OK. The local test never exercises the proxy → container network path. The container was bound to 127.0.0.1, so Railway's ingress proxy (which is external to the container) could not reach it.

**How to avoid:** Always include an external HTTP probe (`curl https://api.vigilhub.io/...`) in the --runtime verification mode.

### Pitfall 2: RAILWAY_ENVIRONMENT does not exist

**What goes wrong:** Docs and community posts sometimes say `RAILWAY_ENVIRONMENT=production`. The official Railway variables reference (fetched 2026-04-22) does NOT list a bare `RAILWAY_ENVIRONMENT`. Only `RAILWAY_ENVIRONMENT_NAME` and `RAILWAY_ENVIRONMENT_ID` exist.

**How to avoid:** Use `RAILWAY_SERVICE_ID` as the detection signal — it is listed in the official docs as always injected, is a UUID (unambiguous), and is not user-configurable.

### Pitfall 3: `|| true` placement matters in multi-pipe chains

**What goes wrong:** Appending `|| true` after only the last command in a chain doesn't help if pipefail fires on an earlier command. In `cmd1 | cmd2 | cmd3 || true`, if `cmd2` exits 1, pipefail still aborts the assignment before `|| true` is reached.

**How to avoid:** `|| true` must appear at the end of the entire `$(...)` command substitution. Verified working: `VAR=$(cmd1 | cmd2 | cmd3 || true)`.

### Pitfall 4: Updating code but not the comment

**What goes wrong:** `vigil-core/src/index.ts` lines 183-184 contain a comment that explicitly says "Railway prod leaves VIGIL_BIND_HOST unset → defaults to 127.0.0.1 behind Railway's proxy." This claim is now proven false. Leaving the comment creates the next 502 outage.

**How to avoid:** The comment must be updated as part of Fix 1 — new comment should explain that Railway is detected via `RAILWAY_SERVICE_ID` and `0.0.0.0` is forced automatically.
