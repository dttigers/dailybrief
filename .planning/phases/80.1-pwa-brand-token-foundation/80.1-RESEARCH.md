## Summary

The PWA is built on Tailwind CSS v4.2.2 (the `@tailwindcss/vite` plugin, not v3). In v4, there is no
`tailwind.config.js` — all theming is done via the `@theme` directive in `src/index.css`. The file
currently contains only `@import "tailwindcss";`, so adding brand tokens is a single focused CSS edit.

The color migration scope is substantial. Every component uses hardcoded `slate-*` and `indigo-*`
classes for backgrounds, borders, and text. There are also `blue-*`, `purple-*`, `amber-*`,
`green-*`, `yellow-*`, `pink-*`, `rose-*`, and `teal-*` classes used for category/status indicators.
The brand spec replaces the `slate/indigo` UI chrome with Vigil teal and warm neutrals, while
status-only colors (blue, amber, green) are mapped to their brand equivalents. Category-color
semantics (task=blue, therapy=purple, etc.) must be preserved using brand-safe values.

The font-weight violation scope is also real: 26 occurrences of `font-semibold` and `font-bold`
exist across components and pages. All must become `font-medium` (weight 500). One structural change
is required in `ThoughtRow.tsx`: therapy classification tags currently use `rounded-full` (pill shape)
but the brand spec requires them to use a square badge shape — they are "routing" tags, not category
tags.

**Primary recommendation:** Define all brand tokens in `src/index.css` via `@theme`, load Inter from
Google Fonts in `index.html`, then do a systematic find-and-replace of off-brand Tailwind classes
across all 12 component files and 10 page files.

## Architecture Patterns

### Tailwind v4 CSS-First Theme Configuration

In Tailwind v4 there is NO `tailwind.config.js`. All tokens are defined in the entry CSS file via
`@theme`. [VERIFIED: tailwindcss.com/docs/theme]

```css
/* src/index.css */
@import "tailwindcss";

/* Load Inter from Google Fonts */
@import url('https://fonts.googleapis.com/css2?family=Inter:wght@400;500&display=swap');

@theme {
  /* Override default font-sans with Inter */
  --font-sans: 'Inter', ui-sans-serif, system-ui, sans-serif;

  /* Vigil Teal palette */
  --color-teal-50: #E1F5EE;
  --color-teal-100: #9FE1CB;
  --color-teal-400: #1D9E75;
  --color-teal-600: #0F6E56;
  --color-teal-800: #085041;

  /* Vigil Warm Neutrals */
  --color-gray-50: #F1EFE8;
  --color-gray-100: #D3D1C7;
  --color-gray-400: #888780;
  --color-gray-900: #2C2C2A;
  --color-surface: #F9F9F8;

  /* Status Accents — info/warning/success only */
  --color-info-50: #E6F1FB;
  --color-info-400: #378ADD;
  --color-warning-50: #FAEEDA;
  --color-warning-400: #BA7517;
  --color-success-50: #EAF3DE;
  --color-success-400: #639922;
}
```

Source: [CITED: https://tailwindcss.com/docs/theme]

Once defined, these generate utilities like `bg-teal-600`, `text-gray-900`, `bg-info-50`, etc.

The existing `teal-*` Tailwind default colors will be overridden by our custom `--color-teal-*`
definitions. The default Tailwind `gray-*` scale will be partially overridden by our warm neutral
definitions. [CITED: https://tailwindcss.com/docs/customizing-colors — "adding tokens to an existing
namespace extends it; using `--color-*: initial` clears the whole namespace"]

## Common Pitfalls

### Pitfall 1: Tailwind v3 Config Approach
**What goes wrong:** Creating a `tailwind.config.ts` or `tailwind.config.js` file and using `theme: { extend: { colors: {...} } }` syntax.
**Why it happens:** Tailwind v3 muscle memory — the project is on v4, which uses CSS-first `@theme`.
**How to avoid:** All tokens go in `src/index.css` via `@theme` directive. No JS config file.
**Warning signs:** If a `tailwind.config.*` file appears, it's the wrong approach for this stack.

### Pitfall 2: Overriding Full Color Namespace
**What goes wrong:** Using `--color-*: initial` in `@theme` which wipes ALL Tailwind default colors (including `red-*` used for error states).
**Why it happens:** Following "disable defaults" examples from docs.
**How to avoid:** Do NOT use `--color-*: initial`. Add tokens additively. The brand red used for delete buttons and error states should remain available.
**Warning signs:** `bg-red-600` (Delete button in BulkActionBar) stops rendering.

### Pitfall 3: Missing font-display=swap
**What goes wrong:** Inter loads slowly, causing FOUT (flash of unstyled text) on slow connections.
**Why it happens:** Missing `display=swap` in Google Fonts URL.
**How to avoid:** Always include `&display=swap` in the font URL: `?family=Inter:wght@400;500&display=swap`.

### Pitfall 4: Dark Theme vs Brand Light Theme Mismatch
**What goes wrong:** The brand guidelines show a light-background UI (Surface #F9F9F8, Gray 900 text). The existing PWA is dark (slate-950 backgrounds, slate-100 text). Direct token substitution without deciding on theme direction produces an incoherent result.
**Why it happens:** The brand spec and the existing implementation have incompatible default themes.
**How to avoid:** The planner must decide: implement light theme (matching brand spec) or dark theme (current PWA style) using brand tokens. The Success Criteria says "replace hardcoded slate/blue with brand token classes" — it does not mandate switching to light theme. Using warm-dark equivalents (gray-900 backgrounds, gray-50 text) is the safest interpretation.
**Warning signs:** Phase description says "brand-compliant values from BRAND-GUIDELINES.md" but doesn't specify light/dark.

### Pitfall 5: font-semibold Regex Missing font-bold
**What goes wrong:** Search-and-replace catches `font-semibold` but misses `font-bold` (used in WorkOrderRow and AuthPage).
**Why it happens:** Two separate Tailwind weight violations.
**How to avoid:** Search for both `font-semibold` AND `font-bold` separately. Both map to `font-medium`.

## Open Questions

1. **Light vs Dark Theme**
   - What we know: Brand guidelines show a light surface (#F9F9F8) design. Current PWA is dark (slate-950 bg).
   - What's unclear: The phase goal says "replace hardcoded slate/blue" but doesn't specify switching to light theme.
   - Recommendation: Planner should default to dark-themed brand tokens (gray-900 bg, gray-50 text) as the lowest-friction interpretation. If user wants the full brand light theme, that's a larger UX change better scoped separately. Flag in plan tasks.

2. **`accent-indigo-500` on checkboxes**
   - What we know: ThoughtsPage.tsx uses `accent-indigo-500` for checkbox color.
   - What's unclear: Whether `accent-teal-600` will render correctly across browsers.
   - Recommendation: Replace with `accent-teal-600` — the CSS `accent-color` property is well-supported (94%+ browsers as of 2026).
