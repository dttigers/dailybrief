# Phase 79.1: Code Review Report

**Reviewed:** 2026-04-14T00:00:00Z
**Depth:** standard
**Files Reviewed:** 6
**Status:** issues_found

## Summary

This phase closes the OAuth schema gap by adding `scopes` and `account_email` columns to `oauth_tokens`, persisting granted scopes in the callback, and reading them in the status endpoint. The migration, callback wiring, and status logic are all structurally sound. Three issues require attention before this can be considered production-safe.

The most significant issue is a **schema/migration drift**: the migration enforces `NOT NULL DEFAULT '[]'` on `scopes` while the Drizzle schema declares it nullable — this creates a type lie at the TypeScript layer. The second issue is that `account_email` is **never populated** despite being the stated goal of this phase (no `id_token` decode anywhere in the callback). The third is a **conflict between the migration default and the upsert conflict-update path**, which omits `scopes` when the row already exists.

---

## Warnings

### WR-01: `scopes` column nullable in schema.ts but NOT NULL in migration

**File:** `vigil-core/src/db/schema.ts:183`
**Issue:** The migration (`0008_add_oauth_scopes_and_account_email.sql`, line 1) adds `scopes jsonb DEFAULT '[]'::jsonb NOT NULL`, but the Drizzle schema declaration is `jsonb("scopes").$type<string[]>()` — no `.notNull()`, no `.default([])`. This means:
1. Drizzle's TypeScript types infer `scopes` as `string[] | null`, forcing defensive null-coalescing everywhere downstream.
2. If Drizzle ever re-generates or auto-migrates the column, it will drop the `NOT NULL` constraint, causing a silent regression.
3. The google-status endpoint already compensates with `rows[0].scopes ?? []` (line 48 of google-status.ts), which is correct but only necessary because the schema doesn't express the truth.

**Fix:** Align the schema declaration with the migration:
```typescript
// vigil-core/src/db/schema.ts, line 183
scopes: jsonb("scopes").$type<string[]>().notNull().default([]),
```
This makes the TS type `string[]` (not `string[] | null`), removes the need for null-coalescing downstream, and keeps schema.ts as the authoritative source of truth.

---

### WR-02: `account_email` is never written — column is dead on arrival

**File:** `vigil-core/src/routes/google-auth.ts:140-174`
**Issue:** The `account_email` column was added to the schema and migration as a stated deliverable of this phase, but the callback never extracts or persists it. The `GoogleAuthDeps.dbUpsertFn` signature (line 19-25) does not include an `account_email` parameter. The production upsert (lines 154-172) does not set `accountEmail`. No `id_token` decode, no `getPayload()` call, and no `verifyIdToken()` anywhere in the file. The column will remain `NULL` for all new OAuth connections.

This is not just a missing feature — if any downstream code reads `accountEmail` expecting a value (e.g., for display or de-duplication), it will silently get `null`.

**Fix:** Extract the email from Google's `id_token` during the callback. The `google-auth-library`'s `OAuth2Client.verifyIdToken` is the safest path since it validates the token signature:
```typescript
// After line 134 (expiresAt assignment), add:
let accountEmail: string | null = null;
if (tokens.id_token) {
  try {
    const ticket = await client.verifyIdToken({
      idToken: tokens.id_token,
      audience: clientId!,
    });
    accountEmail = ticket.getPayload()?.email ?? null;
  } catch {
    // non-fatal: email is optional metadata
  }
}
```
Then pass `accountEmail` through `dbUpsertFn` and persist it in the upsert `set` block:
```typescript
set: {
  encryptedRefreshToken,
  accessToken: storedAccessToken,
  expiresAt: storedExpiresAt,
  scopes,
  accountEmail,
  updatedAt: new Date(),
},
```
Also update the `GoogleAuthDeps.dbUpsertFn` signature and the test mock accordingly.

### WR-03: Conflict-update path does not update `scopes` — re-auth silently drops scope changes

**File:** `vigil-core/src/routes/google-auth.ts:163-171`
**Issue:** The `onConflictDoUpdate` set block (lines 163-170) updates `encryptedRefreshToken`, `accessToken`, `expiresAt`, and `updatedAt`, but **omits `scopes`**. On a re-authentication (existing `google` row), the stored scopes will not be updated to reflect the newly granted scopes. The `calendarSelections` column is also absent from the conflict-update set, though that is likely intentional (preserve user selections).

After a user re-authenticates (e.g., to add Gmail scope), `google-status` will continue reading the old scope set from the database, making it appear as if the new scope was not granted.

**Fix:** Add `scopes` to the conflict-update set:
```typescript
.onConflictDoUpdate({
  target: oauthTokens.provider,
  set: {
    encryptedRefreshToken,
    accessToken: storedAccessToken,
    expiresAt: storedExpiresAt,
    scopes,               // <- add this
    // accountEmail,      // <- add once WR-02 is resolved
    updatedAt: new Date(),
  },
});
```

## Info

### IN-01: Migration default `'[]'::jsonb NOT NULL` is inconsistent with how INSERT uses it

**File:** `vigil-core/drizzle/0008_add_oauth_scopes_and_account_email.sql:1`
**Issue:** The column default (`'[]'::jsonb`) is only useful for rows inserted without explicitly providing `scopes`. The production insert path always provides `scopes: []` explicitly (line 159 of google-auth.ts), so the migration default is never exercised in practice. This is harmless but worth noting as a future reader may rely on the default for backfilling — it is correct for existing rows.

No action required unless WR-01 is fixed (at which point the Drizzle-level `.default([])` will cover the same ground in new migrations).

---

### IN-02: Test GA-04 covers scopes written but no test covers the `account_email` write path

**File:** `vigil-core/src/routes/google-auth.test.ts:91-100`
**Issue:** `GA-04-scopes-stored` correctly verifies that both scopes reach the DB upsert function. There is no parallel test that verifies `account_email` is extracted and passed to `dbUpsertFn`. Once WR-02 is resolved, a test like `GA-09-account-email-stored` should be added that provides a mock `id_token` and asserts the decoded email is passed through.

This is purely a coverage gap, not a test reliability issue.
