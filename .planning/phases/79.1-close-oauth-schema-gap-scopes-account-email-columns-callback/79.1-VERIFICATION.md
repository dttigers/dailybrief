### Gaps Summary

Three gaps prevent full goal achievement. Two are blockers; one is a plan-required behavior that was not ported:

**Gap 1 (Blocker) ‚Äî `openid` + `email` absent from consent scope list:** The `accountEmail` decode and persist code is structurally correct and tests pass against a mock `id_token`. But `REQUESTED_SCOPES` in `google-auth.ts` only contains `calendar.readonly` and `gmail.readonly`. Google requires `openid` (and optionally `email`) in the consent request to include an `id_token` in the token exchange response. Without this, the `tokens.id_token` check at line 143 will never be true in production, and `accountEmail` will always be stored as `null`. Fix: add `'openid'` and `'email'` to `REQUESTED_SCOPES`.

**Gap 2 (Blocker) ‚Äî `email` field absent from `/v1/google/status` response:** Plan 02 required the status endpoint to return `{ calendar, gmail, email }` after a fresh OAuth round-trip. The `google-status.ts` `GoogleStatusResponse` interface only has `{ calendar, gmail }`. The DB query does not select `accountEmail`, the response object never includes it. This is a complete miss, not a stub. Fix: extend the `dbSelectFn` type and DB select to include `accountEmail`, extend `GoogleStatusResponse` to include `email?: string | null`, and include it in the JSON response.

**Gap 3 (Warning) ‚Äî Back-compat branch for empty-scopes legacy rows not implemented:** Plan 02 truth #4 required that `scopes=[]` rows return `{ calendar: 'connected', gmail: 'needs_auth' }`. The new `google-status.ts` returns `{ calendar: 'needs_auth', gmail: 'needs_auth' }` for empty scopes (correct behavior for no-token, but incorrect for the legacy re-auth scenario). Pre-79.1 OAuth rows will show calendar as needing re-auth after this update deploys. This may be intentional (force re-auth to populate scopes), but it deviates from the stated plan requirement. Suggest either adding the back-compat branch or explicitly removing the requirement from the plan.

All three gaps come from the architectural split (calendar-auth.ts ‚Üí google-auth.ts + google-status.ts) landing without completing the google-status.ts feature set to match the plan's specified behavior.
