# Phase 79.1 Plan 02: OAuth Callback + Status Wiring Summary

Wired the OAuth callback in `vigil-core/src/routes/calendar-auth.ts` to persist granted scopes (parsed from `tokens.scope`) and account email (decoded from `id_token`) into the new Wave-1 columns, and simplified `/v1/google/status` to read them directly while keeping a back-compat branch for legacy rows â€” closing the Phase 81 UAT Test 4 Gmail-status gap.

## What Was Built

**Task 1 â€” calendar-auth.ts callback + status wiring (commit `b1d9a9d`):**

Five targeted changes in `vigil-core/src/routes/calendar-auth.ts`:

1. **Extended `Tokens` interface** with `id_token?: string | null` and `scope?: string | null` fields (Google's library surfaces them as optional; they had been untyped before).

2. **Added `openid` + `email` to consent scope list** at `GET /auth/google`. Without these, Google omits the `id_token` response field â€” the only server-to-server source of the authenticated user's email.

3. **Extended `dbUpsertFn` DI signature** with two trailing args: `scopes: string[]` and `accountEmail: string | null`. Append-only ordering preserves existing test-mock call sites (tests that don't override `dbUpsertFn` still work).

4. **Callback success-path derivation** (after `encryptToken`):
   - `grantedScopes` derived from `tokens.scope.split(/\s+/).filter(s => s.length > 0)` â€” typed guard on string, empty array fallback. T-79.1-01 mitigation: scopes come exclusively from Google's server-to-server response, never from query params.
   - `accountEmail` derived by splitting `tokens.id_token` on `.`, base64url-decoding part[1] with `Buffer.from(parts[1]!, "base64url")`, and JSON-parsing the payload to read `email` claim. Wrapped in try/catch so malformed id_tokens degrade to `null` rather than breaking the flow. T-79.1-02 mitigation: email is never read from query params; T-79.1-06 accepted (no sig verify â€” TLS channel is trust anchor).
   - Default `dbUpsertFn` lambda updated to persist both new fields on both `insert` and `onConflictDoUpdate` branches.

5. **`/google/status` default `statusFn` simplified** â€” replaced the `as unknown` cast with a typed `select({ provider, scopes, accountEmail })`. Response-derivation block (calendar/gmail/email) unchanged; back-compat branch for `granted.length === 0` intentionally preserved for legacy pre-79.1 rows that will exist in production until the Wave-1 migration runs and the user re-authorizes.

**T-79.1-03 PII logging check:** No new `console.*` statements reference `grantedScopes`, `accountEmail`, or `tokens.id_token`. Only `err.message` is logged on catch (pre-existing behavior).
