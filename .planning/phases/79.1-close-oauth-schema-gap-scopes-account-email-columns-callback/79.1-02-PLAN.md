```typescript
      // T-01 mitigation: grantedScopes comes from Google's server-to-server token
      // response, never from request params. Google returns `scope` as a
      // space-separated string; split on whitespace (one-or-more) to get the array.
      const grantedScopes: string[] =
        typeof tokens.scope === "string" && tokens.scope.length > 0
          ? tokens.scope.split(/\s+/).filter((s) => s.length > 0)
          : [];

      // T-02 mitigation: account email is decoded from the id_token JWT claim,
      // which Google signs. We do a payload decode (not signature verify) because
      // the id_token arrived over the authenticated server-to-server token
      // exchange — the channel itself is the trust anchor. If id_token is absent
      // (e.g. openid scope not granted), accountEmail stays null (nullable column).
      let accountEmail: string | null = null;
      if (typeof tokens.id_token === "string" && tokens.id_token.length > 0) {
        const parts = tokens.id_token.split(".");
        if (parts.length === 3) {
          try {
            const payloadJson = Buffer.from(parts[1]!, "base64url").toString("utf8");
            const payload = JSON.parse(payloadJson) as { email?: unknown };
            if (typeof payload.email === "string" && payload.email.length > 0) {
              accountEmail = payload.email;
            }
          } catch {
            // Malformed id_token — leave accountEmail null, do not fail the flow.
          }
        }
      }
```

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Google token endpoint -> vigil-core callback | Trusted (server-to-server TLS); tokens.scope and id_token originate here |
| Browser query params -> vigil-core callback | Untrusted; only `code`, `error`, `state` consumed, never email/scopes |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-79.1-01 | Elevation of Privilege (scope escalation) | Callback scope parsing | mitigate | `grantedScopes` is derived exclusively from `tokens.scope` in the token-exchange response (server-to-server). Query params are never read for scope info. A malicious client cannot claim a scope Google did not grant. |
| T-79.1-02 | Spoofing (account_email) | Callback account_email derivation | mitigate | `accountEmail` is decoded from the `id_token` JWT claim returned by Google's token endpoint over the authenticated server-to-server channel. Never sourced from request query params. If id_token is missing/malformed, accountEmail stays null — the flow does not fall back to an untrusted source. |
| T-79.1-03 | Information Disclosure (PII in logs) | Error logging in callback + status | mitigate | Only `err.message` is logged on catch (existing pattern). No new console.log statements reference `grantedScopes`, `accountEmail`, `tokens.id_token`, or the full row. Test asserts absence of new PII log sites via grep. |
| T-79.1-06 | Integrity (id_token signature bypass) | id_token payload decode without signature verification | accept | Trust anchor is the TLS-authenticated token-exchange channel with Google, not the JWT signature. Adding jose/google-auth-library `verifyIdToken()` is a future hardening step (Phase 80+); for v3.1 the server-to-server channel is sufficient per current Google OAuth best-practice guidance for single-tenant personal-use apps. |
