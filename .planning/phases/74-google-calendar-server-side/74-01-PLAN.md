    - Create OAuth2Client with `GOOGLE_CLIENT_ID`, `GOOGLE_CLIENT_SECRET`, `GOOGLE_REDIRECT_URI` env vars
    - Generate a random state nonce: `crypto.randomBytes(16).toString('hex')`

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser -> /v1/auth/google | Untrusted user initiates OAuth redirect |
| Google -> /v1/auth/google/callback | Google sends authorization code via redirect |
| Server -> Google Token Endpoint | Server exchanges code for tokens |
| Server -> PostgreSQL | Encrypted refresh token stored at rest |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-74-01 | Information Disclosure | oauth_tokens table | mitigate | Refresh token encrypted with AES-256-GCM via GOOGLE_TOKEN_ENCRYPTION_KEY env var (D-05) |
| T-74-02 | Information Disclosure | logging | mitigate | Never log access_token, refresh_token, or encryption key — follow sports-service pattern |
| T-74-03 | Tampering | OAuth callback | mitigate | Validate `state` parameter (random nonce, 5-min expiry, one-time use) to prevent CSRF |
| T-74-04 | Spoofing | /v1/auth/google/callback | mitigate | Only accept valid authorization codes from Google; redirect_uri must match registered URI |
| T-74-05 | Information Disclosure | access_token in DB | accept | Short-lived (1h), stored plaintext — encrypting adds complexity for marginal gain |
</threat_model>
