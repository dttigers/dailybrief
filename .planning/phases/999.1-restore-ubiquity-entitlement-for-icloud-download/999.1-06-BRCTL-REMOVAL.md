## Why Deferred (Not Pursued in This Session)

1. **Phase not merged:** `999.1-ubiquity-dry-run` has not been merged to `main`. Running the brctl removal on the dry-run branch would produce a removal commit that either (a) gets squashed/merged along with the main 999.1 changes (conflating bake-clock-start and removal into the same merge — bypassing the entire bake rationale) or (b) remains isolated on a removal branch that then needs a second merge later — adding complexity without benefit.

2. **Bake rationale is sound:** RESEARCH.md explicitly recommends the brctl shim be removed as a *separate* commit so it can be reverted independently if `startDownloadingUbiquitousItem` proves flaky. The 7-day window is designed to catch latent regressions (e.g., edge cases in macOS's fileproviderd/cloudd interaction) before stacking the removal on top. Honoring this window is the prudent engineering choice.

3. **No urgency:** The brctl shim is harmless. Per Plan 05's equivalent-proof rationale, on the operator's Mac (iCloud Drive = "Download Originals to this Mac"), `.icloud` stub placeholder files never appear — meaning `triggerICloudDownloads()` never invokes brctl at all in practice. The brctl block in `processFile` (the hot path) does fork a subprocess, but only when a file arrives with `ubiquitousItemDownloadingStatusKey != .current`. On this Mac configuration, that status is always `.current` for files coming in via Downloads Originals. The empirical subprocess-fork rate is ~0 per day.

4. **On other Mac configurations:** The brctl fork rate equals the per-HEIC-stub arrival rate. This is a single process fork per file — negligible overhead. The removal would be a code simplification, not a performance fix.

## Cost of Leaving brctl in Place

- **One subprocess fork per `.icloud` stub arrival** in `processFile`'s `status != .current` branch.
- **On this Mac (Download Originals):** ~0 stub arrivals per day → ~0 subprocess forks per day. Empirically confirmed by Plan 05 equivalent-proof (brctl log line never appeared in either HEIC drop run).
- **On "Optimize Mac Storage" Macs:** One fork per HEIC stub arrival. This is fine — the fork completes quickly (brctl exits), and the native `startDownloadingUbiquitousItem` call on line 356 runs first regardless (brctl is belt-and-suspenders, not the primary driver).
- **Code smell:** Future readers may not understand why brctl is still present after Phase 999.1 restored the ubiquity entitlement. The comment block on lines 353-355 partially explains it, but Plan 06's comment update would make it clearer.
