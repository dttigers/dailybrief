# Phase 999.1: Restore Ubiquity Entitlement for iCloud Photo Download — Research

**Researched:** 2026-05-19
**Domain:** macOS code signing + Developer ID + iCloud Drive (ubiquity container) entitlements; shell-driven Swift Package Manager release pipeline (no Xcode); LaunchAgent-resident background app
**Confidence:** HIGH on the "what is required" question; MEDIUM on the "does this actually work on a non-Xcode SPM build" question (the entire shipping-app ecosystem assumes Xcode handles the provisioning-profile dance; doing it manually via `codesign --entitlements` + a downloaded `embedded.provisionprofile` is well-documented but the Vigil install pipeline has never exercised it)

## Summary

`com.apple.developer.ubiquity-container-identifiers` is a **restricted entitlement** in Apple terminology — it must be authorized by a provisioning profile embedded in the app bundle at `Contents/embedded.provisionprofile`. This is true even for Developer ID (non-MAS) distribution. The widespread folk-wisdom that "Developer ID apps never need a provisioning profile" is **wrong specifically for iCloud-family entitlements**; it is correct for almost everything else. Apple's own Developer ID support page states verbatim: *"You'll also need to create a Developer ID provisioning profile for apps using advanced capabilities such as CloudKit."* This applies equally to `ubiquity-container-identifiers` because both belong to the same iCloud capability family on developer.apple.com — enabling iCloud at the App-ID level provisions both entitlements together.

The runtime failure mode when the entitlement is *claimed* but *not authorized by a profile* is **process termination by amfid with error -67050** ("Killed because of its use of com.apple.developer.ubiquity-container-identifiers entitlement"). The runtime failure mode when the entitlement is *not claimed at all* — Vigil's current state — is **silent no-op of `startDownloadingUbiquitousItem` and a `nil` return from `url(forUbiquityContainerIdentifier:)`** — exactly what the debug doc empirically observed. So the phase has two failure modes to navigate between: doing nothing (current pain) vs. claiming the entitlement without proper provisioning (worse pain — Monitor would refuse to launch).

**Primary recommendation:** Build this phase as a **5-checkpoint, all-or-nothing dry-run on a throwaway branch** (`phase-999.1-spike`). Do not touch `main` until Checkpoint 5 (Monitor launches under launchd, signature verifies, ubiquity URL is non-nil, brctl no longer fires) passes. The single irreversible-feeling step is one-time App-ID + iCloud-container creation at developer.apple.com — but it's actually fully reversible (disable iCloud capability on the App ID; profile becomes invalid; the next install reverts to empty-entitlements territory). Keep the brctl shim (commit `fff8a15`) in the codebase as belt-and-suspenders for UBIQ-06 deferral. Removing brctl is a separate, trivial follow-on PR that should be its own commit so it can be reverted independently if ubiquity proves flaky.

### Pattern 1: Restricted-Entitlement Sign Flow (the entire phase)

**What:** For each restricted entitlement an app claims, the embedded provisioning profile must list it as authorized. Mismatch → amfid kills the process at launch.
**When to use:** Any iCloud-family entitlement, Apple Pay, Sign In With Apple, Push, Associated Domains. Most other entitlements (App Sandbox, hardened runtime opt-outs, personal-info access like calendars/contacts) are **not** restricted and need no profile.
**Example:**
```bash
# Source: TN3125 + developer.apple.com/support/developer-id
# After building .app bundle and before final codesign:
cp Entitlements/DailyBriefMonitor.provisionprofile \
   "$MONITOR_APP/Contents/embedded.provisionprofile"

codesign --deep --force \
         --sign "$IDENTITY" \
         --options runtime \
         --identifier "com.jamesonmorrill.dailybriefmonitor" \
         --entitlements "$REPO_DIR/Entitlements/DailyBriefMonitor.entitlements" \
         "$MONITOR_APP"

# Verify both the signature AND that the profile is properly embedded:
codesign --verify --verbose "$MONITOR_APP"
codesign -d --entitlements - "$MONITOR_APP" | grep ubiquity
security cms -D -i "$MONITOR_APP/Contents/embedded.provisionprofile" | \
    plutil -p - | grep -A 20 Entitlements
```

### Pattern 2: Entitlements XML for Ubiquity (the actual plist value)

**What:** The entitlements file must list iCloud containers with the `iCloud.` prefix (not the `$(TeamIdentifierPrefix)` macro).
**When to use:** Always for `ubiquity-container-identifiers`. The `$(TeamIdentifierPrefix)` macro is **only used by Xcode** as a build-time substitution; `codesign` invoked directly from a shell script does **not** substitute it [VERIFIED: empirical reports + Apple forum 118773]. Hardcode the literal value.

**Example:**
```xml
<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE plist PUBLIC "-//Apple//DTD PLIST 1.0//EN" "http://www.apple.com/DTDs/PropertyList-1.0.dtd">
<plist version="1.0">
<dict>
    <key>com.apple.developer.ubiquity-container-identifiers</key>
    <array>
        <string>iCloud.com.jamesonmorrill.dailybriefmonitor</string>
    </array>
</dict>
</plist>
```

**Note on `nil` container:** `FileManager.default.url(forUbiquityContainerIdentifier: nil)` returns the URL for the **first** container listed in the entitlement. Since Vigil only has one container, passing `nil` (as the current code does) is correct and will return the iCloud.com.jamesonmorrill... URL [CITED: Apple Foundation docs].

### Pattern 3: Listening for Materialization Without `startDownloadingUbiquitousItem`

**What:** Two complementary patterns currently exist in `FolderWatcherService`:
- `triggerICloudDownloads()` (line 187): scans for `.icloud` hidden placeholder files (e.g., `.IMG_1234.jpg.icloud`)
- `processFile()` ubiquity branch (line 348): handles visible files with status != `.current`

iCloud Drive in modern macOS (since ~Big Sur) emits **visible stubs** much more often than hidden placeholders, so `processFile()` is the hot path. The phase changes nothing about which path runs — only whether the `try? startDownloadingUbiquitousItem` call on line 356 actually does work.

### Anti-Patterns to Avoid

- **Using `$(TeamIdentifierPrefix)` in the entitlements XML when signing via shell**: `codesign` does not expand it; the value lands in the signature literally, which the profile then disagrees with, and amfid kills the app. Hardcode `iCloud.com.jamesonmorrill.dailybriefmonitor`.
- **Leaving the provisioning profile in source control**: it embeds your Team ID and is tied to a single Apple Developer account. Gitignore it; check in a `.example` placeholder with documentation.
- **Embedding the profile AFTER `codesign --deep`**: the outer signature won't cover it; on launch the OS sees a bundle whose contents don't match the seal. Copy the profile in BEFORE `codesign`.
- **Adding `com.apple.developer.icloud-services` "just in case"**: that's the CloudKit entitlement Phase 58 explicitly removed. It is a **restricted entitlement requiring a separate provisioning capability** and adding it without also enabling CloudKit on the App ID puts you right back in Phase 58's launch-failure territory. Vigil does not use CloudKit; do not re-add it.
- **Skipping `codesign --verify --verbose` in install.sh**: the existing script does this (line 210); keep it as a fast-fail. If verification fails, do NOT proceed to `launchctl bootstrap` — you would be loading a binary you already know is broken.
- **Manually editing the downloaded `.provisionprofile`**: it's a signed binary plist (CMS-wrapped). Any byte-edit invalidates it.

## Common Pitfalls

### Pitfall 1: amfid Kills the Process with -67050 on First Launch

**What goes wrong:** `codesign --verify` passes (the *signature* is structurally valid), but at runtime amfid validates that every entitlement in the binary appears in `embedded.provisionprofile`'s allowed-entitlements list. If `ubiquity-container-identifiers` is in the entitlements XML but not in the profile, the process is killed within ~50ms of `posix_spawn`, before `main()` runs.

**Why it happens:** The provisioning profile is the source of truth for which restricted entitlements are *permitted*. Claiming an entitlement is one thing; being allowed to claim it is another. [CITED: TN3125, Apple forum 18443]

**How to avoid:**
- Generate the profile AFTER enabling the iCloud capability on the App ID
- Run `security cms -D -i Entitlements/DailyBriefMonitor.provisionprofile | plutil -p -` and confirm `Entitlements` dict contains `com.apple.developer.ubiquity-container-identifiers`
- Run a test launch from Terminal first (`open ~/.local/bin/DailyBriefMonitor.app`) BEFORE letting launchd own the process — Terminal surfaces the killed-by-amfid message in `Console.app` immediately

**Warning signs:** `launchctl print gui/$UID/com.jamesonmorrill.dailybriefmonitor` shows `last exit code = 9` or `= -1` and `state = not running` immediately after `launchctl bootstrap`. `Console.app` filter on `amfid` shows "Killed: 9 (entitlement) ... -67050".

### Pitfall 2: TeamIdentifierPrefix Macro Doesn't Expand

**What goes wrong:** Entitlements XML contains `$(TeamIdentifierPrefix).com.jamesonmorrill.dailybriefmonitor`; install.sh signs; codesign embeds the literal string `$(TeamIdentifierPrefix).com.jamesonmorrill.dailybriefmonitor` (not `5H57ADQS8G.com.jamesonmorrill.dailybriefmonitor`) in the signature. The provisioning profile authorizes `5H57ADQS8G.com.jamesonmorrill.dailybriefmonitor`. They don't match → amfid kill, see Pitfall 1.

**Why it happens:** `$(TeamIdentifierPrefix)` is an Xcode build-setting variable expanded at Xcode build time. The standalone `codesign` tool, invoked by a shell script, treats it as a literal string. [VERIFIED: multiple developer reports; Apple forum thread 118773]

**How to avoid:** Hardcode the iCloud container identifier as `iCloud.com.jamesonmorrill.dailybriefmonitor`. The container identifier is **not** team-prefixed at the entitlement layer — that prefix is added when the Apple Developer Portal generates the *fully-qualified* identifier behind the scenes. The entitlement XML uses the friendly `iCloud.*` form.

**Warning signs:** `codesign -d --entitlements - <app>` shows a literal `$(...)` in the output.

### Pitfall 3: Profile Embedded AFTER codesign

**What goes wrong:** Profile is copied to `Contents/embedded.provisionprofile` after `codesign --deep` runs. On the next launch macOS sees a sealed bundle plus a file that wasn't part of the seal → `codesign --verify` fails OR Gatekeeper refuses to launch.

**Why it happens:** `codesign --deep` computes a SHA-256 over every file in the bundle. Any post-sign addition breaks the seal.

**How to avoid:** Copy the profile **before** the outer `codesign --deep` call in `install.sh`. Specifically: insert the `cp` between the existing Safari .appex embedding block (ends line 197) and the `codesign --deep` block (starts line 204).

**Warning signs:** `codesign --verify --verbose "$MONITOR_APP"` reports "a sealed resource is missing or invalid" or "resource added".

### Pitfall 4: brctl Race After Adding Ubiquity

**What goes wrong:** Native `startDownloadingUbiquitousItem` now actually does something, AND the brctl shim still fires. brctl `exit 0` happens almost instantly because the file is already in `.downloading` state from the native call. Probably benign, but produces a noisy log line that obscures whether the native call did the work or brctl did.

**Why it happens:** Both code paths still exist in `FolderWatcherService.swift:356-377` after the entitlement lands. The brctl shim was specifically added to compensate for the native API's silent no-op.

**How to avoid:**
- During UBIQ-03 acceptance, **temporarily comment out the brctl block** to prove `startDownloadingUbiquitousItem` is the actual driver. If the file still materializes, native is working.
- For the optional UBIQ-06 cleanup, **delete** the brctl block entirely. Don't leave it commented out — that confuses future readers.
- If you decide to keep brctl as belt-and-suspenders, change its log line to "FolderWatcherService: brctl shim (post-native): %@" so logs distinguish.

**Warning signs:** Both `dispatched brctl download for ...` AND a successful native materialization log in the same second. Materialization works but you can't tell which code path won the race.

### Pitfall 5: Cert Identifier vs Profile Mismatch

**What goes wrong:** `install.sh` line 207 sets `--identifier "com.jamesonmorrill.dailybriefmonitor"`. The provisioning profile is generated against the same App ID. Good. BUT if you accidentally pick a *different* App ID at portal step 4 (e.g., a typoed copy), the profile's bound App ID won't match the codesign identifier and you'll get a mismatch.

**Why it happens:** Apple Developer Portal lets you create as many App IDs as you want; getting the wrong one is easy.

**How to avoid:** When downloading the profile (portal step 4), open it in Quick Look or run `security cms -D -i <file>` immediately and verify the `Name` field and `Entitlements.application-identifier` match `5H57ADQS8G.com.jamesonmorrill.dailybriefmonitor` exactly. Add this as a sanity check at the top of `install.sh` (see Validation Architecture below).

**Warning signs:** `codesign --verify --verbose` says "entitlements not authorized by profile" or amfid kills with -67050 even though the entitlement XML looks correct.

### Pitfall 6: Bare-Binary Copy at $INSTALL_DIR/DailyBriefMonitor

**What goes wrong:** install.sh line 109 copies the bare binary to `$INSTALL_DIR/DailyBriefMonitor` (not the .app bundle). Line 213-218 signs it. But there's no `Contents/embedded.provisionprofile` next to a bare binary. If anything ever launches that bare binary, amfid kills it with -67050.

**Why it happens:** Comment on line 108 says "Keep the bare binary for backward compat (UpdateService, bootstrap.sh)". So it's deliberately there. But it can't carry the ubiquity entitlement.

**How to avoid:** Three options, pick one in the plan:
1. Keep the bare binary but **sign it with a separate, empty entitlements file** (no ubiquity). Then UpdateService and bootstrap.sh continue to work, and the LaunchAgent runs the .app variant which has ubiquity.
2. Delete the bare binary entirely; audit `UpdateService.installBuiltBinaries()` and `Scripts/bootstrap.sh` to use the .app path.
3. Sign the bare binary with the same entitlements + place `embedded.provisionprofile` next to it as `DailyBriefMonitor.provisionprofile` — but the OS only looks for the profile at the `.app/Contents/` location, so this WILL NOT WORK for a bare binary; the bare binary cannot use restricted entitlements at all.

Recommendation: option 1 (split entitlements files) is least disruptive. Create `Entitlements/DailyBriefMonitor-bare.entitlements` (empty dict) and use it for the bare-binary signing on line 217. Document this.

**Warning signs:** `UpdateService` triggers a relaunch via the bare binary instead of the .app and the menu bar item silently disappears (process killed at launch).

### Pitfall 7: Profile in Git

**What goes wrong:** `.provisionprofile` checked into the public repo. It's not super-sensitive (it's a signed identifier of an App ID), but it embeds the Team ID and is bound to one Apple account, making the repo confusingly entangled with one developer's identity.

**Why it happens:** Easy to forget to gitignore a new file in `Entitlements/` when that directory already has a tracked `.entitlements` file.

**How to avoid:**
- Add `Entitlements/*.provisionprofile` to `.gitignore` in the first plan
- Check in `Entitlements/DailyBriefMonitor.provisionprofile.example.txt` with placeholder instructions
- `install.sh` should `[[ -f Entitlements/DailyBriefMonitor.provisionprofile ]] || { echo "ERROR: profile missing — see README"; exit 1 }` early

**Warning signs:** `git status` shows the new file as untracked; `cat .gitignore | grep provisionprofile` returns empty.

```bash
# Sanity-check the profile: must be a CMS-signed plist, must not be expired,
# must list the ubiquity entitlement.
security cms -D -i "$PROFILE_SRC" > /tmp/profile.plist 2>&1 \
    || { echo "ERROR: profile is not a valid CMS file." >&2; exit 1; }

if ! /usr/libexec/PlistBuddy -c "Print :Entitlements:com.apple.developer.ubiquity-container-identifiers" \
        /tmp/profile.plist >/dev/null 2>&1; then
    echo "ERROR: profile does not authorize ubiquity-container-identifiers." >&2
    exit 1
fi

# Embed into the .app bundle
cp -f "$PROFILE_SRC" "$MONITOR_APP/Contents/embedded.provisionprofile"
echo "  Embedded provisioning profile."
```

## Open Questions

1. **Will enabling iCloud on the existing App ID `com.jamesonmorrill.dailybriefmonitor` invalidate the notarization "App ID + Developer ID cert" binding that the brctl-only build relies on today?**
   - What we know: Apple docs (`/help/account/identifiers/enable-app-capabilities/`) state "Provisioning profiles that contain a modified App ID become invalid. You'll need to regenerate the provisioning profiles that use that App ID." But Vigil currently uses NO provisioning profile (it's a Developer-ID-signed app with no profile at all), so there is nothing to invalidate.
   - What's unclear: Whether enabling iCloud as a capability on the App ID has any retroactive side-effect on notarization tickets already issued for current builds.
   - Recommendation: Treat this as a CHECKPOINT in the plan. After step 3 of the portal flow, before generating the new profile, reinstall the **current main-branch** build (no ubiquity yet) and confirm it still launches cleanly. If it doesn't, you've discovered a regression that's actually pre-existing in App ID config. Roll back: uncheck iCloud capability on the App ID. The plan can include this as a "App-ID capability toggle is reversible" verification.

2. **Does the bare-binary copy at `$INSTALL_DIR/DailyBriefMonitor` (install.sh line 109) get launched by anything?**
   - What we know: Comment on line 108 says "for backward compat (UpdateService, bootstrap.sh)". UpdateService is mentioned in Phase 51-04-PLAN notes. bootstrap.sh exists somewhere in `Scripts/`.
   - What's unclear: Whether either still calls the bare-binary path in 2026, or whether they've already migrated to the .app path.
   - Recommendation: Have the planner add a sub-task: `grep -rn "DailyBriefMonitor" Sources/ Scripts/ | grep -v ".app"` to enumerate every bare-binary reference. Then decide on Pitfall 6's three options.

3. **Will `launchctl bootstrap` race the new code-signature evaluation, in the same way that Phase 134's `launchctl bootout`/`bootstrap` sometimes hits "Input/output error" on re-load?**
   - What we know: The debug doc 2026-04-21T19:40Z entry shows "Bootstrap failed: 5: Input/output error" once on the brctl install, succeeded on retry. Adding a profile to a previously-profile-less bundle changes the codesign payload size and might amplify this race.
   - What's unclear: Whether amfid takes measurably longer to validate a profile-bearing bundle the first time.
   - Recommendation: In the install.sh modification, add a `sleep 1` after `launchctl bootout` and before `launchctl bootstrap` (currently no sleep — line 281-282 are back-to-back). This is a pure safety net; cost is negligible.

4. **Is the iCloud container identifier `iCloud.com.jamesonmorrill.dailybriefmonitor` available at developer.apple.com, or has someone else (perhaps a copy of "DailyBrief" or "Vigil" app) already claimed it under a different team?**
   - What we know: iCloud container identifiers are globally unique across all Apple developer accounts. They're also visible at `/help/account/identifiers/create-an-icloud-container/` only if you own them.
   - What's unclear: Whether the obvious identifier choice is free for Jameson Morrill's team.
   - Recommendation: As the very first task of the spike, log in to portal and try to register the container. If it's taken, choose `iCloud.com.jamesonmorrill.vigil` or similar. The container identifier and the App ID can have different names.
