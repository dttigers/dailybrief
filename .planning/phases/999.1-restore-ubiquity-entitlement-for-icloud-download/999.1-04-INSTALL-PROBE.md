## UpdateService Re-Sign Cross-Check (Settles Checker Warning #8)

**Source read:** `Sources/DailyBriefMonitor/UpdateService.swift` lines 145-200.

`UpdateService.installBuiltBinaries()` COPIES the bare binaries from `.build/release/` via
`FileManager.default.copyItem` (line 166). It does NOT invoke `codesign` at any point in the
function. Specifically:
- Line 148 enumerates the binary names `["DailyBrief", "DailyBriefMonitor"]`
- Lines 149-150 build `src` and `dst` paths under `.build/release/` and `$INSTALL_DIR`
- Line 166 calls `FileManager.default.copyItem(atPath: src, toPath: dst)` — copy only, no signing

**Therefore:** An Update-click does NOT re-sign the bare binary. The installed bare binary
(`~/.local/bin/DailyBriefMonitor`) inherits whatever signature `install.sh` stamped onto the
`.build/release/DailyBriefMonitor` copy. Since `install.sh` now signs the bare binary with
`DailyBriefMonitor-bare.entitlements` (empty dict, per Plan 03 Edit 4), every Update-click
preserves the empty entitlements on the bare binary.

**No risk:** UpdateService cannot mutate the bare binary's signature to accidentally claim ubiquity
(which would cause amfid -67050 kills since there is no `embedded.provisionprofile` neighbor at
`$INSTALL_DIR/DailyBriefMonitor`).

This eliminates checker warning #8 (option a: Update-click-trigger task). The question is settled
by source-read of UpdateService.swift:145-200; no Update-click trial is required.

Plan 04's checkpoint loop surfaced four issues that Plan 03 did not anticipate. All four were

### Deviation 1 — Cross-Machine Sync Failure / Git Object Store Corruption (Mac-Side)

The operator's working repo at `~/Desktop/Local AI/dailybrief` ended up with a git object database

phase. All subsequent install and probe work was performed from the fresh clone.

**Impact:** No code changes required. Mac-side state was fully consistent after re-clone.

### Deviation 2 — Entitlement Format Hotfix #1 (commit `a6acd3d`) — Team Prefix Required for amfid Wildcard Match

**Issue:** Plan 03 used the friendly container ID form `iCloud.com.jamesonmorrill.dailybriefmonitor`
per RESEARCH.md Pitfall 2 guidance. At runtime, amfid returned `-413` "No matching profile found".

**Root cause:** The Developer ID provisioning profile authorizes a team-scoped wildcard
`5H57ADQS8G.*` in its `ubiquity-container-identifiers` entitlement list. The friendly form
`iCloud.com.jamesonmorrill.dailybriefmonitor` does NOT begin with `5H57ADQS8G.`, so it does not
match the wildcard — amfid rejects the app.

**Fix:** Changed entitlement in `Entitlements/DailyBriefMonitor.entitlements` to the manually-expanded
literal `5H57ADQS8G.iCloud.com.jamesonmorrill.dailybriefmonitor`.

**Note on RESEARCH.md:** Pitfall 2 says "use friendly form (no TeamIdentifierPrefix macro)".
This is partially correct — the TeamIdentifierPrefix MACRO must not appear literally in the
entitlement XML (the shell `codesign` call does not expand Xcode build-setting macros). But for
Developer ID profiles, the team prefix must be present as a literal in the entitlement value.
Pitfall 2 should be split into:
- **Pitfall 2a (unchanged):** `$(TeamIdentifierPrefix)` macro does not expand — write a literal
- **Pitfall 2b (new):** For Developer ID profiles, the literal must include the team prefix
  (`5H57ADQS8G.iCloud...`), not the friendly form (`iCloud...`)

### Deviation 3 — Entitlement + Info.plist Hotfix #2 (commit `10e4fd9`) — bird Requires NSUbiquitousContainers in Info.plist

**Issue:** With the team-prefixed entitlement (Deviation 2 fix), amfid passed but bird/cloudd
ignored the app entirely: no entry in System Settings → iCloud → Apps Syncing to iCloud Drive;
no `iCloud~com~jamesonmorrill~dailybriefmonitor` container directory appeared in
`~/Library/Mobile Documents/`; `url(forUbiquityContainerIdentifier:)` returned nil.

**Intermediate experiment (commit `001a892`):** Listing BOTH the friendly form AND the team-prefixed
form in the entitlement array. This re-broke amfid: its policy is that ALL array entries must be
authorized by the profile's allow-list. The friendly form (`iCloud.com.jamesonmorrill.dailybriefmonitor`)
is NOT in the profile's allow-list, so amfid rejected the app again.

**Final fix (commit `10e4fd9`):** Keep entitlement team-prefixed-only (for amfid). Add
`NSUbiquitousContainers` dict to `install.sh`'s generated `Contents/Info.plist` (for bird/cloudd),
using the FRIENDLY container ID (`iCloud.com.jamesonmorrill.dailybriefmonitor`) as the dict key,
with fields: `NSUbiquitousContainerIsDocumentScopePublic = true`,
`NSUbiquitousContainerName = DailyBriefMonitor`,
`NSUbiquitousContainerSupportedFolderLevels = Any`.

After this fix, bird provisioned the container directory immediately on the next install.

**Canonical Apple pattern (Developer ID + iCloud Drive):**
- **Entitlement (signing/amfid channel):** team-prefixed literal `5H57ADQS8G.iCloud.com...`
- **Info.plist NSUbiquitousContainers (bird/cloudd channel):** friendly form `iCloud.com...` as the dict key

### Deviation 4 — Probe Script Bugs (commits `6fe657a` + `6e6b5af`)

Two bugs discovered in `verify-ubiquity.sh` during probe runs:

**Bug 4a (commit `6fe657a`) — UBIQ-04b Gatekeeper check downgraded from FAIL to advisory WARN.**
`spctl -a -vv` checks Apple notarization, which is orthogonal to the ubiquity entitlement chain.
Notarization would fail on `main`'s build as well (Vigil uses a LaunchAgent deployment model that
bypasses Gatekeeper for daemon launch). Downgraded to advisory so the probe correctly reflects the
entitlement chain status without being blocked by an unrelated notarization deferral.

**Bug 4b (commit `6e6b5af`) — UBIQ-04 `codesign --verify --verbose` SIGPIPE under `set -euo pipefail`.**
The original check used `codesign --verify --verbose ... | grep -q 'valid on disk'`.
Under `set -euo pipefail`: `grep -q` exits after the first match; codesign's write to the now-closed
pipe returns SIGPIPE; pipefail propagates non-zero; the `if !` inverted check fired the FAIL branch.
Fix: capture codesign output into a shell variable first, then grep against the variable.
