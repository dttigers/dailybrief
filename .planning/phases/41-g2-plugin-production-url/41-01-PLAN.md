<objective>
Make the G2 plugin's API URL and API key configurable via Vite environment variables, add Bearer token authentication to all API requests, and package the plugin as .ehpk for Even Hub distribution.

Purpose: The G2 plugin currently hardcodes `http://localhost:3001/v1` and sends no auth headers. Production API at Railway requires both the correct URL and a valid Bearer token. This plan makes the plugin production-ready.
Output: Configurable API client with auth, production .env, and .ehpk package file.
</objective>

<task type="auto">
  <name>Task 1: Add configurable API URL and Bearer auth to G2 plugin</name>
  <files>vigil-g2-plugin/src/api.ts, vigil-g2-plugin/.env, vigil-g2-plugin/.env.production</files>
  <action>
1. Create `vigil-g2-plugin/.env` for local development:
   ```
   VITE_API_URL=http://localhost:3001/v1
   VITE_API_KEY=
   ```

2. Create `vigil-g2-plugin/.env.production` for production builds:
   ```
   VITE_API_URL=https://vigil-core-production.up.railway.app/v1
   VITE_API_KEY=
   ```
   Note: The actual API key value must be filled in by the user before building (it's a secret — `vk_` prefixed key from Phase 38).

3. Update `vigil-g2-plugin/src/api.ts`:
   - Replace hardcoded `const BASE_URL = 'http://localhost:3001/v1'` with `const BASE_URL = import.meta.env.VITE_API_URL || 'http://localhost:3001/v1'`
   - Add `const API_KEY = import.meta.env.VITE_API_KEY || ''`
   - Create a helper function `authHeaders()` that returns `{ 'Authorization': \`Bearer \${API_KEY}\`, 'Content-Type': 'application/json' }` when API_KEY is set, or just `{ 'Content-Type': 'application/json' }` when empty (local dev without auth)
   - Update all three fetch calls (fetchSummary, fetchBrief, fetchAffirmation) to include `headers: authHeaders()` in fetch options
   - fetchAffirmation already has Content-Type in body — ensure headers merge correctly

4. Add `vigil-g2-plugin/.env*.production` and `vigil-g2-plugin/.env` to .gitignore if not already covered (check existing .gitignore patterns — `*.env` or `.env` may already be there).

Do NOT add a Vite env.d.ts file — keep it simple. The `import.meta.env` typing works out of the box with Vite's client types.
  </action>
  <verify>
cd vigil-g2-plugin && npm run build succeeds without errors. Inspect dist/assets/*.js to confirm BASE_URL is not hardcoded to localhost (should show the env var value or fallback).
  </verify>
  <done>api.ts reads URL and key from Vite env vars with localhost fallback. Auth headers included in all API calls when key is set. .env files created for dev and production.</done>
</task>

<task type="auto">
  <name>Task 2: Add pack script and build production .ehpk</name>
  <files>vigil-g2-plugin/package.json</files>
  <action>
1. Add scripts to `vigil-g2-plugin/package.json`:
   - `"build:prod": "tsc && vite build --mode production"` — builds with .env.production values
   - `"pack": "evenhub pack app.json dist"` — packages dist/ into .ehpk
   - `"release": "npm run build:prod && npm run pack"` — one-command build + pack

2. Add `*.ehpk` to `vigil-g2-plugin/.gitignore` (create the file if it doesn't exist) so packaged binaries aren't committed.

3. Run `cd vigil-g2-plugin && npm run build:prod` to verify the production build succeeds.

4. Run `cd vigil-g2-plugin && npm run pack` to verify .ehpk is created.

Note: The .ehpk will have a placeholder API key (empty string) since .env.production has `VITE_API_KEY=`. This is expected — the user fills in their key before running `npm run release` for real distribution. The build will still succeed and the .ehpk will be structurally valid.
  </action>
  <verify>
- `ls vigil-g2-plugin/*.ehpk` shows the packaged file exists
- `cd vigil-g2-plugin && npm run build:prod` exits 0
- package.json contains build:prod, pack, and release scripts
  </verify>
  <done>Package scripts added. Production build succeeds. .ehpk file is generated by pack command. .ehpk files gitignored.</done>
