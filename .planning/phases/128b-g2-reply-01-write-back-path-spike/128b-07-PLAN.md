C-1 runbook outline (operator-driven; the plan documents the steps, operator executes):

  1. Open a new shell. Verify environment:
     - `which tmux` (must exist; expected /usr/bin/tmux or equivalent)
     - `tmux -V` (expected 3.4 or compatible)
     - `which claude` (expected ~/.local/bin/claude on Linux; /opt/homebrew/bin/claude or similar on macOS)
     - `claude --version` (expected 2.1.141 or current; record actual)
     - `echo "$CLAUDE_SESSION_ID"` (expected empty unless operator is currently inside a claude session — that's the D-T5 trigger to use a NEW tmux pane below)

  2. Start a disposable tmux session for the live test:
     - `tmux new-session -d -s vigil-claude-c1-spike-$(date +%s) "claude --model haiku"` (or another model per operator preference; haiku is the cheapest)
     - Note the session name (e.g., `vigil-claude-c1-spike-1715712345`)
     - `tmux attach -t <session-name>` (operator drives the claude session interactively)

  3. Drive claude to a `needs_input`-like pause:
     - Ask claude to run a tool that requires permission (e.g., "Run the bash command `echo hello > /tmp/c1-test.out`")
     - Claude will surface a Bash permission dialog (`needs_input` pause) — DO NOT respond inside the pane

  4. In a SEPARATE shell (still on the same workstation), run ONE of the per-path probes against the live session. Recommended: use the PATH E REGRESSION SCRIPT against this live session (the autonomous-plan probes use their own disposable sessions; C-1 confirms Path E works against the operator's-style live session too):
     - `tmux send-keys -t vigil-claude-c1-spike-<timestamp> Enter` — this dismisses the permission dialog the operator just triggered
     - Observe in the original pane (the one running claude): does the gated tool run? Does claude continue?

  5. Verify the round-trip:
     - The Bash tool should have written `/tmp/c1-test.out` containing `hello` (proof that permission was granted via the injected Enter)
     - 60 seconds later, ask claude another question (e.g., "What is 13 multiplied by 17?") — expect `221` response within ~3 seconds
     - This mirrors the spike 001 L4 round-trip + 60s health probe pattern

  6. Cleanup:
     - `tmux kill-session -t vigil-claude-c1-spike-<timestamp>`
     - `rm /tmp/c1-test.out`

  7. Record observations in `.planning/phases/128b-g2-reply-01-write-back-path-spike/128b-MEASUREMENTS.md` `## Operator C-1 wallclock` appendix AND in `128b-07-SUMMARY.md`. Use the redaction-safe variable names per D-G1 if any are needed (typically the appendix is plain prose + the tmux session name).
