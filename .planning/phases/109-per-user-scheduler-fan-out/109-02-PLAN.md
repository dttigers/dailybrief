<objective>
Scope the `/prioritize` route's filesystem cache key by authenticated userId,
so two users with overlapping caseNumbers cannot share a cache line. Closes
SCHED-01 success criterion #3.

**Purpose:** `prioritize.ts::getCacheKey()` currently returns
`wo-priority-${today}-${hash}.json` — the hash derives from sorted caseNumbers
alone. If User A and User B both prioritize a work order list that shares
`caseNumber: "CS0353601"`, the cache collides and one user's AI-ranked result
leaks into the other user's response. This is a real cross-user data leak
(CIA-I: integrity and confidentiality). Phase 109 scopes the cache-key by
the authenticated userId — `c.get("userId")` is guaranteed non-null because
`/v1/prioritize` is registered after the global `bearerAuth` dispatcher at
`index.ts:151`.

**Output:** Modified `prioritize.ts` with `getCacheKey(userId, workOrders)`
signature + `wo-priority-${userId}-${today}-${hash}.json` filename; new
`prioritize.test.ts` with a cache-key isolation unit test (CACHE-01).

**Scope boundary:** Scheduler fan-out is Plan 01. Calendar fold-in is Plan 03.
This plan touches only `prioritize.ts` and a new test file.

**Migration note (D-10):** No sweep for old unscoped cache files. The existing
filename already embeds `today` (server TZ midnight), so pre-migration files
become unreachable naturally at the first midnight rollover after deploy.
Zero startup branching, zero route-level cleanup.
</objective>

// Run with: npx tsx --test src/routes/prioritize.test.ts

import { test } from "node:test";
import assert from "node:assert/strict";
import { getCacheKey, type WorkOrder } from "./prioritize.js";

// ── Fixture ─────────────────────────────────────────────────────────────────

function makeWorkOrders(): WorkOrder[] {
  return [
    {
      caseNumber: "CS0353601",
      store: "Store 1234",
      shortDescription: "HVAC not cooling",
      trade: "HVAC",
      location: "Back room",
      equipment: "Rooftop unit",
      priority: "High",
      contact: "John Doe",
      state: "Open",
    },
    {
      caseNumber: "CS0353598",
      store: "Store 5678",
      shortDescription: "Leaking faucet",
      trade: "Plumbing",
      location: "Men's restroom",
      equipment: "Faucet",
      priority: "Medium",
      contact: "Jane Roe",
      state: "Open",
    },
  ];
}

// ── CACHE-01: cross-user isolation (D-15 literal) ──────────────────────────

test("CACHE-01: getCacheKey yields different filenames for different userIds with identical workOrders", () => {
  const wo = makeWorkOrders();
  const keyForUser1 = getCacheKey(1, wo);
  const keyForUser2 = getCacheKey(2, wo);
  assert.notEqual(
    keyForUser1,
    keyForUser2,
    `cache filenames must differ by userId; got identical key ${keyForUser1} for both users`,
  );
  // Additional belt-and-suspenders: userId appears literally in the filename.
  assert.ok(keyForUser1.includes("-1-"), `user 1 cache key must contain '-1-'; got ${keyForUser1}`);
  assert.ok(keyForUser2.includes("-2-"), `user 2 cache key must contain '-2-'; got ${keyForUser2}`);
});

// ── CACHE-02: filename shape matches D-08 pattern ──────────────────────────

test("CACHE-02: getCacheKey filename matches the wo-priority-{userId}-{YYYY-MM-DD}-{hex}.json pattern", () => {
  const key = getCacheKey(1, makeWorkOrders());
  // Pattern: wo-priority-<userId:digits>-<YYYY-MM-DD>-<32-hex>.json
  const pattern = /^wo-priority-\d+-\d{4}-\d{2}-\d{2}-[a-f0-9]{32}\.json$/;
  assert.match(key, pattern, `cache filename shape mismatch; got ${key}`);
  assert.ok(key.startsWith("wo-priority-1-"), `filename must start with 'wo-priority-1-'; got ${key}`);
  assert.ok(key.endsWith(".json"), `filename must end with '.json'; got ${key}`);
});

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| /prioritize request → cache filesystem | Two users with overlapping caseNumbers must not share a cache line. Before Phase 109: they did. After Phase 109: filename includes userId, so cache reads/writes are partitioned per user. |
| c.get("userId") → getCacheKey | userId comes from bearerAuth middleware (index.ts:105-122, registered globally at line 151 before route registration at line 151). Any future refactor that removes or reorders bearerAuth would cause `c.get("userId")` to return undefined → the filename becomes `wo-priority-undefined-…` which would be one cache line shared across ALL anon requests. This is a latent regression risk. |
| cache file on disk → userId leak | The userId appears literally in the filename on the server's local disk. On Railway, the filesystem is ephemeral and not publicly readable — userId is already non-secret (visible in every JWT claim, every API response header). No privacy regression. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-109-P-01 | Information Disclosure | cross-user cache line sharing | mitigate | Filename now contains userId. CACHE-01 test asserts `getCacheKey(1, wo) !== getCacheKey(2, wo)` for identical work orders. This is the primary phase threat and the mitigation is the whole point of the plan. |
| T-109-P-02 | Tampering | auth-wiring regression → `undefined` userId in filename | mitigate (partial) | CONTEXT §"Unauthenticated /prioritize writes" called out this gap. Runtime defence-in-depth (explicit `if (!userId) return 401`) is deferred per D-09 rationale — `bearerAuth` is a global dispatcher and a regression there would break far more than /prioritize. Instead: CACHE-01 regex in CACHE-02 asserts `/^wo-priority-\d+-/` — the `\d+` requires the userId segment to be digits; if userId ever became `undefined`, the filename would be `wo-priority-undefined-…` and CACHE-02 would fail in CI. This is the load-bearing regression guard. |
| T-109-P-03 | Information Disclosure | logging userId in error paths | accept | The existing 500-error path returns `err.message` only (line 107); no new logging added. userId is already non-secret. |
| T-109-P-04 | Denial of Service | unbounded cache file growth | accept | Pre-existing. Filenames include `today`, so day N's files become unreachable at TZ midnight. A future cron-sweep is an orthogonal concern (out of scope per D-10). |
| T-109-P-05 | Spoofing | /prioritize without bearer token | mitigate | Already mitigated by the global bearerAuth dispatcher at index.ts:151 (registered BEFORE `app.route("/v1", prioritize)`). Not a Phase 109 regression. |
