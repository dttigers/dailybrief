  <files>Sources/DailyBriefMonitor/Settings/SettingsView.swift, Sources/DailyBriefMonitor/Settings/SettingsViewModel.swift</files>
  <action>
  **SettingsViewModel changes:**
  Add new published properties:
  - `emailAuthType: String` (bound to config.email.authType, default "app_password")
  - `oauth2ClientId: String` (bound to config.email.oauth2ClientId)
  - `oauth2TenantId: String` (bound to config.email.oauth2TenantId)
  - `oauth2RefreshToken: String` (bound to config.email.oauth2RefreshToken — display as masked/readonly indicator, not editable directly)

  Update `init()` to load these from config. Update `save()` to persist them back.

  **SettingsView changes (Email / IMAP tab):**
  Add an "Authentication" section between Connection and Credentials:

  ```
  Section("Authentication") {
      Picker("Auth Type", selection: $viewModel.emailAuthType) {
          Text("App Password").tag("app_password")
          Text("OAuth2 (Microsoft 365)").tag("oauth2")
      }
      .pickerStyle(.segmented)
  }
  ```

  Conditionally show fields based on authType:
  - If "app_password": show existing "IMAP Password" SecureField (current behavior)
  - If "oauth2": show:
    - TextField "Client ID" bound to oauth2ClientId
    - TextField "Tenant ID" bound to oauth2TenantId
    - HStack with Text showing "Refresh Token: ✓ Configured" or "✗ Not configured" based on whether oauth2RefreshToken is non-empty
    - Text hint: "Run `dailybrief email-auth` in terminal to authenticate"

  Keep all other existing fields (IMAP Host, Port, TLS, Email Address, Search Pattern, Lookback Days) visible regardless of auth type — they apply to both.

  Do NOT add an "Authenticate" button that launches the device code flow from the Settings UI — the CLI command handles this. The Settings UI is for configuration, the CLI handles the interactive auth flow.
  </action>
