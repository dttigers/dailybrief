# Phase 127.5 — G2 Input Gesture Audit

**Date:** 2026-05-12
**Requirement:** AUDIT-G2-INPUT-01
**Auditor scope:** 30-min code review of vigil-g2-plugin/src/screens/companion.ts + main.ts + navigation.ts + work-orders.ts

## Verdict

**Verdict:** REACTIVATE

Single-press on G2 hardware is currently dropped by FOUR independent guard bugs in `vigil-g2-plugin/src/main.ts:221-260`, every one of which matches the EVEN-SKILLS.md:38 protobuf-zero-omission prediction verbatim. The plumbing is NOT correct; the `?? 0` nullish-coalesce guard is missing on every branch that would route a single-press. Commit `bfa3e0b3` ("single-press reactivated") only modified `.planning/PROJECT.md` and `.planning/research/EVEN-SKILLS.md` — it did NOT touch `vigil-g2-plugin/src/`, so the on-hardware code path is unchanged from Phase 124 D-08's pre-research state. Phase 133 G2-ACTION-01/02 and G2-REPLY-02/03 CAN ship on single-press once a ≤10-line patch lands in `main.ts` + `navigation.ts`. The patch itself is deferred to the Phase 133 implementer because the UX semantics of "what does CLICK_EVENT mean on each screen" is the gesture-grammar decision Phase 133 owns, not this audit.

## Code-level trace (question a)

Four branches in `vigil-g2-plugin/src/main.ts:221-260` decide whether an event reaches a handler. Each one is broken on hardware in a way consistent with EVEN-SKILLS.md:38 ("Any field with a zero/default value ... will be `undefined`, not the zero value").

**NAV_EVENTS membership at `main.ts:66-70`:**

```typescript
const NAV_EVENTS = new Set([
  OsEventTypeList.SCROLL_TOP_EVENT,      // 1
  OsEventTypeList.SCROLL_BOTTOM_EVENT,   // 2
  OsEventTypeList.DOUBLE_CLICK_EVENT,    // 3
])
```

`CLICK_EVENT (= 0)` is absent from the set. Even if every guard below were correctly nullish-coalesced, no NAV branch could fire for single-press because the membership check at `main.ts:233`, `main.ts:239`, `main.ts:247` would return `false`.

**Branch 1 — list-item CLICK_EVENT at `main.ts:224-228`:**

```typescript
if (
  event.listEvent?.eventType === OsEventTypeList.CLICK_EVENT &&
  event.listEvent.currentSelectItemIndex != null &&
  bridge
) {
```

The strict `=== OsEventTypeList.CLICK_EVENT` (`=== 0`) comparison rejects `eventType === undefined`. Per EVEN-SKILLS.md:38, real G2 hardware delivers `eventType` as `undefined` for value-0 events, so this branch is dead-on-hardware. The secondary guard `currentSelectItemIndex != null` is similarly broken: per EVEN-SKILLS.md:38 the first item arrives as `undefined`, so item-0 taps would also fail even if `eventType` somehow squeaked through. This is the exact failure mode the `project_g2_tap_expand_broken` memory recorded for Phase 45's list-item taps ("v0.7.3 sim revealed eventType is undefined on real listEvents").

**Branch 2 — list NAV at `main.ts:233`:**

```typescript
if (event.listEvent?.eventType && NAV_EVENTS.has(event.listEvent.eventType) && bridge) {
```

The truthy `event.listEvent?.eventType` short-circuits when `eventType === 0` (CLICK_EVENT's numeric value is falsy) and also when `eventType === undefined`. Two distinct paths to silent drop.

**Branch 3 — text NAV at `main.ts:239`:**

```typescript
if (event.textEvent?.eventType && NAV_EVENTS.has(event.textEvent.eventType) && bridge) {
```

Same truthy-guard bug. The Companion body container is a `TextContainerProperty` with `isEventCapture: 1` (`companion.ts:386-399`), so this is the branch a single-press on Companion would flow through. The guard rejects both `undefined` and `0`.

**Branch 4 — sys NAV at `main.ts:245-249`:**

```typescript
if (event.sysEvent && bridge) {
  const eventType = event.sysEvent.eventType
  if (eventType && NAV_EVENTS.has(eventType)) {
```

Same truthy-guard bug. Relevant to lifecycle events; also relevant if the SDK delivers Companion taps via `sysEvent` rather than `textEvent` (per EVEN-SKILLS.md:23 the canonical channel for CLICK_EVENT is `sysEvent.eventType` for non-list containers).

**Summary of (a):** Zero of four branches handles `eventType === undefined` or `eventType === 0` correctly. The `?? 0` guard that EVEN-SKILLS.md:38 explicitly mandates is missing everywhere.

## Behavioral trace (question b)

Single-press on the Companion screen, step by step:

1. Hardware fires a single press while the Companion is visible.
2. Per EVEN-SKILLS.md:20-23, the SDK packages this as either `sysEvent.eventType === undefined` (canonical CLICK_EVENT channel for non-list containers) OR `textEvent.eventType === undefined` (because the Companion body is a text container with `isEventCapture: 1` per `companion.ts:386-399`). Both shapes carry `eventType === undefined`, NOT `eventType === 0`.
3. `main.ts:221` enters the `onEvenHubEvent` callback.
4. **`main.ts:224`** — list-item CLICK_EVENT branch: `event.listEvent` is undefined (this is a text container, not a list), guard fails; fall through.
5. **`main.ts:233`** — list NAV branch: `event.listEvent?.eventType` is undefined, falsy guard fails; fall through.
6. **`main.ts:239`** — text NAV branch: `event.textEvent?.eventType` is undefined, falsy guard fails; fall through. **THIS is the line where a Companion single-press is silently dropped if the SDK routed via `textEvent`.**
7. **`main.ts:245-247`** — sys NAV branch: `event.sysEvent` is present, `const eventType = event.sysEvent.eventType` resolves to `undefined`, the `if (eventType && ...)` guard's truthy check fails. **THIS is the line where a Companion single-press is silently dropped if the SDK routed via `sysEvent`.**
8. No further handlers. The callback returns. No log, no telemetry, no banner update. Event is invisible to the plugin.

The Companion D-08 docstring at `companion.ts:22-26` ("DOUBLE_CLICK is the only Companion tap event ... single-tap is sim-only on G2") is consistent with this trace AS AN OBSERVATION but not as a root cause — the root cause is the missing `?? 0` guard, not an SDK limitation. The docstring's "single-tap is sim-only" claim is mechanically explained by the bug, not by hardware behavior.

## Counter-evidence triangulation

Two independent data points triangulate the same failure mode:

1. **work-orders.ts list-item taps.** The list container at `work-orders.ts:87-105` has `isEventCapture: 1`. Per the `project_g2_tap_expand_broken` memory entry: "Phase 45's CLICK_EVENT-based list-item navigation was sim-only ... v0.7.3 sim revealed `eventType` is undefined on real listEvents." This is the empirical confirmation of EVEN-SKILLS.md:38 from Vigil's own hardware history.

2. **Companion single-tap absence.** Phase 124 D-08 deferred single-tap on Companion because "not reliably plumbed on hardware." The current `main.ts:224, 233, 239, 247` code is structurally identical to the Phase 45 code that silently dropped list-item taps. Same bug, same hardware, same outcome.

Both failures are the SAME failure mode: a `=== 0` or truthy `&&` check on a protobuf-zero-omitted field. The Phase 124 D-08 conclusion ("hardware doesn't deliver single-tap") was a symptomatic read of a code bug. The canonical SDK skills (EVEN-SKILLS.md:38, :165-168) and the Phase 45 retro both predict that fixing the guard makes single-press fire on hardware.

The contrasting positive case — Companion DOUBLE_CLICK_EVENT works on hardware as of 2026-05-10 (`project_g2_companion_doubletap_hardware_verified` memory) — confirms that `DOUBLE_CLICK_EVENT (= 3)` is non-falsy, so the truthy guard at `main.ts:247` accidentally passes it through. This is consistent with the bug being specifically a value-0/undefined problem, not a general "single-press unreachable" problem.

## Phase 133 gesture-grammar direction

**G2-ACTION-01 (WORK_ORDERS list, mark complete):** **single-press tap-select + second single-press to confirm.** Wire `listEvent.eventType ?? 0 === CLICK_EVENT && (listEvent.currentSelectItemIndex ?? 0)` at `main.ts:224` and route to a new "armed → confirm" state machine in `navigation.ts`. The natural Even SDK gesture for list selection is `listEvent.currentSelectItemIndex` (EVEN-SKILLS.md:165-167). Add a 1-second debounce + visible "tap again to complete" footer hint per EVEN-SKILLS.md:165. Reference: `vigil-g2-plugin/src/screens/work-orders.ts:87` (the list container that already has `isEventCapture: 1`).

**G2-ACTION-02 (REMINDERS, mark complete):** **single-press tap-select + second single-press to confirm.** Same pattern as G2-ACTION-01; if REMINDERS uses a list container, reuse the work-orders.ts:87 pattern verbatim. If REMINDERS uses a text container, route via `textEvent.eventType ?? 0 === CLICK_EVENT` at `main.ts:239`.

**G2-REPLY-02/03 (Companion HUD reply mode):** **single-press cycles 5 prefabs + DOUBLE_CLICK confirms.** The Companion body at `companion.ts:386-399` is a single text container — there is no per-item selection. Use single-press as the cycle primitive (low cognitive cost, single finger, matches "tap to see the next option") and reserve DOUBLE_CLICK as the confirm gate (matches the existing D-08 contract at `companion.ts:22-26` that DOUBLE_CLICK is the Companion's "commit" gesture). Phase 133 implementer wires: single-press → advance prefab index → rebuild Companion; DOUBLE_CLICK → submit current prefab → exit reply mode. DO NOT use swipe-up/down for cycling — those are already bound to inter-screen navigation in `navigation.ts:243-247` and re-binding them on Companion would block the user's ability to leave the screen.

## Minimum patch (optional — only if Verdict = REACTIVATE and patch is ≤10 lines)

N/A — patch deferred to Phase 133 implementer.

Rationale: the plumbing fix is ≤10 lines (add `CLICK_EVENT` to `NAV_EVENTS` at `main.ts:66-70`; replace four guard expressions at `main.ts:224, 233, 239, 246-247` with `?? 0` nullish-coalesced reads). BUT adding `CLICK_EVENT` to `NAV_EVENTS` routes single-press to `handleNavEvent` (`navigation.ts:184-257`), whose current `switch` statement has no `CLICK_EVENT` branch (the `default: return` at `navigation.ts:252-253` silently swallows it). Without a corresponding `CLICK_EVENT` semantics decision in `handleNavEvent` — which is exactly the gesture-grammar decision Phase 133 owns — the patch ships pure plumbing with zero behavioral improvement, and a future regression test would have nothing to assert on. Landing the plumbing without the UX is worse than not landing it: it would prove single-press fires (good) and then drop it on the floor (bad), making a future Phase 133 implementer think the plumbing is still broken.

Recommendation: Phase 133's first plan should be a 2-task plan — Task 1 lands the plumbing fix (the ≤10-line `?? 0` patch); Task 2 wires the UX semantics in `navigation.ts:handleNavEvent` and adds the regression test. This keeps the plumbing fix auditable as its own commit while preventing the plumbing-without-UX failure mode.
