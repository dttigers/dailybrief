# Phase 125 — UI Design Contract

> Visual and interaction contract for the Quiet mode + remaining polish riders + plugin v0.3.0 ship phase.
> Generated by gsd-ui-researcher, verified by gsd-ui-checker.
>
> **Two UI surfaces in this phase:**
>
> 1. **PWA Settings → G2 Plugin section** (NEW) — Quiet-mode toggle row mirroring Phase 116
>    CAL-01 / SPORTS-01 source-picker shape. Tailwind v4 + Inter, no shadcn (project is
>    hand-rolled).
> 2. **G2 Companion HUD header rightSide glyph** (NEW) — A `Q` glyph indicating
>    Quiet mode is active. Lives next to the existing offline `!` and `N/M` cycle
>    indicator established by Phase 124. Even Hub SDK text primitives only — no DOM,
>    no font selection, no color (greyscale 0–15).
>
> The two surfaces speak the same `users.quiet_mode` truth via the SSE
> `quiet_mode_changed` event type (CONTEXT D-02). Plugin filter applies at
> HUD-write time per CONTEXT D-02; PWA toggle is the only user-facing
> control.

### Quiet-Mode Toggle Row

**Markup pattern (mirrors existing `sports-checkbox-${league}` row at lines 1059–1068):**

```tsx
<div data-testid="quiet-mode-row" className="space-y-1.5">
  <label className="flex items-center gap-2 cursor-pointer py-1">
    <input
      type="checkbox"
      checked={quietModeEnabled}
      onChange={() => handleQuietModeToggle()}
      className="w-4 h-4 rounded border-gray-400/30 bg-gray-900/80 accent-teal-600 cursor-pointer"
      data-testid="quiet-mode-checkbox"
      aria-describedby="quiet-mode-help"
    />
    <span className="text-sm text-gray-100">Quiet mode</span>
  </label>
  <p id="quiet-mode-help" className="text-xs text-gray-500 pl-6">
    Only urgent events (need-input prompts and failures) reach your glasses while this is on.
    Other events are queued and delivered when you turn it off.
  </p>
```

**Critical rules locked here:**

- Use `<input type="checkbox">` — NOT a custom switch component. Rationale: every
  other toggle in `SettingsPage.tsx` (CAL-01 calendars, SPORTS-01 leagues) is a
  checkbox; introducing a switch breaks the visual pattern and creates a
  precedent the planner would have to defend.
- `accent-teal-600` is load-bearing — matches the brand-mark teal `#0F6E56`
  defined at `vigil-pwa/src/index.css` line 12, and matches the existing
  CAL-01 / SPORTS-01 checkbox accent.
- Helper copy in a sibling `<p>`, not a `<label>` child — keeps the clickable
  area a single line (label + checkbox), prevents help-text mis-clicks from
  toggling state.
- `aria-describedby` ties the help text to the checkbox for screen-reader users.

### Toggle Interaction Contract

```typescript
// State (component scope)
const [quietModeEnabled, setQuietModeEnabled] = useState<boolean>(false)
const [quietModeStatus, setQuietModeStatus] = useState<'loading' | 'ok' | 'error'>('loading')
const [quietModeError, setQuietModeError] = useState<string | null>(null)
const lastSavedQuietModeRef = useRef<boolean>(false)
const quietModeSaveTimerRef = useRef<number | null>(null)

// Toggle handler (verbatim shape of handleCalendarToggle at line 588)
function handleQuietModeToggle() {
  // Optimistic flip
  const next = !quietModeEnabled
  setQuietModeEnabled(next)

  // 400ms debounced PUT
  if (quietModeSaveTimerRef.current !== null) {
    window.clearTimeout(quietModeSaveTimerRef.current)
  }
  quietModeSaveTimerRef.current = window.setTimeout(async () => {
    quietModeSaveTimerRef.current = null
    try {
      await setQuietMode(next)
      lastSavedQuietModeRef.current = next
    } catch {
      // Rollback + error toast
      setQuietModeEnabled(lastSavedQuietModeRef.current)
      showToast({
        variant: 'error',
        body: "Couldn't save quiet mode — try again",
      })
    }
  }, 400)
}
```

**Locked timing values:**

- **Debounce:** 400ms — verbatim Phase 116 D-21 / Phase 115 D-08 value.
- **Toast duration:** Inherited from `useToast` default; do NOT override.

## Surface 2: G2 Companion HUD — Quiet-Mode Header Glyph

### Glyph Decision (locks a CONTEXT.md "Specific Idea")

**Decision:** Surface a `Q` glyph in the Companion HUD header `rightSide` when
Quiet mode is active. **Locked. The "optional" framing in CONTEXT.md
"Specific Ideas" is resolved here as YES — ship it.**

**Rationale (saving the planner from re-deciding):**

1. **User confusion is real.** Without a glyph, a user wearing G2 with Quiet
   mode on sees a HUD that goes silent. They can't distinguish "DND on, events
   queued" from "nothing happening" or "SSE disconnected" or "watcher crashed".
   The `!` offline glyph already exists for the third case (Phase 124
   UI-SPEC §"Offline Indicator"); adding `Q` for the first case completes the
   set. Without it, the user has no on-glasses signal that they themselves
   muted the HUD via the PWA toggle.
2. **Glyph budget is preserved.** Phase 124 UI-SPEC §"N/M Indicator" caps
   `rightSide` at 26 chars. The Phase 124 worst-case ("`! 9/9`" = 5 chars)
   plus `Q` (1 char) plus a space (1 char) = 7 chars — well under cap.
3. **Character choice: `Q` (ASCII 0x51), NOT `🌙`.** ASCII is guaranteed in
   the SDK font; `🌙` (U+1F319 CRESCENT MOON) is a multibyte emoji whose
   coverage in the SDK monospace renderer is unverified. **The Phase 124
   UI-SPEC made the same decision for offline indicator (`!` over `⚠`)** —
   we hold the line for visual consistency. CONTEXT.md "Specific Ideas"
   suggests `🌙 / Q`; we lock `Q`.
4. **No new SDK API.** `buildVigilHeader(containerID, name, rightSide?)`
   already accepts a custom right-side string (Phase 124 D-09 site). The
   plugin's `companion.ts:computeRightSide()` (lines 277–285) is the only
   modification site.

### Updated `rightSide` Priority Order (extends Phase 124 UI-SPEC §"Offline Indicator")

Phase 124 established the priority. **Phase 125 ADDS the `Q` glyph as a prefix-token slot,
without breaking the existing `! N/M` priority.** The full ordered priority is:

| # | Conditions | `rightSide` value | Char count |
|---|-----------|-------------------|-----------|
| 1 | Quiet ON, SSE down, ≥2 sessions | `Q ! 2/3` | 7 |
| 2 | Quiet ON, SSE down, ≤1 session | `Q !` | 3 |
| 3 | Quiet ON, SSE up, ≥2 sessions | `Q 2/3` | 5 |
| 4 | Quiet ON, SSE up, ≤1 session | `Q` | 1 |
| 5 | Quiet OFF, SSE down, ≥2 sessions | `! 2/3` | 5 (Phase 124) |
| 6 | Quiet OFF, SSE down, ≤1 session | `!` | 1 (Phase 124) |
| 7 | Quiet OFF, SSE up, ≥2 sessions | `2/3` | 3 (Phase 124) |
| 8 | Quiet OFF, SSE up, ≤1 session | `undefined` → HH:MM clock | 8 (Phase 124) |

**Construction rule (locked):**

```typescript
function computeRightSide(): string | undefined {
  const quiet = isQuietMode() ? 'Q' : ''
  const offline = sseConnected ? '' : '!'
  const sessions =
    activeSessions.length >= 2
      ? `${currentSessionIndex + 1}/${activeSessions.length}`
      : ''
  // Order: quiet → offline → sessions, joined with single space
  const combined = [quiet, offline, sessions].filter(Boolean).join(' ')
  return combined || undefined
}
```

This is a **strict superset** of Phase 124's `computeRightSide` —
appending one more `.filter(Boolean)` participant in front. Existing
Phase 124 unit tests for offline+N/M continue to pass. No regression risk
to the Phase 124 D-14 byte-identical screenshot gate, because the home
screen does not surface `rightSide` at all (only Companion does — Phase 124
UI-SPEC §"Offline Indicator" explicitly limits the indicator to Companion).

### `Q` Glyph Width Math (CHARS_PER_LINE budget verification)

`buildVigilHeader` (lines 38–41) builds:

```
${WORDMARK}${' '.repeat(leftPad)}${right}
```

Where `WORDMARK = 'VIGIL'` (5 chars), `CHARS_PER_LINE = 32`, and
`leftPad = max(1, 32 - 5 - right.length)`.

| `right` value | `right.length` | `leftPad` | Total |
|--------------|---------------|-----------|-------|
| `Q` | 1 | 26 | 32 ✓ |
| `Q 2/3` | 5 | 22 | 32 ✓ |
| `Q !` | 3 | 24 | 32 ✓ |
| `Q ! 9/9` | 7 | 20 | 32 ✓ |

All cases fit comfortably. Maximum `right.length` we can handle without
wrapping is 26 chars; we top out at 7. **No overflow risk.**

### Visual Differentiation (greyscale only)

Per Phase 124 UI-SPEC §"Color", the HUD has no per-state border-color
changes. The `Q` glyph is text-only, rendered the same brightness as
the rest of the header right-side. This matches the established `!` and
`2/3` rendering — uniform glyph treatment is the established contract.

**No banner state is added.** Quiet mode does NOT show a body-content
banner or change the 3-line HUD body. It changes ONLY the header
`rightSide`. The body still renders the active session's label / state /
last-event-message exactly as in Phase 124.

### What Quiet Mode Suppresses (HUD-Write Filter)

Per CONTEXT.md D-02: filter is at HUD-write time, not at SSE-receive time.
Concretely in `companion.ts`:

| Event type | Quiet OFF behavior | Quiet ON behavior |
|-----------|-------------------|-------------------|
| `needs_input` | applyAgentEvent → banner `[NEEDS INPUT]` (persistent) | unchanged — banner shows |
| `task_failed` | applyAgentEvent → banner `[TASK FAILED]` (persistent) | unchanged — banner shows |
| `task_complete` | applyAgentEvent → toast `[DONE]` 3s | applyAgentEvent updates cache; **HUD body NOT rebuilt with toast**; on quiet→off the held event replays per server-side suppression queue |
| `milestone` | applyAgentEvent → toast `[MILESTONE]` 3s | applyAgentEvent updates cache; **HUD body NOT rebuilt with toast** |
| `heartbeat` | state-line refresh only | applyAgentEvent updates cache; **HUD body NOT rebuilt** |

**Critical:** The plugin-side filter is a defense-in-depth safety net.
The server-side suppression queue (CONTEXT D-04) is the primary mechanism
— most non-allowlist events never reach the plugin while Quiet is on. The
plugin filter exists because (a) `quiet_mode_changed` and `agent_events`
race during reconnect is possible (synthetic state-bootstrap frame in D-03
narrows but doesn't eliminate), and (b) the synthetic frame may be in a
fast-path local cache that gets out of sync if `quiet_mode_changed → true`
arrives between two events.

## Navigation + Interaction Contract

### PWA

- **Click target:** Entire `<label>` (checkbox + "Quiet mode" text) is
  clickable per the `cursor-pointer` class. Helper copy below the label
  is **not** in the click target — it's a sibling `<p>` (intentional —
  prevents accidental toggles).
- **Keyboard:** Native checkbox a11y. `Tab` reaches checkbox. `Space`
  toggles. `Enter` does nothing (browser-default). No custom keyboard handler.
- **Screen reader:** `aria-describedby="quiet-mode-help"` ties the helper
  copy to the checkbox.
- **Toggle race protection:** 400ms debounce drops in-flight rapid toggles
  per the verbatim CAL-01 pattern.

### G2

- **Quiet mode does NOT change tap-event behavior.** DOUBLE_CLICK on
  Companion still ack's the banner (rule 1) → cycles session (rule 2)
  → goes Home (rule 3) per Phase 124 UI-SPEC §"DOUBLE_CLICK on Companion".
- **No new tap-event semantics introduced.** The Quiet-mode toggle is
  PWA-only; the user cannot toggle Quiet mode from the glasses.
- **SCROLL_BOTTOM / SCROLL_TOP** unchanged from Phase 124.
- **Header rebuild on `quiet_mode_changed`:** When `quiet_mode_changed`
  arrives via SSE, plugin updates module-level `quietMode` ref AND
  triggers `rebuildCurrentScreen` if currentScreen === Screen.COMPANION.
  This makes the `Q` glyph appear/disappear within ~100ms of PWA toggle.
