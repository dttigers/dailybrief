<task type="auto">
  <name>Task 1: Fix config startup error with default config creation</name>
  <files>Sources/JarvisCore/Config/ConfigLoader.swift</files>
  <action>
**Bug:** ConfigLoader.load() throws ConfigError.fileNotFound when ~/.config/dailybrief/config.json doesn't exist. The CLI crashes, and the menu bar app silently degrades with no AI features. There's no way for the menu bar app to auto-create a config.

**Fix:** Add a `loadOrCreate()` method to ConfigLoader that returns a default config when the file doesn't exist, and saves it so subsequent loads work. Keep the existing `load()` throwing behavior for the CLI (which has --setup guidance).

Add this method to ConfigLoader:
```swift
/// Loads existing config or creates a default config file if none exists.
/// Used by the menu bar app to ensure config always exists on startup.
public static func loadOrCreate() throws -> AppConfig {
    if FileManager.default.fileExists(atPath: configPath) {
        return try load()
    }
    // Create default config
    let defaultConfig = AppConfig(
        email: .init(emailAddress: "", appPassword: ""),
        reminders: .init(),
        sports: .init(),
        ai: .init(claudeApiKey: ""),
        pdf: .init(),
        printing: .init()
    )
    try save(defaultConfig)
    return defaultConfig
}
```

This ensures:
- Menu bar app always gets a valid config (empty but parseable)
- User can then fill in values via Settings UI
- CLI still throws with helpful "Run with --setup" message
- Config file is created with all default values so Settings UI can load and save correctly
  </action>

<task type="auto">
  <name>Task 2: Fix settings window sizing and tab content clipping</name>
  <files>Sources/DailyBriefMonitor/Settings/SettingsView.swift, Sources/DailyBriefMonitor/AppDelegate.swift</files>
  <action>
**Bug:** SettingsView sets `.frame(width: 850, height: 500)` but AppDelegate creates the NSWindow at 600x420. This causes horizontal and vertical clipping. The Sports tab with 4 expandable league pickers and the Calendar tab with dynamic calendar lists exceed the fixed height.

**Fix 1 — Sync window size with view frame in AppDelegate.openSettings():**

Change the NSWindow contentRect from `width: 600, height: 420` to `width: 880, height: 560`. The extra 30px width and 60px height account for window chrome/padding.

Also add `.resizable` to the styleMask so users can manually resize if needed:
```swift
styleMask: [.titled, .closable, .resizable],
```

**Fix 2 — Make SettingsView tabs scrollable:**

In SettingsView.swift, wrap each tab's Form content in a ScrollView to handle variable-height content. The Sports tab is the worst offender — when all 4 leagues are enabled, the pickers overflow.

Change the outer frame from `.frame(width: 850, height: 500)` to `.frame(width: 850, minHeight: 500)` to allow the content to use available space without hard-capping at 500.

For each tab body (sportsTab, calendarTab, foldersTab, etc.), wrap the Form in ScrollView:
```swift
private var sportsTab: some View {
    ScrollView {
        Form {
            // existing content
        }
    }
}
```

Apply ScrollView wrapping to the tabs that have dynamic/tall content: sportsTab, calendarTab, foldersTab, emailTab. Simple tabs (aiTab, pdfTab, printingTab, remindersTab, insightsTab) can stay as-is since they're small.

Do NOT change the tab structure or tab labels — only wrap the existing Form content.
  </action>
