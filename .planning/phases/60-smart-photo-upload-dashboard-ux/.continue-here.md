## Critical Anti-Patterns

| Pattern | Description | Severity | Prevention Mechanism |
|---------|-------------|----------|---------------------|
| Masked HTTP errors via generic catchall | `mapPhotoError`'s httpStatus default returned "Couldn't process photo — see logs" for any unmapped HTTP code. Hid a 404 (missing /v1 suffix in Base URL) for 2 hours of debugging. | advisory | FIXED in 60-02 — 401/404/429 cases added. For future error-mapping code: ALWAYS surface the status code somewhere user-visible on unknown codes, OR log it loudly on stderr. Generic catchalls are debugging landmines. |
| `Scripts/install.sh` "Build complete! (0.19s)" silent skip | Release build cache skipped compilation silently when it thought nothing changed. Source edits never reached the installed binary. `rm -rf .build/release` did nothing (it's a symlink); only `rm -rf .build` forced a real rebuild. | advisory | When in doubt: `rm -rf .build && ./Scripts/install.sh`. Logged for a future polish fix to install.sh itself. |
| `launchctl bootstrap` fails silently in install.sh | If DailyBriefMonitor is already loaded, `launchctl bootstrap` returns "5: Input/output error" and does nothing. install.sh doesn't `bootout` first. | advisory | After install.sh: `killall DailyBriefMonitor` to trigger KeepAlive respawn, OR launch manually: `~/.local/bin/DailyBriefMonitor &`. Logged for polish fix. |
| Vigil API Base URL requires `/v1` suffix | `VigilAPIClient` does not auto-prepend the version prefix. A Base URL of `http://localhost:3001` (no `/v1`) produces 404 on every endpoint. Hit this during Phase 60 human-verify. | advisory | The dashboard now shows a specific 404 banner text mentioning "Base URL may be missing /v1 suffix". Logged as a polish candidate to auto-prepend in `VigilAPIClient` and/or validate on Settings save. |

None are `blocking` — Phase 60 is closed and Phase 61 starts fresh. These are session-learned gotchas worth knowing but don't gate resumption.
