<objective>
Replace the silent permission-reset failure with persistent Developer ID Application code signing in the install.sh flow. This plan (a) cleans the incompatible CloudKit entitlements out of DailyBriefMonitor.entitlements that would otherwise block Developer ID signing, and (b) adds cert-resolution, hard-fail, sign, and verify steps to install.sh so every rebuild produces binaries with a stable designated requirement — the mechanism that lets macOS TCC preserve granted Full Disk Access / Automation / Accessibility grants across rebuilds.

Purpose: Stop burning a round of TCC re-granting on every rebuild. This is the single biggest daily friction point for the solo dev loop right now and is the reason v2.4 opens with SIGN instead of PHOTO.

Output:
- DailyBriefMonitor.entitlements stripped of CloudKit keys (dead code per research)
- install.sh with cert guard + codesign calls + verification, fails loud on missing cert
- A signed-cert verification checkpoint where the user confirms the binaries are signed with Developer ID Application and TCC permissions survive a rebuild
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| developer workstation → macOS keychain | install.sh queries the login keychain via `security find-identity` to locate the Developer ID cert. The private key never leaves the keychain — codesign accesses it via Apple's keychain API. |
| macOS TCC subsystem → installed binaries | TCC inspects the designated requirement of the installed binary to decide whether a permission grant from a previous install still applies. Ad-hoc or unsigned binaries = different DR every rebuild = TCC treats as new app. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-58-01 | Spoofing | install.sh cert resolution | mitigate | Use `security find-identity -v -p codesigning` and grep for `Developer ID Application` string — the `-v` flag excludes invalid/expired certs; grep prefix scoping excludes `Mac Developer` / `Apple Development` certs which have different intermediate chains and would produce non-matching DRs. Pick `head -1` — single-developer scenario, deterministic first-match. |
| T-58-02 | Tampering | Unsigned binary silently produced and installed | mitigate | Hard fail on empty IDENTITY (D-04). Never fall back to `--sign -`. Post-sign `codesign --verify --verbose` catches the rare case where codesign exits 0 but the signature is invalid (e.g., keychain GUI prompt timed out). Verify failure propagates non-zero exit via `|| { echo ...; exit 1; }`. |
| T-58-03 | Tampering | CloudKit entitlements block signing, causing silent fallback | mitigate | Task 1 strips CloudKit keys from DailyBriefMonitor.entitlements before Task 2 ever runs a codesign command. Acceptance criteria verify the keys are gone (`grep -c icloud` = 0). |
| T-58-04 | Information Disclosure | Remediation message leaks keychain details | accept | Remediation only names the `security import` command + cert path placeholder. It does not print cert fingerprints, keychain contents, or existing identity data. The output is instructional, not diagnostic — no sensitive info exposed. |
| T-58-05 | Denial of Service | `--timestamp` flag fails offline and blocks signing | mitigate | Explicitly omit `--timestamp` (58-RESEARCH.md Pitfall 6). Local-machine signing does not need a trusted timestamp — that requirement only applies to notarization, which is out of scope for this phase and this milestone. |
| T-58-06 | Elevation of Privilege | First-run keychain partition list prompt could be socially engineered | accept | The partition list GUI prompt is a macOS security feature, not a vulnerability. User must manually click "Always Allow" once per machine — this is a deliberate trust gesture, documented in the Task 3 checkpoint. No script-level mitigation available (setting partition list programmatically requires the keychain password, which is a worse security posture). |
| T-58-07 | Repudiation | Signed binary lacks audit trail | accept | Developer ID signature embeds team identity and timestamp of signing (not trusted timestamp, but filesystem mtime). This is sufficient for a solo-dev workstation. Audit trail is out of scope for single-user tooling. |
</threat_model>
