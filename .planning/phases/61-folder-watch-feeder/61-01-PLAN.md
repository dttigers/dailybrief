Create `Sources/DailyBriefMonitor/FolderWatcherService.swift` containing a new `FolderWatcherService` actor. This is the core of Phase 61.

**Architecture:**

The service is a Swift `actor` (not `@Observable class` — actor protects shared mutable state from the DispatchSource callback thread). It must be constructed with these dependencies injected via init:

```swift
public actor FolderWatcherService {
    // Dependencies — injected by AppDelegate
    private let imageService: APIImageDescriptionService
    private let transcriptionService: TranscriptionService
    private let captureService: CaptureService

    // State
    private var watchedSources: [(source: DispatchSourceFileSystemObject, fd: Int32)] = []
    private var knownFiles: Set<String> = []   // Track filenames already queued/processed to avoid duplicates
    private var _failedFiles: [(url: URL, reason: String)] = []
    private var isRunning = false
    private var processingTask: Task<Void, Never>?
    private var pendingQueue: [URL] = []       // FIFO queue for sequential processing (D-08)

    public init(
        imageService: APIImageDescriptionService,
        transcriptionService: TranscriptionService,
        captureService: CaptureService
    )
```

Note: The service takes `APIImageDescriptionService` directly (not through `PhotoProcessingAPI` protocol) because `PhotoProcessingAPI` is defined in DashboardViewModel.swift in the same DailyBriefMonitor target — calling `processPhoto` directly on the concrete actor avoids protocol extraction complexity. The `processPhoto(imageURL:preview:forcePaperType:)` convenience overload on `APIImageDescriptionService` already handles reading file data + mediaType detection + `prepareImage` compression, so the watcher can use it directly.

**Public interface for menu bar observation (WATCH-06):**

**Start/stop lifecycle:**

```swift
    public func start()   // Called from AppDelegate.applicationDidFinishLaunching
    public func stop()    // Called from AppDelegate.applicationWillTerminate
```

`start()` implementation:
1. Read `AppConfig.shared.folderWatching` — if `enabled == false`, return immediately (no-op).
2. Expand `~` in `audioFolderPath` and `imageFolderPath` using `NSString(string:).expandingTildeInPath`.
3. For each path, call `FileManager.default.createDirectory(at:withIntermediateDirectories:true)` to ensure the directory exists.
4. Also create the `done/` subdirectory inside each watched directory upfront.
5. For each directory, open with `open(path, O_EVTONLY)` (NOT `O_RDONLY` — Pitfall 3: O_RDONLY prevents volume unmount).
6. Create `DispatchSource.makeFileSystemObjectSource(fileDescriptor: fd, eventMask: .write, queue: watcherQueue)` where `watcherQueue` is a private serial `DispatchQueue(label: "com.vigil.folderwatcher")`.
7. Set event handler: `source.setEventHandler { [weak self] in Task { await self?.handleDirectoryChange(directoryURL) } }` — the `Task` bridges from DispatchQueue to actor isolation (Pitfall 4).
8. Set cancel handler: `source.setCancelHandler { close(fd) }` — prevents fd leak (Pitfall 3).
9. Call `source.resume()`.
10. Store `(source, fd)` tuple in `watchedSources`.
11. Do an initial scan of each directory to pick up any files that were dropped while the app was not running.
12. Log startup via `NSLog("FolderWatcherService: watching %@ and %@", audioPath, imagePath)`.

`stop()` implementation:
1. Cancel all sources: `for (source, _) in watchedSources { source.cancel() }`.
2. Clear `watchedSources`.
3. Cancel `processingTask`.

**File type classification (D-07):**

```swift
    private enum FileKind { case audio, image }

    private static let imageExtensions: Set<String> = ["jpg", "jpeg", "png", "heic", "tiff", "bmp"]
    private static let audioExtensions: Set<String> = ["wav", "m4a", "mp3", "caf"]

    private static func classify(_ url: URL) -> FileKind? {
        let ext = url.pathExtension.lowercased()
        if imageExtensions.contains(ext) { return .image }
        if audioExtensions.contains(ext) { return .audio }
        return nil
    }
```

These extension sets match D-07 exactly. Note they differ slightly from DashboardViewModel (which also has gif, webp, aiff, heif, tif) — the watcher uses the D-07 spec exactly.

**Directory scan (called after each VNODE_WRITE event):**

```swift
    private func handleDirectoryChange(_ directoryURL: URL)
```

1. Call `FileManager.default.contentsOfDirectory(at: directoryURL, includingPropertiesForKeys: [.isRegularFileKey], options: [.skipsHiddenFiles])`.
2. Filter results: exclude any URL whose path contains `/done/` (Pitfall 1 — prevents re-processing moved files). Specifically: `url.pathComponents.contains("done")` or `url.deletingLastPathComponent().lastPathComponent == "done"`.
3. Filter: only files where `classify(url) != nil` (D-07 — silently ignore unsupported types).
4. Filter: only files not already in `knownFiles` set (prevents duplicate processing on repeated VNODE events).
5. For each new file: add filename to `knownFiles`, append URL to `pendingQueue`.
6. If `processingTask` is nil or finished, start a new `processingTask` that drains `pendingQueue` (see below).

**Sequential FIFO processing loop (D-08):**

```swift
    private func startProcessingLoop()
```

Creates a `Task` stored in `processingTask`:
```swift
processingTask = Task {
    while !pendingQueue.isEmpty {
        let url = pendingQueue.removeFirst()
        await processFile(url)
    }
    processingTask = nil
}
```

**Wait-for-stable debounce (D-06):**

```swift
    private func waitForStable(_ url: URL) async throws -> Bool
```

Poll `FileManager.default.attributesOfItem(atPath: url.path)[.size]` at 1-second intervals. After two consecutive reads return the same size AND size > 0, the file is stable. Return `true`. If the file disappears during polling (user moved it), return `false`. Max 30 iterations (30 seconds) to avoid infinite wait on a continuously-written file — after 30s, treat as stable.

**File processing (routes to correct pipeline):**

```swift
    private func processFile(_ url: URL)
```

1. Call `waitForStable(url)`. If returns `false` (file disappeared), remove from `knownFiles` and return.
2. Verify file still exists (may have been moved by user between queue and processing).
3. Classify the file via `Self.classify(url)`.
4. Based on kind:

**Image processing (WATCH-02, D-09):**
```swift
case .image:
    // Use the URL-based convenience overload — it handles Data reading,
    // mediaType detection, and prepareImage compression internally.
    let _ = try await imageService.processPhoto(
        imageURL: url,
        preview: false,           // D-09: headless commit, no preview
        forcePaperType: nil       // D-10: backend auto-coerces low-confidence to "lined"
    )
```

**Audio processing (WATCH-03, D-11):**
```swift
case .audio:
    let text = try await transcriptionService.transcribe(audioURL: url)
    let _ = try await captureService.capture(text, source: .voice)
```

5. On success: call `postProcess(url)` (see below).
6. On failure: catch the error, append `(url: url, reason: shortErrorDescription)` to `_failedFiles`. Log via `NSLog("FolderWatcherService: failed to process %@: %@", url.lastPathComponent, error.localizedDescription)`. Do NOT move or delete the file (WATCH-06).

**Error reason mapping:** Map common errors to short user-friendly strings:
- `ProcessPhotoError.http(let status, _)` → "API error (HTTP \(status))"
- `ProcessPhotoError.transport(_)` → "Network error"
- `TranscriptionError.notAvailable` → "Speech recognition unavailable"
- Default → `error.localizedDescription` truncated to 80 chars

**Post-processing (D-04, D-05):**

```swift
    private func postProcess(_ url: URL) throws
```

1. Read `AppConfig.shared.folderWatching.autoDeleteAfterProcessing`.
2. If `true`: `try FileManager.default.removeItem(at: url)`.
3. If `false`: move to `done/` subfolder:
   - `let doneDir = url.deletingLastPathComponent().appendingPathComponent("done")`
   - `try FileManager.default.createDirectory(at: doneDir, withIntermediateDirectories: true)`
   - Build destination: `doneDir.appendingPathComponent(url.lastPathComponent)`
   - On name collision (D-05): append counter — `filename-2.ext`, `filename-3.ext`, etc.
   - `try FileManager.default.moveItem(at: url, to: destination)`
4. Remove from `knownFiles` set after successful post-processing.
5. Also clean the file from `_failedFiles` if it was previously there (D-03 auto-clear).

**Error state auto-clear (D-03):**

After each successful processing + post-processing, scan `_failedFiles` and remove entries whose URL no longer exists in the watched directory. This handles the case where a user manually deletes or moves a failed file — the error count drops and the menu bar icon returns to normal.

Also in `handleDirectoryChange`: after scanning, check if any URLs in `_failedFiles` no longer exist on disk and remove them.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| local filesystem -> FolderWatcherService | Untrusted files dropped into watched directory by any process |
| FolderWatcherService -> Vigil Core API | Image data sent over HTTPS to backend |
| FolderWatcherService -> local SFSpeechRecognizer | Audio data processed locally |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-61-01 | Tampering | File extension vs content mismatch | accept | Extension allowlist (D-07) filters types; prepareImage in APIImageDescriptionService validates actual image data and rejects corrupt payloads with HTTP 400. Low risk — user's own filesystem. |
| T-61-02 | DoS | Oversized image file | mitigate | prepareImage in APIImageDescriptionService applies 5 MB compression cap (Phase 60 WR-02). No additional guard needed in watcher — inherited automatically. |
| T-61-03 | Elevation of Privilege | Symlink in watched directory pointing to sensitive file | mitigate | contentsOfDirectory with .isRegularFileKey filters; additionally verify `url.resourceValues(forKeys: [.isSymbolicLinkKey]).isSymbolicLink != true` before processing. |
| T-61-04 | Information Disclosure | File contents sent to external API | accept | Same trust model as dashboard upload — user explicitly drops files into watched folder. API uses HTTPS + bearer auth. |
| T-61-05 | DoS | Rapid file drops overwhelming API | mitigate | Sequential FIFO processing (D-08) — one file at a time. No concurrent API calls. |
</threat_model>
