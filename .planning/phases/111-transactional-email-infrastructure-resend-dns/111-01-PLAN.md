  <how-to-verify>
    1. Log in to https://dash.cloudflare.com, select `vigilhub.io`, navigate to the DNS tab.
    2. Add Record 1 — DKIM CNAME:
       - Type: CNAME
       - Name: `resend._domainkey`
       - Target: (paste the exact target captured in Task 1 — e.g., `resend._domainkey.us-east-1.resend.com`)
       - Proxy status: **DNS only (gray cloud)** — CRITICAL, see CONTEXT §specifics Cloudflare gotcha. If left as "Proxied" the CNAME resolves to Cloudflare edge IPs and DKIM verification fails silently.
       - TTL: Auto
       - Save.
    3. Add Record 2 — SPF TXT:
       - Type: TXT
       - Name: `@` (root domain — Cloudflare will display this as `vigilhub.io`)
       - Content: `v=spf1 include:_spf.resend.com -all`
       - TTL: Auto
       - Save. (Note: TXT records have no proxy option in Cloudflare — N/A.)
    4. Add Record 3 — DMARC TXT:
       - Type: TXT
       - Name: `_dmarc`
       - Content: `v=DMARC1; p=none; rua=mailto:jamesonmorrill1@gmail.com`
       - TTL: Auto
       - Save.
    5. After all three saved, run these `dig` commands (Cloudflare DNS propagates fast — usually < 60 seconds; retry if first attempt shows NXDOMAIN):

       - `dig TXT _dmarc.vigilhub.io +short` → output contains the literal string `v=DMARC1; p=none; rua=mailto:jamesonmorrill1@gmail.com` (same resolver-quirk note as SPF — do NOT false-FAIL on surrounding quotes or whitespace)

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Cloudflare DNS → public internet | TXT/CNAME records are world-readable (DNS is public); DMARC rua=mailto: exposes the user's personal email |
| Resend dashboard → vigilhub.io zone control | Whoever controls Resend API keys can send mail AS vigilhub.io; DKIM signing makes spoofing by others effectively impossible |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-111-01 | Spoofing | Third-party attacker sends mail AS noreply@vigilhub.io to phish Vigil users | mitigate | SPF `-all` hard-fail + DKIM CNAME pointing to Resend-only DKIM keys means any sender that is not Resend fails DMARC alignment. DMARC `p=none` + rua reports surface spoof attempts to jamesonmorrill1@gmail.com; tighten to `p=quarantine` after 1-2 weeks of clean reports (deferred, see CONTEXT.md) |
| T-111-02 | Information Disclosure | DMARC `rua=mailto:jamesonmorrill1@gmail.com` publishes user's personal email in a world-readable TXT record (dig queryable by anyone) | accept | Low-value leak — the email is already public via several other channels (GitHub commits, domain registration, etc.); alternative would be to stand up a separate DMARC report inbox which is scope creep for day 1. Mitigation: if the inbox ever starts receiving spam from the public rua address, rotate to a dedicated forwarder |
| T-111-03 | Tampering | Proxied DKIM CNAME in Cloudflare (orange cloud) resolves to Cloudflare edge IPs, breaking DKIM verification → emails land in spam, possibly enabling downstream phishing | mitigate | Task 2 acceptance criteria explicitly requires gray-cloud DKIM CNAME; dig verification asserts target ends in `.resend.com`, NOT a bare IP. If the check fails, Task 2 does not pass — cannot silently ship broken DKIM |
| T-111-04 | Denial of Service | SPF `-all` hard-fail means a temporary Resend outage could cause legitimate mail to be rejected by receivers that strictly enforce SPF | accept | Tradeoff acknowledged — hard-fail is the correct posture given Resend is the only outbound sender (D-03). A Resend outage halting password-reset emails is a downtime incident, not a security failure; failover to a second provider is deferred scope |
| T-111-05 | Information Disclosure | Resend dashboard credentials (password + 2FA) compromise = attacker can rotate API key + send AS vigilhub.io | transfer | Inherent to SaaS transactional email; mitigated by Resend's account security (password + 2FA mandatory). Out-of-scope for this phase; user owns Resend account credential hygiene |
</threat_model>
