# Phase 999.2: CaptureBar — Multi-Line Input Support - Research

**Researched:** 2026-05-19
**Domain:** React 19 + TypeScript controlled-textarea UI, single-file surgical edit
**Confidence:** HIGH

## Summary

This is a small, well-scoped UI change: swap `<input type="text">` → `<textarea>` in **one file** (`vigil-pwa/src/components/CaptureBar.tsx`), give it the same keyboard grammar that already exists in `ThoughtRow.tsx` edit mode (Phase 100 EDIT-01: plain `Enter` = newline, `Cmd/Ctrl+Enter` = submit, `Escape` = blur/cancel), implement a bounded auto-grow (1→6 lines, then internal scroll), and set `enterKeyHint="enter"` so mobile soft-keyboards render a return key with newline-insert semantics. **No new packages, no API changes, no migration** — `createThought` already POSTs `content` verbatim as JSON, which preserves `\n` end-to-end, and the display side already renders multi-line via `whitespace-pre-line` shipped in Phase 115 POLISH-01 D-16.

The driving incident is Phase 115 UAT 2 (2026-04-28): a paste of a three-line reflection silently collapsed because HTML's `<input>` strips newlines on paste per the WHATWG spec. The success criterion is a UAT replay of exactly that flow.

**Primary recommendation:** Convert the single CaptureBar element to a controlled textarea using the established ThoughtRow keyboard pattern. Implement auto-grow via a `useRef<HTMLTextAreaElement>` plus a `useLayoutEffect` (or post-onChange imperative) that sets `el.style.height = '0px'` then `el.style.height = el.scrollHeight + 'px'`, capped by CSS `max-h-36` (≈ 9rem = 6 lines at this font size). Add a single `useEffect` that resets height to baseline when `input === ''`. Ship one focused test file (`CaptureBar.test.tsx`) with five behaviors (paste preserves `\n`, Cmd+Enter submits, plain Enter inserts newline + does NOT submit, whitespace-only rejected, post-submit height resets). No `<form>` wrapper is needed (CaptureBar is not inside one today — verified by grep). No deferred or speculative work.

## User Constraints (from CONTEXT.md)

### Locked Decisions
- **D-01** Submit semantic: Plain `Enter` inserts a newline. `Cmd+Enter` (macOS) / `Ctrl+Enter` (Linux/Windows) submits. Matches `ThoughtRow.tsx:294-308` (Phase 100 EDIT-01). Trailing-newline pastes do not accidentally submit.
- **D-02** Submit affordance: The existing **Save button** remains the primary discoverable submit affordance. Stays enabled whenever `input.trim()` is non-empty and `!isSubmitting` — identical disable logic to today.
- **D-03** Auto-grow: Textarea starts at one line (visually identical to today's `<input>` height) and auto-grows up to **6 lines (~9rem)**. Beyond 6 lines, textarea caps and scrolls internally. Implementation hint: `rows={1}` + `height = scrollHeight` adjustment on `onChange`, capped via CSS `max-h-36` (or equivalent).
- **D-04** Post-submit reset: When the textarea is cleared after a successful submit, height resets back to the single-line baseline.
- **D-05** Mobile return key: `enterKeyHint="enter"` (and `inputMode` left at default for text) so the on-screen return key inserts a newline. Mobile submit happens via the Save button. Same component, same rules on every platform.
- **D-06** Placeholder: Stays terse: `Capture a thought...` (unchanged). Save button carries the visible submit affordance; `Cmd+Enter` is a power-user layer on top. No persistent helper text, no kbd badge, no inline tooltip in the placeholder.
- **D-07** Whitespace guard: Keep the existing `input.trim()` guard at `CaptureBar.tsx:15-16`. **No new guard logic needed.**
- **D-08** Interior newlines: `"line1\n\nline2"` must be preserved verbatim — only leading/trailing whitespace is trimmed (`String.prototype.trim()` does exactly this; no regex substitution).
- **D-09** Element swap is `<input type="text">` → `<textarea>`. All other CaptureBar concerns (fire-and-forget triage at lines 27-40, error display, sticky-bottom container, focus ring, Tailwind class palette) are preserved as-is unless changing them is necessary to make the textarea look identical at single-line height.

### Claude's Discretion

### Pattern 1: Controlled textarea with submit-on-modified-Enter
**What:** A React-controlled `<textarea>` that uses `onKeyDown` to intercept only the `Cmd/Ctrl+Enter` combo for submit; everything else (including plain Enter) goes to the default textarea behavior (insert newline).

**When to use:** Capture surface where the dominant interaction is short single-line text but multi-line must be possible.

**Example (canonical Vigil pattern from ThoughtRow.tsx:294-308):**
```typescript
// Source: vigil-pwa/src/components/ThoughtRow.tsx:294-308 (verified in-repo)
function handleKeyDown(e: React.KeyboardEvent<HTMLTextAreaElement>) {
  if (e.key === 'Escape') {
    e.preventDefault()
    // (in ThoughtRow this also reverts draft + ends edit; CaptureBar would just blur)
  } else if (e.key === 'Enter' && (e.metaKey || e.ctrlKey)) {
    e.preventDefault()
    handleSave()  // CaptureBar equivalent: handleSubmit()
  }
  // Plain Enter: allow newline in textarea (do nothing)
}
```

### Pattern 2: scrollHeight auto-grow recipe
**What:** Reset textarea height to `0`, then read `scrollHeight`, then set height to that value. Browser handles the rest (CSS `max-h-*` caps it; `overflow-y-auto` adds the internal scrollbar when content exceeds the cap).

**When to use:** When the textarea needs to grow with content but cap at a visual threshold.

**Why reset to 0 first:** Without the reset, `scrollHeight` reflects the LAST rendered height, not the height the content actually needs. Setting to `0px` first forces the browser to recompute based purely on content.

**Pitfall:** Running this on every keystroke can cause layout thrash. The cost is negligible for a single short textarea (it's not a list of 1000 rows), but use `useLayoutEffect` (not `useEffect`) so the height adjustment happens BEFORE the browser paints — otherwise users see a 1-frame jitter on each keystroke.

**Example:**
```typescript
import { useLayoutEffect, useRef } from 'react'

const textareaRef = useRef<HTMLTextAreaElement>(null)

useLayoutEffect(() => {
  const el = textareaRef.current
  if (!el) return
  el.style.height = '0px'
  el.style.height = `${el.scrollHeight}px`
}, [input])
```

The CSS class `max-h-36 overflow-y-auto resize-none` does the rest: caps at ~9rem (6 text lines at the project's base size), shows an internal scrollbar past that, disables manual resize handle.

### Anti-Patterns to Avoid
- **Don't add a `<form>` wrapper around the textarea.** CaptureBar is not in a form today (verified by `grep "form" ThoughtsPage.tsx` — no `<form>` element anywhere near `<CaptureBar>`). Adding a form would activate browser-native submit-on-Enter, which conflicts directly with D-01.
- **Don't use `onSubmit` on a `<form>`.** Same reason. The submit gesture in this phase is keyboard-only-via-onKeyDown and click-via-Save-button, both already covered by the existing `handleSubmit` function.
- **Don't `setState` inside `useLayoutEffect` to track height.** The height is purely a DOM concern; storing it in React state would re-render every keystroke for no behavioral benefit and create an infinite loop if the effect dep on state.
- **Don't reach for `ResizeObserver` for auto-grow.** The textarea size is driven by its OWN content, not by an external container resize. A ResizeObserver would fire AFTER the user types (not before), causing an extra frame of jitter. `useLayoutEffect` keyed on the `input` state is the right tool.
- **Don't replace `input.trim()` with a regex.** `trim()` is already correct per D-08. A regex like `/\s+/g → ' '` would collapse interior newlines — exactly the bug this phase exists to prevent.
- **Don't add a `name` attribute thinking the textarea needs it.** This is a controlled component with explicit `value` and `onChange`; there is no form to serialize.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Keyboard submit detection | Custom event-bus listening for global `keydown` | `onKeyDown` handler on the textarea itself (matches ThoughtRow pattern) | Element-scoped is simpler, doesn't leak when the component unmounts, doesn't conflict with other inputs on the page (e.g., search bar at top of ThoughtsPage). |
| Auto-grow without scrollHeight | Manual line-count via `(value.match(/\n/g) ?? []).length` and a per-line pixel constant | `el.style.height = el.scrollHeight + 'px'` recipe | scrollHeight accounts for line wrapping, font-size changes, zoom level, and proportional fonts; a line-count approach breaks the moment a single typed line wraps. |
| Mobile newline UX | Polyfill or library | `enterKeyHint="enter"` native HTML attribute [VERIFIED: WHATWG HTML LS; supported in iOS Safari 13.4+, Android Chrome 77+] | Native attribute, zero JS, zero cost. |
| Multi-line clipboard parsing | `onPaste` handler that intercepts and re-inserts text | Nothing — let the browser handle paste natively into the textarea | `<textarea>` accepts multi-line clipboard content by default. Intercepting paste would be a regression. |
| Whitespace-only rejection | Regex `/^\s*$/` | `String.prototype.trim()` then `!trimmed` check | Identical behavior; trim is already the codebase convention (CaptureBar.tsx:15-16, ThoughtRow.tsx:189). |

**Key insight:** This phase is the rare one where **almost every "should we build X" question has the answer "no — the browser already does it."** Resist the urge to add `react-textarea-autosize`, an `onPaste` handler, a form wrapper, or a custom keyboard manager. The browser's native textarea behavior plus 4–6 lines of glue (key handler + scrollHeight effect + enterKeyHint) is the entire solution.

## Common Pitfalls

### Pitfall 1: jsdom returns `scrollHeight = 0`
**What goes wrong:** Tests written against `el.scrollHeight` or against `el.style.height` produce noise or false negatives.
**Why it happens:** jsdom doesn't compute layout. Every element's `scrollHeight`/`clientHeight`/`offsetHeight` is `0` by default.
**How to avoid:** Don't assert on height in unit tests. Test that the value is correct, that submit fires, that the input clears post-submit. Manually verify the visual auto-grow in dev mode and in UAT.
**Warning signs:** A test passes locally with strict `style.height === '24px'` assertions but breaks the next time anyone runs `vitest` on a different machine.

### Pitfall 2: `fireEvent.paste` doesn't actually paste
**What goes wrong:** A test does `fireEvent.paste(textarea, { clipboardData: ... })` and asserts the textarea now contains `"line1\nline2"`, but the value is unchanged.
**Why it happens:** `fireEvent.paste` only dispatches the paste event; it does NOT mutate the input value (that's the browser's job after the default paste handler runs, which jsdom does not implement fully).
**How to avoid:** Use `userEvent.paste(textarea, 'line1\nline2')` from `@testing-library/user-event` — it both fires the event AND inserts the text. This is the established Vigil pattern (already used in `AuthPage.test.tsx`).
**Warning signs:** Test is named "paste preserves newlines" but the only assertion is on `fireEvent.paste` being called, not on the resulting value.

### Pitfall 3: Plain Enter still submits because something else swallows it
**What goes wrong:** User reports "I can't add a newline — pressing Enter just submits."
**Why it happens:** Either (a) the new `<textarea>` was accidentally placed inside a `<form>` (browser submits on Enter) or (b) an `onKeyDown` was left with `if (e.key === 'Enter') { e.preventDefault(); handleSubmit(); }` from the old `<input>` (CaptureBar.tsx:47-52 today) and was not rewritten to require the metaKey/ctrlKey modifier.
**How to avoid:** Confirm no `<form>` wrapper exists (verified: ThoughtsPage.tsx has none). Rewrite the keydown handler exactly per the ThoughtRow pattern in Pattern 1 above.
**Warning signs:** D-01 test ("plain Enter inserts newline, does NOT submit") is the trap-test for this regression — never delete it.

### Pitfall 4: Height doesn't reset after submit
**What goes wrong:** User pastes a long block of text, submits, and the textarea stays expanded (showing 6 empty lines).
**Why it happens:** The auto-grow `useLayoutEffect` keyed on `[input]` only runs when `input` changes. Setting `input = ''` does cause it to re-run, but if the effect uses `scrollHeight` of an empty textarea, the result may still be more than baseline due to padding measurement quirks.
**How to avoid:** The recipe `el.style.height = '0px'; el.style.height = el.scrollHeight + 'px'` handles the empty case correctly because an empty textarea's `scrollHeight` equals the height of one line of text (padding + line-height). Verify with manual UAT.
**Warning signs:** Visual jitter or a slowly-shrinking textarea after submit. If it doesn't snap immediately, the reset path is broken.

### Pitfall 5: `enterKeyHint="enter"` does nothing on desktop
**What goes wrong:** A reviewer notes that pressing Enter on desktop didn't change behavior after adding `enterKeyHint="enter"`.
**Why it happens:** `enterKeyHint` is a hint to **on-screen keyboards only** (iOS Safari, Android Chrome, Windows on-screen keyboard). Desktop physical keyboards ignore it entirely. This is correct behavior.
**How to avoid:** Document this in the test name or comment. Manual mobile UAT is the only meaningful validation.
**Warning signs:** A "desktop Enter behavior" test asserting on `enterKeyHint` — that test is testing the wrong layer.

### Pitfall 6: Triage call receives unfamiliar `\n` and misclassifies
**What goes wrong:** A captured thought with multi-line content gets a wrong category from the triage endpoint.
**Why it happens:** Existing `triageThought(trimmed)` (CaptureBar.tsx:28) sends the full content (now with interior `\n`) to `/v1/triage`. The endpoint is an LLM call — LLMs handle multi-line input fine, but if anyone tuned the prompt for single-line inputs, multi-line could shift category distributions.
**How to avoid:** No code change required, but flag for human verification during UAT. If triage misbehavior shows up, the fix lives in the server-side triage prompt, not in this phase.
**Warning signs:** UAT testers report "my multi-line reflection got classified as a task." Surface this in the UAT script.

### Pitfall 7: Even Hub-style silent no-op on rebuild
**What goes wrong:** Devs commit the textarea swap, refresh the PWA in their browser, and don't see the change.
**Why it happens:** Service worker caches the old bundle.
**How to avoid:** During testing, hard-reload (Cmd+Shift+R / Ctrl+Shift+R) or use a fresh incognito session. Not a code concern, but worth a one-liner in the UAT runbook.
**Warning signs:** "Looks the same as before" feedback from a tester who skipped the hard reload.

### Example 1: Updated CaptureBar.tsx (full target shape — illustrative, not prescriptive)
```typescript
// Source: derived from in-repo patterns: CaptureBar.tsx (current) + ThoughtRow.tsx:94, 294-308, 387-395
import { useLayoutEffect, useRef, useState } from 'react'
import { createThought, triageThought, updateThought, type ThoughtApiResponse } from '../api/client'

interface CaptureBarProps {
  onCapture: (thought: ThoughtApiResponse) => void
  onCategoryUpdate: (id: number, category: string) => void
}

export default function CaptureBar({ onCapture, onCategoryUpdate }: CaptureBarProps) {
  const [input, setInput] = useState('')
  const [isSubmitting, setIsSubmitting] = useState(false)
  const [error, setError] = useState<string | null>(null)
  const textareaRef = useRef<HTMLTextAreaElement>(null)

  // D-03 / D-04: auto-grow up to max-h-36, reset to baseline when input clears.
  useLayoutEffect(() => {
    const el = textareaRef.current
    if (!el) return
    el.style.height = '0px'
    el.style.height = `${el.scrollHeight}px`
  }, [input])

  async function handleSubmit() {
    const trimmed = input.trim()             // D-07: empty / whitespace-only rejected
    if (!trimmed || isSubmitting) return

    setIsSubmitting(true)
    setError(null)

    try {
      const thought = await createThought(trimmed)
      onCapture(thought)
      setInput('')                            // triggers the useLayoutEffect to reset height
      setIsSubmitting(false)

      // Fire-and-forget triage (unchanged from current implementation)
      triageThought(trimmed)
        .then((result) =>
          updateThought(thought.id, {
            category: result.category,
            ...(result.tags ? { tags: result.tags } : {}),
            ...(result.therapyClassification ? { therapyClassification: result.therapyClassification } : {}),
          }).then(() => onCategoryUpdate(thought.id, result.category)),
        )
        .catch(() => { /* swallow 503/502 — thought stays uncategorized */ })
    } catch (err) {
      setError(err instanceof Error ? err.message : 'Failed to save thought')
      setIsSubmitting(false)
    }
  }

  function handleKeyDown(e: React.KeyboardEvent<HTMLTextAreaElement>) {
    if (e.key === 'Enter' && (e.metaKey || e.ctrlKey)) {  // D-01
      e.preventDefault()
      handleSubmit()
    } else if (e.key === 'Escape') {                       // Discretion: blur without clearing
      e.preventDefault()
      e.currentTarget.blur()
    }
    // Plain Enter: do NOT preventDefault — textarea inserts a newline by default.
  }

  return (
    <div className="sticky bottom-0 bg-gray-900 border-t border-gray-900/40 p-4">
      <div className="flex gap-2 max-w-4xl mx-auto items-end">  {/* items-end keeps Save button aligned with bottom of growing textarea */}
        <textarea
          ref={textareaRef}
          value={input}
          onChange={(e) => setInput(e.target.value)}
          onKeyDown={handleKeyDown}
          placeholder="Capture a thought..."                  // D-06
          disabled={isSubmitting}
          rows={1}                                            // D-03 baseline
          enterKeyHint="enter"                                // D-05
          className="flex-1 bg-gray-900/80 border border-gray-400/30 rounded-lg px-4 py-2.5 text-gray-50 placeholder-gray-400 focus:outline-none focus:ring-2 focus:ring-teal-600 resize-none max-h-36 overflow-y-auto"
        />
        <button
          onClick={handleSubmit}
          disabled={input.trim() === '' || isSubmitting}     // D-02 unchanged
          title="⌘+Enter"                                    // Discretion: cheap tooltip
          className="px-4 py-2.5 bg-teal-600 hover:bg-teal-400 disabled:opacity-50 disabled:cursor-not-allowed rounded-lg text-white font-medium transition-colors"
        >
          Save
        </button>
      </div>
      {error && (
        <p className="text-red-400 text-sm mt-1 max-w-4xl mx-auto">{error}</p>
      )}
    </div>
  )
}
```

Notes on the example above:
- **`items-end`** added to the inner flex container — without it, the Save button would re-center vertically as the textarea grows. With `items-end`, the button stays pinned at the bottom of the growing textarea.
- **`useLayoutEffect`** not `useEffect` — fires before paint, prevents flicker.
- **`rows={1}`** baseline (not `rows={2}` or `rows={3}`) per D-03 "visually identical to today's `<input>` height."
- **Platform-detected tooltip:** The example uses `title="⌘+Enter"`. A nicer-but-optional variant: `title={navigator.platform.includes('Mac') ? '⌘+Enter' : 'Ctrl+Enter'}`. Either is fine; planner's call.

## Open Questions

1. **Should the Escape behavior also clear the textarea?**
   - What we know: ThoughtRow's Escape reverts the draft (because edit mode has an "original value to revert to"). CaptureBar has no "original" — it's always either empty or in-progress unsaved input.
   - What's unclear: Whether `Escape` blurring without clearing is the right default vs. `Escape` clearing the input.
   - Recommendation: Default to **blur without clearing** (low risk — accidental Escape doesn't destroy a long paste). Planner may override; documented in CONTEXT.md Discretion section. Add a one-line test pinning whichever choice the planner makes.

2. **Does the Save button benefit from a `Cmd+Enter` tooltip on touch devices?**
   - What we know: `title` attribute on mobile Safari/Chrome shows on long-press but is not a great UX. Adding `title="⌘+Enter"` on a touchscreen is at worst harmless.
   - What's unclear: Whether the Vigil aesthetic prefers a clean button with no tooltip vs. desktop discoverability.
   - Recommendation: Include the tooltip. It's one attribute. Low cost, easy to revert.

3. **Should the textarea autofocus on page load?**
   - What we know: The current `<input>` does NOT autofocus (verified: no `autoFocus` attribute at line 57). ThoughtRow's edit-mode textarea DOES autofocus (line 394), but that's for an explicitly-opened edit session.
   - What's unclear: Whether this phase should change the focus behavior.
   - Recommendation: **Do NOT add autofocus.** D-09 says preserve everything not in scope; focus behavior is not in scope. Keep current "click to focus" interaction.
