<objective>
Convert the Thoughts CaptureBar from a single-line `<input type="text">` to a controlled `<textarea>` with auto-grow (1→6 lines, then internal scroll), the established Vigil keyboard grammar (plain Enter inserts a newline, Cmd/Ctrl+Enter submits, Escape blurs), `enterKeyHint="enter"` for mobile soft-keyboards, and a Save button that remains the primary discoverable submit affordance. Add a new vitest unit test file `CaptureBar.test.tsx` pinning the seven behaviors (CAP-MULTI-01..05 + D-04 post-submit height/value reset + D-07 whitespace-only guard).

Purpose: Close the capture-side gap that surfaced during Phase 115 UAT 2 (2026-04-28) when a pasted three-line reflection silently collapsed because HTML's `<input>` strips newlines on paste per the WHATWG spec. The display side (ThoughtRow `whitespace-pre-line` at line 399) and the entire transport (`JSON.stringify` → `/v1/thoughts` POST → DB → SELECT → render) already preserve `\n` end-to-end; only the capture entry point lags.

Output:
- `vigil-pwa/src/components/CaptureBar.tsx` — single-file edit (element swap + auto-grow effect + keydown rewrite + 1 attribute + 1 flex utility).
- `vigil-pwa/src/components/CaptureBar.test.tsx` — new test file with seven vitest cases.

No new packages. No API changes. No migration. No server work.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| user keyboard / clipboard → CaptureBar textarea (DOM) | User-controlled string crosses into React state. React's controlled-component pattern is the boundary owner. |
| CaptureBar textarea → `createThought` JSON body | Same string crosses the network boundary as a JSON-escaped value via `JSON.stringify`. Identical to the pre-phase boundary (only the element producing the value changed). |
| CaptureBar textarea → `triageThought` LLM prompt | Same string is forwarded to the server-side triage endpoint, which builds an LLM prompt. Pre-existing boundary; multi-line content is a new content shape but not a new boundary. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-999.2-01 | Tampering | `<textarea>` value → display via `ThoughtRow.tsx:401` `{thought.content}` JSX interpolation | accept | React's default JSX interpolation escapes HTML automatically (no `dangerouslySetInnerHTML`); Phase 115 POLISH-01 already proved `whitespace-pre-line` renders the user's literal newlines without script injection. No new mitigation needed — this is the same rendering path the existing `<textarea>` edit mode already uses. |
| T-999.2-02 | Tampering | `JSON.stringify({content, source: 'text'})` POST body — multi-line content | accept | `JSON.stringify` escapes `"`, `\`, and control chars (including `\n`, `\r`, `\t`) per RFC 8259. There is no string the user can type into the textarea that breaks out of the JSON string literal. Verified spec-level guarantee. |
| T-999.2-03 | Information Disclosure | `setError(err.message)` rendering HTTP failure message in the red `<p>` element | accept | The error path surfaces only the HTTP status from `vigilFetch` (`Failed to create thought: 500`), NOT the user's input. Existing pattern preserved verbatim per D-09. No PCM/PII redaction concerns because the content originates from the user themselves. |
| T-999.2-04 | Tampering | Newline-bearing prompt sent to fire-and-forget `triageThought` endpoint (server-side LLM call) | accept | The triage endpoint has accepted arbitrary user text since Phase 84+; multi-line content is a new SHAPE but not a new boundary. Server-side prompt hardening is orthogonal to this phase. RESEARCH Pitfall 6 explicitly flags category-drift as a UAT observability concern with the fix scoped to a follow-on server-side prompt phase, NOT to this phase. |
| T-999.2-05 | Elevation of Privilege (low) | User uses devtools to remove `resize-none` and drag the textarea past the 6-line visual cap | accept | The CSS `max-h-36` enforces the visual cap independent of `resize-none`. Server accepts any `content` length; no data-integrity concern. Even if the user bypasses both, the only consequence is they see more of their own typed content — no privilege boundary is crossed. |
| T-999.2-06 | Denial of Service (low) | User pastes an extremely large multi-line block (e.g., 100k chars) | accept | The PWA does not impose a client-side content cap today (`<input>` had no `maxLength`). Server-side `/v1/thoughts` accepts any content length per existing contract. Adding a cap would be a scope-reduction (D-09 forbids it); if the server starts rejecting large bodies, the existing `setError` path already surfaces the failure. No new attack surface. |

**No new threats are introduced by this phase.** All threats above were already present with the `<input>` element; the only delta is that the user can now type/paste content shapes that include `\n`, which is a content-shape change, not a trust-boundary change. The `<textarea>` element is a strict subset of the input element's attack surface (no `type="file"`, no `formaction`, no `pattern` parser quirks).

**Package legitimacy:** Not applicable — this phase installs zero new packages. All dependencies (`react`, `@types/react`, `vitest`, `@testing-library/*`, `tailwindcss`) are pre-existing and have been in active use across many shipped phases. RESEARCH §"Package Legitimacy Audit" explicitly skips the audit per protocol.
