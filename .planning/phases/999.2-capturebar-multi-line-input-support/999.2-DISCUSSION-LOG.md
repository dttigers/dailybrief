## Submit semantic

| Option | Description | Selected |
|--------|-------------|----------|
| Cmd/Ctrl+Enter saves (Recommended) | Plain Enter inserts newline; Cmd+Enter (macOS) / Ctrl+Enter (Linux/Windows) submits. Matches ThoughtRow edit-mode (Phase 100 EDIT-01) — one Vigil convention. Paste-with-newlines just works. Slight learning curve for new users; Save button still works. | ✓ |
| Enter saves, Shift+Enter newline | Chat-app convention (Slack, ChatGPT, iMessage). Familiar. BUT: breaks parity with ThoughtRow edit-mode (Cmd+Enter there, Enter here = inconsistent), and a paste that ends with a trailing `\n` would immediately submit unless you suppress Enter handling during composition events. | |
| Enter saves always; multi-line only via paste | Enter always submits (current behavior). Newlines only enter the field via paste. Simplest mental model for capture-bar, but you can't type a multi-line thought — only paste one. | |

**User's choice:** Cmd/Ctrl+Enter saves (Recommended)
**Notes:** Aligns CaptureBar with the established `ThoughtRow.tsx:294-308` edit-mode keyboard grammar. Eliminates trailing-newline-paste-auto-submit footgun.

## Auto-grow vs fixed height

| Option | Description | Selected |
|--------|-------------|----------|
| Auto-grow 1–6 lines, then scroll (Recommended) | Starts at single line (matches current visual). Grows up to ~6 lines (~9rem) as user types/pastes. Beyond that, scrolls internally. Sticky bottom layout adjusts naturally. Implemented via `rows={1}` + JS height = scrollHeight on input, capped via max-height CSS. | ✓ |
| Auto-grow unbounded | Textarea grows to fit all content with no cap. A 50-line paste pushes the list way up. Maximum readability of what you're about to submit; can dominate the viewport on long thoughts. | |
| Fixed 1-line, internal scroll | Textarea stays at one line visually; multi-line content scrolls inside. Zero layout shift. Hides the fact that there ARE multiple lines — user might forget the paste contained newlines. | |

**User's choice:** Auto-grow 1–6 lines, then scroll (Recommended)
**Notes:** Keeps sticky-bottom layout sane for large pastes while still surfacing that multiple lines are present.

## Mobile return key

| Option | Description | Selected |
|--------|-------------|----------|
| Return inserts newline (Recommended) | Set `enterKeyHint="enter"` so the return key inserts a newline (matching desktop plain-Enter). Submit on mobile happens via the Save button (already there). Consistent with chosen submit semantic; paste-with-newlines on mobile works trivially. | ✓ |
| Return submits on mobile (split behavior) | Set `enterKeyHint="send"`. Return submits on mobile (more natural for one-handed capture), but desktop and mobile behave differently — same component, different rules. Users who paste multi-line on mobile would have to remove trailing newline. | |
| No hint — let browser pick | Don't set enterKeyHint at all. iOS/Android default "return" key inserts a newline (current textarea default). Less explicit but works. You give up the chance to label the key as "send" if you wanted that later. | |

**User's choice:** Return inserts newline (Recommended)
**Notes:** Single keyboard grammar across desktop and mobile. Explicit `enterKeyHint="enter"` prevents browsers from picking "send" by default in some contexts.

## Placeholder & affordance hints

| Option | Description | Selected |
|--------|-------------|----------|
| Terse placeholder, keep Save button (Recommended) | Placeholder stays `Capture a thought...`. The Save button remains the discoverable submit affordance. Cmd+Enter is a power-user shortcut layered on top, discovered via tooltip on the button. Minimal visual chrome, matches current Vigil aesthetic. | ✓ |
| Hint in placeholder | Placeholder becomes `Capture a thought (⌘+Enter to save)` on desktop. Maximizes discoverability of the keyboard shortcut. Slightly noisier; placeholder disappears once user types, so the hint is fleeting. | |
| Hint below input, dim text | Tiny dim text under the input: `⌘+Enter to save · Enter for newline`. Persistent affordance hint. Adds ~16px vertical chrome to a previously-minimal capture row. | |
| Save button tooltip + kbd badge | Save button gets a `title="⌘+Enter"` tooltip plus a small `⌘⏎` keyboard badge inside or beside the button. Discoverable without text noise; some icon-design work for the badge. | |

**User's choice:** Terse placeholder, keep Save button (Recommended)
**Notes:** Preserves Vigil's minimal capture aesthetic. Optional `title` tooltip on the Save button is left to planner discretion (noted in CONTEXT.md "Claude's Discretion").

## Claude's Discretion

- **Resize handle:** Default to `resize-none` on the new textarea (auto-grow would fight a manual resize). Planner may override.
- **Escape key handling:** No precedent in CaptureBar. Default: blur the textarea without clearing content (so accidental Escape doesn't wipe a long paste). Planner may decide otherwise.
- **Save button tooltip:** Optional `title="⌘+Enter"` (platform-detected). Cheap discoverability win; planner can include or skip.
- **Tests:** No CaptureBar test file exists. Planner should add `CaptureBar.test.tsx` covering paste-preserves-newlines, Cmd+Enter submits, plain Enter inserts newline, whitespace-only rejected, height resets after submit.

## Deferred Ideas

- **Save-button keyboard badge UI (`⌘⏎` chip):** Considered under affordance — rejected for terse placeholder. Revisit if shortcut proves undiscoverable.
- **Draft persistence (auto-save to localStorage on input change):** Natural follow-on for long multi-line thoughts. Own phase.
- **Markdown rendering in display:** Out of scope — `whitespace-pre-line` is the only display-side transform.
- **Auto-split paste into per-line thoughts:** Out of scope and conflicts with this phase's whole point (preserve newlines).
