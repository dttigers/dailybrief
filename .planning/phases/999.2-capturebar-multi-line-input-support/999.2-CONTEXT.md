# Phase 999.2: CaptureBar — Multi-Line Input Support - Context

**Gathered:** 2026-05-19
**Status:** Ready for planning

<domain>
## Phase Boundary

Fix the Thoughts CaptureBar so multi-line input (typed or pasted) preserves `\n` characters into the persisted thought body. Today the element is `<input type="text">` at `vigil-pwa/src/components/CaptureBar.tsx:57`, which strips newlines on paste per HTML spec — and there is no way to type a newline at all. Display side already renders multi-line correctly via Phase 115 POLISH-01 D-16 (`whitespace-pre-line` on `ThoughtRow.tsx:399 <p>`); this phase fixes the capture entry point so the data actually carries newlines downstream.

In scope:
- Swap `<input>` → `<textarea>` in CaptureBar
- Submit gesture (keyboard semantic) and mobile return-key behavior
- Auto-grow sizing behavior of the textarea
- Placeholder/affordance hint policy
- Whitespace-only guard parity (must still reject `"\n\n\n"`)

Out of scope (deferred to other phases):
- Display rendering of multi-line content (already shipped in Phase 115)
- Edit-mode behavior in `ThoughtRow.tsx` (already correct via Phase 100 EDIT-01)
- Markdown rendering, rich text, attachments, draft persistence
- Any API/DB schema change — `createThought` already accepts any string

</domain>

<decisions>
## Implementation Decisions

### Submit semantic
- **D-01:** Plain `Enter` inserts a newline. `Cmd+Enter` (macOS) / `Ctrl+Enter` (Linux/Windows) submits. Matches the established Vigil convention from `ThoughtRow.tsx` edit mode (Phase 100 EDIT-01, see `ThoughtRow.tsx:294-308`) — one keyboard grammar across capture and edit. Trailing-newline pastes do not accidentally submit.
- **D-02:** The existing **Save button** remains the primary discoverable submit affordance. It stays enabled whenever `input.trim()` is non-empty and `!isSubmitting` — identical disable logic to today.

### Auto-grow behavior
- **D-03:** Textarea starts at one line (visually identical to today's `<input>` height) and auto-grows up to **6 lines (~9rem)** as the user types/pastes. Beyond 6 lines, the textarea caps at 6 lines and scrolls internally. Implementation hint for planner: `rows={1}` + a ResizeObserver-free `height = scrollHeight` adjustment on the `onChange` handler, capped via CSS `max-h-36` (or equivalent). This keeps the sticky-bottom layout from being shoved off-screen by very large pastes while still showing the user the bulk of what they're about to submit.
- **D-04:** When the textarea is cleared after a successful submit, height resets back to the single-line baseline.

### Mobile return key
- **D-05:** Set `enterKeyHint="enter"` (and `inputMode` left at default for text) so the on-screen return key inserts a newline, matching desktop plain-Enter. Mobile submit happens via the Save button. Same component, same rules on every platform — no split desktop/mobile behavior.

### Placeholder & affordance
- **D-06:** Placeholder stays terse: `Capture a thought...` (unchanged from today). The Save button carries the visible submit affordance; `Cmd+Enter` is a power-user layer on top. No persistent helper text, no kbd badge, no inline tooltip in the placeholder — preserves the minimal Vigil aesthetic.

### Whitespace / empty-string guard
- **D-07:** Keep the existing `input.trim()` guard at `CaptureBar.tsx:15-16`. `trim()` already strips all leading/trailing whitespace including newlines, so `"\n\n\n"` collapses to `""` and is rejected exactly like today. **No new guard logic needed** — this is mentioned only so the planner does not invent a redundant one.
- **D-08:** Interior newlines (e.g., `"line1\n\nline2"`) must be preserved verbatim — only leading/trailing whitespace is trimmed. This is already the behavior of `String.prototype.trim()`; reaffirming so no one swaps in a regex that collapses internal whitespace.

### Forced (non-discussion) implementation choices
- **D-09:** Element swap is `<input type="text">` → `<textarea>`. All other CaptureBar concerns (fire-and-forget triage at lines 27-40, error display, sticky-bottom container, focus ring, Tailwind class palette) are preserved as-is unless changing them is necessary to make the textarea look identical at single-line height.

### Claude's Discretion
- **Resize handle:** `ThoughtRow` edit-mode textarea uses `resize-y`. For CaptureBar (auto-grow up to 6 lines, then internal scroll), default to **`resize-none`** — manual resize would fight the auto-grow logic. Planner may override if a clear reason emerges.
- **Escape key:** No precedent in CaptureBar today. Reasonable default: `Escape` blurs the textarea without clearing content (so an accidental Escape doesn't wipe a long paste). Planner may decide otherwise — low stakes, easy to revisit.
- **Save button tooltip:** Optional small enhancement — `title="⌘+Enter"` (or platform-detected `Ctrl+Enter`) on the Save `<button>`. Cheap discoverability win, but explicitly NOT placed in the placeholder or as visible helper text per D-06. Planner can include or skip.
- **Tests:** No CaptureBar test file exists today. Planner should add a focused test (`vigil-pwa/src/components/CaptureBar.test.tsx`) covering: paste preserves `\n`, Cmd+Enter submits, plain Enter inserts newline (does not submit), whitespace-only is rejected, textarea resets to baseline height after submit. RTL + Vitest already set up in the project.

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `vigil-pwa/src/components/ThoughtRow.tsx` edit-mode `handleKeyDown` (lines 294-308) — the exact keyboard semantic CaptureBar should mirror. Not literally importable (different state shape) but the logic transfers directly: check `e.key === 'Enter' && (e.metaKey || e.ctrlKey)` for submit; let plain Enter fall through.
- Existing `input.trim()` guard at `CaptureBar.tsx:15-16` — covers the empty-and-whitespace-only requirement (CAP-MULTI-04) without modification.
- Existing fire-and-forget triage at `CaptureBar.tsx:27-40` — completely unaffected by this change; preserve verbatim.

### Established Patterns
- Tailwind utility classes for inputs: `bg-gray-900/80 border border-gray-400/30 rounded-lg px-4 py-2.5 text-gray-50 placeholder-gray-400 focus:outline-none focus:ring-2 focus:ring-teal-600` — keep these on the new `<textarea>` so it looks identical at single-line height.
- Sticky-bottom capture container: `sticky bottom-0 bg-gray-900 border-t border-gray-900/40 p-4` — auto-grow textarea will adjust this naturally because the container's height is content-driven.
- Submit-disable predicate: `input.trim() === '' || isSubmitting` — unchanged.
- Error display: `<p className="text-red-400 text-sm mt-1 ...">` below the row — unchanged.

### Integration Points
- `onCapture(thought)` callback into parent — unchanged. Content with `\n` flows up identically.
- `triageThought(trimmed)` — receives `trimmed` (which now may contain interior newlines). Triage endpoint is text-classification; newlines are harmless. No backend change required.
- `createThought` POST body — `JSON.stringify` will encode `\n` as `\\n` in JSON, decode back to `\n` on the server. Standard.

</code_context>

<specifics>
## Specific Ideas

- The driving real-world incident: during Phase 115 UAT 2 (2026-04-28) the user pasted a three-line reflection into the capture bar and watched the newlines silently disappear. That UAT moment IS the success criterion for this phase — a successful UAT replay is "paste a three-line reflection, see three lines in the thought list."
- One Vigil keyboard grammar across capture and edit (Cmd/Ctrl+Enter saves, plain Enter for newline, Escape blurs/cancels) — this is the through-line the user keeps choosing.

</specifics>

<deferred>
## Deferred Ideas

- **Save button keyboard-badge UI (`⌘⏎` chip inside the button):** Came up under affordance — rejected in favor of terse placeholder + optional `title` tooltip. Revisit if user feedback shows the Cmd+Enter gesture is undiscoverable.
- **Draft persistence (auto-save to localStorage on input change):** Not asked for, but a natural follow-on once long multi-line thoughts are common. Own phase.
- **Rich text / Markdown rendering in display:** Out of scope — `whitespace-pre-line` is the only display-side transform. Markdown belongs in its own phase.
- **Voice / paste-detection that auto-creates separate thoughts per line:** Out of scope and would conflict with this phase's whole point (preserve newlines AS newlines).

</deferred>
