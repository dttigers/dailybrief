### Pattern 2: `concurrently` Invocation

**Verified flags for `concurrently` 9.2.1:** [VERIFIED: github.com/open-cli-tools/concurrently README]

```bash
concurrently \
  --kill-others \
  --names 'core,pwa' \
  --prefix-colors 'blue,magenta' \
  'npm --prefix vigil-core run dev' \
  'npm --prefix vigil-pwa run dev'
```

**Flag semantics:**
- `--kill-others`: When EITHER process exits (for any reason — crash or Ctrl+C), kill the other. This is the right choice for dev: if tsx crashes, vite should stop too.
- `--kill-others-on-fail`: Only kills others on non-zero exit — leaves vite running if tsx exits cleanly. NOT the right default for dev (you want both to stop on Ctrl+C).
- `--names 'core,pwa'`: Prefix each log line with `[core]` or `[pwa]`.
- `--prefix-colors 'blue,magenta'`: Chalk color names; blue = vigil-core, magenta = vigil-pwa.
- Ctrl+C forwarding: automatic — `concurrently` forwards SIGINT to all child processes by default. No `--handle-input` needed. [VERIFIED: concurrently 9.2.1 README]

### Pattern 3: `launchctl bootout` Retirement (D-09)

**Current state on iMac (VERIFIED live):** The daemon IS registered in the Aqua GUI session. `launchctl list com.jamesonmorrill.vigilcore` shows PID (was running as 8368 at research time) but `LastExitStatus=256`, meaning it attempted to start and failed (dist/index.js missing or env vars absent). The plist file exists at `~/Library/LaunchAgents/com.jamesonmorrill.vigilcore.plist`. [VERIFIED: live `launchctl list` on iMac 2026-04-21]

**Two valid `bootout` forms on macOS 15:** [VERIFIED: `launchctl bootout --help` on macOS 15.7.5]

```bash
# Form 1: plist path (documented in CONTEXT.md)
launchctl bootout gui/$UID ~/Library/LaunchAgents/com.jamesonmorrill.vigilcore.plist

# Form 2: service target (more explicit, preferred)
launchctl bootout gui/$UID/com.jamesonmorrill.vigilcore
```

Both forms are valid. The service-target form (Form 2) is preferred because it doesn't depend on the plist still existing on disk (you can delete the plist first and then bootout, but that order is wrong — bootout first, then delete plist).

**CRITICAL GOTCHA:** The plist has `LimitLoadToSessionType = Aqua`. This means:
- `bootout` MUST run in the user's GUI (Aqua) desktop session.
- If the dev script runs `bootout` via SSH or a cron job, it will fail silently or error.
- The retirement step in `dev-setup.sh` must be interactive or document that the user runs it in Terminal.app on the Mac desktop.

`launchctl unload` is the deprecated predecessor — `bootout` is the current form and works on macOS 12+. [VERIFIED: macOS 15 `launchctl` help output]

**Correct retirement sequence:**
```bash
# 1. Bootout (while plist still exists on disk)
launchctl bootout gui/$UID ~/Library/LaunchAgents/com.jamesonmorrill.vigilcore.plist
# 2. Delete the plist
rm ~/Library/LaunchAgents/com.jamesonmorrill.vigilcore.plist
```

### Pattern 4: Homebrew `postgresql@16` Install Behavior

**Formula state:** `postgresql@16` (16.13) is available in Homebrew but NOT currently installed on this iMac. `postgresql@17` (17.9) is also available. The data directory (`/usr/local/var/postgresql@16`) does NOT exist on this machine yet. [VERIFIED: `brew info postgresql@16` + `brew list` 2026-04-21]

**Install side effects:** When `brew install postgresql@16` runs, Homebrew automatically runs `initdb` to create the default cluster:

```
initdb --locale=en_US.UTF-8 -E UTF-8 /usr/local/var/postgresql@16
```

This is idempotent — if the data directory already exists, `initdb` is skipped. [VERIFIED: Homebrew caveats output]

**Formula is keg-only:** Not symlinked into `/usr/local`. Binaries live at `/usr/local/opt/postgresql@16/bin/`. The `dev-setup.sh` must use full path or add to PATH:

```bash
export PATH="/usr/local/opt/postgresql@16/bin:$PATH"
```

**`brew services start postgresql@16`:** Starts Postgres and marks it to restart at login. This is sticky — it persists across reboots. D-16 says do NOT auto-start on every dev invocation. The `dev-setup.sh` should call it once during first-time setup; subsequent `npm run dev` invocations should NOT call it again. The pre-flight check (D-10) verifies Postgres is running and emits the fix command if not.

**Port collision risk:** `postgresql@16` uses port 5432 by default. `postgresql@17` also uses 5432. If both are installed and started, there will be a port conflict. Since `postgresql@17` is not installed, there is no collision risk today. [VERIFIED: `brew list` shows neither formula installed]

### Pattern 5: Pre-Flight Port Check

**Most reliable macOS one-liner for `:3001` occupancy:** [VERIFIED: live test on iMac]

```bash
lsof -iTCP:3001 -sTCP:LISTEN 2>/dev/null
```

- `/usr/sbin/lsof` exists on macOS at that path.
- Returns empty + exit 0 if port is free; returns process info + exit 0 if port is occupied.
- The pre-flight script should check the output, not just the exit code:

```bash
OCCUPANT=$(lsof -iTCP:3001 -sTCP:LISTEN 2>/dev/null)
if [[ -n "$OCCUPANT" ]]; then
  echo "ERROR: port 3001 is already in use:"
  echo "$OCCUPANT"
  echo "Fix: lsof -iTCP:3001 -sTCP:LISTEN | tail -n +2 | awk '{print \$2}' | xargs kill"
  exit 1
fi
```

`nc -z localhost 3001` is an alternative but less informative (no PID). `lsof` is preferred.

### Pattern 6: `tsx --env-file` — Critical Fix

**PITFALL:** The current `vigil-core/package.json` dev script is:
```json
"dev": "tsx watch src/index.ts"
```

`tsx 4.x` does NOT auto-load `.env` files. [VERIFIED: tsx help output shows --env-file is a flag that must be passed explicitly] This means when a developer runs `npm --prefix vigil-core run dev` (or `npm run dev` in vigil-core/), `process.env.DATABASE_URL` will be UNDEFINED unless the shell has already exported it. With D-17, the local `.env` will have the localhost DATABASE_URL — but it won't be picked up automatically.

**Required fix to `vigil-core/package.json`:**
```json

```

`--env-file` was added in Node 20.6 and is supported in Node 20+ (Railway uses Node 20 per Dockerfile) and Node 25 (local machine). [VERIFIED: `node --help | grep env-file` output]

**drizzle-kit** (`npm run db:migrate`) DOES auto-load `.env` from the current working directory — it uses its own dotenv integration. No change needed for migration scripts. [CITED: orm.drizzle.team/docs/drizzle-kit-migrate]

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |

## Common Pitfalls

### Pitfall 1: `tsx watch` silently ignores `.env`
**What goes wrong:** Developer runs `npm run dev` in vigil-core. DATABASE_URL is not set. `connection.ts` logs a warning (`DATABASE_URL not set — PostgreSQL features unavailable`) and sets `db = null`. The server starts on 3001 but every DB query silently fails. Looks like the server is running but POST /v1/thoughts returns 500.
**Why it happens:** `tsx watch` does not auto-load `.env`. This is Node-native behavior — `.env` is never auto-loaded without explicit `--env-file`.
**How to avoid:** Update `vigil-core/package.json` dev script to `tsx --env-file=.env watch src/index.ts`.
**Warning signs:** `[vigil-core] DATABASE_URL not set` in terminal output.

### Pitfall 2: `launchctl bootout` in SSH session fails for Aqua services
**What goes wrong:** Dev runs `dev-setup.sh` via SSH (e.g., from MacBook Pro). `bootout gui/$UID/com.jamesonmorrill.vigilcore` errors: `No such process` or `Could not find specified service`.
**Why it happens:** The plist has `LimitLoadToSessionType = Aqua`. Aqua services only exist in the GUI login session. SSH sessions run in a different domain.
**How to avoid:** Document in `dev-setup.sh` that the daemon retirement step must run in a Terminal.app window on the local Mac desktop, not via SSH. Consider making it an explicit manual step with clear instruction.
**Warning signs:** `launchctl bootout` exits non-zero without a useful error message.

### Pitfall 3: Homebrew keg-only PATH
**What goes wrong:** After `brew install postgresql@16`, running `psql` fails with `command not found`. `createdb vigil_dev` fails.
**Why it happens:** `postgresql@16` is keg-only and not symlinked to `/usr/local`. Binaries are at `/usr/local/opt/postgresql@16/bin/`.
**How to avoid:** `dev-setup.sh` must either use full paths or prepend to PATH:
```bash
export PATH="/usr/local/opt/postgresql@16/bin:$PATH"
```
**Warning signs:** `command not found: psql` after `brew install postgresql@16` succeeds.

### Pitfall 4: Daemon PID is 8368 but LastExitStatus=256 — bootout still required
**What goes wrong:** Developer sees `LastExitStatus=256` (daemon failed to start) and assumes no bootout is needed. After `brew services start postgresql@16`, Postgres starts on 5432 fine. But at next login, launchd tries to restart `vigilcore` again (it's still registered), which may now succeed if dist/index.js appears, hijacking port 3001.
**Why it happens:** A daemon with `LastExitStatus!=0` is still registered. launchd will retry it on the next boot/session.
**How to avoid:** Always run `bootout` + delete plist even if the daemon is currently failed/not-running. [VERIFIED: current daemon state on iMac 2026-04-21]

### Pitfall 5: VITE_API_URL vs VITE_API_BASE naming mismatch
**What goes wrong:** Plans reference `VITE_API_URL` (as written in CONTEXT.md D-11). The actual env var in the codebase is `VITE_API_BASE`. Setting the wrong var name has no effect — PWA defaults to `api.vigilhub.io` prod and still calls prod.
**Why it happens:** CONTEXT.md used `VITE_API_URL` but the actual Vite env var is `VITE_API_BASE` (confirmed in `vigil-pwa/src/api/client.ts` and current `vigil-pwa/.env.local`). [VERIFIED: live file read 2026-04-21]
**How to avoid:** Use `VITE_API_BASE` in all plans and scripts. Current `.env.local` already has the correct name and value (`VITE_API_BASE=http://localhost:3001`).

### Pitfall 6: `seed-local.ts` user must pre-exist before `generate-key.ts` pattern
**What goes wrong:** `seed-local.ts` attempts to insert an API key for a user, but the user row doesn't exist yet. Foreign key constraint fails.
**Why it happens:** `generate-key.ts` shows this pattern: it looks up the user by email, fails if not found, and instructs "create the user via POST /v1/auth/register first". The seed script can't call an HTTP endpoint.
**How to avoid:** `seed-local.ts` must INSERT the user row directly (not via HTTP), then insert the API key referencing the inserted user's `id`. Pattern:
```typescript
// 1. Upsert user
const [user] = await db.insert(users).values({ email, passwordHash }).onConflictDoUpdate(...).returning();
// 2. Insert vk_ API key referencing user.id
await db.insert(apiKeys).values({ userId: user.id, name, keyHash, keyPrefix });
```
Idempotency: use `onConflictDoNothing()` or `onConflictDoUpdate()` for all seed rows.

### Pitfall 7: `VIGIL_ALLOWED_EMAILS` must include seed user email
**What goes wrong:** Local vigil-core starts with the seed user email in the DB, but `VIGIL_ALLOWED_EMAILS` doesn't include it. Registration via POST /v1/auth/register returns 403.
**Why it happens:** `JWT_SECRET + VIGIL_ALLOWED_EMAILS` are fail-closed (per memory note `project_vigil_core_env_gates.md`). Local `.env` must list the seed user email.
**How to avoid:** `dev-setup.sh` template for local `.env` must include `VIGIL_ALLOWED_EMAILS=<seed-email>`.

### Pitfall 8: Anthropic budget cap is workspace-level, not per-key
**What goes wrong:** Dev creates a new API key and expects a per-key $20 cap. The key has no cap. A dev loop bursts requests and hits hundreds of dollars.
**Why it happens:** Anthropic does NOT support per-API-key spending limits. Limits are workspace-level only. [VERIFIED: support.claude.com/workspaces docs 2026-04-21]
**How to avoid:** Create a separate **workspace** in Anthropic Console (Settings > Workspaces) for local dev. Set the workspace spend limit to $20/month (Workspace > Limits tab > Change Limit). Generate the dev API key inside that workspace. The $20 cap applies to all keys in that workspace. [CITED: support.claude.com/en/articles/9796807-creating-and-managing-workspaces]
**Planning implication:** D-18 says "dev-scoped Anthropic API key with $20 budget cap" — this requires creating a workspace, not just a key. The setup instructions must include this workspace-creation step.

## Code Examples

### `dev-setup.sh` Core Skeleton (idempotent, fail-loud pattern)

```bash
#!/usr/bin/env bash
# scripts/dev-setup.sh — First-time local dev bootstrap (idempotent)
# Usage: bash scripts/dev-setup.sh
# NOT a replacement for scripts/bootstrap.sh (which is 1Password-dependent and stale)
set -euo pipefail

SCRIPT_DIR="$(cd "$(dirname "$0")" && pwd)"
REPO_ROOT="$(cd "$SCRIPT_DIR/.." && pwd)"
PG_BIN="/usr/local/opt/postgresql@16/bin"

red()   { printf '\033[31m%s\033[0m\n' "$*"; }
green() { printf '\033[32m%s\033[0m\n' "$*"; }
info()  { printf '[dev-setup] %s\n' "$*"; }

# 1. Retire daemon (if still registered)
if launchctl list com.jamesonmorrill.vigilcore 2>/dev/null | grep -q 'Label'; then
  info "Retiring com.jamesonmorrill.vigilcore launchd daemon..."
  launchctl bootout "gui/$UID/com.jamesonmorrill.vigilcore" || true
fi
PLIST="$HOME/Library/LaunchAgents/com.jamesonmorrill.vigilcore.plist"
if [[ -f "$PLIST" ]]; then
  rm "$PLIST"
  green "  Plist deleted: $PLIST"
fi

# 2. Install postgresql@16 if missing
```

### `pre-flight-check.sh` Skeleton (D-10)

```bash
#!/usr/bin/env bash
# Pre-flight check for npm run dev — fail loud, print exact fix, exit 1
set -euo pipefail

FAIL=0
red()  { printf '\033[31m%s\033[0m\n' "$*"; }
green(){ printf '\033[32m%s\033[0m\n' "$*"; }
info() { printf '[preflight] %s\n' "$*"; }

check_daemon_gone() {
  info "Check 1: vigilcore daemon not registered"
  if launchctl list com.jamesonmorrill.vigilcore 2>/dev/null | grep -q 'Label'; then
    red "  FAIL — daemon still registered. Fix:"
    red "    launchctl bootout gui/\$UID/com.jamesonmorrill.vigilcore"
    red "    rm ~/Library/LaunchAgents/com.jamesonmorrill.vigilcore.plist"
    FAIL=1
  else
    green "  PASS — daemon not registered"
  fi
}

check_port_free() {
  info "Check 2: port 3001 is free"
  OCCUPANT=$(lsof -iTCP:3001 -sTCP:LISTEN 2>/dev/null)
  if [[ -n "$OCCUPANT" ]]; then
    red "  FAIL — port 3001 is in use:"
    red "  $OCCUPANT"
    red "  Fix: lsof -iTCP:3001 -sTCP:LISTEN | tail -n +2 | awk '{print \$2}' | xargs kill"
    FAIL=1
  else
    green "  PASS — port 3001 is free"
  fi
}

check_postgres_running() {
  info "Check 3: postgresql@16 is running"
  if ! pg_isready -h localhost -q 2>/dev/null; then
    red "  FAIL — Postgres not running. Fix:"
    red "    brew services start postgresql@16"
    FAIL=1
  else
    green "  PASS — Postgres is running"
  fi
}

check_env_local() {
  info "Check 4: vigil-core/.env has localhost DATABASE_URL"
  ENV_FILE="$(dirname "$0")/../vigil-core/.env"
  if ! grep -q "DATABASE_URL=postgresql://localhost" "$ENV_FILE" 2>/dev/null; then
    red "  FAIL — vigil-core/.env DATABASE_URL is not pointing at localhost"
    red "  Fix: run bash scripts/dev-setup.sh to reset .env"
    FAIL=1
  else
    green "  PASS — DATABASE_URL points at localhost"
  fi
}

check_daemon_gone
check_port_free
check_postgres_running
check_env_local

if [[ "$FAIL" -ne 0 ]]; then
  red "preflight: FAILED — fix the above and re-run npm run dev"
  exit 1
fi
green "preflight: all checks passed — starting dev servers"
exit 0
```

## Open Questions

1. **`pg_isready` path after install**
   - What we know: `pg_isready` will be at `/usr/local/opt/postgresql@16/bin/pg_isready` (keg-only path)
   - What's unclear: The pre-flight check script will need to use the full path or update PATH
   - Recommendation: All pre-flight scripts use `PG_BIN="/usr/local/opt/postgresql@16/bin"` constant at the top

2. **`seed-local.ts` idempotency semantics**
   - What we know: D-06 says "Idempotent or drop-and-reseed semantics — planner decides"
   - What's unclear: Whether `onConflictDoNothing` (safe, preserves local state) or full drop-and-reseed (always clean) is better
   - Recommendation: `onConflictDoNothing` — preserves local edits made during dev; run `dev-reset.sh` explicitly when you want a clean slate

3. **`sync-anthropic-key.sh` exact changes for D-18**
   - What we know: The script currently syncs from `config.json` canonical → `.env`, plist, Railway; it will flag dev/prod split as drift
   - What's unclear: Whether to retire the script, skip the plist sync (plist is gone), or add a `--local` mode
   - Recommendation: After plist retirement (D-09), the sync script should skip the plist sync step (use `--skip-launchagent` or remove that code block). Add a check: if `vigil-core/.env` differs from `config.json`, that is EXPECTED and should NOT be reported as drift.
