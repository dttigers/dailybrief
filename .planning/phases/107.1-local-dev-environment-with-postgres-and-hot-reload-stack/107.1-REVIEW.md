# Phase 107.1: Code Review Report

**Reviewed:** 2026-04-21T19:00:00Z
**Depth:** standard

## Summary

Phase 107.1 delivers a cohesive local-dev bootstrap (`dev-setup.sh`), a pre-`concurrently` health gate (`preflight-check.sh`), a seed script with idempotent-by-design inserts, a destructive reset (`dev-reset.sh`), and a RUNBOOK-level documentation pass. Overall quality is high — the scripts consistently use `set -euo pipefail`, print explicit fix commands on failure, and avoid auto-heal magic. Secret handling is solid: no hard-coded API keys, `.env` properly gitignored with timestamped backups on overwrite, and `sync-anthropic-key.sh` correctly plumbs `ANTHROPIC_API_KEY` via argv (not interpolated shell strings) into both the python processor and the Railway CLI.

One destructive-path gap warrants pre-merge attention: `dev-reset.sh` can be coerced into dropping a remote database because it does not bind `dropdb` to `localhost` and does not run the preflight gate before the destructive step. Three consistency/completeness warnings (missing JWT_SECRET placeholder detection, no DB identity echo before drop, preflight gate skipped in reset path) and several info-level observations round out the review.

The known `work_orders` schema-drift issue is documented in the RUNBOOK and deferred-items.md — flagged out of review scope.

## Critical Issues

### CR-01: `dev-reset.sh` can drop a REMOTE database if `PGHOST` is set in the user's shell

**File:** `Scripts/dev-reset.sh:19-21`
**Issue:** The destructive `dropdb --if-exists vigil_dev` and subsequent `createdb vigil_dev` commands do not pass `-h localhost`. If the invoking shell has `PGHOST` exported (common on machines that point `psql` at a staging/prod Postgres, including MacBook Pro per the per-machine memory note), both commands will target the REMOTE host — silently dropping and recreating a remote `vigil_dev` database with zero confirmation of which host is about to be nuked. The confirmation prompt on line 11 says "DROP and rebuild vigil_dev" but never echoes the target host, so the operator has no way to catch the mistake. This is the same class of risk `preflight-check.sh` guards against for `DATABASE_URL` in the `npm run dev` path, but `dev-reset.sh` bypasses that gate entirely.

Relatedly: the confirmation is `y/Y` single-character acceptance — fine for a pure-local script, but combined with the missing host binding makes the blast radius larger than the operator can see.

**Fix:**
```bash
# At top of script, after the PATH export:
export PGHOST=localhost  # belt-and-braces pin — dropdb/createdb/psql all honor this

# And before the confirm prompt, echo the exact target so the operator sees it:
echo "Target host: localhost"
echo "Target database: vigil_dev"
printf "This will DROP and rebuild vigil_dev on localhost (all local data is lost). Continue? [y/N] "
read -r CONFIRM
[[ "$CONFIRM" != "y" && "$CONFIRM" != "Y" ]] && { echo "aborted."; exit 0; }

# And run the preflight gate BEFORE the drop to catch a Railway DATABASE_URL in .env,
# mirroring `npm run dev`:
bash "$SCRIPT_DIR/preflight-check.sh" || { echo "preflight failed — refusing to reset"; exit 1; }

# Then:
dropdb -h localhost --if-exists vigil_dev
createdb -h localhost vigil_dev
```

## Warnings

### WR-01: `dev-setup.sh` does not detect an unreplaced placeholder `JWT_SECRET`

**File:** `Scripts/dev-setup.sh:100-107`
**Issue:** Step 8 (lines 100–107) warns the operator if `ANTHROPIC_API_KEY` is still `sk-ant-dev-workspace-key-here` or empty, but there is no equivalent check for `JWT_SECRET`. The `.env.example` default (`replace-me-with-32-plus-random-bytes-for-local-dev-only`) is 56 characters — it passes the 32-char floor enforced by `src/utils/jwt.ts` — so a user who runs `dev-setup.sh` and never edits `vigil-core/.env` will ship with a KNOWN JWT_SECRET value. For local-only dev the impact is bounded, but it's a footgun because it silently works instead of failing loud like the rest of this phase's design.

**Fix:**
```bash
# Add alongside the ANTHROPIC_VAL check:
JWT_VAL=$(grep -E '^JWT_SECRET=' "$CORE_ENV" | head -1 | cut -d= -f2- || true)
if [[ "$JWT_VAL" == "replace-me-with-32-plus-random-bytes-for-local-dev-only" ]] || [[ -z "$JWT_VAL" ]]; then
  MANUAL_NOTES+=("Replace JWT_SECRET in vigil-core/.env with a fresh random value:")
  MANUAL_NOTES+=("    openssl rand -hex 32")
fi
```

### WR-02: `dev-reset.sh` skips the `preflight-check.sh` gate before destroying data

**File:** `Scripts/dev-reset.sh:11-26`
**Issue:** `npm run dev` (package.json:6) runs `preflight-check.sh` before starting `concurrently`, which is the correct pattern — fail loud with an exact fix command before touching anything. `dev-reset.sh` performs strictly MORE destructive operations (dropdb → createdb → migrate → seed) yet runs no preflight. In particular, check 4 (`DATABASE_URL points at localhost`) is precisely what you want verified before a reset. Combined with CR-01 this is how a user loses prod data. Listed separately because even after fixing CR-01 (`PGHOST=localhost`), the `db:migrate` and `seed:local` steps still consume `DATABASE_URL` from `.env`, so a Railway URL in `.env` would cause the migration and seed to hit prod even though `dropdb` was correctly scoped to localhost.

**Fix:** Invoke preflight at the top of `dev-reset.sh` (see CR-01 fix snippet). This catches a Railway `DATABASE_URL` before any destructive work runs.

### WR-03: `dev-setup.sh` does not abort when `vigil-core/.env` contains a Railway hostname

**File:** `Scripts/dev-setup.sh:75-80`
**Issue:** When an existing `.env` already contains `rlwy.net` / `railway.app` / `proxy.rlwy`, the script prints a yellow WARNING and appends a manual note, then continues. It then proceeds to run `db:migrate` (line 93) and `seed:local` (line 97), both of which consume `DATABASE_URL` via `tsx --env-file=.env`. If the operator misses the warning (or if color escapes are stripped by a non-TTY terminal), the script will migrate and seed against a Railway host. The final preflight at step 9 catches this too late — after the destructive migrate and seed have already run.

**Fix:**
```bash
if grep -qE '(rlwy\.net|railway\.app|proxy\.rlwy)' "$CORE_ENV"; then
  red "[dev-setup] FAIL — vigil-core/.env contains a Railway hostname."
  red "  Backup saved at $BACKUP."
  red "  Fix: edit $CORE_ENV — DATABASE_URL should be postgresql://localhost:5432/vigil_dev"
  red "  Then re-run: bash scripts/dev-setup.sh"
  exit 1
fi
```

### WR-04: `sync-anthropic-key.sh` hard-codes stdin termination inside the `--include-config-env` heredoc with `"$KEY"` passed as argv — but the key is still printed on command failure

**File:** `Scripts/sync-anthropic-key.sh:85-101`
**Issue:** The python heredoc receives the key as `sys.argv[2]` (correct — avoids shell interpolation injection). However, if the `python3` process crashes mid-write (SIGINT, SIGKILL, OOM), the `ANTHROPIC_API_KEY` line is partially written to `$ENV_PATH` without a trailing newline, and the prior contents are truncated because `p.write_text()` opens in `w` mode. There is no atomic-write-then-rename pattern, so the operator's `~/.config/dailybrief/.env` could be left in a corrupt half-written state. Because this file is consumed by the Mac-app path (not vigil-core), a corrupted `.env` is harder to notice than the vigil-core `.env` backed up by `dev-setup.sh`. Note: this is pre-existing behavior — Plan 06 only amended whether the block runs, not the write semantics — but it's in scope because the phase's secret-handling story documents `sync-anthropic-key.sh` as the canonical heal command.

**Fix:**
```python
# In the python heredoc, write to a temp file in the same directory then os.replace:
import os, tempfile
dirpath = os.path.dirname(path) or "."
fd, tmp = tempfile.mkstemp(dir=dirpath, prefix=".env.tmp.")
try:
    with os.fdopen(fd, "w") as f:
        f.write("\n".join(out) + "\n")
    os.replace(tmp, path)  # atomic on POSIX
except Exception:
    os.unlink(tmp)
    raise
```

## Info

### IN-01: `seed-local.ts` prints the raw `vk_` key to stdout unconditionally

**File:** `vigil-core/scripts/seed-local.ts:146-149`
**Issue:** On first-run (no existing api_keys rows), the raw `vk_<64 hex>` key is written to stdout via `console.log`. This is the only way the operator can capture the unrecoverable key, so the behavior is correct — but the key ends up in shell history via scrollback, in CI logs if the script is ever run in CI, and potentially in editor terminal buffers. Acceptable for local dev fixture, but worth flagging so the pattern isn't copied to a higher-stakes surface later.
**Fix:** Consider writing the key to a well-known dotfile (`vigil-core/.seed-key` with `0600` perms, gitignored) in addition to stdout — so the operator has a recoverable-on-disk copy and the stdout output can be throwaway. Not blocking.

### IN-02: `seed-local.ts` hard-codes `SEED_PASSWORD = "dev-password-1234"`

**File:** `vigil-core/scripts/seed-local.ts:39`
**Issue:** The literal password is embedded in source (grep-discoverable across the repo). For a local-dev fixture this is correct — it has to be deterministic so the operator can log in — but it should be explicitly marked as "local dev only, do NOT use this password in any other context" in a comment that greps will catch. Currently the comment on the same line says "local-only, replaceable via /v1/auth/login flow", which is accurate but understates the "do not ship to prod" aspect.
**Fix:** Expand the comment:
```typescript
// LOCAL DEV ONLY. This password is committed to source so the seed is deterministic
// across machines. If you see this string anywhere outside vigil-core/scripts/seed-local.ts,
// it's a bug — rotate via POST /v1/auth/login.
const SEED_PASSWORD = "dev-password-1234";
```

### IN-03: `dev-setup.sh` and `dev-reset.sh` assume Intel Homebrew path `/usr/local/opt/postgresql@16/bin`

**File:** `Scripts/dev-setup.sh:11`, `Scripts/dev-reset.sh:8`, `Scripts/preflight-check.sh:10`, `Scripts/dailybrief-doctor.sh:228`
**Issue:** `PG_BIN="/usr/local/opt/postgresql@16/bin"` is hard-coded to the Intel Homebrew prefix. Per the MacBook Pro memory note this is correct for Jameson's current two machines (iMac + MacBook Pro are both x86_64 with `/usr/local` brew), but Apple Silicon uses `/opt/homebrew/...`. If a future machine is M-series, every script breaks silently (the `-x` guards will trip with an actionable error, so "silently" is overstated — but it's still a per-script hard-code that diverges from `$(brew --prefix postgresql@16)`).
**Fix:**
```bash
PG_BIN="$(brew --prefix postgresql@16 2>/dev/null)/bin"
# OR keep the hard-code but centralize it in one file sourced by all four scripts
```

### IN-04: README.md uses lowercase `scripts/` but the physical directory is `Scripts/`

**File:** `README.md:71, 119, 120`
**Issue:** `README.md` line 71 instructs `bash scripts/dev-setup.sh` (lowercase s), but `ls` shows `Scripts/` (capital S). On macOS with the default case-insensitive APFS volume this works; on Linux CI or a case-sensitive macOS volume it would fail. The RUNBOOK is also mixed: `bash scripts/dev-setup.sh` (line 129, 165) lowercase. `package.json:6` also uses lowercase `scripts/preflight-check.sh`. Four out of five references are lowercase; the physical directory is the outlier.
**Fix:** Rename `Scripts/` → `scripts/` via `git mv` to match all documentation and the orchestrator script. Single-case is less error-prone.

### IN-05: `dailybrief-doctor.sh` health-check row parses the JSON body as a positional argv, which rejects bodies containing literal single quotes

**File:** `Scripts/dailybrief-doctor.sh:193-208`
**Issue:** The health response body is captured into `HEALTH_RESPONSE` and passed to `python3 -c '...' "$HEALTH_RESPONSE"`. The python then does `json.loads(sys.argv[1])`. This is safer than shell-interpolating it into the script body, BUT note the comment on line 186 claims "This is the ONE place where parsing the JSON body is correct — because the row is informational only." That's fine as a scoping rationale, but the code doesn't handle the case where the body contains a NUL byte or exceeds `ARG_MAX` (unlikely for a /health response, but matches the ARG_MAX memory note about photo endpoints). Purely defensive — health responses are small and under the script's control.
**Fix:** No change required; documenting for future auditors. If the health response ever grows, pipe stdin instead of argv:
```bash
HEALTH_INFO=$(printf '%s' "$HEALTH_RESPONSE" | /usr/bin/python3 -c '
import json, sys
d = json.loads(sys.stdin.read())
...')
```
