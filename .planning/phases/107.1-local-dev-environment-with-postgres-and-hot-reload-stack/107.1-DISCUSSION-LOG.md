## Daemon vs dev-server conflict

### Daemon disposition

| Option | Description | Selected |
|--------|-------------|----------|
| Retire it on all dev machines | Bootout + remove plist (iMac matches MacBook pattern) | ✓ |
| Auto-bootout wrapper in `npm run dev` | Keep plist; dev script handles unload/reload | |
| Keep daemon, repoint at local Postgres | Daemon + dev share local DB; bootout still required | |
| Move dev to a different port (3002) | Daemon keeps :3001; PWA + OAuth redirects change | |

**Claude's finding that anchored the decision:** Mac apps all point at `https://api.vigilhub.io/v1` ([Sources/JarvisCore/Config/AppConfig.swift:28](Sources/JarvisCore/Config/AppConfig.swift#L28)) — daemon serves no real traffic today.

### Pre-flight safety

| Option | Description | Selected |
|--------|-------------|----------|
| Pre-flight check prints fix command | Explicit; user runs fix manually | ✓ |
| Auto-heal — script fixes what it can | Less friction; loses visibility into what was "wrong" | |

### Dev port

| Option | Description | Selected |
|--------|-------------|----------|
| 3001 (same as prod/daemon) | No PWA/OAuth reconfiguration | ✓ |
| Different port (e.g., 3002) | Frees :3001 for potential daemon return; more config churn | |

### `dist/` maintenance

| Option | Description | Selected |
|--------|-------------|----------|
| No — remove `dist/` from daemon workflow | Daemon retired; `dist/` no longer a fresh-artifact concern | ✓ |
| Yes — something else depends on dist/ | Unknown caller would block the cleanup | |
