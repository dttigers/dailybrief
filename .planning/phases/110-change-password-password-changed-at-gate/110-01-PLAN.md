<objective>
Add the `users.password_changed_at` TIMESTAMPTZ NOT NULL column to the schema
source-of-truth and ship the migration that lands it cleanly on the live DB.
This is the foundation for AUTH-09 — the column is the truth source the
bearerAuth gate (Plan 02) reads on every JWT request and the change-password
endpoint (Plan 02) writes on every successful change. Phase 112 (AUTH-10
forgot-password) will also write to this column on reset.

**Purpose:** Plans 02 and 03 cannot start until this column exists in the live
DB — `bearerAuth` would crash on the SELECT, `change-password` would fail to
write, and the test suite would compile-error against a missing column on the
drizzle-inferred type. This plan is structurally Wave 1 because Plans 02 and 03
both import from `users` schema and both need the column live before any test
runs against a real DB.

**Output:** A modified `vigil-core/src/db/schema.ts` with the new column placed
immediately after `updatedAt` (D-01); a new migration file
`vigil-core/drizzle/0015_add_password_changed_at.sql` following the Phase
102/108 5-step template (D-02); the migration applied to the live DB via
`npm run db:migrate` (CRITICAL [BLOCKING] — `npm run build` would falsely pass
without this step because drizzle-kit infers types from schema.ts directly,
masking a missing-column runtime failure).

**Scope boundary:** No endpoint changes (Plan 02). No middleware changes (Plan
02). No PWA changes (Plan 03). This plan touches schema files and runs
migration only.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| migration SQL → Postgres catalog | DDL is executed by the role attached to DATABASE_URL. The 3 statements (ADD COLUMN, UPDATE, SET NOT NULL) are minimal-scope; no DROP, no TRUNCATE, no permission grant. |
| backfill UPDATE → users.created_at | The backfill copies an EXISTING column's value to a NEW column on the same row. No external input; no SQL injection surface (no parameter binding from user input). |
| post-migration deploy → existing user sessions | The strict-less-than gate (Plan 02 D-05) combined with `password_changed_at = created_at` backfill (D-03) guarantees zero existing JWTs are invalidated by the deploy itself. This is a load-bearing semantic: D-03 is the mitigation. |
