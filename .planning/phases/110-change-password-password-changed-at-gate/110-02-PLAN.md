<objective>
Build the AUTH-09 enforcement layer: the bearerAuth JWT-iat gate that
invalidates pre-change tokens, and the POST /v1/auth/change-password endpoint
that updates the password hash + password_changed_at and re-issues a fresh JWT
so the calling client stays logged in. Both pieces must ship in the same plan
so the gate's "Session expired" body has at least one endpoint whose response
exercises it under test.

**Purpose:** Plan 01 added the column to the DB. This plan turns that column
into actual security primitive: every JWT request now passes through a
strict-less-than iat gate, and authenticated users can change their password
with the calling session staying valid. The D-14 ordering (signToken AFTER
db.update) is load-bearing — getting it wrong creates a race window where the
just-issued token would bounce against its own write. A test pins the ordering.

**CRITICAL ARCHITECTURAL NOTE (D-09 'index.ts:151 pattern'):** The
change-password handler MUST live on a router that is mounted in `index.ts`
**AFTER** the global `bearerAuth` dispatcher at line 116 (i.e., in the
protected block alongside `prioritize` at line 152). The existing `auth`
router (vigil-core/src/routes/auth.ts) is mounted at index.ts:109 — BEFORE
the dispatcher — because /auth/register and /auth/login are intentionally
public. Appending change-password to that router would make change-password
structurally PUBLIC and `c.get("userId")` would be undefined → silent auth
bypass. This plan therefore creates a NEW router file
`vigil-core/src/routes/change-password.ts` (Option A — mirrors Phase 109
prioritize.ts verbatim) and adds a single mount line in index.ts's protected
block.
