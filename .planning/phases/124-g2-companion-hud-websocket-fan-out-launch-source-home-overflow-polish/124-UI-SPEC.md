# Phase 124 — UI Design Contract

> Visual and interaction contract for the G2 Companion HUD + WebSocket fan-out + launch-source/home-overflow polish phase.
> Generated by gsd-ui-researcher, verified by gsd-ui-checker.
>
> **IMPORTANT PLATFORM NOTE:** This phase targets the Even Realities G2 glasses display via the Even Hub SDK.
> The display is 576×288 px, 16-level greyscale (0=black, 15=white), rendered via `TextContainerProperty`
> primitives — NOT a browser DOM. There is no HTML, CSS, Tailwind, shadcn, or font selection.
> "Typography" and "color" sections in this contract express SDK field values, not web tokens.

| Property | Value |
|----------|-------|

## Container ID Allocation

**Critical finding:** The SDK's `containerTotalNum: 1~12` limit applies **per page/screen rebuild**, not as a global total across all screens. Each `CreateStartUpPageContainer` / `RebuildPageContainer` call declares only the containers for that screen. The comment in `constants.ts` ("max 12 total across all screens") is incorrect — it is 12 per screen, not 12 globally.

**Decision:** Allocate new Companion IDs 13, 14, 15. Since IDs are per-page (only the active screen's containers exist at any moment), reusing ID space is not required and would add complexity. The dynamic `RebuildPageContainer` fallback noted in CONTEXT.md Discretion is NOT needed.

| ContainerId Constant | Value | Screen | Role |
|----------------------|-------|--------|------|
| HOME_HEADER | 1 | Home | Unified VIGIL header |
| HOME_BODY | 2 | Home | Task count + top priority body |
| HOME_FOOTER | 3 | Home | Nav hint footer |
| WORK_ORDERS_HEADER | 4 | Work Orders | Unified VIGIL header |
| WORK_ORDERS_LIST | 5 | Work Orders | List container |
| WORK_ORDERS_FOOTER | 6 | Work Orders | Nav hint footer |
| AFFIRMATION_HEADER | 7 | Affirmation | Unified VIGIL header |
| AFFIRMATION_BODY | 8 | Affirmation | Affirmation text body |
| AFFIRMATION_FOOTER | 9 | Affirmation | Nav hint footer |
| TASK_DETAIL_HEADER | 10 | Task Detail | Unified VIGIL header |
| TASK_DETAIL_BODY | 11 | Task Detail | Task content body |
| TASK_DETAIL_FOOTER | 12 | Task Detail | Nav hint footer |
| **COMPANION_HEADER** | **13** | **Companion** | **Unified VIGIL header (with N/M indicator)** |

**Screen const:** `Screen.COMPANION = 'companion'`
**Carousel slot:** 2 (between HOME and WORK_ORDERS)
**New SCREEN_ORDER:** `[HOME, COMPANION, WORK_ORDERS, AFFIRMATION]`

**COMPANION_BODY (ContainerId 14)**

#### Body content composition (3-line HUD)

Lines are joined with `\n`. No DIVIDER row inside the body — the `borderWidth: 1` border provides visual separation.

**Line 1 — Session label** (top):
- Truncate session `label` field to 30 chars max. If truncated, append `…` (Unicode U+2026). This leaves 2 chars of breathing room within CHARS_PER_LINE=32 before the header indicator is needed.
- Truncation is left-biased: take the first 30 chars. Do NOT do directory-aware right-segment truncation in this phase (deferred per CONTEXT.md).
- Example: `vigil-watch — Claude Code…`

**Line 2 — State** (middle):
Exact strings from the 5 Vigil event types:

| Event received | Middle line text |
|----------------|-----------------|
| `needs_input` | `waiting for input` |
| `task_complete` | `done` |
| `task_failed` | `failed` |
| `milestone` | `running` |
| `heartbeat` | `running` |
| No events yet (initial) | `idle` |

**Line 3 — Last event message** (bottom):
- Use the `message` field from the most recent agent event (max 280 chars from API).
- Truncate to 32 chars + `…` for display (no marquee — SDK `TextContainerProperty` has no scroll field).
- If `message` is empty or absent: show the event name as fallback (e.g., `heartbeat`, `task_complete`).

**Full body string template:**
```
{line1_session_label}\n{line2_state}\n{line3_message}
```

No blank lines between the three content lines — the 210px body with paddingLength 8 has sufficient visual separation from the border.

#### Banner overlay states

The "banner" is implemented by replacing the `COMPANION_BODY` content with a banner-priority string, not by adding a 4th container. Use `RebuildPageContainer` to update only the body container when a banner state activates.

**Persistent banner (needs_input or task_failed):**
```
[NEEDS INPUT]           ← or [TASK FAILED]
{session_label_30chars}
{message_32chars}
```
- ALL-CAPS banner label on line 1, inside square brackets.
- Session label (truncated to 30) on line 2.
- Message (truncated to 32) on line 3.
- Persists until ack via DOUBLE_CLICK (D-08 rule 1) OR a superseding event arrives for that session.
- `needs_input` and `task_failed` have equal banner priority. If both are active simultaneously on different sessions, the most-recently-received event's session is displayed (D-09 cycling applies).

**Toast banner (task_complete or milestone):**
```
[DONE]                  ← or [MILESTONE]
{session_label_30chars}
{message_32chars}
```
- Same 3-line structure as persistent banner.
- Auto-clears after **3 seconds** via `setTimeout`. Decision: 3s is the spec-verbatim value from the milestone spec table ("3-second toast"). This is the locked value; do not make it configurable in this phase.
- Implementation: `setTimeout(() => rebuildCompanionBody(normalState), 3000)`.

**Heartbeat (no banner):**
- Does NOT show a banner.
- Updates line 2 (state) to `running` if not already.
- Rebuilds only `COMPANION_BODY` content with updated state.

#### N/M Indicator (header rightSide)

When ≥2 active sessions exist, replace the HH:MM clock in the header `rightSide` with a session position indicator.

**Format:** `N/M` where N = current session index (1-based) and M = total active sessions.

**Width budget check:** `buildVigilHeader` pads between WORDMARK "VIGIL" (5 chars) and rightSide within CHARS_PER_LINE=32. Maximum rightSide that fits without wrapping: 32 - 5 - 1 (min space) = 26 chars. `N/M` format: max "9/9" = 3 chars — fits easily. With a realistic session count of 1–9, format is always ≤5 chars.

**When exactly 1 active session:** `rightSide` is omitted → header falls back to HH:MM AM/PM clock (standard behavior).
**When 0 active sessions (empty state):** `rightSide` is also omitted → HH:MM clock.
**When ≥2 sessions:** `rightSide = '${currentIdx+1}/${total}'` (e.g., `'2/3'`).

### 2. Home Screen Polish (G2-POLISH-07 — AGENT-HUD-01 co-resident)

**File:** `vigil-g2-plugin/src/screens/home.ts` — single-file change.

**New body content composition:**

```typescript
const bodyContent = [
  `* ${pendingCount} tasks pending`,
  '',
  'TOP PRIORITY:',
  topPriority,
].join('\n')
```

**Removed from existing body:**
- Blank line after `topPriority`
- `DIVIDER` row (`━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━`)

**The `affirmation` parameter** is removed from `buildHomeContainers` and both exported functions (`buildHomeScreen`, `rebuildHomeScreen`). Callers in `navigation.ts` and `main.ts` drop the `fetchAffirmation()` call from the `Promise.all` when building the home screen.

**Verification gate (D-14):** Two consecutive `VITE_SCREENSHOT_MODE` pipeline runs must produce byte-identical PNGs for the home body region. The plan will include a verification step that captures and diffs screenshots before declaring the fix complete.

### 3. Offline Indicator (D-11 — on Companion and optionally all screens)

**Where it appears:** The `rightSide` parameter of `buildVigilHeader` on the **Companion screen header only** (ContainerId 13). It does not appear on Home, Work Orders, or Affirmation headers — those screens don't have an SSE connection to report on.

**Glyph decision:** Use `'!'` (exclamation mark, ASCII 0x21) rather than `'⚠'` (U+26A0 WARNING SIGN). Rationale: the SDK renders glyphs via an internal monospace font whose Unicode coverage is unverified. ASCII characters are guaranteed to render. The `'⚠'` glyph was proposed in CONTEXT.md D-11 but "or `'!'` if glyph is unreliable" was noted. Lock `'!'` to avoid a hardware-only rendering regression.

**rightSide priority (Companion header):**

1. If SSE disconnected AND ≥2 active sessions: `rightSide = '! 2/3'` (offline indicator + session count, space-separated, max ~7 chars — fits within the 26-char rightSide budget).
2. If SSE disconnected AND ≤1 active session: `rightSide = '!'`.
3. If SSE connected AND ≥2 active sessions: `rightSide = '2/3'`.
4. If SSE connected AND ≤1 active session: omit `rightSide` → HH:MM fallback.

**Clear condition:** On successful SSE reconnect, the `'!'` prefix is removed from `rightSide` and the header is rebuilt.

| Field | Value | Usage |

| `height` — footer | 38 | All screens (250 to 288 = 38px) |

Exceptions: none for this phase. Companion uses the same 40/210/38 split as all existing screens.

## Typography

Typography on G2 is SDK-controlled. The following are text composition rules, not font/size tokens:

| Role | Composition Rule | Max chars |
|------|-----------------|-----------|
| Header line 1 | `VIGIL` + padding + `{rightSide}` — left/right aligned within 32 chars | 32 |
| Header line 2 | `DIVIDER` constant (32× `━`) | 32 |
| Companion line 1 (session label) | Plain text, truncated to 30 chars + `…` if longer | 31 |
| Companion line 2 (state) | One of: `idle`, `running`, `waiting for input`, `done`, `failed` | 15 |
| Companion line 3 (message) | Plain text, truncated to 32 chars + `…` if longer | 33 |
| Banner line 1 (type) | `[NEEDS INPUT]`, `[TASK FAILED]`, `[DONE]`, `[MILESTONE]` | 14 |
| Home body line 1 | `* N tasks pending` | 20 |
| Home body line 3 | `TOP PRIORITY:` | 13 |
| Home body line 4 (task) | First `summary.recent[0]?.content` value, no truncation (SDK wraps) | unlimited |
| Footer text | Nav hint copy — see Copywriting Contract | 40 |

**Line wrapping:** The SDK handles text that exceeds CHARS_PER_LINE by wrapping. For the 3-line companion body, we must NOT rely on SDK wrapping for lines 1–2 (they must be on exactly one visual line each). Lines are kept under 32 chars via explicit truncation in TypeScript before passing to `content`.

## Color

Greyscale only. No hex values — the SDK uses integer 0–15:

| Role | Value | Usage |
|------|-------|-------|
| Body border | 15 (max brightness / white) | `borderColor: 15` on all body containers |
| Header/footer border | 0 (black / no visible border) | `borderColor: 0` on header and footer containers |
| Background | implicit (SDK-controlled, dark) | Not configurable via TextContainerProperty |
| Text | implicit (SDK-controlled, white on dark) | Not configurable |

**Banner visual differentiation:** The banner state does NOT change `borderColor` or `borderWidth` — it changes only the text `content`. The square-bracket notation (`[NEEDS INPUT]`) provides visual priority signaling within the SDK's text rendering constraints. Do not attempt to change border properties per state (would require a rebuild of all 3 containers; content-only rebuild is cheaper and sufficient).

## Navigation + Interaction Contract

### Carousel order (locked — D-05)

```typescript
const SCREEN_ORDER: readonly ScreenName[] = [
  Screen.HOME,        // 0
  Screen.COMPANION,   // 1 (NEW)
  Screen.WORK_ORDERS, // 2
  Screen.AFFIRMATION, // 3
]
```

### DOUBLE_CLICK on Companion (locked — D-08)

Context-sensitive. Evaluated in order:

1. **If banner active** (needs_input or task_failed visible) → dismiss/ack the banner. Body rebuilds to normal 3-line state for the current session.
2. **Else if ≥2 active sessions** → cycle to next session (currentSessionIndex = (currentSessionIndex + 1) % activeSessions.length). Body + header rebuild.
3. **Else** → navigate to HOME (matches existing `handleNavEvent` fallback behavior).

**Implementation site:** `handleNavEvent` switch in `navigation.ts` — add a `currentScreen === Screen.COMPANION` branch BEFORE the generic `DOUBLE_CLICK_EVENT → Screen.HOME` case.

### SCROLL_BOTTOM on Companion

Navigate to next screen in SCREEN_ORDER: `WORK_ORDERS`.

### SCROLL_TOP on Companion

Navigate to previous screen in SCREEN_ORDER: `HOME`.

### DOUBLE_CLICK on HOME (unchanged)

Fires `bridge.shutDownPageContainer(1)` — unchanged from existing behavior. The Companion insertion does NOT change Home's DOUBLE_CLICK handling.

### onLaunchSource registration (G2-POLISH-06 — D-06, D-07)

Registration happens at **module scope** in `main.ts`, before `waitForEvenAppBridge()` resolves. The pattern:

```typescript
// Module scope (before init())
let resolveLaunchSource: (s: LaunchSource) => void
const launchSourcePromise: Promise<LaunchSource> = new Promise(resolve => {
  resolveLaunchSource = resolve
})

const bridge = EvenAppBridge.getInstance()
bridge.onLaunchSource((source) => resolveLaunchSource(source))

// In init():
const [resolvedBridge, source] = await Promise.all([
  waitForEvenAppBridge(),
  Promise.race([
    launchSourcePromise,
    new Promise<LaunchSource>(r => setTimeout(() => r('appMenu'), 500)),
  ]),
])
// Then decide initial screen based on source + active-session check
```

**Landing logic (locked — D-06):**

| Launch source | Active session exists? | Initial screen |
|---------------|------------------------|----------------|
| `glassesMenu` | Yes (≥1 session with `lastEvent.eventTimestamp > now - 5min` AND `event NOT IN ('task_complete', 'task_failed')`) | COMPANION |
| `glassesMenu` | No | HOME |
| `appMenu` | Either | HOME |
| (timeout fallback) | Either | HOME |

## Empty State Contract (D-10)

When Companion screen shows zero active sessions (empty state):

**Body content:**
```
No active sessions
idle
{bottom_line}
```

**Bottom line selection:**
- If `GET /v1/agent-sessions` returns any session within the last 24h: `Last: {label_20chars} {event} {relative_time}` (e.g., `Last: vigil-watch done 14m ago`)
- Else: `No Claude Code activity yet`

**Relative time format (locked):** `{N}m ago` for < 60 minutes, `{N}h ago` for < 24 hours. Do NOT use "just now" (ambiguous on a glance display) — use `0m ago` if less than 60 seconds. No "today" boundary logic.

**Blank line convention:** No blank lines between the 3 empty-state lines. Same 3-line body structure as active state.

## SSE State Machine Contract

The SSE shim (`src/lib/sse-client.ts`) maintains connection state that drives the offline indicator:

| SSE State | rightSide effect | Reconnect behavior |
|-----------|------------------|--------------------|
| Connected | None (normal N/M or HH:MM) | n/a |
| Disconnected | `'!'` prefix added to rightSide | Exp. backoff: 1s, 2s, 4s, 8s, 16s, cap 30s |
| Reconnecting | `'!'` prefix remains | (waiting) |
| Reconnected | `'!'` prefix cleared, header rebuilt | Last-Event-ID replayed |

**Content preservation on disconnect:** The 3-line body content is NOT wiped on SSE disconnect. Whatever was last displayed stays on screen. The only change is the header rightSide gaining the `'!'` prefix.

## Last-Event-ID Persistence Contract

**Storage key:** `vigil:lastEventId` in WebView `localStorage`.
**Type:** string (stringified integer from `agent_events.id` DB column).
**Initial value:** absent (first connect replay is bounded to last 24h by server).
**Update:** After each SSE frame successfully parsed, `localStorage.setItem('vigil:lastEventId', id)`.
**On reconnect:** SSE shim reads `localStorage.getItem('vigil:lastEventId')` and sends as `Last-Event-ID` header on the reconnect fetch.
**Clear condition:** Not cleared in this phase (bearer key change flow is Phase 125+).
