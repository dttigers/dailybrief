<task type="auto">
  <name>Task 1: Extend CompletionStore to 3-state tracking</name>
  <files>Sources/DailyBrief/Services/CompletionStore.swift</files>
  <action>
  1. Add a WorkOrderStatus enum inside CompletionStore:
     ```
     enum WorkOrderStatus: String, Codable {
         case open
         case inProgress
         case done
     }
     ```
  
  2. Change storage format from Set<String> (case numbers) to [String: WorkOrderStatus] (case number → status).
     File path stays: ~/.config/dailybrief/completed_workorders.json
  
  3. Add backward compatibility in load(): 
     - First try decoding as [String: String] (new format, map values to WorkOrderStatus)
     - If that fails, try decoding as [String] (old format) and map each to .done
     - This migrates the old format seamlessly on first load
  
  4. Update methods:
     - `save(_ statuses: [String: WorkOrderStatus])` — encode and write
     - `setStatus(_ caseNumber: String, _ status: WorkOrderStatus)` — load, update, save
     - `status(for caseNumber: String) -> WorkOrderStatus` — returns .open if not tracked
     - `markComplete(_ caseNumber: String)` — calls setStatus(caseNumber, .done) (backward compat)
     - `markIncomplete(_ caseNumber: String)` — calls setStatus(caseNumber, .open)
     - `listByStatus(_ status: WorkOrderStatus) -> [String]` — filter and return case numbers
     - Keep `listCompleted() -> [String]` as alias for listByStatus(.done)
  
  5. The PDF generation filter should continue working: work orders with status .done are excluded from the daily brief (same behavior as before, just using new data structure).
  
  6. Build to verify: swift build
  </action>
  <verify>swift build succeeds with no errors</verify>
  <done>CompletionStore supports 3-state workflow, backward-compatible with old JSON format</done>
</task>
