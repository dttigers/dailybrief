# Phase 107.2: Code Review Report

**Reviewed:** 2026-04-22T14:00:00Z
**Depth:** standard
**Files Reviewed:** 6
**Status:** issues_found

## Summary

Phase 107.2 changes are tightly scoped and security-conscious: the `VIGIL_BIND_HOST`
env var defaults to safe `127.0.0.1`, the new prod CORS guard fail-closes at boot
if `CORS_ORIGINS` is unset, and the Vite proxy approach correctly avoids exposing
browser-facing CORS complexity for cross-machine dev. The preflight check's firewall
probe uses the correct `socketfilterfw --getglobalstate | grep` form noted in
RESEARCH Pitfall 3. No critical security issues found.

Three warnings worth addressing before merge:

1. **Nullish-coalescing miss in `vite.config.ts`** — `VITE_DEV_API_TARGET=` (empty
   string) falls through `??` and hands an empty string to the Vite proxy, which
   then silently fails at request time rather than at boot.
2. **CRLF / whitespace fragility in `preflight-check.sh`** — `${BIND_VALUE// /}`
   only strips ASCII spaces, not tabs or trailing `\r` from CRLF-edited `.env`
   files. Will spuriously fail Check 5 if someone edits `.env` in a Windows-line-
   ending editor.
3. **`.env.example` ships `VIGIL_BIND_HOST=0.0.0.0` uncommented** — the inline
   comment ("Default (unset) = 127.0.0.1 = localhost-only") contradicts the value
   on the very next line. Copy-to-`.env` immediately lands on the exposed-bind
   path, not the safe default the doc describes.

Info items are mostly doc drift and pattern redundancy in `.gitignore`.

## Warnings

### WR-01: VITE_DEV_API_TARGET empty-string fallthrough

**File:** `vigil-pwa/vite.config.ts:15`
**Issue:** `env.VITE_DEV_API_TARGET ?? 'http://localhost:3001'` uses nullish
coalescing, which only falls through on `null` / `undefined`. `loadEnv()` returns
an empty string (`""`) when the `.env.local` file contains `VITE_DEV_API_TARGET=`
(key present, no value). That empty string is truthy-enough to satisfy `??`, so
`apiTarget` becomes `""`, and Vite's proxy registers a target of empty string.
No error at startup — the failure manifests as mysterious proxy errors on the
first `/v1` request. Given `.env.local.example:24` explicitly shows the pattern
`VITE_DEV_API_TARGET=http://localhost:3001`, a user editing that line to clear
it (intending to use the default) hits this silent failure.
**Fix:**
```ts
// Treat empty string the same as unset — logical-OR instead of nullish coalescing.
const apiTarget = env.VITE_DEV_API_TARGET || 'http://localhost:3001'
```
Optional belt: validate it parses as a URL and fail loud on malformed input:
```ts
const apiTarget = env.VITE_DEV_API_TARGET || 'http://localhost:3001'
try { new URL(apiTarget) } catch {
  throw new Error(`VITE_DEV_API_TARGET is not a valid URL: "${apiTarget}"`)
}
```

### WR-02: preflight-check.sh strips spaces only, not tabs/CR

**File:** `Scripts/preflight-check.sh:103`
**Issue:** `BIND_VALUE="${BIND_VALUE// /}"` removes ASCII space characters but
not tabs, non-breaking spaces, or trailing `\r` (CR) which CRLF-edited `.env`
files drop onto the end of every value. If someone opens `vigil-core/.env` in a
Windows-line-ending editor (or uses `echo` over SSH without `-n` on some shells),
`BIND_VALUE` becomes `"0.0.0.0\r"`. The subsequent `[[ "$BIND_VALUE" == "0.0.0.0" ]]`
check fails, flow falls through to the "not a supported value" FAIL branch, and
the user gets a confusing "VIGIL_BIND_HOST='0.0.0.0\r' is not a supported value"
error. Same fragility applies to `DB_URL` parsing at line 69 but that predates
this phase — only flagging the new Check 5 site.
**Fix:** Strip all whitespace including `\r`:
```bash
# Strip all whitespace (spaces, tabs, CR) and surrounding quotes.
BIND_VALUE=$(grep -E '^VIGIL_BIND_HOST=' "$ENV_FILE" 2>/dev/null | head -1 | cut -d= -f2- || true)
BIND_VALUE="${BIND_VALUE//[$' \t\r\n']/}"          # any whitespace
BIND_VALUE="${BIND_VALUE#\"}"; BIND_VALUE="${BIND_VALUE%\"}"   # strip "..."
BIND_VALUE="${BIND_VALUE#\'}"; BIND_VALUE="${BIND_VALUE%\'}"   # strip '...'
```
The quote-stripping also defuses the related footgun where a user writes
`VIGIL_BIND_HOST="0.0.0.0"` (dotenv-style) and the script reads literal
`"0.0.0.0"` including quotes.

### WR-03: `.env.example` ships `VIGIL_BIND_HOST=0.0.0.0` uncommented, contradicting its own doc

**File:** `vigil-core/.env.example:56-65`
**Issue:** The comment block (lines 56–64) repeatedly describes the safe default:
"Default (unset) = 127.0.0.1 = localhost-only (prod-safe behind Railway proxy)",
then line 65 sets `VIGIL_BIND_HOST=0.0.0.0` — the exposed-bind value. A new
developer running `scripts/dev-setup.sh` (which copies `.env.example` → `.env`)
silently lands on 0.0.0.0 bind by default, not the documented safe default.
Preflight Check 5 then reports "PASS — 0.0.0.0 bind …" which is consistent but
misleading given the doc comment's framing. Either:
- (a) The example should leave the var commented and require opt-in, OR
- (b) The doc comment should be updated to acknowledge the example DOES pre-opt-in.

Given Phase 107.2's explicit goal is cross-machine dev access, option (b) is
the simpler fix; option (a) is safer for anyone cloning the repo who doesn't
intend to expose their dev stack on Tailscale.
**Fix (recommended — option a, safer default):**
```bash
# DO NOT set this on Railway. [existing comment block unchanged]
# Uncomment to enable cross-machine Tailscale/LAN dev access; leave commented
# for localhost-only (the safe default).
#VIGIL_BIND_HOST=0.0.0.0
```
**Alternative (option b — match current behavior, fix the doc):** replace the
"Default (unset) = 127.0.0.1" sentence with "This template pre-enables cross-
machine access; comment the line below out to revert to localhost-only."

**File:** `vigil-core/src/index.ts:59`
**Issue:** `(process.env["JWT_SECRET"] as string).length < 32` — the preceding
`!process.env["JWT_SECRET"]` short-circuit already narrows the type to `string`
on the right side of `||`. The `as string` cast is dead weight and mildly
misleading (suggests the author wasn't sure about narrowing).
**Fix:**
```ts
if (!process.env["JWT_SECRET"] || process.env["JWT_SECRET"].length < 32) {
```

### IN-02: CORS fallback line-number comment drift

**File:** `vigil-core/src/index.ts:67`
**Issue:** Comment references "line ~74" for the `origin: corsOrigins ?? "*"`
fallback; actual location after all the 107.2-era imports is line 84. The `~`
softens this, but future maintainers will waste seconds scanning the wrong area.
**Fix:** Update the comment to `"line ~84"`, or drop the line number and rely
on "below" / "next to the cors() call" phrasing that doesn't drift.

### IN-03: Prod CORS guard depends on NODE_ENV being set

**File:** `vigil-core/src/index.ts:69`
**Issue:** The fail-closed guard fires only when `process.env.NODE_ENV === "production"`.
Railway sets `NODE_ENV=production` by default for Node deploys, so in practice
this works. But if NODE_ENV is ever unset/misspelled in Railway (e.g., someone
adds `NODE_ENV=prod` or deletes it), the guard no-ops and the `origin: corsOrigins ?? "*"`
fallback ships wildcard CORS silently. This is one layer of defense-in-depth;
tightening is optional.
**Fix (optional hardening):** Default to fail-closed unless explicitly in dev:
```ts
// Require CORS_ORIGINS unless NODE_ENV is explicitly 'development' or 'test'.
const env = process.env.NODE_ENV;
const isDev = env === "development" || env === "test";
if (!isDev && !process.env.CORS_ORIGINS) {
  console.error("FATAL: CORS_ORIGINS must be set outside dev/test — refusing to boot with wildcard CORS");
  process.exit(1);
}
```
This also covers staging / preview deploys where NODE_ENV may be unset.

### IN-04: CORS_ORIGINS whitespace-only value bypasses the guard

**File:** `vigil-core/src/index.ts:69,77-79`
**Issue:** If someone sets `CORS_ORIGINS="   "` (whitespace-only) in Railway,
`!"   "` is `false`, so the prod guard does NOT fire. Then `split(",").map(trim)`
yields `[""]`, which is passed to Hono's `cors({ origin: [""] })`. Empty-string
origin matches no requests → every browser request gets blocked. Loud failure
at request time, but the "FATAL refuse to boot" semantics the rest of the file
upholds are violated.
**Fix:** Trim + check length after trim:
```ts
const rawCors = process.env.CORS_ORIGINS?.trim();
if (process.env.NODE_ENV === "production" && !rawCors) {
  console.error("FATAL: CORS_ORIGINS must be set (non-empty) in production");
  process.exit(1);
}
```

### IN-05: VITE_DEV_API_TARGET default in example file duplicates code default

**File:** `vigil-pwa/.env.local.example:24`
**Issue:** `VITE_DEV_API_TARGET=http://localhost:3001` in the example is the
same string the `vite.config.ts:15` fallback uses. Committing it means anyone
copying the example to `.env.local` locks in that value explicitly — if the
code default ever changes (e.g., to `http://127.0.0.1:3001` to dodge the IPv6
resolution quirk also documented in RESEARCH Pitfall 1), users who copied the
example will keep hitting the old hostname. Cheap to avoid.
**Fix:** Comment the line and mention it's the default:
```bash
# Default is http://localhost:3001 — uncomment to override.
#VITE_DEV_API_TARGET=http://localhost:3001
```

### IN-06: .gitignore pattern sprawl for .env variants

**File:** `.gitignore:10-17,30`
**Issue:** Overlapping / redundant env-ignore rules introduced over time:
- `.env` (10) is a subset of `*.env` (11)
- `.env.local` (12) and `.env.*.local` (13) are subsets of `.env*.local` (30)
- `.env.local.bak*` (16) is a subset of `*.env.local.bak*` (17) — both added
  this phase per the diff
- `.env.bak.*` (14) and `*.env.bak.*` (15) are similarly nested

Not a bug — git handles overlapping ignores fine — but the sprawl makes it hard
to tell what the intended policy is, and increases the chance a later contributor
"simplifies" by removing the wrong rule.
**Fix (optional cleanup, keep as-is if risk-averse):**
```gitignore
# Env files — never commit. Covers all dotenv variants and editor backups.
.env
.env.*
*.env
*.env.*
```
That single block subsumes every current rule (10-17, 30). Validate by running
`git check-ignore -v <path>` against the files that must stay ignored before
deleting the old rules.
