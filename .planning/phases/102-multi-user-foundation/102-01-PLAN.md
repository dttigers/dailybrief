```sql
DROP INDEX IF EXISTS "uq_ai_cache_type";

ALTER TABLE "app_settings"  ALTER COLUMN "user_id" SET NOT NULL;
ALTER TABLE "app_settings"  ADD CONSTRAINT "app_settings_user_id_users_id_fk"  FOREIGN KEY ("user_id") REFERENCES "users"("id") ON DELETE RESTRICT;
CREATE INDEX IF NOT EXISTS "idx_app_settings_user_id"  ON "app_settings"  ("user_id");
ALTER TABLE "app_settings"  DROP CONSTRAINT IF EXISTS "app_settings_pkey";
ALTER TABLE "app_settings"  ADD CONSTRAINT "app_settings_pkey" PRIMARY KEY ("user_id", "key");
```

Note: Every `ADD CONSTRAINT <name> FOREIGN KEY ...` lacks an `IF NOT EXISTS` because Postgres does not support that syntax for constraints. Idempotency is handled by the preceding `DROP CONSTRAINT IF EXISTS` where needed — but FK-not-yet-added cannot be dropped. To make the FK-add idempotent, wrap each in a DO block:

```sql
DO $$ BEGIN
  ALTER TABLE "api_keys" ADD CONSTRAINT "api_keys_user_id_users_id_fk" FOREIGN KEY ("user_id") REFERENCES "users"("id") ON DELETE RESTRICT;
EXCEPTION WHEN duplicate_object THEN NULL; END $$;
```

Apply this DO-block-with-duplicate_object-exception pattern for each of the 11 FK ADDs AND for the app_settings_pkey re-add. This keeps the migration re-runnable even if a prior partial attempt added some constraints.
