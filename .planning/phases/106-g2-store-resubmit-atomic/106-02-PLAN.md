```typescript
/** Handle navigation events from temple touchpad / R1 ring */
export async function handleNavEvent(
  eventType: OsEventTypeList,
  bridge: EvenAppBridge,
): Promise<void> {
  // Task detail is a sub-screen, not in SCREEN_ORDER — handle explicitly
  if (currentScreen === Screen.TASK_DETAIL) {
    if (eventType === OsEventTypeList.SCROLL_TOP_EVENT) {
      await navigateTo(Screen.WORK_ORDERS, bridge)
    } else if (eventType === OsEventTypeList.SCROLL_BOTTOM_EVENT) {
      await navigateTo(Screen.AFFIRMATION, bridge)
    } else if (eventType === OsEventTypeList.DOUBLE_CLICK_EVENT) {
      await navigateTo(Screen.HOME, bridge)
    }
    return
  }

  // G2-02: home-screen double-tap hands off to the host-rendered exit-confirm dialog.
  // Per D-01, we do NOT render a custom confirmation UI. Per RESEARCH Pitfall 3,
  // we fire-and-forget — the SDK's Promise<boolean> semantics are undocumented,
  // and lifecycle transitions (confirm/cancel) arrive via existing FOREGROUND_*
  // listeners in main.ts (lines 75-82). exitMode=1 per D-01.
  if (
    currentScreen === Screen.HOME &&
    eventType === OsEventTypeList.DOUBLE_CLICK_EVENT
  ) {
    void bridge.shutDownPageContainer(1)
    return
  }

  let target: ScreenName

  switch (eventType) {
    case OsEventTypeList.SCROLL_BOTTOM_EVENT:
      target = getNextScreen(currentScreen)
      break
    case OsEventTypeList.SCROLL_TOP_EVENT:
      target = getPrevScreen(currentScreen)
      break
    case OsEventTypeList.DOUBLE_CLICK_EVENT:
      target = Screen.HOME
      break
    default:
      return
  }

  await navigateTo(target, bridge)
}
```

**Why the new branch comes AFTER the task-detail branch:** Task-detail is a sub-screen handled explicitly (not in `SCREEN_ORDER`). Placing the home-branch BEFORE task-detail would short-circuit when a user double-taps in task-detail (because `currentScreen` would flip to `HOME` during navigation — not applicable here since we read `currentScreen` before mutation, but ordering-as-specified is the safe contract).

**Why the branch comes BEFORE the generic switch:** The existing `DOUBLE_CLICK_EVENT` case in the switch (`target = Screen.HOME`) is now dead-code for home (home can't `navigateTo(Screen.HOME)` from home — it would be a no-op refresh). Leaving the case intact keeps the switch symmetrical and preserves the behavior for work-orders/affirmation (D-02 "Trigger scope is the home screen ONLY. Preserve today's handleNavEvent behavior on work-orders/affirmation/task-detail").

**Why `void` and not `await`:** Per RESEARCH Pitfall 3 — the SDK d.ts comment says `Promise<boolean>` "return `true` 表示成功" (true = success) but doesn't define whether "success" means "dialog shown" or "user confirmed". Awaiting and branching on the boolean would create simulator-vs-hardware divergence. Lifecycle transitions arrive via existing `FOREGROUND_ENTER_EVENT` / `FOREGROUND_EXIT_EVENT` listeners in main.ts:75-82.

**Do NOT modify any other function in navigation.ts.** Specifically: leave `Screen`, `SCREEN_ORDER`, `getNextScreen`, `getPrevScreen`, `buildScreen`, `navigateTo`, `refreshCurrentScreen`, `navigateToTaskDetail`, `currentScreen` all exactly as-is. Do NOT add a new import — `OsEventTypeList` and `EvenAppBridge` are already imported (lines 4-7).
