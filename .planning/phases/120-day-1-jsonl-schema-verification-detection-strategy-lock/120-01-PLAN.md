## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| local-repo → github.com (public) | Anything committed to `main` becomes world-readable on push |
| user GitHub identity → `dttigers` org | `gh repo create` acts under the user's authenticated session; misuse risk is creating the wrong repo or wrong org |
| user shell → `gh` CLI | OAuth token in `gh auth status` controls write access to dttigers org |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-120-01 | I (Information disclosure) | .gitignore on public repo | mitigate | Append explicit `vk_*`, `watch.toml`, `.env`, `.env.*`, `/verification-log/` rules in Task 2 BEFORE Plan 120.02 ever writes a verification log. The `/verification-log/` directory rule is preemptive — Plan 120.02 may be tempted to commit raw JSONL excerpts directly into the repo working tree; gitignoring that path forces Plan 120.03 to consciously copy the curated subset into the README appendix instead of bulk-committing user content. |
| T-120-02 | I (Information disclosure) | LICENSE copyright line | accept | LICENSE is GitHub-seeded with the user's GitHub display name; this is intentional public attribution per D-04 and matches existing vigil-core posture. No PII risk. |
| T-120-03 | E (Elevation of privilege) | `gh auth status` token scope | mitigate | Task 1 is a checkpoint:human-action — if the existing token cannot create a repo under `dttigers`, the user re-auths interactively in their own terminal. Executor never types tokens directly. |
| T-120-04 | T (Tampering) | Repo name collision (existing dttigers/vigil-watch) | mitigate | Task 1 explicitly checks `gh repo view dttigers/vigil-watch` BEFORE `gh repo create`; halts on collision rather than overwrite. |
