    ```typescript
    /**
     * Capture a product event. No-op when posthog singleton is null (Phase 103 D-10 gate).
     * D-01..D-04: Properties whose NAME is in BLOCKED_PROPERTY_NAMES are dropped silently
     * (case-sensitive match). For each dropped property, a `posthog_property_blocked`
     * meta-event is emitted with {event_name, property_name}. The user's event still emits
     * with the surviving allowed properties — drop-the-property, not drop-the-event (D-02).
     * Signature unchanged from Phase 103 (D-03) — zero call-site churn.
     */
    export function trackEvent(
      userId: number | string,
      event: string,
      properties: Record<string, string | number | boolean | null | undefined> = {},
    ): void {
      // Partition properties into (allowed, blocked) in one pass.
      const allowed: Record<string, string | number | boolean | null | undefined> = {};
      const blocked: string[] = [];
      for (const key of Object.keys(properties)) {
        if (BLOCKED_PROPERTY_NAMES.has(key)) {
          blocked.push(key);
        } else {
          allowed[key] = properties[key];
        }
      }

      // D-02: Emit one meta-event per blocked property name. These fire BEFORE the
      // user's event so a downstream PostHog consumer always sees the canary first.
      // No-op if posthog === null (D-10 gate).
      for (const blockedName of blocked) {
        posthog?.capture({
          distinctId: String(userId),
          event: "posthog_property_blocked",
          properties: {
            event_name: event,
            property_name: blockedName,
          },
        });
      }

      // D-02: User's event still emits with surviving properties (drop the bad
      // property, not the entire event). Even if `allowed` is empty, the event
      // itself is still useful — `thought_created` with no props beats no event.
      posthog?.capture({
        distinctId: String(userId),
        event,
        properties: allowed,
      });
    }
    ```

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| route handler → trackEvent | Untrusted (developer-authored) property maps cross here. The wrapper is the last line of defense before egress to PostHog. |
| trackEvent → posthog-node SDK → PostHog Cloud | Network egress to a third party (https://us.i.posthog.com). Anything that lands in `properties` leaves the customer trust boundary. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-105-01 | Information disclosure | trackEvent properties (user-generated thought / chat / transcript text) | mitigate | D-04 BLOCKED_PROPERTY_NAMES Set drops `content`, `body`, `text`, `message`, `description`, `title`, `note`, `transcript` BEFORE `posthog.capture()`. Test 1+2 in this plan assert the literal Set membership and case-sensitivity. |
| T-105-02 | Information disclosure | trackEvent properties (developer accidentally adds NEW string field with a non-blocked name, e.g. `summary`) | accept | Type signature `string \| number \| boolean \| null \| undefined` is the first signal in code review. The runtime guard catches the documented leak names. New leak names get added to the Set as they are spotted (CONTEXT.md D-04: "easy to extend"). The `posthog_property_blocked` meta-event surfaces actual leaks for retroactive cleanup. |
| T-105-03 | Information disclosure | identifyUser properties (route handler accidentally passes user-generated text) | mitigate | identifyUser uses the SAME property type as trackEvent (string\|number\|boolean\|null\|undefined). Plan 03's call site explicitly passes only `{ email, createdAt }` per D-09 allowlist — no other properties added. Code review enforces. |
| T-105-04 | Repudiation | posthog_property_blocked meta-event emission | accept | The meta-event is exactly what proves a leak attempt happened. Without it, drops would be silent. With it, every drop is auditable in PostHog. This IS the audit log. |
| T-105-05 | Tampering | BLOCKED_PROPERTY_NAMES Set mutated at runtime by a route handler | mitigate | TypeScript const + `ReadonlySet<string>` annotation in the consumed type prevents mutation. The Set lives at module scope — no runtime injection path. Code review catches any `BLOCKED_PROPERTY_NAMES.add(...)` or `.delete(...)` call. |
| T-105-06 | Denial of service | Massive properties object causes O(n) Set lookups in trackEvent | accept | Object.keys(properties) is bounded by the route handler's intent — Phase 105 call sites pass 3-5 properties each (D-08, D-12, D-14). Set.has is O(1). No real DoS surface. |
| T-105-07 | Elevation of privilege | trackEvent / identifyUser bypassed by direct `posthog.capture` call from a route file | mitigate | Phase 103 D-14 lock: route files MUST NOT import the singleton. Code review + the `// MUST NOT import { posthog } directly` comment at the top of posthog.ts. Plan 02 + 03 call sites are required to use `trackEvent` / `identifyUser` only. Future leak prevention: an ESLint rule could ban `import { posthog }` outside `analytics/` (deferred). |
