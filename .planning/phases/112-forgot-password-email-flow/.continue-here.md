## Critical Anti-Patterns

| Pattern | Description | Severity | Prevention Mechanism |
|---------|-------------|----------|---------------------|
| Argon2 timing assumption | RESEARCH §A2 assumed argon2 verify is 100-200ms. Live measurement on `@node-rs/argon2` (native Rust binding) shows ~18-19ms. Without compensating logic, hit-path's small DB ops dominate and skew enumeration-safety timing comparisons. | advisory | If Plan 112-03 (or any future phase) adds timing-parity logic, MEASURE first via a one-shot bench script before committing to a threshold. Don't trust pre-execution time estimates. |
| Rate-limit accumulation in timing tests | Test 3 of Plan 112-02 made 7 sequential requests against the same handler instance; the per-IP rate limit fired after request #5, short-circuiting calls 6+ to ~0.4ms and skewing the median below the hits median (false ratio of 56x). | advisory | Timing tests that make multiple requests against rate-limited endpoints MUST reset the rate-limit state between calls (e.g., `__resetBucketsForTest()` inside `timeOne()`). Real production traffic doesn't share a process with the limiter so the production semantics are unaffected. |
| 503 on null DB breaks enumeration safety | Original 112-02 handler returned 503 when `db` was null. This leaks DB-availability through response shape (200 vs 503), breaking the D-03 invariant that all paths must return identical body+status. | advisory | Treat null-db as miss-path (run dummy argon2 + return enum-safe 200). Production should never see null-db, but defense in depth keeps the response surface uniform under any failure mode. |
