## TL;DR

Use the Even Realities G2 glasses as an ambient notification + status layer for
Claude Code sessions running on Morrill House (and the user's local Mac). The
glasses won't display code or terminal output — they'll tap on the temple when
Claude Code needs the user, finishes a task, or hits something interesting.

Pairs existing strengths: VS Code Claude Code extension on Morrill House
(already in daily use), Vigil Core API on Railway (live at `api.vigilhub.io`),
Even Hub SDK (already integrated for v0.2.0 plugin). This is mostly glue.

## Architecture

Two-layer architecture from v0.2.0 stays. Adding one new component on each
layer plus one new daemon.

| Component | Purpose |
|---|---|
| `vigil-watch` (macOS daemon, Swift) | Watches `~/.claude/projects/<project>/<session>.jsonl` files for state changes. POSTs events to Vigil Core. |
| `POST /v1/agent-events` (Vigil Core) | New endpoint. Receives events, fans out to subscribed glasses clients. **Scoped per `userId`** (mirror SCHED-01). |
| `GET /v1/agent-sessions` (Vigil Core) | Returns currently-tracked sessions and last known state. |
| `/v1/agent-stream` (existing WebSocket) | Add `agent-events` as a new event type on the existing channel — no new connection needed from clients. |
| Companion view (G2 client) | New HUD screen. Shows current session state, last event, and pending agent count. Tap-to-acknowledge on temple. |

**Data flow:** `[Claude Code in VS Code] → [JSONL files] → [vigil-watch] → [Vigil Core API] → [G2 client] → [HUD]`

## Event types (5)

| Event | Trigger | HUD treatment |
|---|---|---|
| `needs_input` | Claude Code is asking a question and waiting | Persistent banner + gentle haptic. Highest priority. |
| `task_complete` | Session ended normally, no errors | 3-second toast with task summary line |
| `task_failed` | Non-zero exit, test failure, or error string detected | Persistent banner with error type |
| `milestone` | User-defined string match (e.g. "deploy succeeded", "all tests passed") | 3-second toast |
| `heartbeat` | No new JSONL lines for ≥60s while running | Subtle dot indicator on dashboard. No alert. |

## vigil-watch Daemon (Swift, macOS)

**Pivot from rev 1:** Originally specced as a Rust PTY-wrapping CLI for tmux.
Reality: user works in the **VS Code Claude Code extension**, not tmux.
Rev 2 pivots to a Swift launchd daemon watching JSONL files on disk.

### Day-1 verification gate (BLOCKING)

The detection strategy assumes Claude Code writes session state as JSON Lines
to `~/.claude/projects/<project-id>/<session-id>.jsonl`, based on community
reverse-engineering of the open-source `vscode-claude-status` extension.

**Day 1 of execute-phase MUST verify before any production code:**
1. Open VS Code, start a Claude Code session.
2. Tail `~/.claude/projects/<latest>/<session>.jsonl` while interacting.
3. Document actual JSONL line types observed.
4. Confirm or correct the mapping table below.
5. Pay special attention to: how tool approval prompts appear, what fields
   indicate "awaiting input," how session end is signaled, error structure on
   tool failure.

**Adjust spec mapping in code if reality differs.** Don't proceed to Phase 2
coding until mapping is confirmed.

### Internal architecture

Three responsibilities (actors or Combine pipeline):

1. **Watcher** — `FSEventStream` rooted at `~/.claude/projects/`, debounces
   rapid-fire events. Emits `(filePath, lastReadOffset)` tuples.
2. **Parser** — Maintains `filePath → byte offset` dictionary. Seeks to
   `lastReadOffset`, reads to EOF, splits on newline. Persists offsets to
   `~/Library/Application Support/vigil-watch/offsets.json` for restart safety.
3. **Emitter** — POSTs to Vigil Core with retry/backoff. In-memory queue
   (max 100) for events generated while Core is unreachable. On `SIGTERM`
   from launchd: drains queue with 5-second timeout, then exits cleanly.

### Expected JSONL line types (assumed — verify Day 1)

| JSONL type (assumed) | Meaning | Vigil event emitted |
|---|---|---|
| `user` | User submitted a prompt | (none — track session liveness) |
| `assistant` | Claude responded | (none — informational) |
| `tool_use` (with `awaiting_approval`) | Claude wants to run a tool, waiting for approval | `needs_input` |
| `tool_result` (success) | Tool executed cleanly | (none — resets silence timer) |
| `tool_result` (error) | Tool errored | `task_failed` (deduped) |
| `session_end` / `stop` | Conversation ended cleanly | `task_complete` |
| (no new lines for N seconds) | Session quiet but not ended | `heartbeat` |
| (custom regex match in any text) | User-defined milestone string seen | `milestone` |

### Detection rules

- `needs_input`: emit at most once per 30s per session. Deduplicates against
  rapid-fire approval prompts.
- `task_failed` mid-session: emit on first `tool_result` error per session,
  then suppress until either `task_complete` or `session_end` is seen.
- `milestone`: each `--milestone` pattern fires at most once per session
  lifetime.
- `heartbeat`: emit when JSONL file has no new lines for ≥ `heartbeatSeconds`
  (default 60). After emission, suppress until new lines appear.
- `task_complete` vs `task_failed` precedence: if both could apply (errors
  followed by clean stop), prefer `task_failed`.

### CLI surface

```
vigil-watch daemon                       # default, run by launchd
vigil-watch run --verbose                # foreground mode, logs to stdout
vigil-watch tail <session-id>            # parsed events for one session, no posting
vigil-watch test                         # synthetic event to verify Core connectivity
vigil-watch install                      # install launchd plist + start at login
vigil-watch uninstall                    # remove launchd plist
vigil-watch status                       # daemon state, queue depth, last event
```

### Configuration

Path: `~/.config/vigil/watch.toml`. Created on first run with defaults.

```toml
[core]
api_url = "https://api.vigilhub.io"        # NOTE: spec said morrillhouse.com — corrected to live URL
api_key = ""                                # falls back to VIGIL_API_KEY env var
retry_seconds = 5
max_retries = 3

[detection]
heartbeat_seconds = 60
needs_input_debounce_seconds = 30
milestone_patterns = []                     # optional regexes against assistant message text

[paths]
projects_dir = "~/.claude/projects"

[behavior]
backfill_on_start = false
host_label = "macbook-pro"                  # appended to host field in events
```

### Event payload contract

POST to `${api_url}/v1/agent-events` with `Authorization: Bearer ${api_key}`.

```json
{
  "session_id": "claude-<jsonl-filename-without-extension>",
  "event": "needs_input",
  "message": "Claude wants to run: rm -rf node_modules",
  "timestamp": "2026-05-05T20:34:12Z",
  "label": "vigil-vscode-extension",
  "host": "macbook-pro",
  "exit_code": null
}
```

**Field rules:**
- `session_id`: derived from JSONL filename. Stable across daemon restarts.
- `label`: from project directory name (Claude Code derives from workspace folder).
- `host`: from `Host.current().localizedName`, plus optional `host_label` config override.
- `message`: human-readable summary. Truncate to 280 chars.

### Tech choices

- **Language:** Swift. Already maintain Jarvis (~9,400 lines Swift); macOS APIs
  (FSEvents, launchd) are first-class.
- **Dependencies:** `Foundation` (URLSession, JSON), `CoreServices` (FSEvents),
  `os.log` (logging), `swift-argument-parser` (CLI). Avoid heavyweight Swift packages.
- **Repo:** `github.com/dttigers/vigil-watch`. Swift Package, MIT.
- **Build:** `swift build -c release` → `cp .build/release/vigil-watch /usr/local/bin/`.
- **Distribution Phase 1:** Local builds on each Mac. No notarization needed.
- **Distribution Phase 2 (later):** GitHub Actions arm64 release binaries.

### Fallback strategies (if JSONL approach fails Day 1)

In order:

- **Fallback A — Notification observation:** `NSDistributedNotificationCenter`
  or read macOS Notification Center DB for VS Code notifications mentioning
  "Claude." Requires Full Disk Access; less reliable signal.
- **Fallback B — VS Code extension:** TypeScript extension hooking
  `vscode.window` events and integrated terminal. More code, separate
  distribution channel, doesn't run when VS Code is closed.
- **Fallback C — Process inspection:** Watch for `claude` CLI process in `ps`,
  observe open file descriptors and write times. Coarse signal — basically
  only "running" vs "not running."

Document Day-1 findings and chosen path in repo README before writing
production code.

## G2 Companion HUD (plugin v0.3.0)

New HUD screen accessible from the Vigil plugin tile in the Even Realities app.

### Layout

- **Top line:** session label (truncated to fit)
- **Middle line:** current state (idle / running / waiting / done)
- **Bottom line:** last event message (scrolling if too long)

### Interaction

- Double tap on temple: context-sensitive — ack banner if shown, else cycle through active sessions if ≥2, else navigate Home (Phase 124 D-08).
- Single tap (CLICK_EVENT) and long press (LONG_PRESS_EVENT): DEFERRED via SEED-011 — sim-only on G2 hardware in v0.0.9 SDK; original three-tap-variant intent preserved for re-activation when SDK exposes reliable events.

### Quiet mode

Quiet mode is honored on the HUD via a PWA Settings toggle that pipes through Vigil Core to a quiet_mode_changed SSE event consumed by the plugin (Phase 125 / D-01..D-04 — SDK exposes no Focus API as of @evenrealities/even_hub_sdk@0.0.9). When Quiet mode is enabled, only needs_input and task_failed reach the HUD; other events queue silently in a server-side suppression queue and replay in chronological order on toggle-off.

## Risks and open questions

- **PTY/JSONL detection:** Day-1 verification is load-bearing. Have fallback
  plan ready.
- **needs_input false-positive rate:** Start permissive on detection patterns,
  tighten later.
- **HUD real estate is tiny:** Resist showing too much. The whole point is
  glanceable.
- **Even Hub approval timing:** v0.2.0 plugin still in store-review. If
  approval lands during this build, retest plugin behavior on whatever the
  store-published version is.

## Success criteria

- Kick off a Claude Code session in VS Code on Morrill House, walk away,
  reliably get a tap on the temple when it needs the user.
- False positive rate on `needs_input` is low enough that user doesn't
  disable the feature within a week.
- Full demo flow can be recorded in under 60 seconds for portfolio.
- Vigil v0.2.0 G2 plugin → v0.3.0 ships to Even Hub developer portal within
  this milestone.

## Notes for downstream phases

- Day 1 is verification, not coding. Don't skip it.
- Build incrementally: file watcher → parser → emitter. Get each working in
  isolation before combining.
- Prefer Apple frameworks over third-party Swift packages. Keep dependency
  tree shallow.
- User values reliability over scope. Ship 5 reliable event types before
  adding a 6th.
- Surface ambiguities in Day-1 findings document, not as silent design
  decisions in code.
- **api_url drift:** Spec PDFs said `vigil-core.morrillhouse.com`. Reality is
  `api.vigilhub.io`. Use the latter.
- **Auth model:** Vigil Core uses bearer JWT (multi-user since v3.4) and vk_
  prefixed SHA-256-hashed keys. vigil-watch should use a vk_ key (simpler
  for daemon use case).
- **Per-user scoping:** `/v1/agent-events` and `/v1/agent-sessions` MUST be
  scoped by `userId`. Mirror SCHED-01 fan-out pattern. Single-user today,
  but the structural constraint prevents future cross-user leakage.
